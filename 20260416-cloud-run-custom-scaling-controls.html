📋 概要

Cloud Run のオートスケーラーに、カスタム CPU 使用率ターゲットおよびカスタム同時実行数ターゲットを指定できる「スケーリングコントロール」機能が Preview として登場しました。従来の固定 60% ターゲットから脱却し、ワークロード特性に応じて 10% ~ 95% の範囲でターゲット値をカスタマイズできます。

さらに、いずれか一方のスケーリングドライバーを無効化して単一メトリクスでのスケーリングが可能になり、オプトインするだけでインスタンス数が少ないサービスでも精度の高いスケーリングが動作する改善モデルが有効になります。

  • Metrics Explorer で推奨インスタンス数を可視化

🛠 アーキテクチャ図

Cloud Run オートスケーラーは CPU 使用率と同時実行数の 2 つのドライバーから推奨インスタンス数を算出し、より多くのインスタンスを必要とする方の値を採用します。スケーリングコントロールにより、各ドライバーのターゲット値をカスタマイズしたり、一方を無効化できます。

💡 主要機能

💻 カスタム CPU 使用率ターゲット

CPU 使用率ターゲットを 0.1 (10%) ~ 0.95 (95%) の範囲で指定可能。全 CPU の 1 分間平均使用率がモニタリングされます。デフォルトは 0.6 (60%)。

👥 カスタム同時実行数ターゲット

同時実行数ターゲットを 0.1 ~ 0.95 の範囲で指定可能。最大同時実行数に対する使用率として算出され、1 分間/10 分間の平均が使用されます。

🔧 スケーリングドライバーの選択的無効化

CPU or 同時実行数のいずれかを disabled に設定可能。CPU バウンドなら同時実行数を無効化、I/O バウンドなら CPU を無効化して最適なスケーリングを実現。

🚀 高精度オートスケーリングモデル

オプトインするだけで改善されたスケーリングモデルが有効に。デフォルト 60% を維持してもインスタンス数が少ないサービスの精度が向上。カスタムターゲットとの差分が 10% 閾値を超えるとスケーリング発動。

YAML アノテーション

YAML Cloud Run サービス定義 (カスタムスケーリング設定) 
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  annotations:
    run.googleapis.com/launch-stage: BETA
  name: my-service
spec:
  template:
    metadata:
      annotations:
        # カスタム CPU ターゲット (0.1 - 0.95)
        run.googleapis.com/scaling-cpu-target: '0.75'
        # カスタム同時実行数ターゲット (0.1 - 0.95)
        run.googleapis.com/scaling-concurrency-target: '0.50'

📝 設定方法

📌 前提条件: Google Cloud プロジェクト作成済み / Cloud Run API 有効化 / gcloud CLI 認証済み / gcloud beta コンポーネントインストール済み

Step 1: オプトイン (デフォルトターゲット維持)

デフォルト値 (60%) を維持する場合でも、オプトインすることで改善されたスケーリングモデルが有効になります。

Bash オプトイン (デフォルトターゲット維持) 
gcloud beta run services update SERVICE_NAME \
  --scaling-cpu-target=0.6 \
  --scaling-concurrency-target=0.6

Step 2: カスタムターゲットの設定

Bash CPU 75% / 同時実行数 50% に設定 
gcloud beta run services update SERVICE_NAME \
  --scaling-cpu-target=0.75 \
  --scaling-concurrency-target=0.50

Step 3: スケーリングドライバーの選択的無効化

Bash CPU のみでスケーリング / 同時実行数のみでスケーリング 
# CPU のみでスケーリング (同時実行数ドライバーを無効化)
gcloud beta run services update SERVICE_NAME \
  --scaling-concurrency-target=disabled

# 同時実行数のみでスケーリング (CPU ドライバーを無効化)
gcloud beta run services update SERVICE_NAME \
  --scaling-cpu-target=disabled

Step 4: 設定の確認

Bash スケーリング設定の確認 
gcloud run services describe SERVICE_NAME

出力の Target CPU utilization と Target concurrency utilization を確認。Console では Revisions > Containers > Autoscaling metrics でも確認できます。

Step 5: デフォルト値への復元 (オプトアウト)

Bash オプトアウト 
gcloud beta run services update SERVICE_NAME \
  --scaling-cpu-target=default \
  --scaling-concurrency-target=default

🎯 メリット

💼 ビジネス面

💲コスト最適化 - ターゲットを高めに設定してインスタンスの稼働率を向上。遊休リソースを最小化
💪SLA 準拠の可用性確保 - ターゲットを低めに設定してバッファを確保。スパイク時もレイテンシ SLA を維持
🏆ワークロード別の最適化 - サービスごとに異なるスケーリング戦略で、コストと性能を個別に最適化

🛠 技術面

🎯スケーリング精度の向上 - 特に少数インスタンスサービスにおける予測可能性が大幅改善
🔌スケーリングドライバーの選択 - 単一メトリクスによるシンプルで予測可能なスケーリングモデル
📈段階的な調整 - Metrics Explorer の recommended_instances メトリクスで動作を確認しながら最適化

⚠️ デメリット・制約事項

⚠️ Preview ステータス: 本番環境での使用は「Pre-GA Offerings Terms」が適用されます。機能仕様が変更される可能性があります。

制限事項

  • ❌ CPU と同時実行数の両方を同時に無効化することはできない
  • ❌ ターゲット値は小数点以下 2 桁までに制限
  • ❌ メモリ使用率に基づくスケーリングは非サポート

考慮すべき点

  • ⚠️ 低ターゲット (10% 付近) → インスタンス数大幅増加、コスト上昇のリスク
  • ⚠️ 高ターゲット (90% 以上) → スケーリング遅延、突発的トラフィック時のレイテンシ悪化リスク
  • ⚠️ 高 CPU ターゲット設定時は許容閾値の幅が広がり、一括スケーリング動作の可能性
  • ⚠️ シングルスレッド + マルチ CPU の場合、CPU ベースのスケーリングが不正確 → 同時実行数ベースを推奨
  • ⚠️ ターゲット変更はリビジョン新規作成を伴う → traffic splitting での段階テストを推奨

💻 ユースケース

🤖 UC1: CPU バウンドな ML 推論サービスのコスト最適化

シナリオ: 画像認識 API を Cloud Run でホスティング。CPU 集約的で同時実行数は低いが、デフォルト 60% CPU ターゲットではインスタンスが過剰にスケールアウト。

Bash CPU 80% / 同時実行数無効化 
gcloud beta run services update ml-inference-api \
  --scaling-cpu-target=0.80 \
  --scaling-concurrency-target=disabled
✨ 効果: CPU 80% まで許容しインスタンス数を抑制。CPU のみのシンプルなスケーリングでコスト最大 25% 削減の可能性。

🌐 UC2: I/O バウンドな API ゲートウェイの高可用性確保

シナリオ: 外部 API への中継サービス。リクエストの大部分が I/O wait で CPU 使用率は低い。トラフィックスパイク時にも低レイテンシを維持したい。

Bash 同時実行数 30% / CPU 無効化 
gcloud beta run services update api-gateway \
  --scaling-cpu-target=disabled \
  --scaling-concurrency-target=0.30
✨ 効果: 同時実行数 30% でスケールアウト開始 → 70% 分のバッファが常に確保され、安定した低レイテンシを維持。

💥 UC3: バースト性の高い Webhook 受信サービス

シナリオ: 外部 SaaS からの Webhook 受信。通常時はトラフィックが少ないが、バッチ処理完了時に短時間で大量リクエスト到着。

YAML Webhook 受信サービスの設定例 
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  annotations:
    run.googleapis.com/launch-stage: BETA
  name: webhook-receiver
spec:
  template:
    metadata:
      annotations:
        run.googleapis.com/scaling-cpu-target: '0.40'
        run.googleapis.com/scaling-concurrency-target: '0.25'
        autoscaling.knative.dev/minScale: '1'
        autoscaling.knative.dev/maxScale: '50'
    spec:
      containers:
        - image: gcr.io/my-project/webhook-receiver:latest
✨ 効果: 低めのターゲットで早期スケールアウト。minScale: 1 と組み合わせ、バースト時のコールドスタート遅延を最小化。

🔗 関連サービス・機能

⚡Cloud Run インスタンスオートスケーリング - スケーリングコントロールはメトリクスベーススケーリングの動作を制御。オンデマンドスケーリング (ゼロからのスケール) とは独立して動作
📈Cloud Run 最大/最小インスタンス数設定 - スケーリングコントロールと併用してコストの上限制御やコールドスタート回避を実現
👥Cloud Run 同時実行数設定 (max concurrency) - インスタンスあたりの最大同時リクエスト数の設定。スケーリングコントロールの同時実行数ターゲットとは別の設定
🔒Cloud Run マニュアルスケーリング - オートスケーリングの代替として固定インスタンス数での運用が必要な場合に利用
📊Cloud Monitoring (Metrics Explorer) - run.googleapis.com/scaling/recommended_instances メトリクスでスケーリングドライバーの動作を可視化・分析
