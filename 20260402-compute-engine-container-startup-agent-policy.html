⚙️ Compute Engine: コンテナ起動エージェントの組織ポリシー制約

Compute Engine Preview
📅 2026-04-02
Container startup agent org policy constraint

📋 概要

Compute Engine において、非推奨となったコンテナ起動エージェント (konlet) の使用を組織レベルで制御するための新しいマネージド制約 constraints/compute.managed.disableVmsWithContainerStartupAgent が Preview として提供されました。

この制約を適用することで、gce-container-declaration メタデータを使用する Compute Engine インスタンスの新規作成をブロックできます。 特に注目すべき点として、ドライランモードでの適用にも対応しており、リソースの作成をブロックせずに非推奨メタデータの使用状況を監査ログで特定できます。

🔄 Before / After 比較

❌ Before (アップデート前)

  • 組織レベルで gce-container-declaration の使用を一括禁止する手段がなかった
  • 非推奨機能の使用プロジェクト特定には各プロジェクトで個別にコマンド実行が必要
  • 段階的な制限の仕組みがなく、移行計画の策定が困難

✅ After (アップデート後)

  • 組織ポリシー制約で組織/フォルダ/プロジェクト単位で一括制御が可能に
  • ドライランモードで既存ワークフローに影響なく使用状況を監査
  • MIG を含む包括的な制御が実現

⚡ 主要機能

🔒 マネージド制約によるコンテナ起動エージェントの使用制限

→ ブール型のマネージド制約 constraints/compute.managed.disableVmsWithContainerStartupAgent を適用することで、gce-container-declaration メタデータキーを持つ Compute Engine インスタンスの新規作成を防止
→ インスタンステンプレートに該当メタデータキーが含まれている場合、そのテンプレートを使用したインスタンス作成もブロック対象

⚙️ マネージドインスタンスグループ (MIG) への適用

→ gce-container-declaration メタデータキーを含むインスタンステンプレートを使用する MIG に対しても制約が適用される
→ 新規の MIG 作成やスケールアウト時に非推奨メタデータの使用を防止

🔍 ドライランモード対応

→ 組織ポリシーをドライランモードで適用し、実際のリソース作成をブロックせずに違反を検出
→ 監査ログを通じて、非推奨メタデータを使用しているプロジェクトやワークフローを特定
→ ドライランの結果を確認した上で、本番ポリシーへの昇格が可能

🔧 設定方法

💡 前提条件: 組織ポリシー管理者ロール (roles/orgpolicy.policyAdmin) が付与されていること。ドライランモードの監査ログを確認するには、ログ閲覧者ロールが必要です。

Step 1: 影響を受けるインスタンスの特定

Bash 非推奨メタデータを使用している VM を一覧表示 
# プロジェクト内で gce-container-declaration メタデータを使用している VM を一覧表示
gcloud compute instances list \
  --filter="metadata.items.key:gce-container-declaration"

Step 2: ドライランモードで組織ポリシーを設定

YAML ドライランモードのポリシー定義 
# policy-dryrun.yaml
name: organizations/ORGANIZATION_ID/policies/compute.managed.disableVmsWithContainerStartupAgent
dryRunSpec:
  rules:
    - enforce: true
Bash ドライランモードでポリシーを適用 
# ドライランモードでポリシーを適用
gcloud org-policies set-policy policy-dryrun.yaml \
  --update-mask=dryRunSpec

Step 3: 監査ログで違反を確認

Google Cloud Console の「組織ポリシー」ページで、該当する制約の「ドライラン」タブから違反ログを確認できます。

Step 4: 本番ポリシーとして適用

YAML 本番ポリシーの定義 
# policy-enforce.yaml
name: organizations/ORGANIZATION_ID/policies/compute.managed.disableVmsWithContainerStartupAgent
spec:
  rules:
    - enforce: true
Bash 本番ポリシーを適用 
# 本番ポリシーを適用
gcloud org-policies set-policy policy-enforce.yaml \
  --update-mask=spec
⚠️ 注意: 本番ポリシーを適用すると、gce-container-declaration メタデータを使用する VM の新規作成がブロックされます。必ずドライランモードで影響を確認してから適用してください。

👍 メリット

💼 ビジネス面

→ コンプライアンスの強化: 非推奨機能の使用を組織全体で統一的に管理でき、計画的な移行が可能

🛠️ 技術面

→ 段階的な移行: ドライランモードと本番モードの 2 段階で安全に移行を推進
→ 包括的な制御: 個別 VM、インスタンステンプレート、MIG を含めた制御が可能
→ 階層的なポリシー管理: 組織、フォルダ、プロジェクトの階層に沿った継承・上書き

⚠️ デメリット・制約事項

制限事項

  • ❌ 本機能は Preview のため、SLA の対象外であり、本番環境での使用は推奨されない場合がある
  • ❌ 既存の VM には影響しないため、既にデプロイ済みのコンテナ起動エージェント VM は引き続き動作する
  • ❌ 制約は新規作成のみを対象としており、既存 VM のメタデータ更新には適用されない

考慮すべき点

  • 💡 ドライランモードの監査ログ確認には適切な IAM ロールとログ設定が必要
  • 💡 移行先 (startup scripts, cloud-init, Cloud Run, Batch, GKE) を事前に検討しておくこと
  • 💡 MIG が制約の対象となるため、自動スケーリング中のインスタンス作成にも影響

🎯 ユースケース

🏢 大規模組織での非推奨機能の段階的廃止

シナリオ: 数十のプロジェクトを持つ大規模組織で、コンテナ起動エージェントを使用している VM を把握し、2026 年 7 月 31 日のデッドラインまでに移行を完了したい。

Bash 段階的な廃止手順 
# 1. まずドライランモードで制約を組織全体に適用
gcloud org-policies set-policy policy-dryrun.yaml \
  --update-mask=dryRunSpec

# 2. 監査ログで影響を受けるプロジェクトを特定
# (Cloud Console の組織ポリシーページでドライラン違反ログを確認)

# 3. 各プロジェクトでの移行完了後、本番ポリシーを適用
gcloud org-policies set-policy policy-enforce.yaml \
  --update-mask=spec

効果: 影響範囲を事前に把握した上で、プロジェクトごとに移行を進め、最終的に組織全体で非推奨機能の使用を禁止できる。

🛡️ セキュリティチームによるガバナンス強化

シナリオ: セキュリティチームが、新規プロジェクトで非推奨のデプロイ方法が使用されることを防ぎたい。

効果: 組織ポリシー制約を適用することで、新規プロジェクトでの非推奨機能の使用を自動的にブロックし、推奨されるコンテナデプロイ方法 (startup scripts, cloud-init, Cloud Run, Batch, GKE) の採用を促進できる。
