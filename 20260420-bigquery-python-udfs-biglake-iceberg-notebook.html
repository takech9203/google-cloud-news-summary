📌 概要

BigQuery に対して 3 つの重要なアップデートが発表されました。Python UDF エコシステムの成熟、オープンフォーマットデータレイクからの移行パスの拡充、データ分析の入門体験の改善という 3 つの方向で BigQuery プラットフォームを強化します。

🔄 Before / After 比較

❌ Before

  • Python UDF は Pandas DataFrame のみサポート (Arrow ネイティブ処理不可)
  • Python UDF のリソース使用状況を Cloud Monitoring で監視できない
  • コンテナ同時リクエスト数を CREATE FUNCTION で制御できない
  • Python UDF の実行コストが INFORMATION_SCHEMA.JOBS から確認できない
  • 外部カタログからの BigLake Iceberg メタデータ移行は手動
  • ノートブックギャラリーは Preview 段階

✅ After

  • Apache Arrow RecordBatch によるゼロコピーに近い列指向バッチ処理が選択可能
  • Cloud Monitoring で CPU/メモリ/同時リクエスト数をリアルタイム監視
  • container_request_concurrency で 1~1000 の範囲で同時実行数を制御
  • external_service_costs カラムで Python UDF コストを可視化
  • BigQuery Migration Service でメタデータ移行を自動化
  • ノートブックギャラリーが GA (SQL, Python, Spark テンプレート)

⚡ 主要機能 1: Python UDF 機能強化 (Preview)

🏹

Apache Arrow ベクトル化

Apache Arrow の RecordBatch インターフェースで、Pandas のオーバーヘッドを回避したゼロコピーに近い列指向バッチ処理が可能に。pyarrow.compute モジュールによる高速演算を利用できます。

📈

Cloud Monitoring 統合

CPU 使用率、メモリ使用率、最大同時リクエスト数の 3 メトリクスが Cloud Monitoring にエクスポート。Metrics Explorer やダッシュボードから監視可能。

🔧

同時実行制御

container_request_concurrency オプションでコンテナあたりの同時リクエスト数を 1~1000 で制御。デフォルトは 80。

💰

コスト可視化

INFORMATION_SCHEMA.JOBS の external_service_costs カラムと Job API の ExternalServiceCosts フィールドでコストを確認。

💻 コードサンプル: Apache Arrow ベクトル化 UDF

Apache Arrow の pyarrow.compute モジュールを使用した高速なベクトル化 UDF の実装例です。

SQL Arrow ベクトル化 UDF の作成 
CREATE FUNCTION `PROJECT_ID.DATASET_ID`.multiplyVectorizedArrow(
  x FLOAT64, y FLOAT64
)
RETURNS FLOAT64
LANGUAGE python
OPTIONS(
  runtime_version="python-3.11",
  entry_point="vectorized_multiply_arrow"
)
AS r'''
import pyarrow as pa
import pyarrow.compute as pc

def vectorized_multiply_arrow(batch: pa.RecordBatch):
  x = batch.column('x')
  y = batch.column('y')
  return pc.multiply(x, y)
''';
SQL コンテナリソース設定の例 
CREATE FUNCTION `PROJECT_ID.DATASET_ID`.resizeImage(image BYTES)
RETURNS BYTES
LANGUAGE python
OPTIONS(
  entry_point='resize_image',
  runtime_version='python-3.11',
  packages=['Pillow==11.2.1'],
  container_memory='2Gi',
  container_cpu=2,
  container_request_concurrency=100
)
AS r"""
import io
from PIL import Image

def resize_image(image_bytes):
  img = Image.open(io.BytesIO(image_bytes))
  resized_img = img.resize((256, 256), Image.Resampling.LANCZOS)
  output_stream = io.BytesIO()
  resized_img.convert('RGB').save(output_stream, format='JPEG')
  return output_stream.getvalue()
""";

🔧 設定手順

ステップ 1: 必要な API の有効化

Bash 
gcloud services enable \
  biglake.googleapis.com \
  bigquerydatatransfer.googleapis.com \
  bigquerymigration.googleapis.com \
  secretmanager.googleapis.com \
  storagetransfer.googleapis.com

ステップ 2: 移行の作成

Google Cloud Console で Migration > Services ページに移動し、「Register or Migrate Open Lakehouse」セクションの「Create migration」をクリック。カタログタイプ、リージョン、ソースシステムの URL を設定して移行を実行します。

💡 前提条件: BigLake API、BigQuery Data Transfer API、BigQuery Migration API、Secret Manager API、Storage Transfer API が有効化されていること。BigQuery Admin ロール (roles/bigquery.admin) が付与されていること。サービスエージェントに Storage Transfer Admin、Service Usage Consumer、Storage Admin、BigLake Admin ロールが付与されていること。BigLake カタログが作成済みであること。

🔧 使い方

ステップ 1: ノートブックギャラリーを開く

BigQuery Studio のエディタペインで「SQL query」の横にあるドロップダウン矢印をクリックし、「Notebook > All templates」を選択します。

ステップ 2: テンプレートからノートブックを作成

ギャラリーからテンプレートを選択し、「Use this template」をクリックして実行可能なノートブックに変換します。

🔧 設定方法: Python UDF ベクトル化

💡 前提条件: BigQuery プロジェクトとデータセットが作成済みであること。bigquery.routines.create 権限を持つ IAM ロールが付与されていること。

ステップ 1: Apache Arrow ベクトル化 UDF の作成

SQL ベクトル化 UDF の作成例 
CREATE FUNCTION `my_project.my_dataset`.my_vectorized_udf(
  col1 FLOAT64, col2 FLOAT64
)
RETURNS FLOAT64
LANGUAGE python
OPTIONS(
  runtime_version="python-3.11",
  entry_point="my_function",
  max_batching_rows=1000
)
AS r'''
import pyarrow as pa
import pyarrow.compute as pc

def my_function(batch: pa.RecordBatch):
  return pc.add(batch.column('col1'), batch.column('col2'))
''';

ステップ 2: Cloud Monitoring でメトリクスを確認

Cloud Monitoring の Metrics Explorer で BigQuery Managed Routine Invocation リソースタイプを選択し、CPU 使用率やメモリ使用率を確認します。BigQuery のジョブ詳細画面の「Cloud Monitoring dashboard」リンクからもアクセスできます。

✅ メリット

ビジネス面

✓ データレイクハウス移行の加速: 外部カタログからのメタデータ移行自動化により、既存のデータレイクを BigLake ベースのレイクハウスアーキテクチャに効率的に統合
✓ コスト可視化による予算管理: Python UDF のコストが INFORMATION_SCHEMA.JOBS で確認可能になり、チームごとの利用コスト配分や予算管理が実現
✓ データ分析の民主化: ノートブックギャラリーのプリビルトテンプレートにより、初心者でも迅速にデータ分析を開始可能

技術面

✓ 処理パフォーマンスの向上: Apache Arrow RecordBatch によるゼロコピーに近い列指向バッチ処理で Pandas のオーバーヘッドを回避
✓ 運用監視の強化: Cloud Monitoring 統合により、リソース使用状況をリアルタイムに把握し、ボトルネック特定やキャパシティプランニングが容易に
✓ 同時実行の最適化: container_request_concurrency オプションにより、スループットとレイテンシのバランスを最適化

⚠️ デメリット・制約事項

制限事項

⚠️ Preview 段階: Python UDF の全機能は Preview 段階であり、SLA の対象外。本番ワークロードでの利用には注意が必要です。
✗ BigLake Iceberg メタデータ移行はワンタイム同期のみで、継続的な同期は非対応。移行後のソースとデスティネーション双方への書き込みはデータ損失のリスク
✗ BigLake Iceberg メタデータ移行は 1 回あたり最大 10,000 テーブルまで。大規模ワークロードは複数回に分割する必要
✗ Python UDF は python-3.11 ランタイムのみサポート。JSON、RANGE、INTERVAL、GEOGRAPHY データ型は非対応
✗ Python UDF コンテナは最大 4 vCPU / 16 GiB まで。VPC ネットワーク、Assured Workloads、CMEK は非対応
✗ Python UDF の結果はキャッシュされない (非決定的と見なされるため)

考慮すべき点

  • 💡 Python UDF イメージストレージの 10 GiB/プロジェクト/リージョンのクォータに注意。不要な UDF は DROP FUNCTION で削除してクォータを解放
  • 💡 ミューテーションレート (30 回/分/プロジェクト/リージョン) を超えないよう CI/CD パイプラインでの UDF デプロイ頻度を管理
  • 💡 Arrow ベクトル化 UDF は Pandas UDF と異なるインターフェースを持つため、既存 Pandas UDF からの移行には関数シグネチャの変更が必要
  • 💡 メタデータ移行後の BigLake テーブルには BigLake 料金が適用される

🎯 ユースケース

🔍 ユースケース 1: 大規模テキストデータの高速処理

シナリオ: 数億行のログデータに対して、正規表現によるパターン抽出やテキスト変換を行う Python UDF を使用。従来の行単位処理ではスループットが不足。

効果: Apache Arrow のゼロコピー列指向処理により、Pandas 経由と比較してデータのシリアライゼーションオーバーヘッドが削減され、大規模バッチ処理のスループットが向上。

SQL 大規模テキスト処理の実装例 
CREATE FUNCTION `project.dataset`.extract_patterns(text STRING)
RETURNS STRING
LANGUAGE python
OPTIONS(
  runtime_version="python-3.11",
  entry_point="extract",
  max_batching_rows=5000,
  container_cpu=4,
  container_memory='8Gi',
  container_request_concurrency=200
)
AS r'''
import pyarrow as pa
import pyarrow.compute as pc

def extract(batch: pa.RecordBatch):
  texts = batch.column('text')
  return pc.utf8_replace_slice(texts, 0, 0, "processed_")
''';

🏢 ユースケース 2: オンプレミスデータレイクの Google Cloud 移行

シナリオ: オンプレミスの Apache Hive Metastore に数千テーブルのメタデータを持つデータレイクを、Google Cloud の BigLake レイクハウスアーキテクチャに移行。

効果: BigQuery Migration Service を使用してメタデータを自動移行することで、テーブル定義やスキーマ情報を手動で再作成する必要がなくなり、移行にかかる時間と労力が大幅に削減。移行後は BigQuery からの統合クエリや Iceberg テーブルの管理が可能に。

🎓 ユースケース 3: データ分析チームのオンボーディング

シナリオ: 新しいデータアナリストがチームに参加する際、BigQuery の使い方やデータ分析のベストプラクティスを効率的に学習させたい。

効果: ノートブックギャラリーのプリビルトテンプレートを使用し、SQL の基本クエリ、Python によるデータ探索、BigQuery DataFrames の活用、生成 AI 機能の利用などを段階的に学習。実行可能なサンプルコードが含まれており、すぐに手を動かして学べる。

💰 料金

Python UDF

Python UDF は追加料金なしで提供されます。課金が有効な場合:

  • BigQuery Services SKU に基づいて課金
  • 料金は Python UDF 呼び出し時のコンピュートおよびメモリ消費量に比例
  • UDF コンテナイメージのビルド・リビルドにかかるリソースも課金対象
  • 外部またはインターネットへのネットワーク Egress が発生する場合、Cloud Networking の Premium Tier Egress 料金が適用

BigLake Iceberg メタデータ移行

  • メタデータの転送自体は無料
  • 移行後のデータには BigLake 料金が適用

📋 まとめ

今回のアップデートは、BigQuery の Python UDF エコシステムの成熟 (パフォーマンス、監視、コスト管理)、オープンフォーマットデータレイクからの移行パスの拡充、そしてデータ分析の入門体験の改善という 3 つの方向で BigQuery プラットフォームを強化するものです。

🚀 特に注目すべきポイント:
  • Python UDF の Apache Arrow ベクトル化と Cloud Monitoring 統合は、大規模データ処理パイプラインを運用するチームにとって即座に活用すべき機能
  • BigLake Iceberg メタデータ移行は、既存の Hive Metastore ベースのデータレイクを持つ組織が Google Cloud レイクハウスへの移行を検討する契機
  • ノートブックギャラリー GA は、データ分析チームの生産性向上とオンボーディング効率化に直結
