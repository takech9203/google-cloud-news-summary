📋 概要

Compute Engine のマネージド インスタンス グループ (MIG) における インスタンス フレキシビリティ ポリシーが拡張されました。インスタンス テンプレートで指定された 最小 CPU プラットフォームとディスク定義を インスタンス選択ごとにオーバーライドできるようになり、異なる CPU アーキテクチャやストレージ構成の マシンタイプを単一の MIG 内で柔軟に運用できます。

🔄 Before / After 比較

❌ アップデート前

  • マシンタイプのみオーバーライド可能
  • CPU プラットフォームはテンプレートに固定
  • ディスク構成もテンプレートに固定
  • 異なるアーキテクチャ (Intel/AMD/Arm) は別々の MIG が必要

✅ アップデート後

  • マシンタイプ + CPU + ディスクをオーバーライド可能
  • 各インスタンス選択に minCpuPlatform を指定可能
  • 各インスタンス選択に disks を定義可能
  • x86 と Arm を単一 MIG 内で混在可能

🏗️ アーキテクチャ図

インスタンス フレキシビリティ ポリシーにより、MIG は複数のインスタンス選択の中から rank (優先度) とリソース可用性に基づいて最適なマシンタイプを自動選択します。

flowchart TD
    A[MIG インスタンス フレキシビリティ ポリシー] --> B[instanceFlexibilityPolicy]
    B --> C["instanceSelection 1\nrank: 1 - 最優先"]
    B --> D["instanceSelection 2\nrank: 2 - 次優先"]
    B --> E["instanceSelection 3\nrank: 3 - フォールバック"]

    C --> C1["machineTypes:\nc3-standard-8, c3-standard-16"]
    C --> C2["minCpuPlatform:\nIntel Sapphire Rapids"]
    C --> C3["disks: PD-SSD 100GB"]

    D --> D1["machineTypes:\nn2-standard-8, n2-standard-16"]
    D --> D2["minCpuPlatform:\nIntel Ice Lake"]
    D --> D3["disks: PD-SSD 200GB"]

    E --> E1["machineTypes:\nt2a-standard-8"]
    E --> E2["minCpuPlatform:\nAmpere Altra"]
    E --> E3["disks: PD-Balanced 100GB"]

    C1 & D1 & E1 --> F["MIG が可用性に基づき\n最適なマシンタイプを自動選択"]
    F --> G[VM インスタンス作成]

⚡ 主要機能

🖥️ 1. 最小 CPU プラットフォームのオーバーライド

→インスタンス選択ごとに minCpuPlatform フィールドを指定可能
→インスタンス テンプレートの CPU プラットフォーム設定を上書きし、各マシンタイプに適切なプラットフォームを設定
→対応例: Intel Sapphire Rapids, Intel Ice Lake, AMD Milan, Ampere Altra

💾 2. ディスク定義のオーバーライド

→インスタンス選択ごとに disks フィールドでディスク構成を定義可能
→ディスクタイプ (PERSISTENT / SCRATCH)、モード (READ_WRITE / READ_ONLY)、デバイス名、ソースなどを指定可能
→マシンタイプの性能特性に合わせた最適なストレージ構成を選択

🌐 3. クロスアーキテクチャ対応

→x86 (Intel/AMD) と Arm (Ampere Altra) のマシンタイプを同一 MIG 内で混在可能
→各アーキテクチャに適した CPU プラットフォームとディスク構成を個別に設定
→リソース可用性の最大化とコスト最適化を同時に実現

📐 技術仕様

instanceFlexibilityPolicy の主要フィールド

フィールド 型 説明
instanceSelections[] map VM 作成時に使用するプロパティを定義する名前付きインスタンス選択
instanceSelections[].machineTypes[] string 完全なマシンタイプ名 (例: n1-standard-16)
instanceSelections[].rank integer 優先度。低い値ほど高い優先度
instanceSelections[].minCpuPlatform string 最小 CPU プラットフォーム名 NEW
instanceSelections[].disks[] object アタッチするディスクの定義 NEW
instanceSelections[].disks[].type enum PERSISTENT または SCRATCH
instanceSelections[].disks[].mode enum READ_WRITE または READ_ONLY
instanceSelections[].disks[].deviceName string デバイス名 (/dev/disk/by-id/google-* に反映)

🔧 設定方法

前提条件

✓Google Cloud プロジェクトで Compute Engine API が有効化されていること
✓適切な IAM 権限 (compute.instanceGroupManagers.update) を持つこと
✓対象リージョンで使用するマシンタイプが利用可能であること

ステップ 1: gcloud CLI で MIG にフレキシビリティを追加

Bash 基本的なマシンタイプの複数指定 
gcloud compute instance-groups managed update INSTANCE_GROUP_NAME \
  --region REGION \
  --instance-selection-machine-types c3-standard-8,c3-standard-16,n2-standard-8

✅ メリット

📈 ビジネス面

→リソース調達の確実性向上: 複数アーキテクチャにまたがるフォールバックで、需要の高い時期でもVM を確保しやすい
→コスト最適化: 優先度付きの構成で、コスト効率の高い構成を優先しつつ代替構成にフォールバック可能

🔧 技術面

→運用の簡素化: 異なるアーキテクチャ用に複数 MIG を管理する必要がなくなる
→柔軟なストレージ: マシンタイプの性能に合わせたディスク構成を選択可能
→マルチアーキテクチャ: Arm 移行を段階的に進めやすい

⚠️ デメリット / 制約事項

制限事項:
  • 現在 Preview ステータス -- 本番環境での利用には SLA が適用されない可能性がある
  • 既存の VM はポリシー変更後も既存のマシンタイプを使い続ける。新しい設定を反映するには既存 VM を削除して MIG をリサイズする必要がある
  • 使用するマシンタイプが対象リージョンでサポートされている必要がある
考慮すべき点:
  • アプリケーションが複数のアーキテクチャ (x86 / Arm) で動作するよう、マルチアーキテクチャ対応のコンテナイメージやバイナリを準備する必要がある
  • 特定のハードウェアが保証されるわけではないため、確実な割り当てが必要な場合はリザベーションとの併用を検討すべき
  • CPU プラットフォームやディスク構成が異なる VM が混在するため、モニタリングとパフォーマンス管理に注意が必要

🎯 ユースケース

1. 大規模バッチ処理の可用性最大化

シナリオ: 大量のデータ処理バッチジョブを実行する際に、特定のマシンタイプが不足して VM のプロビジョニングに失敗するケース。

🥈 n2-standard-16

効果: 最も高性能な Intel Sapphire Rapids を優先しつつ、可用性に応じて Ice Lake や Arm ベースにフォールバックし、バッチジョブの実行を確実に開始できる。

2. Arm 移行の段階的推進

シナリオ: x86 から Arm への移行を進めたいが、全面移行のリスクを抑えたい場合に、同一 MIG 内で両アーキテクチャを混在させて段階的に Arm の比率を高める。

効果: Arm マシンタイプを優先度の高いインスタンス選択に設定し、x86 をフォールバックとすることで、Arm 移行を安全に推進しつつ、Arm リソースが不足した場合は x86 にフォールバック可能。

💰 料金情報

✅ ポイント: インスタンス フレキシビリティ ポリシー自体には追加料金は発生しません。料金は実際に作成された VM のマシンタイプとディスク構成に基づいて標準料金が適用されます。

マシンタイプ別料金比較 (us-central1)

c3-standard-8 8 32 GB 約 $245

※ 料金は継続利用割引 (SUD) 適用前の概算です。実際の料金は利用状況やリージョンにより異なります。

🌍 利用可能リージョン

Preview 機能として、Compute Engine MIG が利用可能なすべてのリージョンで使用できます。ただし、各インスタンス選択で指定するマシンタイプと CPU プラットフォームがそのリージョンで利用可能である必要があります。

💡 確認方法: リージョンごとのマシンタイプの可用性は 利用可能なリージョンとゾーン で確認できます。

📝 まとめ

Compute Engine MIG のインスタンス フレキシビリティ ポリシーが拡張され、最小 CPU プラットフォームとディスク定義のオーバーライドが Preview として利用可能になりました。 異なる CPU アーキテクチャやストレージ構成のマシンタイプを単一の MIG 内で柔軟に運用でき、リソースの可用性向上とコスト最適化が期待できます。 大規模ワークロードの運用や Arm 移行を検討中のユーザーは、この Preview 機能を早期に検証することを推奨します。
