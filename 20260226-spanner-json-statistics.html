📋 概要

Cloud Spanner が PostgreSQL 方言データベース 向けに、各種統計ビューにおける JSON 文字列バージョンのカラム を提供開始。これにより、PostgreSQL 方言データベースでも SPANNER_SYS テーブルからレイテンシ分布やロック統計などの詳細な統計情報を JSON 形式で取得できるようになる。

今回追加された JSON 文字列カラムは、トランザクション統計、クエリ統計、ロック統計 の各ビューに対応しており、これまで GoogleSQL 方言でのみ ARRAY<STRUCT> 型として提供されていた統計情報を、PostgreSQL 方言でも JSON 互換文字列として利用可能にするもの。

🔄 Before / After 比較

❌ Before (アップデート前)

  • TOTAL_LATENCY_DISTRIBUTION (ARRAY<STRUCT> 型) が PostgreSQL 方言で非サポート
  • OPERATIONS_BY_TABLE がサポートされず、テーブルごとの INSERT/UPDATE 統計取得不可
  • LATENCY_DISTRIBUTION がサポートされず、クエリ実行時間ヒストグラム取得不可
  • SAMPLE_LOCK_REQUESTS がサポートされず、ロック競合詳細取得不可

✅ After (アップデート後)

  • TOTAL_LATENCY_DISTRIBUTION_JSON_STRING でコミットレイテンシ分布を JSON 取得
  • OPERATIONS_BY_TABLE_JSON_STRING でテーブル書き込み統計を JSON 取得
  • LATENCY_DISTRIBUTION_JSON_STRING でクエリ実行時間ヒストグラムを JSON 取得
  • SAMPLE_LOCK_REQUESTS_JSON_STRING でロック競合詳細を JSON 取得
  • DISTRIBUTION_PERCENTILE() 関数でパーセンタイルレイテンシ算出が可能に

⚡ 主要機能 (4つの新規 JSON 文字列カラム)

📊 TOTAL_LATENCY_DISTRIBUTION_JSON_STRING

統計ビュー: トランザクション統計

  • コミットレイテンシ (最初のオペレーションからコミット/アボートまで) のヒストグラムを JSON 提供
  • 複数回アボート後のコミットでは各試行のレイテンシを計測
  • 値は秒単位

対象: TXN_STATS_TOP_MINUTE, TXN_STATS_TOP_10MINUTE, TXN_STATS_TOP_HOUR

📝 OPERATIONS_BY_TABLE_JSON_STRING

統計ビュー: トランザクション統計

  • テーブルごとの INSERT/UPDATE 操作の影響を JSON 提供
  • 影響行数 (INSERT_OR_UPDATE_COUNT) と書き込みバイト数 (INSERT_OR_UPDATE_BYTES) を含む
  • テーブル書き込み負荷の可視化に活用可能

対象: TXN_STATS_TOP_MINUTE, TXN_STATS_TOP_10MINUTE, TXN_STATS_TOP_HOUR

⏱️ LATENCY_DISTRIBUTION_JSON_STRING

統計ビュー: クエリ統計

  • クエリ実行時間のヒストグラムを JSON 提供
  • 値は秒単位
  • 失敗クエリ (構文エラー、リトライ後成功含む) の統計も含む

対象: QUERY_STATS_TOP_MINUTE, QUERY_STATS_TOP_10MINUTE, QUERY_STATS_TOP_HOUR

🔒 SAMPLE_LOCK_REQUESTS_JSON_STRING

統計ビュー: ロック統計

  • ロック競合に寄与したサンプルロックリクエスト詳細を JSON 提供
  • 各サンプル: lock_mode, column, transaction_tag の 3 フィールド
  • 1 行あたり最大 20 サンプル

対象: LOCK_STATS_TOP_MINUTE, LOCK_STATS_TOP_10MINUTE, LOCK_STATS_TOP_HOUR

⚙️ 設定方法とコードサンプル

前提条件

1.Cloud Spanner インスタンスが作成済みであること
2.PostgreSQL 方言のデータベースが作成済みであること
3.spanner.databases.read 権限 (例: roles/spanner.databaseReader) が付与されていること

Step 1: トランザクション統計の JSON 文字列カラムを取得

SQL PostgreSQL 方言でのトランザクション統計取得 
-- PostgreSQL 方言でのトランザクション統計取得
SELECT
  interval_end,
  fprint,
  total_latency_distribution_json_string,
  operations_by_table_json_string
FROM spanner_sys.txn_stats_top_minute
WHERE interval_end = (
  SELECT MAX(interval_end)
  FROM spanner_sys.txn_stats_top_minute
)
ORDER BY avg_commit_latency_seconds DESC;

Step 2: クエリ統計のパーセンタイルレイテンシを算出

SQL P50/P95/P99 レイテンシの算出 
-- PostgreSQL 方言での P99 レイテンシ算出
SELECT
  text_fingerprint,
  execution_count,
  avg_latency_seconds,
  SPANNER_SYS.DISTRIBUTION_PERCENTILE(
    latency_distribution_json_string, 50.0
  ) AS p50_latency,
  SPANNER_SYS.DISTRIBUTION_PERCENTILE(
    latency_distribution_json_string, 95.0
  ) AS p95_latency,
  SPANNER_SYS.DISTRIBUTION_PERCENTILE(
    latency_distribution_json_string, 99.0
  ) AS p99_latency
FROM spanner_sys.query_stats_top_minute
WHERE interval_end = (
  SELECT MAX(interval_end)
  FROM spanner_sys.query_stats_top_minute
)
ORDER BY p99_latency DESC
LIMIT 10;

Step 3: ロック統計のサンプルリクエストを JSON で取得

SQL ロック競合のサンプルリクエスト取得 
-- PostgreSQL 方言でのロック統計取得
SELECT
  cast(row_range_start_key AS varchar) AS row_range_start_key,
  lock_wait_seconds,
  sample_lock_requests_json_string
FROM spanner_sys.lock_stats_top_minute
WHERE interval_end = (
  SELECT MAX(interval_end)
  FROM spanner_sys.lock_stats_top_minute
)
ORDER BY lock_wait_seconds DESC;

✅ メリット

💼 ビジネス面

→ PostgreSQL 方言ユーザーの運用効率向上: パフォーマンス分析が本番環境で完結し、運用の効率化とダウンタイム削減に寄与
→ トラブルシューティング時間の短縮: レイテンシ分布やロック競合の詳細をデータベース内で直接取得でき、問題特定までの時間が短縮

🔧 技術面

→ GoogleSQL/PostgreSQL 方言間の機能パリティ向上: GoogleSQL でのみ利用可能だった統計カラム情報を PostgreSQL 方言でも同等に取得可能
→ JSON 形式の汎用性: STRING 型 JSON はアプリケーション側解析が容易、外部監視ツールとの連携も簡便
→ パーセンタイルレイテンシ算出: DISTRIBUTION_PERCENTILE() で P50/P95/P99 等の実用的メトリクスを PostgreSQL 方言から算出可能

⚠️ デメリット・制約事項

制限事項

  • ⚠️ JSON 文字列カラムは STRING 型であり、GoogleSQL の ARRAY<STRUCT> 型とはクエリの記述方法が異なる
  • ⚠️ SAMPLE_LOCK_REQUESTS_JSON_STRING のサンプル数は 1 行あたり最大 20 件に制限
  • ⚠️ ロック競合のサンプリングはランダムであり、競合の片方 (ホルダーまたはウェイター) のみが記録される可能性あり

考慮すべき点

  • 💡 統計データはベストエフォートで収集。内部ネットワーキングの問題等により一部欠落の可能性あり
  • 💡 保持期間は固定 (1分: 6時間、10分: 4日、1時間: 30日)。長期保存には定期的なエクスポートを推奨
  • 💡 GoogleSQL の ARRAY<STRUCT> カラム使用クエリは、PostgreSQL 方言移行時に書き換えが必要

🎯 ユースケース

📈 ユースケース 1: PostgreSQL 方言データベースでのレイテンシ異常検知

シナリオ: PostgreSQL 方言で運用している Spanner データベースで、特定のトランザクションのレイテンシが急増。原因を特定するためにレイテンシ分布を確認。

SQL 直近 1 時間のトランザクションを P99 レイテンシ降順で取得 
SELECT
  fprint,
  commit_attempt_count,
  avg_commit_latency_seconds,
  SPANNER_SYS.DISTRIBUTION_PERCENTILE(
    total_latency_distribution_json_string, 99.0
  ) AS p99_latency,
  SPANNER_SYS.DISTRIBUTION_PERCENTILE(
    total_latency_distribution_json_string, 50.0
  ) AS p50_latency,
  operations_by_table_json_string
FROM spanner_sys.txn_stats_top_hour
WHERE interval_end = (
  SELECT MAX(interval_end)
  FROM spanner_sys.txn_stats_top_hour
)
ORDER BY p99_latency DESC
LIMIT 5;
効果: P99 と P50 のレイテンシ差が大きいトランザクションを特定し、テール・レイテンシの原因を発見。operations_by_table_json_string でどのテーブルへの書き込みが集中しているかも把握可能。

🔒 ユースケース 2: ロック競合の原因分析

シナリオ: PostgreSQL 方言のデータベースでトランザクションのアボート率が上昇。ロック統計を使って競合の原因を特定。

SQL ロック競合のサンプルリクエストを JSON で確認 
SELECT
  cast(row_range_start_key AS varchar) AS row_key,
  lock_wait_seconds,
  sample_lock_requests_json_string
FROM spanner_sys.lock_stats_top_10minute
WHERE interval_end = (
  SELECT MAX(interval_end)
  FROM spanner_sys.lock_stats_top_10minute
)
  AND lock_wait_seconds > 1.0
ORDER BY lock_wait_seconds DESC;
効果: ロック待ち時間が長いキー範囲を特定し、sample_lock_requests_json_string から競合を引き起こしているカラムとトランザクションタグを把握。スキーマ設計やトランザクション分離の改善につなげられる。

💰 料金情報

✅ 追加料金なし: JSON 文字列統計カラムの利用に追加料金は発生しない。統計情報の取得は SPANNER_SYS スキーマへの SQL クエリとして実行され、通常の Spanner 読み取り操作として課金される。

Spanner の料金は使用するエディション (Standard / Enterprise / Enterprise Plus) とコンピュート容量およびストレージ使用量に基づいて課金。詳細は Spanner 料金ページ を参照。
