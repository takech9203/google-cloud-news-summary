📊 BigQuery: Continuous Queries でのステートフル操作のサポート

BigQuery Preview Continuous Queries
2026-04-09
Google Cloud Update

🔍 概要

BigQuery の Continuous Queries (継続クエリ) において、ステートフル操作が Preview として利用可能になりました。 これにより、JOIN 操作やウィンドウ集計関数を使用して、複数の行や時間間隔にわたる情報を保持しながら複雑なリアルタイム分析を実行できるようになります。

従来は Dataflow などの専用ストリーミング処理基盤が必要だったワークロードの一部を、BigQuery の SQL だけで実現できる可能性が広がります。

💡 ポイント: Enterprise または Enterprise Plus エディションのリザベーションが必要です。Preview 段階のため SLA 適用対象外。

🔄 Before / After 比較

❌ Before (アップデート前)

  • Continuous Queries はステートレスな処理のみサポート
  • JOIN 操作が使用できなかった
  • 集計関数 (GROUP BY, HAVING) やウィンドウ関数が使用不可
  • ストリーミングデータの結合には Dataflow 等の別基盤が必要
  • リアルタイムエンリッチメントを BigQuery 単体で実現不可

✅ After (アップデート後)

  • Continuous Queries 内で JOIN 操作が利用可能に
  • ウィンドウ集計でリアルタイム時間ベース集計が可能
  • 複数行にまたがる状態保持ロジックを SQL で記述可能
  • Dataflow なしでストリーミング分析パターンを BigQuery 内で完結
  • 参照テーブルとの結合がリアルタイムで実行可能

⚡ 主要機能

🔗 JOIN 操作のサポート
  • Continuous Queries 内でストリーミングデータと他のテーブルを JOIN で結合可能
  • ストリーミングイベントデータにマスターデータを付加するリアルタイムエンリッチメントが実現可能
  • 複数のストリーミングソースを結合した複合的なリアルタイム分析に対応
🪟 ウィンドウ集計 (Windowing Aggregations) のサポート
  • 時間間隔に基づくデータの集計処理が可能
  • タンブリングウィンドウ (固定長の非重複時間間隔) やホッピングウィンドウ (重複する時間間隔) など一般的なストリーミングウィンドウパターンへの対応
  • GROUP BY 句や集計関数と組み合わせたリアルタイム集約処理
🧠 状態管理 (Stateful Processing)
  • 複数の行にわたる情報を保持しながら処理を実行
  • 時間間隔をまたいだデータの蓄積と分析が可能
  • BigQuery のマネージドインフラ上で状態管理が自動的に行われるため、ユーザーが状態ストアを管理する必要がない

🔧 設定方法

前提条件: Enterprise / Enterprise Plus エディションのリザベーション作成済み、CONTINUOUS ジョブタイプの割り当て設定済み、roles/bigquery.admin または適切な IAM ロールが付与されていること。

ステップ 1: Enterprise リザベーションの作成

Bash Enterprise エディションのリザベーション作成 
# Enterprise エディションのリザベーションを作成
bq mk --reservation \
  --project_id=PROJECT_ID \
  --location=LOCATION \
  --reservation_id=continuous_reservation \
  --slots=100 \
  --edition=ENTERPRISE

ステップ 2: CONTINUOUS ジョブタイプのリザベーション割り当て

Bash CONTINUOUS ジョブタイプの割り当て 
# CONTINUOUS ジョブタイプの割り当てを作成
bq mk --reservation_assignment \
  --project_id=PROJECT_ID \
  --location=LOCATION \
  --reservation_id=continuous_reservation \
  --assignee_id=PROJECT_ID \
  --assignee_type=PROJECT \
  --job_type=CONTINUOUS

ステップ 3: ステートフル Continuous Query の実行

SQL JOIN を使ったリアルタイムエンリッチメント例 
-- bq query --use_legacy_sql=false --continuous=true
INSERT INTO `myproject.dataset.enriched_events`
SELECT
  e.event_id,
  e.event_timestamp,
  e.user_id,
  u.user_name,
  u.segment
FROM APPENDS(
  TABLE `myproject.dataset.raw_events`,
  CURRENT_TIMESTAMP() - INTERVAL 10 MINUTE
) AS e
JOIN `myproject.dataset.users` AS u
  ON e.user_id = u.user_id

💡 ユースケース

🛒 ユースケース 1: リアルタイムデータエンリッチメント

シナリオ: EC サイトのクリックストリームデータにユーザー属性情報や商品マスター情報を JOIN で付加し、リアルタイムでエンリッチメントされたデータを下流システムに提供。

SQL クリックストリームエンリッチメント 
INSERT INTO `myproject.analytics.enriched_clickstream`
SELECT
  c.event_id, c.event_timestamp, c.user_id,
  u.user_segment, u.registration_date,
  c.product_id, p.product_name, p.category
FROM APPENDS(
  TABLE `myproject.analytics.raw_clickstream`,
  CURRENT_TIMESTAMP() - INTERVAL 10 MINUTE
) AS c
JOIN `myproject.master.users` AS u
  ON c.user_id = u.user_id
JOIN `myproject.master.products` AS p
  ON c.product_id = p.product_id

✅ 効果: Dataflow パイプラインなしで BigQuery SQL だけでリアルタイムエンリッチメントを実現。

🌡️ ユースケース 2: リアルタイムウィンドウ集計によるモニタリング

シナリオ: IoT センサーから継続的に取り込まれる温度・湿度データを一定時間ごとに集計し、異常検知やダッシュボード表示用のデータを生成。

✅ 効果: ウィンドウ集計 (例: 5 分間隔のタンブリングウィンドウ) で平均温度や最大値を計算し、Bigtable や Pub/Sub に出力。リアルタイムモニタリングパイプラインを BigQuery 内で完結。

🔒 ユースケース 3: ストリーミング JOIN による不正検知

シナリオ: 金融取引データとブラックリストテーブルを JOIN し、疑わしい取引をリアルタイムで検出して Pub/Sub 経由でアラートシステムに通知。

SQL 不正取引のリアルタイム検知 
EXPORT DATA OPTIONS (
  format = 'CLOUD_PUBSUB',
  uri = 'https://pubsub.googleapis.com/projects/myproject/topics/fraud-alerts'
) AS (
  SELECT TO_JSON_STRING(
    STRUCT(t.transaction_id, t.amount, t.timestamp, b.risk_level)
  ) AS message
  FROM APPENDS(
    TABLE `myproject.finance.transactions`,
    CURRENT_TIMESTAMP() - INTERVAL 5 MINUTE
  ) AS t
  JOIN `myproject.finance.blacklist` AS b
    ON t.account_id = b.account_id
  WHERE t.amount > 10000
)

✅ 効果: 取引データとブラックリストのリアルタイム結合で不正検知レイテンシを大幅短縮。従来 Dataflow で構築していたパイプラインを BigQuery SQL で実装。

✅ メリット

ビジネス面

🏗️ ストリーミング分析基盤の簡素化

Dataflow などの別途基盤なしで BigQuery SQL だけでリアルタイム JOIN / ウィンドウ集計を実現。アーキテクチャの複雑さとコストを削減。

⚡ リアルタイムインサイトの迅速な獲得

ストリーミングデータにマスターデータを即座に結合し、エンリッチメントされた情報をリアルタイムで下流システムに提供。

🎓 SQL スキルの活用

標準 SQL を使用するため、既存 SQL スキルでストリーミング処理を実装可能。専門フレームワークの学習コスト削減。

技術面

🤖 マネージドな状態管理

BigQuery が状態管理を自動で実行。ユーザーが状態ストアの運用やスケーリングを行う必要なし。

🔌 既存エコシステムとの統合

リザベーション、オートスケーリング、アイドルスロット共有など既存の BigQuery リソース管理機能をそのまま利用可能。

📤 多様な出力先

JOIN やウィンドウ集計の結果を BigQuery テーブル、Pub/Sub、Bigtable、Spanner に出力。リバース ETL やイベント駆動アーキテクチャに対応。

⚠️ デメリット・制約事項

制限事項

  • Preview 段階であり、SLA 適用対象外。本番ワークロードでの利用には注意が必要
  • オンデマンド課金モデルではサポートされず、Enterprise / Enterprise Plus リザベーションが必須
  • CONTINUOUS リザベーション割り当ては最大 500 スロットに制限 (上限引き上げは問い合わせ)
  • ユーザーアカウントでの最大実行時間は 2 日間
  • 実行中の Continuous Query の SQL を変更することはできない
  • 出力ウォーターマークラグが 48 時間を超えるとジョブが失敗

考慮すべき点

  • 一時的な問題による再処理で出力に重複データが含まれる可能性あり。下流システムでの冪等性処理を考慮
  • エクスポート先は BigQuery データセットと同一リージョンのリソースのみ対応
  • 列レベル・行レベルのセキュリティ機能は未サポート
  • Preview 段階のため、GA に向けて仕様変更の可能性あり

📌 まとめ

BigQuery Continuous Queries におけるステートフル操作 (JOIN / ウィンドウ集計) のサポートは、BigQuery のリアルタイム処理能力を大幅に拡張する重要なアップデートです。 これまで明示的にサポート対象外とされていた JOIN や集計関数が Preview として利用可能になったことで、Dataflow などの別途ストリーミング基盤を構築せずに、BigQuery の SQL だけでより多くのリアルタイム分析パターンを実現できるようになります。

Enterprise / Enterprise Plus エディションのリザベーションが必要ですが、リアルタイムデータエンリッチメントやストリーミング集計のニーズがあるチームは、Preview 段階のうちに検証を開始することを推奨します。
