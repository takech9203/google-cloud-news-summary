⚡ Flex-start VM Compact Placement Policies GA

AI Hypercomputer Compute Engine GA
2026-03-02

📋 概要

Google Cloud は、スタンドアロン Flex-start VM に対する コンパクト配置ポリシーの適用を一般提供 (GA) として発表しました。 これにより、Flex-start VM を隣接ブロックまたは同一ブロック内に集約配置し、 VM 間のネットワークレイテンシを最小化できるようになります。

Flex-start VM は Dynamic Workload Scheduler (DWS) を活用して最大 7 日間の短期 GPU リソースを 最大 53% オフの割引価格で提供するプロビジョニングモデルです。 今回のアップデートにより、AI/ML ワークロードにおける VM 間通信の低レイテンシ化が実現し、 分散学習やバッチ推論のパフォーマンスが向上します。

🔄 Before / After 比較

❌ Before: 標準配置のみ

✕ スタンドアロン Flex-start VM には配置ポリシーを適用できなかった
✕ VM が同一ゾーン内で離れた場所に分散配置される可能性があった
✕ 密集配置は MIG リサイズリクエスト経由でのみ利用可能だった
✕ 低レイテンシ通信が必要な場合、MIG や予約ベースの方法を選択する必要があった

✅ After: Compact Placement 適用

✓ スタンドアロン Flex-start VM にコンパクト配置ポリシーを適用可能に
✓ VM を隣接ブロック (maxDistance=3) または同一ブロック (maxDistance=2) に配置可能
✓ MIG を使わずとも低レイテンシな VM 間通信を実現
✓ Flex-start の割引 (最大 53%) + コンパクト配置の高性能通信を組み合わせ可能

🔷 主要機能

⚙️ コンパクト配置ポリシーの Flex-start VM への適用

スタンドアロン Flex-start VM に対してコンパクト配置ポリシーを作成・適用可能に。collocation=COLLOCATED を指定してポリシーを作成し、VM 作成時または既存 VM への追加適用の両方に対応します。

📏 maxDistance による配置距離の制御

→ maxDistance=3: 隣接ブロック配置。最大 1,500 VM (A4 の場合)
→ maxDistance=2: 同一ブロック配置。A4 は最大 150 VM、A3 系は最大 256 VM
→ 未指定: ベストエフォートで可能な限り近くに配置 (最大 1,500 VM)

🚀 AI/ML ワークロード向けネットワーク最適化

→ ネットワークホップの削減により VM 間通信レイテンシが低減
→ 分散学習における All-Reduce 通信や勾配同期のパフォーマンス向上
→ MPI ライブラリを使用する HPC タスクの最適化

📊 技術仕様

コンパクト配置ポリシーの maxDistance 設定

maxDistance 配置レベル 最大 VM 数 メンテナンスポリシー
未指定 ベストエフォート 1,500 Migrate / Terminate

📝 設定方法

💡 前提条件: 対象マシンタイプに対するプリエンプティブルクォータが十分に確保されていること。 コンパクト配置ポリシーと VM を同一リージョンに作成すること。 Compute Engine API が有効化されていること。

Step 1: コンパクト配置ポリシーの作成

Bash 配置ポリシーの作成 
gcloud compute resource-policies create group-placement POLICY_NAME \
    --collocation=collocated \
    --max-distance=MAX_DISTANCE \
    --region=REGION

Step 2: Flex-start VM の作成時にポリシーを適用

Bash VM 作成 + ポリシー適用 
gcloud compute instances create VM_NAME \
    --machine-type=MACHINE_TYPE \
    --zone=ZONE \
    --provisioning-model=FLEX_START \
    --max-run-duration=MAX_RUN_DURATION \
    --instance-termination-action=TERMINATION_ACTION \
    --resource-policies=POLICY_NAME

Step 3: 既存の Flex-start VM にポリシーを適用

Bash 既存 VM へのポリシー適用 
# VM を停止
gcloud compute instances stop VM_NAME --zone=ZONE

# ポリシーを適用
gcloud compute instances add-resource-policies VM_NAME \
    --resource-policies=POLICY_NAME \
    --zone=ZONE

# VM を再起動
gcloud compute instances start VM_NAME --zone=ZONE

REST API を使用する場合

JSON REST API リクエスト 
POST https://compute.googleapis.com/compute/v1/projects/PROJECT_ID/regions/REGION/resourcePolicies
{
  "name": "POLICY_NAME",
  "groupPlacementPolicy": {
    "collocation": "COLLOCATED",
    "maxDistance": MAX_DISTANCE
  }
}

🌟 メリット

💼 ビジネス面

コスト効率の高い AI/ML 環境

Flex-start VM の最大 53% 割引とコンパクト配置による低レイテンシを組み合わせ、予約なしで高性能な分散学習環境をコスト効率よく構築できます。

運用の柔軟性向上

MIG リサイズリクエストを使わずとも、スタンドアロン VM で密集配置が可能。小規模から中規模の AI/ML ワークロードを柔軟に管理できます。

🔧 技術面

ネットワークレイテンシの最小化

VM 間のネットワークホップが削減され、分散学習の All-Reduce 通信や勾配同期のパフォーマンスが向上します。

配置の精密制御

maxDistance パラメータにより、ワークロード要件に応じて隣接ブロック (低レイテンシ) と同一ブロック (最低レイテンシ) を選択可能。

スケーラビリティ

maxDistance 未指定/3 で最大 1,500 VM をサポートし、大規模分散学習にも対応します。

⚠️ 制限事項・注意点

🚧 制限事項
✕ Flex-start VM の最大実行時間は 7 日間に制限
✕ A4X Max / A4X マシンタイプは Flex-start VM では使用不可
✕ maxDistance の値が小さいほど VM の配置制約が厳しくなり、リソース確保が困難になる可能性あり
✕ 1 つの Compute Engine リソースに適用できる配置ポリシーは 1 つのみ
✕ 配置ポリシーはリージョナルリソース (同一リージョン内のリソースにのみ適用可能)
🔎 考慮すべき点
→ maxDistance=2 (同一ブロック) では最大 VM 数が 150-256 に制限。大規模ワークロードでは maxDistance=3 またはベストエフォートを検討
→ Flex-start VM はベストエフォートでのリソース割り当て。コンパクト配置と組み合わせるとリソース確保がさらに難しくなる可能性あり
→ maxDistance=2 ではホストメンテナンスポリシーは Terminate のみ対応 (ライブマイグレーション不可)

🎯 ユースケース

🤖 ユースケース 1: 分散モデルファインチューニング

シナリオ: 複数の GPU VM を使用して大規模言語モデルのファインチューニングを 3 日間実行。VM 間の勾配同期には低レイテンシ通信が必要。

Bash 分散ファインチューニングの実装例 
# コンパクト配置ポリシーを作成 (同一ブロック配置)
gcloud compute resource-policies create group-placement fine-tune-policy \
    --collocation=collocated \
    --max-distance=2 \
    --region=us-central1

# Flex-start VM を作成 (A3 High, 8 GPU)
for i in $(seq 1 4); do
  gcloud compute instances create fine-tune-vm-$i \
      --machine-type=a3-highgpu-8g \
      --zone=us-central1-a \
      --provisioning-model=FLEX_START \
      --max-run-duration=259200s \
      --instance-termination-action=DELETE \
      --resource-policies=fine-tune-policy
done

✨ 効果: 4 台の GPU VM が同一ブロック内に配置され、VM 間の通信レイテンシが最小化。Flex-start の割引価格 (最大 53% オフ) で 3 日間のファインチューニングを効率的に実行できます。

🔬 ユースケース 2: HPC シミュレーションのバースト実行

シナリオ: 大規模な科学計算シミュレーションを短期間 (24-48 時間) で実行。MPI 通信のパフォーマンスが結果精度に影響。

Bash HPC シミュレーションの実装例 
# コンパクト配置ポリシーを作成 (隣接ブロック配置)
gcloud compute resource-policies create group-placement hpc-sim-policy \
    --collocation=collocated \
    --max-distance=3 \
    --region=us-east4

# Flex-start VM を作成
gcloud compute instances create hpc-sim-vm-1 \
    --machine-type=h3-standard-88 \
    --zone=us-east4-a \
    --provisioning-model=FLEX_START \
    --max-run-duration=172800s \
    --instance-termination-action=STOP \
    --resource-policies=hpc-sim-policy

✨ 効果: MPI 通信に最適化された低レイテンシ環境で、予約なしに HPC シミュレーションを実行可能。Flex-start の割引により計算コストを削減しながら高性能を維持できます。

💰 料金情報

コンパクト配置ポリシーの作成・削除・適用に追加料金は発生しません。料金は Flex-start VM のコンピューティングリソース (vCPU、メモリ、GPU) に対して課金されます。

Flex-start VM は Dynamic Workload Scheduler (DWS) の料金体系に基づき、対象マシンタイプで最大 53% の割引が適用されます。従量課金 (PAYG) モデルで、VM が実行中の期間のみ課金されます。

マシンタイプ 料金体系 割引率
A4 (GPU) DWS Pricing 最大 53%
A3 Ultra / A3 Mega / A3 High (GPU) DWS Pricing 最大 53%
A2 / G4 (GPU) DWS Pricing 最大 53%
H4D DWS Pricing 最大 53%

🌎 利用可能リージョン

コンパクト配置ポリシーはリージョナルリソースとして作成されます。利用可能なリージョンはマシンタイプにより異なります。

💡 詳細情報: GPU マシンタイプの利用可能リージョン・ゾーンについては GPU のリージョンとゾーン を参照してください。

🔗 関連サービス・機能

⚡ Dynamic Workload Scheduler (DWS)

Flex-start VM の基盤となるスケジューリングエンジン。割引価格でのリソース提供を実現。

🤖 AI Hypercomputer

GPU/TPU を活用した AI/ML ワークロード向けの統合コンピューティングプラットフォーム。Flex-start は AI Hypercomputer の消費オプションの 1 つ。

📦 Workload Policy (MIG 向け)

MIG に対する配置制御。MIG リサイズリクエストで Flex-start VM を使用する場合はワークロードポリシーを利用。

🚀 Cloud Batch / GKE

Cloud Batch はバッチジョブの VM にコンパクト配置ポリシーを自動適用する機能を提供。GKE クラスタでも Flex-start VM を利用可能。
