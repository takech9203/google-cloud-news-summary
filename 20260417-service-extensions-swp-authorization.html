カスタム認可サービスを SWP の処理パスに直接挿入し、多段階の認可制御を実現

📋 概要

Service Extensions において、Secure Web Proxy (SWP) の処理パスにカスタム認可サービスを直接挿入できる Authorization Extensions 機能が Preview として提供開始されました。

組織のアウトバウンド Web トラフィックに対して、Google 提供の組み込み認可エンジンを超えたカスタム認可ロジックを適用できるようになります。さらに、REQUEST_AUTHZ (リクエスト認可) と CONTENT_AUTHZ (コンテンツ認可) の 2 つのポリシープロファイルに対応し、ヘッダーレベルからコンテンツレベルまでの多段階セキュリティ検査を実現します。

🔄 Before / After 比較

❌ Before

  • SWP の認可は組み込み ALLOW/DENY ルールに限定
  • 複雑な認可ロジックの表現が困難
  • 外部認可エンジン (OPA, Cedar 等) との統合に別途プロキシが必要
  • コンテンツベースのセキュリティ検査を SWP 内で行えない
  • プロンプトインジェクション防止や機密データ漏洩検出が不可

✔ After

  • カスタム認可サービスを SWP 処理パスに直接挿入可能
  • REQUEST_AUTHZ でヘッダーベースのカスタム認可を実現
  • CONTENT_AUTHZ でリクエスト/レスポンス全体の深い検査が可能
  • OPA, Cedar 等の外部ポリシーエンジンとシームレスに統合
  • Model Armor 統合で AI ガードレールを自動適用

🛠 アーキテクチャ図

Secure Web Proxy の処理パスにおける Authorization Extensions の動作フローです。REQUEST_AUTHZ が最初に呼び出され、次に組み込みの DENY/ALLOW ポリシー評価、最後に CONTENT_AUTHZ が呼び出されます。各段階で認可が拒否された場合、後続の処理はスキップされクライアントにエラーが返されます。

💡 主要機能

🔒 SWP 向け Authorization Extensions

Secure Web Proxy の処理パスにカスタム認可サービスを直接挿入。ユーザー管理の gRPC バックエンドサービスまたは FQDN ベースサービスをコールアウト先に指定可能。フォワーディングルールあたり 1 つの Authorization Extension をアタッチ。

📄 REQUEST_AUTHZ プロファイル

HTTP リクエストヘッダー情報に基づくアクセス制御。ALLOW, DENY, CUSTOM の 3 つのアクション種別をサポート。リクエストヘッダー到着時に呼び出され、ボディは転送されない。ポリシープロファイル未指定時のデフォルト。

🔎 CONTENT_AUTHZ プロファイル

リクエスト/レスポンスの全コンテンツ (ヘッダー、ボディ、トレーラー) に対する深い検査。FULL_DUPLEX_STREAMED モードでのフル双方向ストリーミング処理。CUSTOM アクションのみサポート。

🤖 Model Armor 統合

CONTENT_AUTHZ プロファイルと Model Armor の統合により AI ガードレールを適用可能。プロンプトインジェクション攻撃の防止、機密データ漏洩の検出、有害コンテンツのフィルタリングを SWP 内で実現。

ext_authz API の CheckRequest / CheckResponse

Protobuf Authorization サービスインターフェース 
// Authorization サービスインターフェース
service Authorization {
  rpc Check(CheckRequest) returns (CheckResponse) {}
}

message CheckRequest {
  // リクエスト属性 (ソース、宛先、ヘッダー等)
  AttributeContext attributes = 1;
}

message CheckResponse {
  // ステータス (OK = 許可、その他 = 拒否)
  google.rpc.Status status = 1;
  oneof http_response {
    // 拒否時のレスポンス
    DeniedHttpResponse denied_response = 2;
    // 許可時のレスポンス (ヘッダー変更可能)
    OkHttpResponse ok_response = 3;
  }
  // 後続のエクステンションへ渡すメタデータ
  google.protobuf.Struct dynamic_metadata = 4;
}

📝 設定方法

📌 前提条件: Google Cloud プロジェクト作成済み / Secure Web Proxy デプロイ済み / Service Extensions API 有効化済み / コールアウトバックエンドサービス構成済み (HTTP/2 対応必須) / 必要な IAM ロール付与済み

Step 1: コールアウトバックエンドサービスの構成

Bash バックエンドサービスの作成 (HTTP/2 プロトコル必須) 
gcloud compute backend-services create authz-callout-backend \
    --project=PROJECT_ID \
    --region=REGION \
    --protocol=HTTP2 \
    --health-checks=HEALTH_CHECK_NAME \
    --load-balancing-scheme=INTERNAL_MANAGED

Step 2: Authorization Extension の作成

Bash AuthzExtension リソースの作成 
gcloud service-extensions authz-extensions create my-authz-extension \
    --project=PROJECT_ID \
    --location=REGION \
    --backend-service=authz-callout-backend \
    --load-balancing-scheme=INTERNAL_MANAGED

Step 3: 認可ポリシーの作成 (REQUEST_AUTHZ)

Bash REQUEST_AUTHZ プロファイルの認可ポリシー 
gcloud network-security authz-policies create my-request-authz-policy \
    --project=PROJECT_ID \
    --location=REGION \
    --action=CUSTOM \
    --custom-provider='{"authzExtension": "projects/PROJECT_ID/locations/REGION/authzExtensions/my-authz-extension"}' \
    --policy-profile=REQUEST_AUTHZ \
    --target='{"loadBalancingScheme": "INTERNAL_MANAGED"}'

Step 4: 認可ポリシーの作成 (CONTENT_AUTHZ) - オプション

Bash CONTENT_AUTHZ プロファイルの認可ポリシー 
gcloud network-security authz-policies create my-content-authz-policy \
    --project=PROJECT_ID \
    --location=REGION \
    --action=CUSTOM \
    --custom-provider='{"authzExtension": "projects/PROJECT_ID/locations/REGION/authzExtensions/my-content-authz-extension"}' \
    --policy-profile=CONTENT_AUTHZ \
    --target='{"loadBalancingScheme": "INTERNAL_MANAGED"}'

🎯 メリット

💼 ビジネス面

🛡セキュリティ態勢の強化 - アウトバウンドトラフィックにコンテキストベースの高度な認可判断を適用し、ゼロトラストセキュリティモデルの実現を加速
📜コンプライアンス要件への対応 - 業界固有の認可要件や規制準拠のカスタムポリシーを SWP 内で直接実施。追加インフラ構築コストを削減
🤖AI セキュリティの統合 - CONTENT_AUTHZ + Model Armor で生成 AI アプリの推論トラフィックに AI ガードレールを一元的に適用可能

🛠 技術面

🔌柔軟な認可アーキテクチャ - OPA, Cedar 等の外部ポリシーエンジンや社内認可システムとシームレスに統合。既存インフラを活用可能
📐多段階の認可評価 - REQUEST_AUTHZ と CONTENT_AUTHZ の 2 段階の拡張ポイントで、ヘッダー/コンテンツ両レベルのセキュリティチェックを適用
🔗メタデータの伝播 - dynamic_metadata フィールドで認可拡張から後続のトラフィック拡張へコンテキスト情報を引き渡し可能
⚙標準プロトコルの採用 - Envoy の ext_proc / ext_authz プロトコルを使用。Envoy エコシステムの豊富なライブラリ/ツールを活用可能

⚠️ デメリット・制約事項

⚠️ Preview ステータス: 本番環境での使用には SLA が適用されません。Pre-GA Offerings Terms が適用されます。GA までに API やリソース定義に破壊的変更が入る可能性があります。

制限事項

  • ❌ フォワーディングルールあたり 1 つの Authorization Extension のみアタッチ可能
  • ❌ ext_authz 使用時はリクエストボディが転送されない
  • ❌ コールアウトバックエンドは HTTP/2 プロトコル必須
  • ❌ gRPC レスポンスメッセージの最大サイズは 128 KB
  • ❌ 特定ヘッダー (X-Forwarded-*, X-Google-*, :method, :authority 等) の操作が制限
  • ❌ プロジェクトあたりリージョン別 Authorization Extensions 数は最大 10

考慮すべき点

  • ⚠️ コールアウト追加によりリクエスト処理のレイテンシが増加 → 認可サービスは SWP と同一リージョンにデプロイ推奨
  • ⚠️ 認可サービスの可用性が SWP 全体の可用性に影響 → 高可用性設計が必要
  • ⚠️ ext_authz 使用時はボディアクセス不可 → コンテンツ検査には CONTENT_AUTHZ (ext_proc) を使用

💻 ユースケース

🔗 UC1: 外部ポリシーエンジンとの統合によるコンテキストベース認可

シナリオ: 企業が Open Policy Agent (OPA) を社内の統一認可エンジンとして使用しており、SWP を通過するアウトバウンドトラフィックにも同じポリシーを適用したい。

Rego OPA ポリシー例: 部門ごとのアウトバウンドアクセス制御 
package swp.authz

default allow = false

allow {
    input.attributes.source.principal == "serviceAccount:finance-app@project.iam.gserviceaccount.com"
    input.attributes.request.http.host == "api.banking-partner.com"
    input.attributes.request.http.method == "GET"
}

allow {
    input.attributes.source.principal == "serviceAccount:hr-app@project.iam.gserviceaccount.com"
    input.attributes.request.http.host == "api.hr-saas.com"
}
✨ 効果: 全社的に統一された認可ポリシーをアウトバウンドトラフィックにも適用でき、ポリシーの一貫性を保持。OPA 側で一元管理できるため運用負荷も軽減。

👥 UC3: マルチ認可システムの統合
