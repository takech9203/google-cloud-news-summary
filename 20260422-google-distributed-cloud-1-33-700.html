📄 概要

Google Distributed Cloud (software only) のバージョン 1.33.700 が VMware 向けおよびベアメタル向けの両方でリリースされました。いずれも Kubernetes v1.33.5-gke.2200 をベースとしています。

VMware 版では Advanced クラスタマイグレーションのリトライ改善が主要テーマです。ベアメタル版では ステイルマウント検知ヘルスチェックが新機能として追加されました。両版とも複数の重要なバグ修正とセキュリティ脆弱性の修正が含まれています。

🔄 Before / After 比較

❌ アップデート前の課題

  • VMware 版でマイグレーション失敗時に手動クリーンアップが必要
  • node-problem-detector の誤配置で containerd が繰り返し再起動
  • ノードプールラベルに -np サフィックスが付与され Pod スケジューリング失敗
  • ベアメタル版で並行タスクが containerd 再起動により失敗
  • Secret/ConfigMap マウントが古い状態のまま残留

✅ アップデート後の改善

  • マイグレーション失敗時に安全にリトライ可能
  • node-problem-detector が非 Advanced クラスタに配置されなくなり安定化
  • ノードプールラベル問題が修正され正常スケジュール回復
  • タスクロック機構で逐次安全実行
  • ステイルマウント検知で不整合を早期発見

🖥️ VMware 版 主要機能・修正 (1.33.700-gke.71)

🆕 Advanced クラスタマイグレーションのリトライ改善 (新機能)

  • 通常クラスタから Advanced クラスタへのマイグレーションが失敗・中断された場合、手動クリーンアップなしで安全にリトライ可能に
  • システムが既存のリソースと一時的な Bootstrap クラスタを自動的に再利用
⚠️ 重要: マイグレーション失敗時に Bootstrap クラスタを削除してはならない。削除するとデータ損失やリカバリ不能になる可能性がある。

🔗 Hub メンバーシップのアップグレード停止問題の修正

非 Advanced から Advanced へのアップグレード時に、Hub メンバーシップの不変フィールドを更新しようとしてアップグレードが停止する問題を解決。クラスタオペレーターが元のメンバーシップフィールドを保持するようになった。

🏷️ ノードプールラベルの -np サフィックス問題の修正

Advanced ユーザークラスタで cloud.google.com/gke-nodepool ラベルに予期しない -np サフィックスが付与される問題を修正。Apigee ワークロードなど、元のプール名を nodeSelector でターゲットする Pod が正常にスケジュールされるようになった。

🔧 node-problem-detector の誤配置修正

非 Advanced VMware クラスタに node-problem-detector が誤って配置され、互換性のないヘルスチェック設定により containerd が繰り返し再起動する問題を修正。ETCD/CRI 障害 (/run/containerd/containerd.sock への接続エラーなど) およびクラスタアップグレードの失敗が解消された。

🛠️ stackdriver.enableVPC フィールドのブロック問題修正

廃止予定の stackdriver.enableVPC フィールドが true に設定されている場合、Advanced クラスタへのアップグレードがブロックされる問題を修正。このフィールドの設定がアップグレード検証プロセスで無視されるようになった。

🔒 脆弱性修正

VMware 版のバージョン履歴に記載された脆弱性が修正された。

🧹 ベアメタル版 主要機能・修正 (1.33.700-71)

🆕 ステイルマウント検知ヘルスチェック (新機能)

  • Secret や ConfigMap のマウントが API サーバーと同期していない「ステイル」状態を検知する新しいヘルスチェックを追加
  • ノード上の全実行中 Pod をイテレーションしてマウントを検証
  • Kubelet のアトミック更新シンリンク構造内のローカルデータを API サーバーのライブオブジェクトおよび更新タイムスタンプと比較
  • 通常の伝搬遅延による誤検知を防ぐため、5 分間の閾値が設定

🔒 並行タスクの逐次ロック実行

  • 同一ノード上の並行タスクが containerd の再起動により失敗する問題を修正
  • タスクがロックされ、逐次実行されるようになった (各ロックは最大 20 分間または成功/失敗まで保持)
  • 並行実行が必要な場合は baremetal.cluster.gke.io/concurrent-machine-update: "true" アノテーションで無効化可能

🗂️ etcd-events のデータディレクトリ問題の修正

マシン初期化フェーズで etcd-events Pod が古いデータディレクトリを読み込み、古いメンバー ID でクラスタに再参加しようとして無限リトライループに陥る問題を修正。失敗時に /var/lib/etcd-events ディレクトリをクリアし、kubeadm-reset にリトライロジックを追加。

⏰ ノードアップグレードのタイムアウト問題の修正

Terminating 状態のネームスペース内に完了した Pod がある場合、ノードアップグレードが無限にハングする問題を修正。ドレインプロセスがターミナルフェーズの Pod に対してエビクションをスキップするよう更新された。

🔐 Metrics API の TLS 検証エラーの修正

証明書ローテーション中に kubectl top、HPA (Horizontal Pod Autoscaling)、VPA (Vertical Pod Autoscaling) が TLS 検証エラーで失敗する問題を修正。CA ローテーション時にリーフ証明書が即座に更新されないことによる一時的な不整合が解消された。

🔒 脆弱性修正

ベアメタル版の脆弱性情報ページに記載された脆弱性が修正された。

🛠️ 設定方法

📚 前提条件:
  • 現在のクラスタバージョンが 1.33.x であること (マイナーバージョンアップの場合は 1.32.x も対象)
  • サードパーティストレージベンダーを使用している場合、本リリースの認定状況を確認
  • アップグレード前にクラスタの診断を実行すること

Step 1: VMware 版クラスタのアップグレード

Bash VMware Admin クラスタのアップグレード 
# クラスタの診断を実行
gkectl diagnose cluster --kubeconfig ADMIN_CLUSTER_KUBECONFIG

# OS イメージのインポート
gkectl prepare \
  --bundle-path /var/lib/gke/bundles/gke-onprem-vsphere-1.33.700-gke.71.tgz \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG

# Admin クラスタのアップグレード
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG

Step 2: ベアメタル版クラスタのアップグレード

Bash ベアメタルクラスタのアップグレード 
# クラスタの診断を実行
bmctl check cluster --cluster CLUSTER_NAME --kubeconfig ADMIN_CLUSTER_KUBECONFIG

# クラスタのアップグレード
bmctl upgrade cluster --cluster CLUSTER_NAME --kubeconfig ADMIN_CLUSTER_KUBECONFIG

Step 3: 失敗したマイグレーションのリトライ (VMware 版、該当する場合)

Bash Bootstrap クラスタを再利用してリトライ 
# 失敗した Admin クラスタのアップグレードをリトライ
# Bootstrap クラスタが存在することを確認してからリトライする
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG \
  --reuse-bootstrap-cluster
⚠️ 重要: --reuse-bootstrap-cluster フラグを使用してリトライすること。Bootstrap クラスタを削除するとデータ損失が発生する可能性がある。

⚠️ 制限事項・考慮点

⏰ GKE On-Prem API 遅延

リリース後、Google Cloud コンソール・gcloud CLI・Terraform で利用可能になるまで 7-14 日かかる

🛠️ gkectl 必須

VMware 版の非 Advanced から Advanced へのアップグレードには gkectl コマンドラインツールが必須。GKE On-Prem API クライアントはサポートされない

🔄 自動変換

非 Advanced クラスタの維持は 1.32 から 1.33 へのアップグレード時のみ。1.33 から 1.34 では自動的に Advanced クラスタに変換される

⚫ タスクロックのパフォーマンス

ベアメタル版のタスクロック機構により、大量の並行タスクが必要な環境ではパフォーマンスに影響する可能性がある

📦 cert-manager 上書き

VMware 版で Advanced クラスタへのマイグレーション時、cert-manager が自動インストールされ既存の cert-manager を上書きする。カスタム構成がある場合は事前にバックアップが必要

🕛 5 分閾値

ベアメタル版のステイルマウント検知は 5 分間の閾値で動作するため、5 分未満の一時的な不整合は検知対象外

🎯 ユースケース

🖥️ ユースケース 1: VMware 環境での Advanced クラスタへの安全なマイグレーション

シナリオ: 大規模な VMware 環境で、多数の非 Advanced クラスタを運用している組織が段階的に Advanced クラスタへ移行する必要がある。過去のバージョンではマイグレーション中の失敗時に手動クリーンアップが必要で、移行作業が長時間化していた。

Bash 段階的マイグレーションの実行例 
# 1. Admin クラスタを 1.33.700-gke.71 にアップグレード
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG

# 2. 失敗した場合、Bootstrap クラスタを保持したままリトライ
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG \
  --reuse-bootstrap-cluster

# 3. User クラスタをアップグレード
gkectl upgrade cluster \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config USER_CLUSTER_CONFIG

✨ 効果: マイグレーション失敗時の復旧時間が大幅に短縮され、移行作業全体のリスクと所要時間が削減される。

🧹 ユースケース 2: ベアメタル環境でのアプリケーション構成データの整合性監視

シナリオ: ベアメタルクラスタ上で金融系アプリケーションを運用しており、Secret に保存された API キーや証明書、ConfigMap に保存された設定値が常に最新であることが求められる環境。Kubelet のキャッシュ問題により、Pod がステイル (古い) 構成データを使用し続けるリスクがある。

✨ 効果: 新しいヘルスチェック機能により、5 分以上の不整合が自動的に検知されるため、ステイルデータによるアプリケーション障害を早期に発見し、迅速に対処できるようになる。
