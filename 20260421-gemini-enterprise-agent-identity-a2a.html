🤖 Gemini Enterprise: Agent Identity 表示 & A2UI/A2A エージェント登録

2026-04-21
Gemini Enterprise Public Preview Agent Identity A2UI / A2A

📋 概要

Gemini Enterprise に 2 つのエージェント管理関連機能が Public Preview として追加されました。

🔐
Agent Identity (SPIFFE ID)
エージェントの認証基盤を可視化
🎨
A2UI プロトコル
リッチ UI コンポーネント対応
🔗
A2A プロトコル
エージェント間通信・タスク委譲

Agent Identity 機能により、管理者はエージェントの SPIFFE ID を確認でき、A2UI/A2A 統合によりリッチ UI 対応のカスタムエージェントを Gemini Enterprise プラットフォーム上で登録・管理できるようになりました。

🔄 Before / After

❌ Before (アップデート前)

  • エージェントの SPIFFE ID をコンソール上で確認する手段がなかった
  • A2UI プロトコルとの統合登録がサポートされておらず、リッチ UI エージェントの構築ワークフローが存在しなかった
  • カスタムエージェントの UI はテキストベースの応答に限定され、カード・フォーム・チャートなどの構造化された視覚要素を生成できなかった

✅ After (アップデート後)

  • Agent Details ページで SPIFFE ID を直接確認可能に
  • A2UI + A2A プロトコルの統合により、リッチ UI 対応エージェントの登録・管理が Gemini Enterprise コンソールから可能に
  • A2UI SDK を使用し、カード・テーブル・フォームなどのインタラクティブ UI コンポーネントを JSON 形式で出力・レンダリング可能に

⚡ 主要機能

🔐 1. Agent Identity (エージェント ID) の表示

Gemini Enterprise の Agent Details ページでエージェントの SPIFFE ID を確認できます。

→SPIFFE ID 形式: principal://TRUST_DOMAIN/NAMESPACE/AGENT_NAME
→Trust Domain は組織レベルで自動プロビジョニング
→X.509 証明書が自動プロビジョニングされ、mTLS による安全な認証が可能
→SPIFFE ID 未公開の場合は Agent Registry リソース ID がフォールバック表示

🎨 2. A2UI (Agent to UI) によるエージェント登録

エージェントが構造化された JSON を出力し、クライアント側でインタラクティブな UI コンポーネントとしてレンダリングします。

→トランスポート非依存: A2A、MCP、REST、WebSocket など任意のプロトコル上で動作
→A2UI SDK (a2ui-agent-sdk) によるスキーマ管理とバリデーション
→MIME タイプ application/json+a2ui で A2A DataPart としてラッピング

サポートされる UI コンポーネント

🃏
Card
情報のサマリー表示
📊
Table
データの比較・一覧
📝
Form
ユーザー入力の受付
📈
Chart
データの可視化
🔘
Button
アクションの実行
🔤
Text / Heading
テキストコンテンツ

🔗 3. A2A (Agent2Agent) プロトコルとの統合登録

異なるビルダーやプラットフォームのエージェント間での発見・協調・タスク委譲を可能にするオープン通信プロトコルです。

→Google Cloud コンソールまたは REST API からエージェント登録が可能
→OAuth 2.0 によるエンドユーザーアクセス制御をサポート
→Agent Card (JSON) でエージェントの能力・スキル・入出力モードを宣言

🔧 技術仕様

SPIFFE ID と Agent Identity

項目 詳細
ID フォーマット principal://TRUST_DOMAIN/NAMESPACE/AGENT_NAME
Trust Domain (組織あり) agents.global.org-ORGANIZATION_ID.system.id.goog
Trust Domain (組織なし) agents.global.project-PROJECT_NUMBER.system.id.goog
証明書 X.509 クライアント証明書 (自動プロビジョニング)
認証方式 mTLS (相互 TLS)
トークン有効期限 デフォルト 1 時間 (証明書期限に依存して短縮される場合あり)
フォールバック Agent Registry リソース ID

📝 設定方法

📌 前提条件:
  • Gemini Enterprise Admin ロール (discoveryengine.agentspaceAdmin) が付与されていること
  • Discovery Engine API が有効化されていること
  • Gemini Enterprise アプリが作成済みであること
  • A2A プロトコルを使用するエージェントがホストされ稼働していること

ステップ 1: Agent Identity の確認

1
Google Cloud コンソールで Gemini Enterprise ページに移動
2
エージェントが登録されたアプリ名をクリック
3
[Agents] をクリック
4
対象エージェント名をクリックして Agent Details ページを開く
5
SPIFFE ID またはフォールバックのリソース ID が表示される

ステップ 2: A2UI/A2A エージェントの登録 (コンソール)

1
Google Cloud コンソールで Gemini Enterprise ページに移動
2
エージェントを登録するアプリ名をクリック
3
[Agents] > [Add Agents] をクリック
4
[Custom agent via A2A] の [Add] をクリック
5
Agent Card JSON を入力 (A2UI 対応の場合は capabilities に A2UI 拡張を含める)
6
[Preview agent details] > [Next] をクリック
7
OAuth 2.0 設定を行うか、[Skip & Finish] でスキップ

ステップ 3: A2UI/A2A エージェントの登録 (REST API)

Bash REST API によるエージェント登録 
curl -X POST \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" \
  "https://ENDPOINT_LOCATION-discoveryengine.googleapis.com/v1alpha/projects/PROJECT_ID/locations/LOCATION/collections/default_collection/engines/APP_ID/assistants/default_assistant/agents" \
  -d '{
    "name": "AGENT_NAME",
    "displayName": "AGENT_DISPLAY_NAME",
    "description": "AGENT_DESCRIPTION",
    "a2aAgentDefinition": {
      "jsonAgentCard": "{\"protocolVersion\":\"v1.0\", ...}"
    }
  }'

ENDPOINT_LOCATION は us、eu、または global を指定します。

✅ メリット

ビジネス面

🏛️エージェントガバナンスの強化 -- Agent Identity により、組織内で稼働するエージェントの認証基盤を一元的に可視化でき、コンプライアンス要件への対応が容易に
🌐エージェントエコシステムの拡張 -- A2UI/A2A 統合により、リッチ UI を持つカスタムエージェントを Gemini Enterprise プラットフォーム上で迅速に展開し、ユーザー体験を向上
🔓ベンダーロックイン回避 -- A2A はオープンプロトコルであり、異なるプラットフォームで構築されたエージェントを統一的に管理可能

技術面

🛡️ゼロトラストセキュリティ -- SPIFFE ベースの Agent Identity により、エージェント間の mTLS 認証が可能。サービスアカウントに依存しない安全な通信を実現
🔌トランスポート非依存の UI 定義 -- A2UI ペイロードは A2A、MCP、REST、WebSocket など任意のプロトコル上で動作するため、既存インフラに柔軟に統合可能
✔️スキーマバリデーション -- A2UI SDK による出力バリデーションにより、LLM が生成する UI コンポーネントの構造的整合性を保証

⚠️ デメリット・制約事項

📋 制限事項
  • Public Preview であり、Pre-GA Offerings Terms が適用される。SLA は提供されない
  • SPIFFE ID はパブリッシャーが公開している場合のみ表示される。未公開の場合はリソース ID へのフォールバック
  • A2A エージェントのセキュリティ保護には、開発者がアプリケーションコード内で REST API を使用して Model Armor を設定する必要がある (コンソールの Model Armor 設定は A2A エージェントには自動適用されない)
  • 利用には Gemini Enterprise Standard、Plus、または Frontline エディションが必要 (Business エディションでは非対応)
💡 考慮すべき点
  • A2A エージェントはユーザー自身でホスト・運用する必要があり、アップタイム管理は利用者側の責任
  • A2UI コンポーネントのレンダリングはクライアント側に依存するため、クライアントの対応状況によって表示が異なる可能性
  • OAuth 2.0 による認証設定はオプションだが、Google Cloud リソースへのアクセスが必要な場合は設定が必要

💡 ユースケース

🔍 ユースケース 1: セキュリティ監査におけるエージェント ID 確認

シナリオ: セキュリティチームが組織内で稼働するすべてのエージェントの認証基盤を棚卸しする際に、各エージェントの SPIFFE ID を確認して IAM ポリシーとの整合性を検証する。

Bash エージェント一覧を取得し SPIFFE ID を確認 
# REST API でエージェント一覧を取得し、SPIFFE ID を確認
curl -X GET \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  "https://global-discoveryengine.googleapis.com/v1alpha/projects/PROJECT_ID/locations/global/collections/default_collection/engines/APP_ID/assistants/default_assistant/agents"

効果: エージェント単位の認証情報を一元管理でき、VPC Service Controls のイングレス/エグレスルールにおいて Agent Identity を使用したきめ細かなアクセス制御が可能に。

📊 ユースケース 2: データ分析ダッシュボードエージェントの構築

シナリオ: データエンジニアリングチームが、BigQuery のデータを分析してチャートやテーブルを含むリッチなダッシュボードを自動生成するエージェントを A2UI で構築し、Gemini Enterprise に登録する。

Python A2UI SDK によるエージェント実装例 
from a2ui.core.schema.manager import A2uiSchemaManager
from a2ui.basic_catalog.provider import BasicCatalog

schema_manager = A2uiSchemaManager(
    catalogs=[BasicCatalog.get_config()]
)

instruction = schema_manager.generate_system_prompt(
    role_description="BigQuery データ分析アシスタント",
    workflow_description="ユーザーのクエリを分析し、結果をチャートやテーブルで表示する",
    ui_description="集計結果にはテーブル、トレンドにはチャート、サマリーにはカードを使用",
    include_schema=True,
    include_examples=True,
    allowed_components=["Heading", "Text", "Card", "Table", "Chart"],
)

効果: エンドユーザーが Gemini Enterprise の Web アプリからテキストで質問するだけで、構造化されたチャートやテーブルを含む分析結果を受け取ることができ、BI ツールを個別に操作する必要がなくなる。

🔗 関連サービス・機能

📌 まとめ

今回の Gemini Enterprise アップデートにより、エージェントのセキュリティガバナンスとカスタムエージェントのUI 体験が大幅に強化されました。

🔐Agent Identity: SPIFFE ベースのゼロトラストアーキテクチャをエージェント管理に導入。VPC Service Controls や IAM と連携した包括的なアクセス制御を実現

📋 推奨アクション: Preview 段階であるため本番利用には注意が必要ですが、エージェント管理基盤の構築を検討している組織は早期に評価を開始することをお勧めします。
