☁️ Cloud Composer: africa-south1 が Composer 3 専用に

Cloud Composer Announcement africa-south1
2026-05-06

📋 概要

Google Cloud は、ヨハネスブルグ (africa-south1) リージョンにおいて Cloud Composer 2 環境の新規作成を終了し、Cloud Composer 3 環境のみをサポートするリージョンへの移行を発表しました。

Cloud Composer 3 は 2024 年 6 月にパブリックプレビューとして発表された次世代のマネージド Apache Airflow サービスであり、インフラストラクチャの完全なテナントプロジェクト管理、エバーグリーンバージョニング、簡素化されたネットワーク設定、パフォーマンスの向上など、多くの改善が含まれています。2026 年 4 月には「Managed Service for Apache Airflow」への名称変更も行われました。

✅ 既存環境への影響なし: africa-south1 リージョンに既に存在する Cloud Composer 2 環境は今回の変更による影響を受けません。既存環境は引き続き動作し、管理・更新が可能です。

🔄 Before / After 比較

❌ Before (変更前)

  • Cloud Composer 2 と 3 の両方を新規作成可能
  • ユーザーが古いバージョンを選択し続ける可能性
  • GKE クラスタがユーザープロジェクト内にデプロイ
  • ネットワーク設定やスケーリング構成が複雑

✅ After (変更後)

  • 新規環境は Cloud Composer 3 のみ作成可能
  • 最新機能を自動的に利用可能
  • インフラがテナントプロジェクトに隠蔽
  • 簡素化されたネットワークと自動スケーリング

⚡ 主要機能

🔒 Cloud Composer 3 専用化

africa-south1 リージョンでは Cloud Composer 2 環境の新規作成が不可に。新規ユーザーは自動的に最新機能を利用可能。

✅ 既存環境への影響なし

既存の Cloud Composer 2 環境は影響を受けず、更新・アップグレード・削除は引き続き可能。

🏢 テナントプロジェクト管理

すべてのインフラストラクチャがテナントプロジェクトに配置され、ユーザーの管理負荷が大幅に軽減。

🔄 エバーグリーンバージョニング

自動更新により常に最新のセキュリティパッチと機能が適用される。

🌐 簡素化されたネットワーク設定

Public/Private IP の切り替えが既存環境でも可能。複雑な PSC 設定が不要に。

🚀 パフォーマンス向上

DAG Processor と Scheduler の分離により、大規模ワークロードへの対応力が向上。

📐 技術仕様: Composer 2 vs Composer 3

項目 Cloud Composer 2 Cloud Composer 3
クラスタ ユーザープロジェクト内の GKE Autopilot テナントプロジェクト内 (隠蔽)

🛠️ 設定方法

前提条件

✓Google Cloud プロジェクトが作成済み
✓Cloud Composer API が有効化済み
✓必要な IAM 権限が付与済み

Step 1: Cloud Composer 3 環境を作成

Bash 
gcloud composer environments create my-composer3-env \
    --location africa-south1 \
    --image-version composer-3-airflow-2.10.5

Step 2: 既存 Composer 2 からの移行 (該当する場合)

Bash 
# 移行前の互換性チェック
gcloud composer environments check-upgrade \
    my-composer2-env \
    --location africa-south1 \
    --image-version composer-3-airflow-2

# 移行スクリプトを使用した移行
python composer_migrate.py \
    --source-env my-composer2-env \
    --target-env my-composer3-env \
    --location africa-south1
💡 移行スクリプト: GitHub で公開されています

✨ メリット

💼 ビジネス面

✓運用コスト削減: フルマネージドインフラにより運用チームの負荷軽減
✓最新機能へのアクセス: Airflow 3 サポート、MCP サーバー連携など

⚙️ 技術面

✓簡素化されたアーキテクチャ: GKE クラスタ管理が不要
✓パフォーマンス向上: DAG Processor 分離、10 倍のワーカーストレージ
✓柔軟なネットワーク: 既存環境でも Public/Private IP 切り替え可能

⚠️ デメリット・制約事項

制限事項:
  • africa-south1 で Cloud Composer 2 の新規環境作成は不可
  • Cloud Composer 2 から 3 へのインプレースアップグレードは非対応 (サイドバイサイド方式での移行が必要)
  • Cloud Composer 3 では KubernetesPodOperator でのカスタムサービスアカウントが非対応
💡 考慮すべき点:
  • 将来的なサポート終了に備えて計画的な移行を推奨
  • 移行前に check-upgrade コマンドで互換性を確認
  • Cloud Composer 3 の料金モデルは Composer 2 と異なるため、コスト比較を事前に実施

🎯 ユースケース

🌍 ユースケース 1: アフリカ地域での新規データパイプライン構築

シナリオ: アフリカ市場に進出する企業が、ヨハネスブルグリージョンでデータパイプラインを構築する場合。

Bash 
# Cloud Composer 3 環境作成 (Airflow 3 対応)
gcloud composer environments create africa-pipeline \
    --location africa-south1 \
    --image-version composer-3-airflow-3.1.7

効果: 最新の Airflow 3 と Cloud Composer 3 の機能をフル活用でき、データレイテンシーを最小限に抑えたパイプラインを構築可能。

🔄 ユースケース 2: 既存 Cloud Composer 2 環境の計画的移行

シナリオ: africa-south1 で運用中の Cloud Composer 2 環境を、新しい Cloud Composer 3 環境に段階的に移行する場合。

効果: 移行スクリプトやスナップショットを使用した移行ガイドにより、ダウンタイムを最小限に抑えた移行が可能。既存環境は影響を受けないため、十分なテスト期間を確保できる。

🌐 利用可能リージョン

africa-south1 は、Cloud Composer 3 専用リージョンに移行した複数のリージョンの一つです。同様の移行は他のリージョンでも段階的に実施されています。

🇿🇦 africa-south1 (ヨハネスブルグ) 🇮🇹 europe-west12 (トリノ) 🇦🇺 australia-southeast2 (メルボルン)

Cloud Composer 3 はすべての Cloud Composer サポートリージョンで利用可能です。
