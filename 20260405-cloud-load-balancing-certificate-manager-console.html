⚠️ Cloud Load Balancing: Certificate Manager 証明書が Google Cloud コンソールで利用可能に

2026-04-05
Cloud Load Balancing Certificate Manager GA

📋 概要

ロードバランサーのプロビジョニング時に、Certificate Manager の証明書を Google Cloud コンソールから直接選択・設定できるようになりました。TLS 証明書の管理とロードバランサーへの展開が、コンソール上のワークフローで一元的に行えるようになります。

Before / After

❌ Before

  • Certificate Manager の証明書をコンソールから直接選択できず、gcloud CLI や API での操作が必要だった
  • コンソールと CLI を行き来する運用で設定ミスのリスクや作業効率の低下
  • 証明書管理とロードバランサー設定が別々のワークフロー

✅ After

  • プロビジョニング画面から直接 Certificate Manager の証明書マップや証明書を選択可能
  • 証明書の選択からロードバランサー構成まで一貫したワークフロー
  • CLI に不慣れなユーザーでも高度な証明書管理機能を活用可能

⚡ 主要機能

🌐 証明書マップの選択 (グローバルロードバランサー向け)

ロードバランサーのプロビジョニング画面で証明書マップを直接選択可能。数千の証明書マップエントリをサポートし、ホスト名ベースの細かな証明書選択が可能です。

対象: Global external ALB, Classic ALB, Global external proxy NLB, Classic proxy NLB

📍 Certificate Manager 証明書の直接選択 (リージョナルロードバランサー向け)

ターゲットプロキシに Certificate Manager 証明書を直接アタッチ。ターゲットプロキシあたり最大 100 の Certificate Manager 証明書を参照可能です。

対象: Regional external ALB, Regional internal ALB, Cross-region internal ALB

🔗 統合されたコンソールワークフロー

証明書の作成・選択からロードバランサーの構成まで、コンソール上で一貫した操作が可能。Google マネージド証明書とセルフマネージド証明書の両方に対応しています。

🔧 設定方法

💡 前提条件: Certificate Manager API が有効であること / roles/certificatemanager.editor 以上の IAM ロール / Certificate Manager で証明書が作成済みであること

Step 1: Certificate Manager で証明書を作成

Bash Google マネージド証明書を DNS 認証で作成 
# Google マネージド証明書を DNS 認証で作成する例
gcloud certificate-manager certificates create my-certificate \
    --domains="example.com,www.example.com" \
    --dns-authorizations=my-dns-auth

Step 2: 証明書マップを作成 (グローバル LB の場合)

Bash 証明書マップとエントリの作成 
# 証明書マップを作成
gcloud certificate-manager maps create my-cert-map

# 証明書マップエントリを作成
gcloud certificate-manager maps entries create my-cert-map-entry \
    --map="my-cert-map" \
    --certificates="my-certificate" \
    --hostname="example.com"

Step 3: ロードバランサーのプロビジョニング時に証明書を選択

⚖ メリット・デメリット

✅ メリット (ビジネス面)

  • コンソール上で証明書管理と LB 設定を一元的に行え、設定時間を短縮
  • GUI ベースの操作により CLI コマンド入力ミスを防止
  • CLI に不慣れなインフラ管理者やセキュリティ担当者も参加しやすい

✅ メリット (技術面)

  • 証明書の選択と LB 構成が単一のコンソール画面で完結
  • 証明書マップにより数千の証明書を管理 (Compute Engine SSL 証明書の 15 件制限を超越)
  • DNS 認証、LB 認証、CA Service 連携など多様な認証方式をコンソールから選択可能

⚠ 制限事項・注意点

  • ロードバランサー認証の場合、SAN フィールドのドメイン数は最大 5 つ (DNS 認証は最大 100)
  • ALL_REGIONS スコープの証明書はロードバランサー認証をサポートしない
  • Global external ALB 等では、Compute Engine SSL 証明書と比較して TLS ハンドシェイクレイテンシが高くなる可能性
  • リージョナル Google マネージド証明書は DNS ベースの認証のみ (Public CA から取得)
  • 大規模本番環境では引き続き IaC (Terraform 等) による管理を推奨

🎯 ユースケース

🌐 ユースケース 1: 新規 Web アプリケーションの HTTPS 設定

シナリオ: 新規に Web アプリケーションをデプロイする際、Global external Application Load Balancer を作成し、HTTPS 通信を有効化する必要がある。

➤ コンソール上でロードバランサーのプロビジョニングと同時に Certificate Manager の証明書マップを選択でき、迅速に HTTPS 対応の環境を構築できる。

🌎 ユースケース 2: マルチリージョンの内部アプリケーション

シナリオ: 社内向けアプリケーションを複数リージョンで展開しており、Cross-region internal Application Load Balancer を使用。各リージョンに対応する TLS 証明書の管理が必要。

➤ コンソールからリージョンごとの Certificate Manager 証明書を直接選択してアタッチでき、複雑なマルチリージョン構成の証明書管理が簡素化される。

💰 料金情報

💡 コンソール統合機能の利用に追加料金はかかりません。

Certificate Manager 自体の使用料金は、証明書の数やタイプに基づきます。Cloud Load Balancing の料金は別途発生します。最新の料金情報は公式の料金ページをご確認ください。
