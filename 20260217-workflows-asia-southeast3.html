📄 Overview

Google Cloud Workflows has been made available in the new asia-southeast3 (Bangkok, Thailand) region. Workflows is a fully managed serverless orchestration platform that can execute Google Cloud services such as Cloud Run, Cloud Run functions, BigQuery, and Cloud Vision AI, as well as any HTTP-based API, in a defined sequence.

With this regional expansion, users in Thailand and the broader Southeast Asia region can deploy and run workflows while meeting data residency requirements. The asia-southeast3 region became generally available on Google Cloud in January 2026, and Workflows joins other services like Compute Engine, Cloud Storage, GKE, and Cloud Build that are progressively adding support for this region.

❌ Before

•The nearest regions for Workflows in Southeast Asia were asia-southeast1 (Singapore) or asia-southeast2 (Jakarta)
•No way to run Workflows within Thailand for data residency requirements
•Network latency from Bangkok to Singapore or Jakarta when executing workflows

✅ After

•Workflows deployment and execution now available in asia-southeast3 (Bangkok)
•Data residency compliant workflow orchestration achievable within Thailand
•Reduced latency through same-region communication for services near Bangkok

🛠️ Setup Guide

💡 Prerequisites:
→Google Cloud project must be active
→Workflows API must be enabled
→Appropriate IAM permissions (workflows.workflows.create) must be granted

Step 1: Enable Workflows API

Bash Enable Workflows API 
gcloud services enable workflows.googleapis.com \
    --project=PROJECT_ID

Step 2: Deploy workflow to asia-southeast3

Bash Deploy workflow 
gcloud workflows deploy my-workflow \
    --source=workflow.yaml \
    --location=asia-southeast3 \
    --project=PROJECT_ID

Step 3: Execute the workflow

Bash Run workflow 
gcloud workflows run my-workflow \
    --location=asia-southeast3 \
    --project=PROJECT_ID

⚠️ Limitations and Considerations

⚠️ Limitations:
•When using HTTP requests or connectors from workflows, the HTTP call origin may differ from the deployment region (Workflows specification)
•asia-southeast3 is a relatively new region, so some Google Cloud services may not yet be available there
💡 Points to Consider:
•Migrating existing workflows from another region to asia-southeast3 requires redeployment
•If using resource location constraints in organization policies, asia-southeast3 must be added to the allowed list

💡 Use Cases

🛒 Use Case 1: E-commerce Order Processing Orchestration in Thailand

Scenario: Automate the process from order reception to inventory check, payment processing, and shipping arrangement for a Thai domestic e-commerce platform using Workflows.

YAML Order Processing Workflow 
main:
  steps:
    - receive_order:
        call: http.post
        args:
          url: https://order-service-xxxxx-as.a.run.app/validate
          body:
            order_id: ${args.order_id}
        result: order_details
    - check_inventory:
        call: http.get
        args:
          url: https://inventory-service-xxxxx-as.a.run.app/check
          query:
            item_id: ${order_details.body.item_id}
        result: inventory_status
    - process_payment:
        switch:
          - condition: ${inventory_status.body.available}
            call: http.post
            args:
              url: https://payment-service-xxxxx-as.a.run.app/charge
              body:
                order_id: ${args.order_id}
                amount: ${order_details.body.total}
            result: payment_result
✅ Effect: Automate business processes with low latency while meeting Thailand's data residency requirements.
