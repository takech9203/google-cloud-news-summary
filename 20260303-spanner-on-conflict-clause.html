Cloud Spanner: GoogleSQL INSERT 文における ON CONFLICT 句のサポート

Cloud Spanner GA
2026-03-03

☁️ 概要

Cloud Spanner の GoogleSQL 方言において、INSERT 文で ON CONFLICT 句がサポートされました。 一意制約違反 (主キーの重複) が発生した場合の動作を宣言的に制御できるようになり、 ON CONFLICT DO NOTHING と ON CONFLICT DO UPDATE の 2 つのアクションが利用可能です。

PostgreSQL や他のデータベースで広く使われている標準的な UPSERT パターンが GoogleSQL でも利用できるようになり、 他のデータベースからの移行やマルチデータベース環境での SQL の共通化が容易になります。

🔄 Before / After 比較

❌ Before (アップデート前)

  • 重複キー処理には Spanner 独自の INSERT OR IGNORE / INSERT OR UPDATE 構文が必要
  • PostgreSQL 方言では ON CONFLICT が使えたが、GoogleSQL では利用不可
  • 他 DB からの移行時に SQL 書き換えが必要

✅ After (アップデート後)

  • ON CONFLICT DO NOTHING で重複行を安全にスキップ可能
  • ON CONFLICT DO UPDATE で標準的な UPSERT 操作が可能
  • PostgreSQL 等との SQL 互換性が向上し、移行が容易に

📊 アーキテクチャ図

ON CONFLICT 句を含む INSERT 文の処理フロー。主キーの重複が検出された場合、指定されたアクションに応じて挿入のスキップまたは既存行の更新が行われます。

flowchart TD
    A["INSERT 文 + ON CONFLICT 句"] --> B{"主キー重複チェック"}
    B -->|重複なし| C["新規行を挿入"]
    B -->|重複あり| D{"ON CONFLICT アクション"}
    D -->|DO NOTHING| E["挿入をスキップ\nエラーなし"]
    D -->|DO UPDATE| F["既存行を更新\nexcluded で値を参照"]
    C --> G["コミット"]
    E --> G
    F --> G

⚙️ 主要機能

1. ON CONFLICT DO NOTHING

✓挿入しようとする行が既に存在する場合、一意制約違反エラーをスローせずにスキップ
✓バッチ挿入時に一部の行が既存でもステートメント全体が失敗しない
✓冪等な書き込み操作の実装に最適

2. ON CONFLICT DO UPDATE

✓重複行がある場合、エラーなしで既存行を更新 (UPSERT)
✓excluded エイリアスで挿入しようとした値を UPDATE の値として参照可能
✓ON UPDATE 式が定義されたカラムは自動的に値が設定される

3. conflict_target の指定

✓主キーカラムを conflict_target として指定
✓複合主キーの場合は、すべての主キーカラムを指定する必要がある

構文の詳細

項目詳細

💻 コードサンプル

ON CONFLICT DO NOTHING: 重複行をスキップ

SQL 重複行の安全な挿入 
-- ON CONFLICT DO NOTHING: 重複行をスキップ
INSERT INTO Singers (SingerId, FirstName, LastName)
VALUES (1, 'Marc', 'Richards')
ON CONFLICT (SingerId) DO NOTHING;

ON CONFLICT DO UPDATE: UPSERT

SQL 重複行を更新 (UPSERT) 
-- ON CONFLICT DO UPDATE: 重複行を更新 (UPSERT)
INSERT INTO Singers (SingerId, FirstName, LastName)
VALUES (1, 'Marc', 'Richards')
ON CONFLICT (SingerId)
DO UPDATE SET
  SingerId = excluded.SingerId,
  FirstName = excluded.FirstName,
  LastName = excluded.LastName;

バッチ挿入 (DO NOTHING)

SQL 複数行の安全な一括挿入 
-- 既存データとの重複を無視して挿入
INSERT INTO Singers (SingerId, FirstName, LastName)
VALUES
  (1, 'Marc', 'Richards'),
  (2, 'Catalina', 'Smith'),
  (3, 'Alice', 'Trentor')
ON CONFLICT (SingerId) DO NOTHING;

複合主キーでの使用

SQL 複合主キーの UPSERT 
-- 複合主キーの場合、すべての主キーカラムを conflict_target に指定
INSERT INTO AlbumInfo (SingerId, AlbumId, AlbumTitle, MarketingBudget)
VALUES (1, 1, 'Total Junk', 100000)
ON CONFLICT (SingerId, AlbumId)
DO UPDATE SET
  SingerId = excluded.SingerId,
  AlbumId = excluded.AlbumId,
  AlbumTitle = excluded.AlbumTitle,
  MarketingBudget = excluded.MarketingBudget;

🛠️ 設定方法

前提条件

✓Cloud Spanner インスタンスおよび GoogleSQL 方言のデータベースが作成済み
✓テーブルに主キーが定義されている
✓適切な IAM 権限 (spanner.databases.write) を持つサービスアカウントまたはユーザー
💡 ヒント: 特別な設定やオプトインは不要です。GoogleSQL 方言のデータベースで即座に利用できます。

✅ メリット

ビジネス面

✓移行コストの削減: ON CONFLICT を使用した SQL の書き換えが不要に
✓開発生産性の向上: 標準 SQL パターンで学習コストが低減、コードの可読性が向上

技術面

✓SQL 互換性の向上: マルチデータベース環境での SQL 共通化が容易に
✓冪等な書き込み: DO NOTHING でリトライ時のエラーハンドリングを簡素化
✓宣言的な UPSERT: 1 つの SQL 文で「存在すれば更新、なければ挿入」を表現

⚠️ 制約事項・注意点

制限事項
✗conflict_target には主キーカラムのみ指定可能 (ユニークインデックスは指定不可)
✗複合主キーの場合、すべての主キーカラムを conflict_target に指定する必要がある
✗DO UPDATE SET 句では、INSERT リストのすべてのカラムを SET 句にも記載する必要がある
✗ON CONFLICT DO UPDATE 内の WHERE 句はサポートされていない
✗Partitioned DML では INSERT 自体がサポートされていないため、ON CONFLICT 句も使用不可
💡 考慮すべき点:
  • 従来の INSERT OR IGNORE / INSERT OR UPDATE 構文も引き続き利用可能
  • DO UPDATE SET の制約 (全カラム指定必須) は PostgreSQL とは異なるため、移行時に注意が必要

🎯 ユースケース

📦 ユースケース 1: 冪等なデータ同期

シナリオ: 外部システムからのデータを定期的に Spanner に同期する際、既存データとの重複を安全に処理したい。

SQL 冪等なデータ同期 
-- 外部システムからのデータを冪等に挿入
INSERT INTO Products (ProductId, Name, Price, LastSyncTime)
VALUES
  (101, 'Widget A', 29.99, CURRENT_TIMESTAMP()),
  (102, 'Widget B', 49.99, CURRENT_TIMESTAMP()),
  (103, 'Widget C', 19.99, CURRENT_TIMESTAMP())
ON CONFLICT (ProductId)
DO UPDATE SET
  ProductId = excluded.ProductId,
  Name = excluded.Name,
  Price = excluded.Price,
  LastSyncTime = excluded.LastSyncTime;

効果: リトライやスケジュール実行時に重複エラーが発生せず、常に最新データが反映される。

📨 ユースケース 2: イベントの重複排除

シナリオ: メッセージキューから受信したイベントを Spanner に記録する際、At-Least-Once 配信による重複イベントを安全に処理したい。

SQL イベント重複排除 
-- イベント ID で重複を排除
INSERT INTO Events (EventId, EventType, Payload, ReceivedAt)
VALUES (@eventId, @eventType, @payload, CURRENT_TIMESTAMP())
ON CONFLICT (EventId) DO NOTHING;

効果: 重複イベントの受信時にエラーが発生せず、アプリケーション側での重複チェックロジックを簡素化できる。

🔗 関連サービス・機能

→Cloud Spanner PostgreSQL 方言: 以前から ON CONFLICT 句がサポートされており、今回 GoogleSQL 方言にも同等の機能が追加された
→INSERT OR IGNORE / INSERT OR UPDATE: GoogleSQL 方言の従来からの重複処理構文。ON CONFLICT 句と同等の機能を提供
→Cloud Spanner Mutation API: DML とは別のデータ変更手段。InsertOrUpdate ミューテーションで同様の UPSERT 操作が可能
→Partitioned DML: 大量データの一括変更に使用。ただし INSERT (および ON CONFLICT) はサポートされていない

📝 まとめ

Cloud Spanner の GoogleSQL 方言で ON CONFLICT 句がサポートされたことにより、標準的な SQL パターンを使用した UPSERT 操作や重複排除が可能になりました。 他のデータベースからの移行の容易化、冪等な書き込み処理の簡素化、コードの可読性向上に寄与します。 既存の INSERT OR IGNORE / INSERT OR UPDATE 構文も引き続き利用可能なため、既存アプリケーションへの影響はありません。 新規開発や移行プロジェクトでは ON CONFLICT 句の採用を検討してください。
