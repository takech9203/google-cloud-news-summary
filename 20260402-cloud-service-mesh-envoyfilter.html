⚖️ Cloud Service Mesh: EnvoyFilter API サポート

Announcement Cloud Service Mesh TRAFFIC_DIRECTOR
2026-04-02

📋 概要

Managed Cloud Service Mesh の TRAFFIC_DIRECTOR 実装において、 EnvoyFilter API の限定的なサポートが追加されました。 これにより、他の Istio API では実現できなかったデータプレーンの拡張機能を利用できるようになります。

EnvoyFilter API は Envoy プロキシの設定を直接カスタマイズするための強力な API であり、 HTTP フィルタチェーンへのフィルタ追加などが可能になります。 特にローカルレートリミットや gRPC-Web フィルタなど、 サービスメッシュのトラフィック制御における重要な拡張機能が利用可能となりました。

⚠️ 注意: この機能はサポートされるフィールドとエクステンションが限定されています。 誤った設定がメッシュの安定性に影響する可能性があるため、他の Istio API で要件を満たせない場合にのみ使用してください。

🔄 Before / After

❌ Before

  • TRAFFIC_DIRECTOR 実装では EnvoyFilter API によるカスタマイズ不可
  • ローカルレートリミットなどの高度なトラフィック制御が利用不可
  • HTTP フィルタチェーンのカスタマイズ手段がなく拡張性に制限

✅ After

  • EnvoyFilter API でデータプレーンの拡張が可能に
  • ローカルレートリミット (LocalRateLimit) を設定可能
  • gRPC-Web フィルタが利用可能に
  • HTTP フィルタの INSERT_FIRST / INSERT_BEFORE 操作に対応

⚡ 主要機能

🔧 EnvoyFilter API の限定サポート

  • TRAFFIC_DIRECTOR コントロールプレーン実装でのみ利用可能
  • configPatches[].applyTo は HTTP_FILTER のみサポート
  • configPatches[].patch.operation は INSERT_FIRST および INSERT_BEFORE (Router フィルタとの組み合わせ) のみサポート

🚦 ローカルレートリミットフィルタ

  • envoy.extensions.filters.http.local_ratelimit.v3.LocalRateLimit をサポート
  • トークンバケット、フィルタの有効化/強制化、レスポンスヘッダの追加が設定可能
  • ダウンストリーム接続ごとのレート制限に対応
  • Rapid、Regular、Stable の全リリースチャンネルで利用可能

🌐 gRPC-Web フィルタ

  • envoy.extensions.filters.http.grpc_web.v3.GrpcWeb をサポート
  • gRPC-Web クライアントからのリクエストを gRPC バックエンドサービスに中継可能
  • 全リリースチャンネルで利用可能

💻 設定例: ローカルレートリミット

以下は、EnvoyFilter を使用してローカルレートリミットを設定する YAML の例です。

YAML EnvoyFilter - Local Rate Limit 
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: local-ratelimit
  namespace: istio-system
spec:
  configPatches:
  - applyTo: HTTP_FILTER
    match:
      listener:
        filter:
          name: envoy.filters.network.http_connection_manager
          subFilter:
            name: envoy.filters.http.router
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.http.local_ratelimit
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.http.local_ratelimit.v3.LocalRateLimit
          stat_prefix: http_local_rate_limiter
          token_bucket:
            max_tokens: 100
            tokens_per_fill: 100
            fill_interval: 60s
          filter_enabled:
            runtime_key: local_rate_limit_enabled
            default_value:
              numerator: 100
              denominator: HUNDRED
          filter_enforced:
            runtime_key: local_rate_limit_enforced
            default_value:
              numerator: 100
              denominator: HUNDRED

🛠 設定方法

📌 前提条件: Cloud Service Mesh が TRAFFIC_DIRECTOR コントロールプレーン実装で有効化されていること。 GKE クラスタに Managed Cloud Service Mesh がプロビジョニングされていること。

Step 1: コントロールプレーン実装の確認

Bash 
# コントロールプレーン実装を確認
kubectl get controlplanerevision -n istio-system -o yaml

TRAFFIC_DIRECTOR 実装であることを確認します。ISTIOD 実装の場合、この機能は利用できません。

Step 2: EnvoyFilter の適用

Bash 
# EnvoyFilter リソースを適用
kubectl apply -f envoyfilter.yaml

Step 3: 設定の検証

Bash 
# EnvoyFilter CR のステータスを確認
kubectl get envoyfilter <NAME> -n <NAMESPACE> -o yaml

# Envoy のコンフィグダンプで設定の伝播を確認
gcloud beta container fleet mesh debug proxy-config \
  <POD_NAME>.<NAMESPACE> \
  --type=listener \
  --membership=<MEMBERSHIP_ID> \
  --location=<MEMBERSHIP_LOCATION> \
  --project=<PROJECT_ID> \
  --output=yaml | grep envoy.filters.http.local_ratelimit -C 5

⚖️ メリット / デメリット

✅ メリット

ビジネス面
  • ローカルレートリミットによりサービスの過負荷を防止
  • マネージドサービスでの高度なデータプレーンカスタマイズが可能に
技術面
  • Istio API の範囲を超えた Envoy ネイティブ機能の直接利用
  • 外部レートリミットサービス不要でローカルにレート制限実施
  • ブラウザベースの gRPC-Web クライアントをシームレスに処理

❌ デメリット / 制約事項

制限事項
  • TRAFFIC_DIRECTOR 実装のみ (ISTIOD 非対応)
  • applyTo は HTTP_FILTER のみ
  • 操作は INSERT_FIRST / INSERT_BEFORE のみ (MERGE, REMOVE 不可)
  • エクステンションはローカルレートリミットと gRPC-Web の 2 種類のみ
考慮すべき点
  • 誤った設定がメッシュ全体の安定性に影響する可能性
  • 標準 Istio API を優先して使用すべき
  • 設定反映に時間がかかる場合がある (2 パス設定コミット)
  • Google サポートはエクステンション設定の正確性を対象外としている

🎯 ユースケース

💰 UC1: マイクロサービスのローカルレートリミット

シナリオ: 決済サービスへのリクエスト急増時に、サービスの過負荷を防ぐためにレート制限を設定

効果: 外部レートリミットサービスを構築せず、各 Envoy サイドカーレベルで 1 分あたり 50 リクエストの制限を適用し、決済サービスの安定性を確保

YAML 決済サービス向けレートリミット 
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: payment-service-ratelimit
  namespace: payment
spec:
  configPatches:
  - applyTo: HTTP_FILTER
    match:
      listener:
        filter:
          name: envoy.filters.network.http_connection_manager
          subFilter:
            name: envoy.filters.http.router
    patch:
      operation: INSERT_BEFORE
      value:
        name: envoy.filters.http.local_ratelimit
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.http.local_ratelimit.v3.LocalRateLimit
          stat_prefix: payment_rate_limiter
          token_bucket:
            max_tokens: 50
            tokens_per_fill: 50
            fill_interval: 60s
          filter_enabled:
            default_value:
              numerator: 100
              denominator: HUNDRED
          filter_enforced:
            default_value:
              numerator: 100
              denominator: HUNDRED
          status:
            code: 429

🌐 UC2: gRPC-Web クライアントのサポート

シナリオ: ブラウザベースの gRPC-Web クライアントから gRPC バックエンドサービスへのリクエストを、サービスメッシュ内で透過的に処理

効果: gRPC-Web プロトコル変換が Envoy サイドカーで自動処理され、 バックエンドサービスの変更なしに Web クライアントからの gRPC 通信が可能に

🔗 関連サービス

🛡️ Cloud Armor

サーバーサイドグローバルレートリミットを提供。EnvoyFilter のローカルレートリミットとは異なり、集中管理型のレート制限を実施可能

📊 Cloud Monitoring

Envoy エクステンションが出力する統計情報を可視化し、フィルタの動作状態を監視可能

⚙️ GKE

Cloud Service Mesh のデータプレーンが動作する基盤。マネージドデータプレーンにより Envoy サイドカーの自動アップグレードが可能

📜 Istio API

VirtualService、DestinationRule など。EnvoyFilter を使用する前に、まず標準 Istio API で要件を満たせないか検討すべき
