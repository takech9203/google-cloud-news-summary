Gemini Enterprise:
Jira / Confluence フェデレーテッドコネクタが GA

Gemini Enterprise GA Federated Search Atlassian
2026-04-03

📋 概要

Gemini Enterprise において、Jira Cloud および Confluence Cloud のフェデレーテッドコネクタが 一般提供 (GA) となりました。Atlassian のプロジェクト管理ツールとナレッジベースのデータを、Gemini Enterprise のエンタープライズ検索基盤からリアルタイムに検索できます。

フェデレーテッド検索は、データを Google Cloud 側にコピーせず、Atlassian API を介して直接データソースを検索する方式です。データの二重管理が不要であり、常に最新のデータに対して検索が実行されます。GA 昇格により本番ワークロードでの利用がサポートされ、SLA の対象となります。

🔄 Before / After 比較

❌ Before (従来)

  • フェデレーテッドコネクタがパブリックプレビューであり、本番利用には SLA が保証されなかった
  • Atlassian データの横断検索にはデータインジェスト方式が必要で、同期やストレージコストが発生
  • Jira イシューや Confluence ドキュメントを探す際に、ツールを切り替えて個別に検索が必要

✅ After (改善後)

  • フェデレーテッドコネクタが GA となり、SLA 付きで本番環境をサポート
  • データインジェスト不要でリアルタイムのフェデレーテッド検索が可能、ストレージコスト不要
  • Gemini Enterprise の統合検索から Jira イシューや Confluence ドキュメントも一元検索可能

🏗️ アーキテクチャ図

⚡ 主要機能

📋 Jira Cloud フェデレーテッドコネクタ

Jira Cloud のイシュー、プロジェクト、コメント、添付ファイル、ワークログなどをリアルタイムに検索可能。OAuth 2.0 (read:jira-user, read:jira-work) による認証。パーミッション対応の検索でアクセス権を持つデータのみ表示。

📝 Confluence Cloud フェデレーテッドコネクタ

Confluence Cloud のページ、スペース、コメントなどをリアルタイムに検索可能。OAuth 2.0 または API トークンによる認証に対応。アイデンティティプロバイダーによるアクセス制御が可能。

🔍 フェデレーテッド検索

データを Google Cloud 側にコピーせず、Atlassian API を介して直接検索。データの二重管理が不要で、常に最新データに対してリアルタイム検索を実行。

🤖 アクション実行

検索だけでなく、Jira イシューの更新やコメント追加などのアクションも実行可能 (write:jira-work スコープが必要)。Gemini の LLM がクエリを書き換えて精度の高い検索結果を返却。

📝 設定方法

前提条件: Gemini Enterprise サブスクリプション (Standard / Plus / Frontline) が有効、Atlassian Cloud 管理者権限、Discovery Engine Editor ロール (roles/discoveryengine.editor) が付与済み、アイデンティティプロバイダーが設定済みであること。

Step 1: Atlassian 側で OAuth 2.0 アプリを作成

Atlassian Developer Console で OAuth 2.0 アプリを作成し、Client ID と Client Secret を取得します。最小限のアプリケーションパーミッションを設定してください。

Step 2: Gemini Enterprise でデータストアを作成

Google Cloud Console > Gemini Enterprise > Data stores > Create data store

  1. ソースセクションで「Jira Cloud」または「Confluence Cloud」を選択
  2. コネクタモードで「Federated search」を選択
  3. 認証設定で Client ID、Client Secret、Instance URI を入力
  4. 「Login」をクリックして Atlassian サインインを完了
  5. 検索対象のエンティティを選択
  6. データストア名を入力して作成を完了

Step 3: アプリに接続

Google Cloud Console > Gemini Enterprise > Apps > アプリを選択 > データストアを接続

作成したデータストアを既存の Gemini Enterprise アプリに接続します。

💡 ユースケース

🐛 ソフトウェア開発チームのナレッジ検索

シナリオ: 開発者がバグ修正に取り組む際、関連する Jira イシューの履歴と Confluence の設計ドキュメントを同時に検索したい。

効果: Gemini Enterprise の統合検索から「認証モジュール タイムアウトエラー」と検索するだけで、関連する Jira イシュー、過去の修正履歴、Confluence の設計ドキュメントが一覧で表示。ツール間の切り替えが不要となり、問題解決までの時間を短縮できます。

📊 プロダクトマネージャーのプロジェクト状況把握

シナリオ: プロダクトマネージャーが複数プロジェクトの進捗状況を横断的に確認し、ステークホルダーへの報告資料を作成したい。

効果: Gemini Enterprise からプロジェクト名やマイルストーンで検索し、Jira のスプリント進捗と Confluence の議事録・仕様書を統合的に把握。Gemini の AI 機能と組み合わせることで、情報の要約やレポート作成も効率化されます。

⚖️ メリット・デメリット

✅ メリット

ビジネス面

  • ✓ 統合検索による生産性向上 -- ツール間の切り替えが不要
  • ✓ データガバナンスの維持 -- データは Atlassian Cloud 側に保持
  • ✓ GA による信頼性保証 -- SLA 対象で本番利用をサポート

技術面

  • ✓ リアルタイム検索 -- インジェストや同期の待ち時間なし
  • ✓ パーミッション対応 -- アクセス権限に基づくフィルタリング
  • ✓ ストレージ不要 -- インデックスストレージクォータを消費しない

⚠️ デメリット・制約

制限事項

  • ! データインジェスト方式と比較して検索品質が劣る可能性がある
  • ! クエリは Atlassian API に送信されるため、Atlassian 側の利用規約が適用
  • ! LLM クエリ書き換え時にセッション内クエリ履歴が Atlassian に送信される可能性

考慮すべき点

  • ● Confluence Cloud は新しい集中型 Atlassian ユーザー管理モデルが必要
  • ● OAuth 2.0 スコープは最小限に設定することを推奨
  • ● 複数のフェデレーテッドデータソースが有効な場合、クエリが全データソースに送信される可能性

💰 料金情報

💡 ポイント: フェデレーテッドコネクタは Gemini Enterprise サブスクリプションの一部として提供されます。フェデレーテッド検索方式ではデータのインジェストが発生しないため、ストレージクォータを消費しません。

エディション別ストレージ容量

エディション ストレージ容量 (ユーザーあたり/月)
Gemini Enterprise Business 25 GiB (プール)
Gemini Enterprise Standard 30 GiB (プール)
Gemini Enterprise Plus 75 GiB (プール)
Gemini Enterprise Frontline 2 GiB (プール)

フェデレーテッド検索ではこれらのストレージ容量を消費しないため、他のデータソースのインジェストに容量を割り当てることができます。

🌐 利用可能リージョン

Gemini Enterprise はグローバル、US、EU のマルチリージョンで利用可能です。ライセンスはプロジェクトとロケーションに紐づくため、複数リージョンで利用する場合は各リージョンに対して個別のライセンスが必要です。

📌 まとめ

Gemini Enterprise の Jira Cloud / Confluence Cloud フェデレーテッドコネクタが GA となったことで、Atlassian 製品のデータを本番環境で安心してエンタープライズ検索に統合できるようになりました。フェデレーテッド方式はデータの複製が不要でリアルタイム検索が可能なため、データガバナンスを維持しつつ生産性を向上させたい組織に適しています。

Atlassian 製品を活用する組織では、Gemini Enterprise との連携を検討し、統合検索基盤の構築を進めることを推奨します。
