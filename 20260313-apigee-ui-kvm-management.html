⚙️ Apigee UI: KVM エントリ管理機能

Apigee UI Feature
2026-03-13

📋 概要

Apigee UI において、環境スコープの Key Value Map (KVM) エントリの表示、追加、編集、削除が可能になりました。 これまで KVM エントリの操作には API または KeyValueMapOperations ポリシーが必要でしたが、 Google Cloud Console の Apigee UI から直接 GUI ベースで管理できるようになります。

API 管理者、開発者、DevOps エンジニアにとって、設定変更やデバッグの作業時間が大幅に短縮されるアップデートです。

🔄 Before / After 比較

❌ Before

✕ curl コマンドや API クライアントの知識が必須で非技術者には困難

✅ After

✓ UI から KVM エントリの表示・追加・編集・削除が可能に
✓ API コールや CLI 不要で GUI ベースの管理を実現
✓ 環境ごとの設定を視覚的に確認、設定ミスの発見が容易に

⭐ 主要機能

👁️

KVM エントリの表示

環境スコープの KVM に含まれるキーと値のペアを一覧表示。マスキング有効時は値がアスタリスクで表示される。

➕

KVM エントリの追加

UI からキーと値のペアを新規追加。API コールなしにエントリを即座に追加可能。

✏️

KVM エントリの編集

既存エントリの値を UI から直接編集。環境固有の設定値を迅速に変更できる。

🗑️

KVM エントリの削除

不要なエントリを UI から削除。確認ダイアログで誤削除を防止。

🔧 技術仕様

KVM のスコープ

スコープ 説明 UI での管理
API プロキシ 特定の API プロキシのみがアクセス可能 API のみ
環境 特定の環境内のすべての API プロキシがアクセス可能 UI および API
組織 すべての環境のすべての API プロキシがアクセス可能 API のみ

KVM の暗号化

🔒 セキュリティ: すべての KVM エントリは Cloud KMS キーを使用して AES256 で暗号化されます。暗号化キーは Apigee 組織のプロビジョニング時に指定された runtimeDatabaseEncryptionKey が使用されます。

KVM マスキング

設定 説明
マスキング有効 UI 上で値がアスタリスク (*****) として表示される
マスキング無効 UI 上で値がプレーンテキストとして表示される

※ マスキングは KVM レベルで設定され、個別のエントリ単位では設定できません。

API を使用した KVM エントリ管理 (参考)

UI に加えて、Apigee API を使用した管理も引き続き可能です。

Bash 
# 環境スコープの KVM エントリ一覧を取得
curl -X GET \
  "https://apigee.googleapis.com/v1/organizations/{ORG}/environments/{ENV}/keyvaluemaps/{KVM_NAME}/entries" \
  -H "Authorization: Bearer $(gcloud auth print-access-token)"

# KVM エントリの追加
curl -X POST \
  "https://apigee.googleapis.com/v1/organizations/{ORG}/environments/{ENV}/keyvaluemaps/{KVM_NAME}/entries" \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my-key",
    "value": "my-value"
  }'

👍 メリット

💼 ビジネス面

→運用効率の向上: API コールや CLI 操作なしに KVM エントリを管理でき、設定変更に要する時間が短縮される
→非技術者のアクセス向上: GUI ベースの操作により、API の知識がない管理者でも KVM エントリの確認・変更が可能に
→インシデント対応の迅速化: 障害時に KVM の設定値を即座に UI で確認・修正でき、MTTR (平均復旧時間) の短縮に寄与

🛠️ 技術面

→操作の可視性向上: KVM エントリの内容を UI で視覚的に確認でき、設定ミスの発見が容易に
→環境間の設定比較が容易: 環境を切り替えながら KVM エントリを確認でき、test/prod の設定差異を把握しやすい
→API とポリシーの二重管理が不要: 単純なエントリの追加・変更であれば、KeyValueMapOperations ポリシーのデプロイや API コールが不要

⚠️ 制約事項・考慮点

⚠️ 制限事項:
  • UI での KVM 管理は環境スコープのみに対応。API プロキシスコープおよび組織スコープの KVM は引き続き API を使用する必要がある
  • 大量のエントリ (150 キーを超える KVM) の操作は API の使用が推奨 (Apigee Edge の場合)
🚨 注意:
  • 本番環境の KVM に対する不用意な変更を防ぐため、IAM ロールによるアクセス制御を適切に設定すること
  • マスキング有効の KVM では UI 上で値が表示されないため、設定内容の確認には API が必要な場合がある
  • KVM エントリの変更は即座に反映されるため、実行中の API プロキシへの影響を事前に評価すること

💡 ユースケース

🎯 ユースケース 1: 環境固有の設定値管理

シナリオ: API プロキシが外部サービスのエンドポイント URL やタイムアウト値を環境ごとに異なる KVM エントリとして保持。開発者が test 環境のエンドポイント URL を変更する必要がある。

KVM 設定例 
KVM: service-config (環境: test)
  backend_url: https://test-api.example.com/v2
  timeout_ms: 5000
  retry_count: 3

効果: UI から直接値を変更でき、API プロキシの再デプロイや API コールが不要。設定変更にかかる時間を数分から数秒に短縮。

🎯 ユースケース 2: フィーチャーフラグの管理

シナリオ: API プロキシが KVM に保存されたフィーチャーフラグを参照し、特定の機能の有効・無効を制御。新機能のロールアウトや緊急の機能無効化を行う。

効果: 運用担当者が GUI から即座にフィーチャーフラグを切り替え可能。API プロキシの再デプロイなしに動作を変更でき、ダウンタイムなしの運用を実現。
