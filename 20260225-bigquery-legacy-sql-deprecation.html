📋 Overview

Google Cloud announced restrictions on Legacy SQL usage in BigQuery. After June 1, 2026, Legacy SQL will be disabled for organizations and projects that did not use it during the evaluation period (November 1, 2025 -- June 1, 2026).

For organizations that used Legacy SQL during the evaluation period, only the specific feature sets that were used will remain available. Feature usage is aggregated at the organization level -- if any project in the organization used a particular feature, that feature remains available across all projects in the organization. Projects not belonging to an organization are managed at the project level.

⚠️ Important: Migration to GoogleSQL is strongly recommended but not mandatory. GoogleSQL supports many features unavailable in Legacy SQL, including DML/DDL statements, CTEs, materialized views, search indexes, and generative AI functions.

🔄 Before / After Comparison

❌ Before (Challenges)

  • Legacy SQL not compliant with SQL 2011 standard -- low compatibility with other database systems
  • DML, DDL, CTEs, materialized views, search indexes, and generative AI functions unavailable in Legacy SQL
  • Legacy SQL cannot leverage BigQuery advanced runtime -- disadvantaged in performance and cost efficiency
  • View definition compatibility issues between the two SQL dialects

✅ After (Improvements)

  • Organizations not using Legacy SQL are automatically transitioned to GoogleSQL-only environments
  • Sufficient grace period provided for organizations actively using Legacy SQL -- enabling phased migration
  • Exemption requests available via Google Forms or email (bq-legacysql-support@google.com)
  • query_dialect field in INFORMATION_SCHEMA.JOBS view enables usage tracking

⭐ Key Features

📅 1. Evaluation Period-Based Feature Availability

Determined based on Legacy SQL usage during the evaluation period (November 1, 2025 -- June 1, 2026). Organizations/projects that never used Legacy SQL will have it disabled after June 1, 2026. Those that used it can continue using only the specific feature sets that were exercised.

🏢 2. Organization-Level Feature Aggregation

Feature usage is aggregated at the organization level. If any project in the organization used a specific feature, that feature is available across all projects within the organization. Projects without an organization are managed individually at the project level. Existing organizations using Legacy SQL can also use it in new projects.

📦 3. Three Feature Set Classifications

  • Basic Language Features: SELECT, FROM, JOIN, WHERE, GROUP BY, etc. -- a single Legacy SQL query enables the entire set
  • Extended Language Features: FLATTEN operator, EACH modifier, table decorators, UDFs, etc. -- individually tracked, each must be explicitly used
  • Function Grouping: Math functions, string functions, window functions, etc. -- using one function in a group enables the entire group

🛠️ Migration Steps

📝 Prerequisites: A BigQuery project or organization must exist, and access to the INFORMATION_SCHEMA.JOBS view is required for checking Legacy SQL usage.
1

Check Legacy SQL Usage

Identify projects using Legacy SQL and job counts. Pay special attention to DEFAULT_LEGACY_SQL jobs where the dialect was not explicitly specified.

SQL Check Legacy SQL usage in the last 30 days 
SELECT
  project_id,
  COUNTIF(query_dialect = 'DEFAULT_LEGACY_SQL') AS default_legacysql_jobs,
  COUNTIF(query_dialect = 'LEGACY_SQL') AS explicit_legacysql_jobs
FROM
  `region-us`.INFORMATION_SCHEMA.JOBS
WHERE
  creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND (query_dialect = 'DEFAULT_LEGACY_SQL' OR query_dialect = 'LEGACY_SQL')
GROUP BY project_id
ORDER BY default_legacysql_jobs DESC;
2

Identify Legacy SQL Queries and Migrate to GoogleSQL

Find specific queries using Legacy SQL and rewrite them in GoogleSQL. Note the key syntax differences.

SQL Identify Legacy SQL queries 
SELECT
  job_id,
  user_email,
  query,
  creation_time
FROM
  `region-us`.INFORMATION_SCHEMA.JOBS
WHERE
  creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 30 DAY)
  AND (query_dialect = 'DEFAULT_LEGACY_SQL' OR query_dialect = 'LEGACY_SQL')
  AND job_type = 'QUERY'
ORDER BY creation_time DESC
LIMIT 100;
3

Set GoogleSQL as the Default SQL Dialect

Change BigQuery's default SQL dialect to GoogleSQL. This ensures queries without a specified dialect automatically run as GoogleSQL.

4

Migrate Views Defined in Legacy SQL

Views defined in Legacy SQL cannot be queried from GoogleSQL and vice versa, so they must be recreated in GoogleSQL.

SQL Identify views defined in Legacy SQL 
SELECT
  table_catalog,
  table_schema,
  table_name,
  view_definition
FROM
  `project_id`.`dataset_id`.INFORMATION_SCHEMA.VIEWS
WHERE
  TRUE; -- Review definitions to identify Legacy SQL views

✅ Benefits

💼 Business Benefits

✓ Cost Optimization: GoogleSQL leverages BigQuery advanced runtime for improved query performance and potential cost reduction
✓ Future Feature Access: All new BigQuery features (generative AI functions, materialized views, search indexes) are provided for GoogleSQL
✓ Talent Accessibility: GoogleSQL complies with SQL 2011 standard, making it easier to leverage personnel with standard SQL skills

⚙️ Technical Benefits

✓ Standard SQL Compliance: SQL 2011 standard -- high compatibility with other database systems
✓ Advanced Query Features: WITH clause composability, flexible subquery placement, ARRAY/STRUCT types, correlated subqueries
✓ DML/DDL Support: INSERT, UPDATE, DELETE, CREATE TABLE available
✓ Accurate COUNT(DISTINCT): GoogleSQL provides accurate and scalable COUNT(DISTINCT) without Legacy SQL's EXACT_COUNT_DISTINCT limitations

⚠️ Limitations & Considerations

🚫 Restrictions

✗ Features not used during the evaluation period become unavailable after June 1, 2026
✗ Views defined in Legacy SQL cannot be queried from GoogleSQL (and vice versa) -- view recreation required
✗ After the evaluation period, new organizations/projects cannot use Legacy SQL
✗ No dedicated tool is provided for auditing specific feature usage

💭 Considerations

⚠ Rarely-used Legacy SQL queries must be executed once during the evaluation period if continued usage is needed
⚠ Legacy SQL usage across all projects in the organization must be checked to understand the impact scope
⚠ When migrating Legacy SQL views, queries referencing those views must also be migrated
⚠ DEFAULT_LEGACY_SQL queries (where dialect was not specified) need identification and resolution

💡 Use Cases

🔍 Use Case 1: Legacy SQL Usage Inventory

Scenario: In a large organization, assess how much Legacy SQL is used across multiple projects and formulate a migration plan.

SQL Organization-wide Legacy SQL usage summary 
SELECT
  project_id,
  COUNTIF(query_dialect = 'DEFAULT_LEGACY_SQL') AS implicit_legacy_jobs,
  COUNTIF(query_dialect = 'LEGACY_SQL') AS explicit_legacy_jobs,
  COUNTIF(query_dialect IN ('GOOGLE_SQL', 'DEFAULT_GOOGLE_SQL')) AS googlesql_jobs,
  MIN(creation_time) AS earliest_legacy_usage,
  MAX(creation_time) AS latest_legacy_usage
FROM
  `region-us`.INFORMATION_SCHEMA.JOBS
WHERE
  creation_time >= TIMESTAMP('2025-11-01')
  AND job_type = 'QUERY'
GROUP BY project_id
HAVING implicit_legacy_jobs > 0 OR explicit_legacy_jobs > 0
ORDER BY implicit_legacy_jobs + explicit_legacy_jobs DESC;

Effect: Enables prioritization of projects requiring migration and estimation of migration effort.

🔄 Use Case 2: Legacy SQL View Migration

Scenario: Migrate views defined in Legacy SQL to GoogleSQL to resolve cross-dialect query compatibility issues.

SQL View migration example (Before / After) 
-- Legacy SQL View (Before)
-- #legacySQL
-- SELECT *, UTC_USEC_TO_DAY(timestamp_col) AS day FROM MyTable

-- GoogleSQL View (After)
CREATE OR REPLACE VIEW `project.dataset.my_view` AS
SELECT *, EXTRACT(DAY FROM timestamp_col) AS day
FROM `project.dataset.MyTable`;

Effect: GoogleSQL views can be combined with DML, CTEs, materialized views, and other modern features.

💰 Pricing

There are no additional charges for the Legacy SQL usage restriction itself. BigQuery's pricing structure remains unchanged.

Migrating to GoogleSQL enables the use of BigQuery advanced runtime, which may reduce query execution costs.

📝 Summary

The BigQuery Legacy SQL usage restriction is a significant change that promotes migration to GoogleSQL. Before the June 1, 2026 deadline, it is recommended to:

  1. Check Legacy SQL usage using the query_dialect field in INFORMATION_SCHEMA.JOBS
  2. Understand the scope of impact across your organization
  3. If using Legacy SQL, ensure all necessary features are exercised during the evaluation period
  4. Plan and execute a systematic migration to GoogleSQL
  5. Prioritize early migration of views defined in Legacy SQL
