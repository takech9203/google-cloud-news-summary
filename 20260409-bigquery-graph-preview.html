📊 BigQuery Graph: GQL によるグラフ分析

BigQuery Preview
2026-04-09

📋 概要

BigQuery Graph は、既存の BigQuery テーブルに格納されたエンティティと関係性データをグラフ (ノードとエッジ) としてモデル化し、Graph Query Language (GQL) を使用して複雑な隠れた関係性を発見できる新機能です。

💡 最大の特徴: 既存のワークフローを変更したりデータを複製する必要がありません。テーブルに格納されたデータから直接グラフを作成し、ISO GQL 標準および ISO Property Graph Queries (SQL/PGQ) 標準に準拠したクエリで分析を行います。

🔄 Before / After 比較

❌ Before

  • ✕関係性の探索にセルフジョインや再帰ジョインが必要
  • ✕SQL でのグラフ探索ロジックが複雑で保守困難
  • ✕専用グラフ DB への ETL が必要 (運用オーバーヘッド)
  • ✕大規模データのグラフ分析にスケーラビリティ課題

✅ After

  • ✓既存テーブルから直接グラフを作成 (複製不要)
  • ✓GQL で関係性を直感的かつ簡潔に記述
  • ✓BigQuery の分散エンジンで大規模グラフ分析
  • ✓ノートブックでグラフ結果を視覚的に確認
  • ✓Spanner Graph と統合しシームレスに連携

⚡ 主要機能

📐 ISO GQL 標準準拠クエリ

ISO GQL 標準および SQL/PGQ 標準に準拠。MATCH、FILTER、RETURN、LET、NEXT などの豊富なステートメントをサポートし、集合演算 (UNION ALL) によるクエリの組み合わせも可能です。

🗄️ 既存テーブルからのグラフ作成

CREATE PROPERTY GRAPH DDL でノードテーブルとエッジテーブルを定義。既存データの複製不要で、テーブルを直接参照してグラフを構成できます。

📈 グラフ可視化

BigQuery Studio、Google Colab、Jupyter Notebook で可視化に対応。%%bigquery --graph マジックコマンドでクエリ結果をグラフ形式で表示し、ノードやエッジをクリックして詳細を確認できます。

🔍 ビルトイン検索機能

ベクトル検索およびフルテキスト検索をグラフと統合。セマンティック検索やキーワード検索をグラフ分析に活用できます。

🔗 Spanner Graph との統合

同一のグラフスキーマとクエリ言語を共有。Spanner での運用ワークロード (OLTP) と BigQuery での分析ワークロードを、再モデリングなしに連携可能です。

📝 GQL の主要ステートメント

ステートメント 概要
GRAPHクエリ対象のプロパティグラフを指定
MATCHグラフパターンに一致するデータを検索
FILTER条件に基づいて結果をフィルタリング
RETURN線形クエリステートメントの終端。結果を返す
LET変数を定義し、後続クエリで使用する値を代入
WITH指定カラムのみを後続に渡す (フィルタリング・リネーム可能)
NEXT複数の線形クエリステートメントをチェーン接続
ORDER BYクエリ結果をソート
LIMIT / OFFSET結果件数の制限およびスキップ
Set operationsUNION ALL 等で複数クエリを結合

プロパティグラフの定義例

SQL CREATE PROPERTY GRAPH 
-- プロパティグラフの定義
CREATE OR REPLACE PROPERTY GRAPH graph_db.FinGraph
  NODE TABLES (
    graph_db.Account,
    graph_db.Person
  )
  EDGE TABLES (
    graph_db.PersonOwnAccount
      SOURCE KEY (id) REFERENCES Person(id)
      DESTINATION KEY (account_id) REFERENCES Account(id)
      LABEL Owns
  );

GQL クエリの実行例

SQL GQL パターンマッチング 
GRAPH graph_db.FinGraph
MATCH (p:Person)-[o:Owns]->(a:Account)
FILTER p.birthday < '1990-01-10'
RETURN p.name

🎯 ユースケース

🏦 金融不正検出

口座間の送金パターンを分析し、マネーロンダリングや不正な資金移動を検出。SQL の再帰クエリよりも直感的かつ効率的に不審なパターンを発見できます。

GRAPH graph_db.FinGraph
MATCH (a:Account {is_blocked: TRUE})
RETURN a
UNION ALL
MATCH (a:Account)<-[:Owns]-(p:Person {id: 2})
RETURN a
NEXT
MATCH (a:Account)-[t:Transfers]->(oa:Account)
WITH DISTINCT oa
RETURN oa.nick_name

🛒 顧客 360 レコメンデーション

顧客の購買履歴・関係性・嗜好をグラフとしてモデル化。類似顧客の購買傾向に基づいたパーソナライズされた商品推薦を実現します。

🏭 サプライチェーントレーサビリティ

部品、サプライヤー、注文、可用性、欠陥情報をグラフ化。特定部品の欠陥発見時に、影響範囲をグラフ探索で迅速に特定できます。

✅ メリット

ビジネス面

✓ データサイロの解消: リレーショナルデータとグラフデータを統一プラットフォームで分析、ETL 不要
✓ 迅速な不正検出: 金融取引の複雑な関係性パターンを GQL で直感的に検出
✓ 顧客理解の深化: 関係性・購買履歴をグラフ可視化し、パーソナライズに活用

技術面

✓ クエリの簡素化: 再帰ジョインが GQL パターンマッチングに置き換わり、可読性と保守性が向上
✓ スケーラビリティ: BigQuery の分散分析エンジンで大規模グラフにも高パフォーマンス
✓ Spanner 互換: OLTP (Spanner) と分析 (BigQuery) のグラフワークロードをシームレスに連携
✓ 標準準拠: ISO GQL / SQL/PGQ 準拠で他グラフ DB との知識移行が容易

⚠️ デメリット・制約事項

制限事項
  • Preview 段階のため、サポートが限定的であり SLA の対象外
  • Enterprise / Enterprise Plus エディションのリザベーションが必要 (Standard やオンデマンドでは不可)
  • グラフ可視化で表示可能なデータ量は最大 2 MB まで
  • 可視化には TO_JSON 関数で JSON 形式で返す必要あり
考慮すべき点
  • GA までにインターフェースや動作が変更される可能性あり
  • サポート・フィードバックは bq-graph-preview-support@google.com へ連絡
  • パス (グラフパス) を返すことで可視化の完全性が向上
