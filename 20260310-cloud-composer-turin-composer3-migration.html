☁️ Cloud Composer: Turin リージョン Composer 3 専用化

Cloud Composer Announcement
2026-03-10

📋 概要

Google Cloud は、Turin (europe-west12) リージョンにおいて Cloud Composer 2 環境の新規作成を終了し、同リージョンを Cloud Composer 3 専用に切り替えることを発表しました。

既存の Cloud Composer 2 環境は引き続き動作しますが、新規作成はできなくなるため、Turin リージョンのユーザーは Cloud Composer 3 への移行を計画する必要があります。Cloud Composer 3 は、簡素化されたネットワーク設定、テナントプロジェクトでのクラスター管理、CeleryKubernetes Executor の採用など、多くのアーキテクチャ改善が含まれています。

💡 ポイント: この変更は Google Cloud が段階的に Cloud Composer 3 への移行を推進している取り組みの一環であり、Turin リージョンがその先行リージョンとなります。

🔄 Before / After 比較

❌ Before (変更前)

✗Composer 2 と Composer 3 の両方で環境作成が可能だったが、バージョン管理が複雑
✗GKE Autopilot クラスターがユーザープロジェクトに展開され、ネットワーク設定が複雑
✗Service Agent に追加の権限が必要

✅ After (変更後)

✓Turin では Composer 3 のみ新規作成可能で、バージョン選択がシンプル
✓テナントプロジェクトベースのアーキテクチャでインフラ管理が簡素化
✓Private IP / Public IP を既存環境で切り替え可能

⚡ 主要機能

🚫 Turin リージョンでの Composer 2 新規作成の終了

2026年3月10日以降、europe-west12 (Turin) リージョンでは Cloud Composer 2 環境を新規作成できなくなります。既存の Cloud Composer 2 環境は引き続き動作します。

☁️ Cloud Composer 3 専用リージョンへの切り替え

Turin リージョンで新しい Composer 環境を作成する場合は Cloud Composer 3 を使用する必要があります。テナントプロジェクトベースのアーキテクチャにより、管理の簡素化と運用効率の向上が期待できます。

🔀 移行パスの提供

Cloud Composer 2 から Cloud Composer 3 へのサイドバイサイド移行がサポートされています。移行スクリプトまたはスナップショットを使用した2つの移行方法が利用可能です。

📊 技術仕様: Composer 2 vs Composer 3

項目 Cloud Composer 2 Cloud Composer 3
クラスター管理 ユーザープロジェクトに GKE Autopilot を展開 テナントプロジェクトで管理
Airflow Executor Celery Executor CeleryKubernetes Executor
ネットワーク設定 Private Service Connect 簡素化 (Public/Private IP 切り替え可)
DAG Processor Scheduler に統合 独立コンポーネントとして分離
Airflow バージョン Airflow 2 Airflow 2, Airflow 3 (Preview)
Service Agent 権限 追加の権限が必要 追加の権限が不要
データベース保持ポリシー 未対応 サポート済み
イメージバージョン形式 composer-2.b.c-airflow-x.y.z composer-3-airflow-x.y.z-build.t

ステップ 1: 互換性チェックの実行

Bash 
# Cloud Composer 2 環境のアップグレードチェックを実行
gcloud composer environments check-upgrade \
  --location europe-west12 \
  --environment COMPOSER_2_ENV_NAME \
  --image-version composer-3-airflow-2.10.2-build.latest

ステップ 2: 移行スクリプトのダウンロードと実行

Bash 
# 移行スクリプトをダウンロード
curl -O https://raw.githubusercontent.com/GoogleCloudPlatform/python-docs-samples/main/composer/tools/composer_migrate.py

# ドライランで移行パラメータを確認
python3 composer_migrate.py \
  --project PROJECT_ID \
  --location europe-west12 \
  --source_environment COMPOSER_2_ENV \
  --target_environment COMPOSER_3_ENV \
  --target_airflow_version 2.10.2 \
  --dry_run

# 移行を実行
python3 composer_migrate.py \
  --project PROJECT_ID \
  --location europe-west12 \
  --source_environment COMPOSER_2_ENV \
  --target_environment COMPOSER_3_ENV \
  --target_airflow_version 2.10.2

ステップ 3: 移行後の確認

Bash 
# 新しい Cloud Composer 3 環境の状態を確認
gcloud composer environments describe COMPOSER_3_ENV \
  --location europe-west12

# DAG の実行状態を確認
gcloud composer environments run COMPOSER_3_ENV \
  --location europe-west12 \
  dags list

✅ メリット

💼 ビジネス面

✓インフラ管理の簡素化: クラスターがテナントプロジェクトで管理されるため、GKE クラスターの直接管理が不要になり運用負荷が軽減
✓最新機能へのアクセス: Airflow 3 (Preview) サポート、データベース保持ポリシー、DAG Processor の独立化など

🔧 技術面

✓簡素化されたネットワーク設定: Private/Public IP の切り替えが既存環境で可能
✓CeleryKubernetes Executor: Celery と Kubernetes Executor の長所を組み合わせ、タスク実行効率が向上
✓追加権限の不要化: Service Agent に追加権限が不要で、セキュリティ設定がシンプルに

⚠️ デメリット・制約事項

制限事項

✗インプレースアップグレードは未サポート。サイドバイサイド移行のみ可能
✗移行スクリプトは同一リージョン・同一プロジェクトでのみ Composer 3 環境を作成可能
✗スナップショットの読み込みは同一バージョンまたはより新しい Airflow にのみ対応
✗KubernetesPodOperator でカスタムサービスアカウント使用時は Composer 3 で非サポート
✗Pod affinity の設定は Composer 3 では利用不可

考慮すべき点

→既存 Composer 2 環境は動作するが、新規作成不可のためスケールアウト計画に影響する可能性
→Composer 3 では一部のプリインストールパッケージのバージョンが異なるため、DAG の互換性を事前検証する必要あり
→Airflow データベースサイズが 20 GB を超える場合は、事前にクリーンアップが必要

🎯 ユースケース

📌 ユースケース 1: 既存 Composer 2 環境からの計画的移行

シナリオ: Turin リージョンで Cloud Composer 2 を運用しているチームが、新規環境の作成制限に対応するため、計画的に Cloud Composer 3 へ移行する。

Bash 
# 1. 互換性チェック
gcloud composer environments check-upgrade \
  --location europe-west12 \
  --environment my-composer2-env \
  --image-version composer-3-airflow-2.10.2-build.latest

# 2. 移行スクリプトで自動移行
python3 composer_migrate.py \
  --project my-project \
  --location europe-west12 \
  --source_environment my-composer2-env \
  --target_environment my-composer3-env \
  --target_airflow_version 2.10.2

効果: DAG、設定、環境変数が自動的に引き継がれ、最小限のダウンタイムで Cloud Composer 3 に移行できる。

📌 ユースケース 2: 新規プロジェクトでの Composer 3 環境構築

シナリオ: Turin リージョンで新しいデータパイプラインプロジェクトを開始する際に、最初から Cloud Composer 3 環境を構築する。

効果: 簡素化されたネットワーク設定とテナントプロジェクトベースのアーキテクチャにより、環境構築が迅速に完了し、インフラストラクチャの管理負荷が軽減される。
