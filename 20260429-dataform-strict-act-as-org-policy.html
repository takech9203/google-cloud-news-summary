🔒 Dataform: Strict Act-As モードのグローバル強制適用とカスタム組織ポリシー制約の GA

Dataform BigQuery Breaking Change GA
2026-04-29
⚠️ Breaking Change: Strict Act-As モードがすべてのリポジトリに対してグローバルに強制適用されました。デフォルトの Dataform サービスエージェントのみでのワークフロー実行は不可となります。カスタムサービスアカウントの設定が必須です。

📋 概要

Dataform において、Strict Act-As モードがすべてのリポジトリに対してグローバルに強制適用されるようになりました。ワークフロー、BigQuery パイプライン、ノートブック、データ準備の実行には、カスタムサービスアカウントまたはユーザー認証情報の使用が必須となります。

同時に、Organization Policy Service のカスタム制約が GAとなり、Dataform の Folder および TeamFolder リソースに対して、より細かな制御が可能になりました。

Before / After 比較

❌ Before (変更前)

  • Strict Act-As モードは任意設定 (手動有効化)
  • デフォルトの Dataform サービスエージェントでワークフロー実行可能
  • 誰がどの SA として実行しているか不明確
  • 組み込み制約に限定された組織ポリシー

✅ After (変更後)

  • Strict Act-As モードがグローバル強制
  • カスタム SA / ユーザー認証情報が必須
  • iam.serviceAccounts.actAs 権限チェック有効
  • CEL 式によるカスタム組織ポリシー制約が利用可能

⭐ 主要機能

🔒 1. Strict Act-As モードのグローバル強制

  • すべての Dataform リポジトリに対して自動的に有効化
  • リポジトリの作成/更新、ワークフロー構成の作成/更新、ワークフロー呼び出し作成、リリース構成更新時に iam.serviceAccounts.actAs 権限チェック実施
  • デフォルト Dataform サービスエージェント単体での実行不可

🔑 2. カスタムサービスアカウントの必須化

  • 新規リポジトリ: 作成時にカスタム SA を選択
  • ワークフロー構成: 構成ごとに SA を指定可能 (未指定時はリポジトリの SA を使用)
  • ワークフロー呼び出し: InvocationConfig で SA を指定可能

🏛️ 3. カスタム組織ポリシー制約 (GA)

  • Dataform の 6 種類のリソースに対してカスタム制約を作成可能
  • CEL (Common Expression Language) による柔軟な条件定義
  • CREATE / UPDATE メソッドに対して ALLOW/DENY アクションを設定

🛠️ 設定方法

💡 前提条件:
  • カスタムサービスアカウントが作成済み
  • SA に BigQuery 実行に必要な権限が付与済み
  • Dataform サービスエージェントにカスタム SA への Service Account User + Service Account Token Creator ロール付与済み

Step 1: カスタムサービスアカウントの作成

Bash SA 作成と BigQuery 権限付与 
# カスタムサービスアカウントを作成
gcloud iam service-accounts create dataform-workflow-sa \
  --display-name="Dataform Workflow Service Account" \
  --project=PROJECT_ID

# BigQuery 関連の権限を付与
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:dataform-workflow-sa@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/bigquery.dataEditor"

gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:dataform-workflow-sa@PROJECT_ID.iam.gserviceaccount.com" \
  --role="roles/bigquery.jobUser"

Step 2: Dataform サービスエージェントへの権限付与

Bash サービスエージェントに actAs 権限を付与 
# Dataform サービスエージェントにカスタム SA への actAs 権限を付与
gcloud iam service-accounts add-iam-policy-binding \
  dataform-workflow-sa@PROJECT_ID.iam.gserviceaccount.com \
  --member="serviceAccount:service-PROJECT_NUMBER@gcp-sa-dataform.iam.gserviceaccount.com" \
  --role="roles/iam.serviceAccountUser"

# Service Account Token Creator ロールも付与
gcloud iam service-accounts add-iam-policy-binding \
  dataform-workflow-sa@PROJECT_ID.iam.gserviceaccount.com \
  --member="serviceAccount:service-PROJECT_NUMBER@gcp-sa-dataform.iam.gserviceaccount.com" \
  --role="roles/iam.serviceAccountTokenCreator"

Step 3: ユーザーへの actAs 権限付与

Bash ワークフロー実行ユーザーへの権限付与 
# ワークフローを実行するユーザーに actAs 権限を付与
gcloud iam service-accounts add-iam-policy-binding \
  dataform-workflow-sa@PROJECT_ID.iam.gserviceaccount.com \
  --member="user:USER_EMAIL" \
  --role="roles/iam.serviceAccountUser"

Step 4: カスタム組織ポリシー制約の作成 (オプション)

YAML constraint-restrict-repository-location.yaml 
name: organizations/ORGANIZATION_ID/customConstraints/custom.restrictRepositoryLocation
resourceTypes:
  - dataform.googleapis.com/Repository
methodTypes:
  - CREATE
  - UPDATE
condition: "resource.name.contains('/locations/us-central1/')"
actionType: ALLOW
displayName: Only us-central1 region is allowed.
description: All resources must be created in the us-central1 region.
Bash 制約の適用 
# カスタム制約を適用
gcloud org-policies set-custom-constraint ~/constraint-restrict-repository-location.yaml

# ポリシーを設定
gcloud org-policies set-policy ~/policy-restrict-repository-location.yaml

⚖️ メリット・デメリット

✅ メリット

ビジネス面

  • 全リポジトリでの権限チェック強制で監査要件対応が容易
  • 組織全体で一貫したリソース管理が可能

技術面

  • 権限の最小化原則を適用しやすい
  • Cloud Logging で実行者の追跡が明確
  • CEL 式による柔軟なポリシー定義

⚠️ デメリット・制約事項

制限事項

  • デフォルト SA のみでの実行は即座に不可
  • カスタムポリシーは既存リソースに遡及適用されない
  • サードパーティリポジトリのコードレビューはユーザー責任

考慮点

  • カスタム SA 未設定のリポジトリはワークフロー失敗の可能性
  • CI/CD パイプラインの SA 設定更新が必要
  • 自動リリースの制限あり

💡 ユースケース

🎯 ユースケース 1: 本番環境のワークフロー実行権限の分離

シナリオ: 開発チームと本番運用チームで異なる SA を使用し、本番 BigQuery データセットへのアクセスを制御

YAML 環境別 SA 設定例 
# 開発用サービスアカウント (限定的な権限)
# dev-dataform-sa@project.iam.gserviceaccount.com
# -> roles/bigquery.dataEditor on dev dataset only

# 本番用サービスアカウント (本番データセットへのアクセス)
# prod-dataform-sa@project.iam.gserviceaccount.com
# -> roles/bigquery.dataEditor on prod dataset

効果: 開発者が誤って本番データを変更するリスクを排除し、権限の最小化原則を徹底

🌏 ユースケース 2: 組織ポリシーによるリージョン制限

シナリオ: データレジデンシー要件により、すべての Dataform リポジトリを特定リージョンに限定

YAML Tokyo リージョン制限の制約定義 
name: organizations/123456789/customConstraints/custom.restrictToJapanRegion
resourceTypes:
  - dataform.googleapis.com/Repository
methodTypes:
  - CREATE
  - UPDATE
condition: "resource.name.contains('/locations/asia-northeast1/')"
actionType: ALLOW
displayName: Only asia-northeast1 (Tokyo) region is allowed.
description: All Dataform repositories must be created in Tokyo region.

効果: 日本国内のデータレジデンシー要件を組織レベルで強制し、設定漏れを防止

💰 料金

Dataform 自体の使用は無料。ワークフロー実行時の BigQuery ジョブには通常の BigQuery 料金が適用されます。Organization Policy Service のカスタム制約の使用にも追加料金は発生しません。
