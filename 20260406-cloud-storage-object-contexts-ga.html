Cloud Storage: Object Contexts

Cloud Storage GA
2026-04-06

Overview

Cloud Storage の Object Contexts 機能が一般提供 (GA) になりました。オブジェクトにカスタムのキーバリューペアを付与し、データの分類、追跡、検索を効率化する機能です。

さらに、Storage Batch Operations を使用して複数オブジェクトのコンテキストを一括で更新できるようになりました。全削除、特定キーの削除、新しいキーバリューペアの追加・更新を単一のバッチジョブで実行できます。

Before / After

Before

  • カスタムメタデータに限られ、構造化コンテキスト管理が困難
  • メタデータベースのフィルタリングが容易ではなくデータ検索に時間がかかる
  • 一括更新には独自スクリプトやワークフローの構築が必要
  • コピー・リライト時のメタデータ引き継ぎを手動管理

After

  • 構造化キーバリューペア + IAM によるアクセス制御
  • コンテキストベースのフィルタリングで効率的にオブジェクトを取得
  • Batch Operations で数百万件のコンテキスト一括更新をサーバーレスで実行
  • コピー・リライト・コンポーズ・移動・リストア時にコンテキストが自動引き継ぎ

Key Features

Object Contexts (GA)

  • オブジェクトにカスタムのキーバリューペアを付与し、データの分類・追跡・検索を実現
  • オブジェクト作成時に付与、または既存オブジェクトに後から追加が可能
  • コピー、リライト、コンポーズ、移動、リストア操作時にコンテキストがデフォルトで保持
  • dropContextGroups JSON API パラメータでコンテキストの引き継ぎ動作を制御可能

コンテキストベースのオブジェクトフィルタリング

  • 特定のコンテキストキーや値でオブジェクトをフィルタリング
  • contexts."KEY":* 構文で特定キーを持つオブジェクトを検索
  • contexts."KEY"="VALUE" 構文でキーと値の組み合わせで検索
  • NOT 演算子による否定フィルタリングにも対応

Storage Batch Operations によるコンテキスト一括更新

  • 全コンテキストのクリア (--clear-all-object-custom-contexts)
  • 特定キーのコンテキスト削除 (--clear-object-custom-contexts)
  • 新しいコンテキストの挿入・更新 (--update-object-custom-contexts)
  • マニフェストファイルまたはプレフィックス指定による対象オブジェクト選択
  • ドライラン機能による事前確認が可能

Technical Specs

IAM Permissions

Permission Description Roles
storage.objects.createContext オブジェクトにコンテキストを付与 Storage Object Admin, User, Creator, Legacy Bucket Writer
storage.objects.updateContext オブジェクトのコンテキストを更新 Storage Object Admin, User, Legacy Object Owner
storage.objects.deleteContext オブジェクトのコンテキストを削除 Storage Object Admin, User, Legacy Object Owner

Context Data Structure

JSON Object Contexts data format 
{
  "contexts": {
    "custom": {
      "customer_id": {
        "value": "cust-78901"
      },
      "payment_status": {
        "value": "unpaid"
      }
    }
  }
}
Note: コンテキストの値は現在、文字列形式のみをサポートしています。

Filtering Syntax

Syntax Description
contexts."KEY":* 指定キーを持つオブジェクトに一致
contexts."KEY"="VALUE" 指定キーと値の組み合わせに一致
NOT contexts."KEY":* 指定キーを持たないオブジェクトに一致
NOT contexts."KEY"="VALUE" 指定キーと値の組み合わせを持たないオブジェクトに一致

Setup Guide

Step 1: Upload with Contexts

Bash Object upload with custom contexts 
gcloud storage cp ./invoice.pdf gs://my-bucket/ \
  --custom-contexts=customer_id=cust-78901,payment_status=unpaid

Step 2: Update Contexts (JSON API)

Bash PATCH request to update contexts 
# Create context JSON
cat > context.json << 'EOF'
{
  "contexts": {
    "custom": {
      "payment_status": {
        "value": "paid"
      }
    }
  }
}
EOF

# Update via PATCH
curl -X PATCH --data-binary @context.json \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json" \
  "https://storage.googleapis.com/storage/v1/b/my-bucket/o/invoice.pdf"

Step 3: Filter by Contexts

Bash List objects filtered by context 
gcloud storage objects list gs://my-bucket/ \
  --metadata-filter='contexts."payment_status"="unpaid"'

Step 4: Batch Operations

Bash Batch update contexts for multiple objects 
gcloud storage batch-operations jobs create my-context-update-job \
  --bucket=my-bucket \
  --included-object-prefixes=invoices/ \
  --clear-object-custom-contexts=old_status \
  --update-object-custom-contexts=review_status=completed,reviewed_by=team-a

Merits / Considerations

Merits

  • Data Governance: IAM 制御されたコンテキスト操作で、データ分類やコンプライアンスラベルの信頼性の高い管理
  • Cost Reduction: Batch Operations による一括更新で開発・運用工数を大幅に削減
  • Search Efficiency: コンテキストベースのフィルタリングで必要なデータの特定が迅速に
  • Data Persistence: コピー・リライト・コンポーズ時にコンテキスト自動保持
  • Multi-cloud: 文字列ベースのキーバリュー形式で、他クラウドのオブジェクトタグとの互換性を確保
  • Fine-grained Access: 作成・更新・削除に個別の IAM パーミッションで最小権限の原則を実現

Considerations

  • コンテキストの値は文字列形式のみサポート (数値やブーリアン型は直接扱えない)
  • GA 移行後はオブジェクトコンテキストに対して追加のストレージ料金が発生する可能性
  • 大量のコンテキスト付与時はストレージコストへの影響を事前に見積もる必要あり
  • 既存のカスタムメタデータワークフローがある場合、移行計画の検討が必要
  • Batch Operations は非同期処理のため、即座にコンテキストが反映されない場合あり

Use Cases

Use Case 1: Compliance Data Classification

Scenario: 医療機関が患者データを Cloud Storage に保存し、HIPAA 準拠のためにデータの機密レベルを管理する必要がある。

Bash Classify and filter PHI data 
# Upload with classification
gcloud storage cp ./patient-record.pdf gs://medical-data/ \
  --custom-contexts=data_classification=PHI,retention_policy=7years

# List PHI data only
gcloud storage objects list gs://medical-data/ \
  --metadata-filter='contexts."data_classification"="PHI"'

Effect: データの機密レベルがオブジェクトに直接紐づき、監査時の対象データ特定が迅速になり、コンプライアンス対応の工数を削減。

Use Case 2: ML Pipeline Data Management

Scenario: ML チームが大量のトレーニングデータを管理し、バッチごとの追跡と処理状態の管理が必要。

Bash Label and track ML training data 
# Batch label training data
gcloud storage batch-operations jobs create label-training-data \
  --bucket=ml-training-data \
  --included-object-prefixes=images/2026-q1/ \
  --update-object-custom-contexts=batch_id=2026_Q1_Run,processing_status=pending

# Check processed data
gcloud storage objects list gs://ml-training-data/ \
  --metadata-filter='contexts."processing_status"="completed"'

Effect: トレーニングデータの処理状態をオブジェクトレベルで追跡でき、パイプラインの再処理防止やデータリネージの確保が容易に。

Use Case 3: Multi-cloud Migration Tag Unification

Scenario: 他のクラウドプロバイダーから Google Cloud へのデータ移行において、既存のオブジェクトタグ体系を維持したい。

Effect: Object Contexts は文字列ベースのキーバリュー形式を採用しているため、AWS S3 のオブジェクトタグなどとの互換性が高く、移行時のメタデータ変換コストを最小限に抑えられる。
