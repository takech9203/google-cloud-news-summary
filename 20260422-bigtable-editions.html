📋 概要

Bigtable Editions が一般提供 (GA) として正式リリースされました。ワークロードの要件に応じて Enterprise と Enterprise Plus の 2 つのエディションを選択でき、パフォーマンス、分析クエリ機能、リソース管理における最適な機能セットとコストバランスを実現します。

2026 年 4 月以前に作成されたすべてのインスタンスは、既存の機能や課金に変更なく自動的に Enterprise エディションに分類されます。

🔄 Before / After

❌ Before (従来)

  • 全インスタンスが単一の機能・価格体系で、ワークロード特性に応じた最適化が不可能
  • サブミリ秒レイテンシにはセルフマネージドキャッシュ (Redis 等) が必要
  • ノードあたりストレージ上限が 32 TB に制限
  • Data Boost で SQL クエリや階層型ストレージ/HDD アクセスが不可

✅ After (Editions)

  • Enterprise / Enterprise Plus からワークロードに最適なエディションを選択可能
  • インメモリティアで最大 120K reads/s、サブミリ秒レイテンシを実現
  • Enterprise Plus で階層型ストレージが 64 TB/node に拡張
  • Enterprise Plus の Data Boost で GoogleSQL クエリ実行が GA

🏗️ アーキテクチャ図

Enterprise エディションは標準的な Bigtable の機能セットを提供し、Enterprise Plus は追加の高度な機能を提供します。エディション間のアップグレード・ダウングレードはいつでも可能ですが、Enterprise Plus からのダウングレードには 48 時間の制限があります。

⚡ 主要機能

🔀 エディション選択モデル

インスタンス作成時または更新時にエディションを選択。Enterprise から Enterprise Plus へのアップグレードは即時反映。ダウングレードは Enterprise Plus 専用機能の無効化後に実行可能 (48 時間制限あり)。

💾 インメモリティア Preview

ハイブリッドストレージノードにより SSD + メモリ層を提供。ノードあたり 8 GB RAM、ベース 40K reads/s から垂直スケーリングで最大 120K reads/s。RDMA でレスポンスタイムを大幅短縮。

📦 拡張階層型ストレージ Preview

ノードあたりの階層型ストレージ上限が Enterprise の 32 TB から Enterprise Plus の 64 TB に拡張。低頻度アクセスデータをより多く保持し、ストレージコストを最適化。

📊 高度な Data Boost GA

GoogleSQL クエリのサポートにより SQL ベースの分析クエリを実行可能。階層型ストレージ / HDD クラスタからのデータ読み取りに対応。本番ワークロードに影響なし。

🔒 自動バックアップクラスタ設定 GA

Enterprise Plus ではレプリケーションされたインスタンスで、どのクラスタで自動バックアップを実行するかを指定可能。バックアップリソースの管理とコスト制御が向上します。

🔧 設定方法

Bigtable API が有効化済みで、Bigtable 管理者ロールが付与されていることが前提です。

Step 1: 新規インスタンスでエディションを選択

Bash Enterprise Plus インスタンス作成 
gcloud bigtable instances create INSTANCE_ID \
  --display-name="My Enterprise Plus Instance" \
  --edition=ENTERPRISE_PLUS \
  --cluster-config=id=CLUSTER_ID,zone=ZONE,nodes=3

Step 2: 既存インスタンスをアップグレード

Bash Enterprise Plus へアップグレード (即時反映) 
gcloud bigtable instances update INSTANCE_ID \
  --edition=ENTERPRISE_PLUS

Step 3: インメモリティアを有効化 (Enterprise Plus のみ)

Google Cloud コンソールでインスタンスの編集画面からインメモリティアを有効にします。

Step 4: ダウングレードする場合

Enterprise Plus 専用機能 (インメモリティアなど) を無効化後に実行。アップグレードから 48 時間以内はダウングレード不可。

Bash Enterprise へダウングレード 
gcloud bigtable instances update INSTANCE_ID \
  --edition=ENTERPRISE

✅ メリット

💰 ビジネス面

  • 最適なコスト構造: 要件に応じた機能と価格のバランスを選択し、過剰コストを抑制
  • キャッシュ層の削減: インメモリティアにより Redis/Memcached の構築・運用が不要に
  • 確約利用割引: 1 年 20%、3 年 40% の CUD で長期コスト最適化

🔧 技術面

  • サブミリ秒レイテンシ: インメモリティアで 1 ms 未満の読み取り、キャッシュ層不要
  • 自動垂直スケーリング: トラフィック急増に自動対応、水平スケーリングとも連携
  • 統合ストレージ: RAM / SSD / 階層型の 3 層を単一 API で管理
  • 高度な分析: Data Boost + GoogleSQL で本番影響なしの分析実行

⚠️ デメリット・制約事項

⚠️ 制限事項
  • インメモリティアは現在 Preview であり、本番環境では Pre-GA 利用規約が適用
  • インメモリティアは単一行ポイント読み取りのみをサポート (スキャン・SQL クエリ非対応)
  • インメモリティアのアプリプロファイルは単一クラスタルーティングのみ対応
  • CMEK クラスタではインメモリティアを利用不可
  • アップグレード後 48 時間はダウングレード不可
  • Enterprise Plus は無料トライアル版では利用不可
💡 考慮すべき点
  • Enterprise Plus は独自のノード SKU を持ち、Enterprise とは異なるノード単価が適用
  • 垂直スケーリング (ベースの 40K reads/s 超過分) には追加の課金乗数が適用
  • インメモリティアのプロビジョニングには最大 30 分かかる場合がある
  • マルチクラスタ構成では各クラスタのメモリティアは独立動作 (最終一貫性)

💡 ユースケース

🎯 リアルタイム広告配信プラットフォーム

ユーザープロファイルや入札データにサブミリ秒でアクセスが必要。従来は Redis クラスタをキャッシュ層として運用し、整合性管理やインフラ運用コストが課題。

→ Enterprise Plus のインメモリティアで Bigtable 単体のサブミリ秒読み取りを実現。Redis 不要でアーキテクチャ簡素化と運用コスト削減。

🌡️ IoT 時系列データの長期保持と分析

IoT デバイスからの大量時系列データを Bigtable に格納。低頻度アクセスの過去データも SSD に保持しストレージコスト増大。過去データへの分析も本番影響なく実行したい。

→ 拡張階層型ストレージ (64 TB/node) で低頻度データをコスト効率よく保持。Data Boost + SQL で本番影響なしの分析クエリ実行。

🏢 マルチテナント SaaS プラットフォーム

複数テナントが共有する SaaS バックエンドとして Bigtable を使用。特定テナントのトラフィック急増によるホットスポットが他テナントの性能に影響。

→ インメモリティアのホットスポット緩和と垂直スケーリングでトラフィック集中を自動吸収。テナント間パフォーマンス分離を改善。

💳 料金

選択したエディションに応じたノード時間、ストレージ、ネットワーク転送で課金。課金モデル自体 (ノード時間、ストレージ、ネットワーク) は従来と同じです。

Enterprise

  • 標準 Bigtable 機能セット
  • ノード時間 + ストレージ + ネットワーク課金
  • 1 年 CUD: 20% 割引
  • 3 年 CUD: 40% 割引

Enterprise Plus

  • 独自ノード SKU (Enterprise と異なる単価)
  • インメモリ垂直スケーリング超過分に課金乗数適用
  • 1 年 CUD: 20% 割引
  • 3 年 CUD: 40% 割引

詳細な料金は Bigtable 料金ページ を参照してください。
