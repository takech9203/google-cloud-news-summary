🚀 主要機能

Rapid Rapid チャネルのバージョン更新

デフォルトバージョンが 1.35.1-gke.1396001 に更新。Kubernetes 1.35 系の最新パッチが含まれ、最新機能の早期検証が可能です。

→新規利用可能: 1.32.12, 1.33.8, 1.34.4, 1.35.1 (2 ビルド)

Regular Regular チャネルのバージョン更新

デフォルトバージョンが 1.34.3-gke.1444000 に更新。K8s 1.35 系が Regular チャネルでも利用可能になり、手動アップグレードによる早期採用が可能です。

→新規利用可能: 1.32.12, 1.33.8, 1.34.4, 1.35.0 (2 ビルド), 1.35.1 (2 ビルド)

Extended Extended チャネルのバージョン更新

デフォルトバージョンが 1.34.3-gke.1444000 に更新。1.30.14 から 1.35.0 までの幅広いバージョンが利用可能で、長期サポート環境向けに旧マイナーバージョンも継続提供されます。

→1.30 系から 1.35 系まで対応 (最大 24 か月サポート)

Stable Stable チャネルのバージョン更新

デフォルトバージョンが 1.33.5-gke.2326000 に更新。Rapid および Regular チャネルで十分に検証されたバージョンのみが提供されます。

→新規利用可能: 1.32.11, 1.33.5, 1.34.3

⚡ 自動アップグレードターゲットの更新

すべてのチャネルで自動アップグレードターゲットが更新されました。マイナーバージョンおよびパッチバージョンの自動アップグレードが順次展開されます。

⚙️ 設定方法

💡 前提条件: Google Cloud プロジェクトで GKE API が有効化済み、gcloud CLI の認証が完了済み、Kubernetes Engine Cluster Admin 以上の IAM 権限が必要です。

ステップ 1: 現在のクラスタバージョンとチャネルを確認

Bash クラスタ情報の確認 
gcloud container clusters list \
    --project PROJECT_ID \
    --format="table(name,location,currentMasterVersion,releaseChannel.channel)"

ステップ 2: 利用可能なバージョンを確認

Bash チャネルの利用可能バージョン一覧 
# 特定チャネルの利用可能バージョンを確認
gcloud container get-server-config \
    --location LOCATION \
    --format="yaml(channels)"

ステップ 3: リリースチャネルを変更する場合

Bash チャネル変更 
gcloud container clusters update CLUSTER_NAME \
    --location LOCATION \
    --release-channel CHANNEL
# CHANNEL: rapid, regular, extended, stable

ステップ 4: 手動でバージョンをアップグレードする場合

Bash 手動アップグレード 
# コントロールプレーンのアップグレード
gcloud container clusters upgrade CLUSTER_NAME \
    --location LOCATION \
    --master \
    --cluster-version VERSION

# ノードプールのアップグレード
gcloud container clusters upgrade CLUSTER_NAME \
    --location LOCATION \
    --node-pool NODE_POOL_NAME \
    --cluster-version VERSION

✅ メリット

ビジネス面

✓運用負荷の軽減: 自動アップグレードにより手動バージョン管理が不要
✓セキュリティリスクの低減: 最新パッチが自動適用され脆弱性リスクを迅速に低減
✓段階的なバージョン採用: チャネルごとの段階提供で本番への影響を最小化

技術面

✓最新 K8s 機能へのアクセス: Rapid で K8s 1.35.1 をいち早く検証可能
✓幅広いバージョン選択肢: Extended で 1.30~1.35 系まで柔軟に選択
✓パッチの早期利用: 手動アップグレードでデフォルト前に先行適用可能

⚠️ デメリット・制約事項

制限事項
✕Rapid チャネルは GKE SLA 対象外 -- 本番環境での使用は非推奨
✕メンテナンスウィンドウ未設定時、任意のタイミングで自動アップグレードが実行される可能性
✕マイナーバージョンアップグレードで K8s API の非推奨化・削除が含まれる場合あり
💡 考慮すべき点
→自動アップグレード前にステージング環境で互換性テストを実施
→メンテナンスウィンドウと除外を適切に設定し、ビジネスクリティカル時間帯を回避
→Extended チャネル利用時もサポート終了日までにアップグレード計画を策定
→複数マイナーバージョンをスキップする手動アップグレードは段階的に実施

💡 ユースケース

🔬 ユースケース 1: 段階的なバージョン検証パイプライン

シナリオ: 開発チームが Kubernetes 1.35 の新機能を検証したい場合、Rapid チャネルの開発クラスタで先行検証し、問題がなければ Regular チャネルの本番クラスタに順次展開します。

Bash 段階的アップグレード例 
# 検証クラスタ (Rapid) を 1.35.1 にアップグレード
gcloud container clusters upgrade dev-cluster \
    --location us-central1 \
    --master \
    --cluster-version 1.35.1-gke.1616000

# 検証完了後、本番クラスタ (Regular) を手動アップグレード
gcloud container clusters upgrade prod-cluster \
    --location us-central1 \
    --master \
    --cluster-version 1.35.1-gke.1396001

効果: 新バージョンの機能やパフォーマンスを検証環境で事前に評価でき、本番環境への影響を最小化しながら最新バージョンを採用できます。

🏦 ユースケース 2: Extended チャネルによる長期バージョン保持

シナリオ: 金融機関の規制要件により、K8s マイナーバージョンのアップグレードには社内監査プロセスが必要で、頻繁なバージョン変更が困難な場合。Extended チャネルで 1.34 系を長期間維持しつつ、パッチアップデートは自動的に適用します。

効果: Extended チャネルの最大 24 か月の長期サポートを活用し、監査プロセスに十分な時間を確保しながら、セキュリティパッチは自動適用でクラスタの安全性を維持できます。

🔗 関連サービス・機能

☸️ GKE リリースチャネル

クラスタのバージョン管理ポリシーを定義。Rapid、Regular、Extended、Stable の 4 チャネルから選択可能。

🕐 GKE メンテナンスウィンドウ

自動アップグレードの実行時間帯を制御。ビジネスクリティカルな時間帯を回避してスケジュール可能。

🔔 GKE クラスタ通知

バージョン更新やアップグレードイベントの通知を受信。Pub/Sub を通じた事前通知に対応。

⚡ GKE Surge Upgrade

ノードプールのアップグレード時に一時的に追加ノードを作成し、可用性を維持しながらローリングアップグレードを実行。
