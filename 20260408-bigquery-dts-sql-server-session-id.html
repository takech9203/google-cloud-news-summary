🗄️ BigQuery Data Transfer Service SQL Server 増分転送 & @@session_id 拡張

Preview: SQL Server 増分転送 GA: @@session_id 拡張 BigQuery Data Transfer Service

📋 概要

BigQuery に関する 2 つのアップデートが発表されました。

1. SQL Server 増分データ転送 (Preview)

BigQuery Data Transfer Service の SQL Server コネクタで増分データ転送が利用可能に。前回転送以降に変更されたデータのみを効率的に転送できます。MySQL, Oracle, PostgreSQL, ServiceNow に続く対応です。

2. @@session_id の UDF / ビュー対応 (GA)

@@session_id システム変数がSQL UDF、テーブル関数、論理ビュー内で利用可能に。セッションコンテキストに基づくロジック実装が容易になりました。

🔄 Before / After 比較

❌ Before (アップデート前)

  • SQL Server からの転送はフルデータ転送のみ。毎回テーブル全体を転送
  • 大規模 DB では転送時間が長く、ネットワーク帯域やリソース消費が大きい
  • @@session_id は UDF / テーブル関数 / 論理ビュー内で参照不可

✅ After (アップデート後)

  • 増分転送で変更データのみを効率的に BigQuery へ転送
  • 主要 RDBMS (MySQL, Oracle, PostgreSQL, SQL Server) 全てで増分転送をサポート
  • @@session_id が UDF / テーブル関数 / 論理ビュー内で利用可能に

⚡ 主要機能

1. SQL Server 増分転送 (Incremental Transfer)

  • -> 前回の転送成功時刻以降に変更されたデータのみを転送
  • -> ウォーターマークカラム (TIMESTAMP 型) で変更を検出
  • -> 初回はフル転送、2 回目以降から増分転送を実施
  • -> オンプレミス / Cloud SQL / AWS / Azure の SQL Server をサポート

2. Append 書き込みモード

新レコードを宛先テーブルに追記。ウォーターマークカラムの指定が必須。既存レコードの重複チェックは行いません。

3. Upsert 書き込みモード

主キーに基づいてレコードの更新/新規挿入を実施。ウォーターマークカラムと主キーの両方の指定が必須。ソースとの整合性を保持。

4. @@session_id の SQL UDF / テーブル関数 / 論理ビュー対応

  • -> SQL ユーザー定義関数 (UDF) 内で @@session_id を参照可能
  • -> テーブル関数内で @@session_id を参照可能
  • -> 論理ビュー内で @@session_id を参照可能
  • -> セッション ID を STRING 型で返却

⚙️ 設定方法

前提条件

  • ✅ BigQuery Data Transfer Service が有効化されていること
  • ✅ roles/bigquery.admin IAM ロールが付与されていること
  • ✅ 転送先の BigQuery データセットが作成済みであること
  • ✅ SQL Server へのネットワーク構成 (ネットワークアタッチメント等) が完了していること

Step 1: SQL Server 増分転送を bq コマンドで作成

Bash bq mk で増分転送構成を作成 
bq mk --transfer_config \
  --target_dataset=mydataset \
  --data_source=sqlserver \
  --display_name='SQL Server Incremental Transfer' \
  --params='{
    "assets": ["mydb/dbo/orders"],
    "connector.authentication.username": "transfer_user",
    "connector.authentication.password": "your_password",
    "connector.database": "mydb",
    "connector.endpoint.host": "192.168.1.100",
    "connector.endpoint.port": 1433,
    "ingestionType": "incremental",
    "writeMode": "WRITE_MODE_UPSERT",
    "watermarkColumns": ["updated_at"],
    "primaryKeys": [["order_id"]],
    "connector.tls.mode": "ENCRYPT_VERIFY_CA_AND_HOST",
    "connector.tls.trustedServerCertificate": "PEM-encoded certificate"
  }'

ソースタイプとして sqlserver を指定し、ingestionType を incremental に設定します。Upsert モードの場合は writeMode と primaryKeys の指定も必要です。

Step 2: @@session_id を SQL UDF 内で使用する例

SQL セッション ID に基づくログ記録用 UDF 
-- セッション ID に基づくログ記録用の UDF
CREATE TEMP FUNCTION GetSessionContext()
RETURNS STRING
AS (
  @@session_id
);

-- テーブル関数内での使用例
SELECT GetSessionContext() AS current_session;

セッション内でこの UDF を呼び出すことで、現在のセッション ID を取得し、監査ログやデバッグに活用できます。

💡 メリット

💼 ビジネス面

  • ✅ SQL Server ユーザーのデータ統合コスト削減 -- フルデータ転送と比較して転送データ量が大幅に削減
  • ✅ Microsoft エコシステムからの移行促進 -- BigQuery へのデータ統合の障壁が低下
  • ✅ セッション管理の強化 -- マルチテナント環境でのセッション管理やアクセス制御が容易に

🔧 技術面

  • ✅ ネットワーク負荷の軽減 -- 変更データのみ転送でインフラ負荷を低減
  • ✅ データベースコネクタの統一 -- 主要 RDBMS 全てで増分転送サポート
  • ✅ UDF の表現力向上 -- セッションコンテキストに依存する複雑なロジックを SQL で表現可能

⚠️ デメリット・制約事項

制限事項

  • ! SQL Server 増分転送は Preview 段階 -- SLA 適用対象外
  • ! 同一 DB への同時転送ジョブ数に制限
  • ! ソーステーブルの DELETE 操作は増分転送で同期されない
  • ! 初回実行後、アセットリスト / 書き込みモード / ウォーターマークカラム / 主キーの変更不可
  • ! 一部の SQL Server データ型は STRING にマッピングされる場合がある

考慮すべき点

  • 💬 Preview のフィードバックは dts-preview-support@google.com に連絡
  • 💬 ウォーターマークカラムの値は単調増加である必要がある
  • 💬 ウォーターマークカラムにインデックス作成でパフォーマンス向上
  • 💬 2027/3/16 にデータ型マッピングの一部更新予定 (datetime2 等が TIMESTAMP から DATETIME に変更)

🎯 ユースケース

1. 基幹業務システム (SQL Server) のデータを BigQuery で分析

シナリオ: ERP システムが SQL Server 上で稼働。数億行規模のトランザクションテーブルを毎回フル転送するのは非現実的。

JSON 増分転送構成例 
{
  "assets": ["erp_db/dbo/sales_transactions"],
  "ingestionType": "incremental",
  "writeMode": "WRITE_MODE_UPSERT",
  "watermarkColumns": ["modified_date"],
  "primaryKeys": [["transaction_id"]]
}

効果: 増分転送で転送時間・コストを大幅削減。Upsert モードでトランザクションのステータス更新も BigQuery に反映。

2. Azure SQL Database から BigQuery へのマルチクラウドデータ統合

シナリオ: Azure 上の SQL Server (Azure SQL Database) で運用しているアプリケーションデータを BigQuery に集約して統合分析。

効果: SQL Server コネクタは Azure でホストされた SQL Server もサポート。増分転送によりクラウド間のデータ転送コストも最小化。

3. @@session_id を活用したマルチテナント UDF

シナリオ: BigQuery セッションを使用してマルチテナント分析基盤を運用。テナントごとのアクセスログ記録のため UDF 内でセッション情報を取得したい。

SQL 監査ログ用 UDF 
CREATE FUNCTION audit_dataset.log_query_access(table_name STRING)
RETURNS STRING
AS (
  CONCAT('session:', @@session_id, ' accessed:', table_name)
);

効果: クエリの監査やデバッグが効率化。テナントごとのセッションに基づいたデータフィルタリングもビューで実装可能。

📝 まとめ

今回のアップデートにより、BigQuery Data Transfer Service の SQL Server コネクタで増分データ転送が Preview として利用可能になり、MySQL, Oracle, PostgreSQL, ServiceNow に続いて主要 RDBMS での増分転送が揃いました。SQL Server は企業の基幹業務システムで広く利用されているため、効率的なデータ統合を求める多くの企業にとって有益な機能です。
