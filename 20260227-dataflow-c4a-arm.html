⚡ Dataflow: C4A マシンシリーズ (Arm) サポート GA

Dataflow Compute Engine GA
2026-02-27

📋 概要

Dataflow における C4A マシンシリーズ (Arm プロセッサ) のサポートが一般提供 (GA) となった。C4A は Google 独自の Axion プロセッサ (Arm Neoverse V2 ベース) を搭載した初の Arm ベース VM シリーズであり、電力効率に最適化された Arm アーキテクチャにより、多くのワークロードでコストパフォーマンスの向上が期待できる。

今回の GA により、Dataflow ユーザーはバッチジョブとストリーミングジョブの両方で C4A マシンタイプをワーカー VM として本番環境で利用できるようになった。

💡 ポイント: Dataflow の料金はマシンタイプファミリーに依存しないため、Arm ベースの C4A を使用することで、同等の x86 ベースマシンと比較してコスト削減を達成できる可能性がある。

🔄 Before / After 比較

❌ アップデート前

  • Arm VM は Tau T2A (Ampere Altra) のみ利用可能
  • 最大 48 vCPU / 192 GB メモリに制限
  • Google Axion プロセッサを Dataflow で活用不可
  • DDR5 メモリ / Titanium ネットワーキング非対応

✅ アップデート後

  • C4A マシンシリーズが GA として利用可能
  • 最大 72 vCPU / 576 GB DDR5 メモリ
  • Google Axion プロセッサの電力効率を活用
  • standard / highcpu / highmem から最適構成を選択
  • Titanium による最大 100 Gbps ネットワーク

🔑 主要機能

💻 C4A マシンタイプ対応

→standard (4 GB/vCPU)、highcpu (2 GB/vCPU)、highmem (8 GB/vCPU) を選択可能
→1 vCPU から 72 vCPU まで幅広いサイズ構成
→バッチジョブとストリーミングジョブの両方に対応

⚡ Google Axion プロセッサ

→Arm Neoverse V2 ベースの Google Axion CPU
→各 vCPU がシングルコアにマッピング (SMT なし) で一貫した性能
→DDR5 メモリによる高帯域幅メモリアクセス

📦 マルチアーキテクチャコンテナ

→Docker Buildx / Cloud Build でマルチアーキテクチャイメージをビルド
→linux/amd64 と linux/arm64 の両方に対応
→Apache Beam SDK 公式イメージはマルチアーキテクチャ対応済み

🛠 設定方法

📋 前提条件:
  • Apache Beam SDK バージョン 2.50.0 以降
  • C4A マシンが利用可能なリージョンを使用
  • カスタムコンテナ使用時はマルチアーキテクチャイメージをビルド済み

Step 1: C4A マシンタイプを指定

Java 
--workerMachineType=c4a-standard-4
Python 
--machine_type=c4a-standard-4
Go 
--worker_machine_type=c4a-standard-4

Step 2: Runner v2 を有効化 (必須)

Bash 
--experiments=use_runner_v2

Step 3: ストリーミングジョブの場合

Java (2.11.0+) 
--enableStreamingEngine

Python SDK 2.21.0+ / Go SDK 2.33.0+ ではデフォルトで有効

Step 4: マルチアーキテクチャコンテナのビルド (カスタムコンテナ使用時)

Bash 
# Docker Buildx を使用した例
docker buildx create --driver=docker-container --use

docker buildx build \
  --platform=linux/amd64,linux/arm64 \
  -t REGISTRY/IMAGE:TAG \
  --push .

実行例: バッチジョブ (Python)

Bash 
python my_pipeline.py \
  --runner=DataflowRunner \
  --project=my-project \
  --region=us-west1 \
  --machine_type=c4a-standard-4 \
  --experiments=use_runner_v2 \
  --temp_location=gs://my-bucket/temp

実行例: ストリーミングジョブ (Java)

Bash 
mvn compile exec:java \
  -Dexec.mainClass=com.example.MyPipeline \
  -Dexec.args=" \
    --runner=DataflowRunner \
    --project=my-project \
    --region=us-west1 \
    --workerMachineType=c4a-standard-4 \
    --experiments=use_runner_v2 \
    --enableStreamingEngine"

⚖ メリット・デメリット

🟢 ビジネス面のメリット

✓コスト削減: Arm アーキテクチャの電力効率により、同等の x86 VM と比較してコストパフォーマンスが向上。料金はマシンタイプファミリーに依存しないため、効率性がそのままコスト削減に直結
✓スケーラビリティの向上: 最大 72 vCPU / 576 GB メモリの構成により、T2A (最大 48 vCPU) と比較して大規模ワークロードに対応可能

🟢 技術面のメリット

✓パフォーマンスの一貫性: 各 vCPU がシングルコアにマッピング (SMT なし) され、コンピュート集約型ワークロードで安定した性能
✓高帯域幅ネットワーク: Titanium ベースの gVNIC により最大 100 Gbps (Tier_1) を実現
✓DDR5 メモリ: 高い帯域幅と低レイテンシでメモリ集約型のデータ処理に有利

🔴 制約事項・考慮点

✗GPU が必要なワークロード (ML 推論の一部など) では Arm VM は使用不可
✗Dataflow Prime と Arm VM の併用は非対応 (Vertical Autoscaling 等は利用不可)
✗Cloud Profiler によるパイプラインプロファイリングが利用不可
✗Right fitting (リソースの自動最適化) は非対応
⚠ その他の考慮点:
  • カスタムコンテナ使用時はマルチアーキテクチャイメージへの移行が必要 (既存 x86 専用コンテナはそのまま動作しない)
  • C4A が利用可能なリージョン・ゾーンに制限あり
  • Runner v2 が必須のため、Runner v1 に依存する既存パイプラインは対応が必要
  • ネイティブライブラリやバイナリを使用する場合、Arm 互換バージョンが必要

🎯 ユースケース

📚 UC1: 大規模 ETL バッチ処理のコスト最適化

シナリオ: 毎日数 TB のデータを BigQuery に投入する ETL パイプラインを運用しており、コンピュートコストの削減が課題

Bash 
python etl_pipeline.py \
  --runner=DataflowRunner \
  --project=my-project \
  --region=us-west1 \
  --machine_type=c4a-standard-8 \
  --experiments=use_runner_v2 \
  --disk_size_gb=50 \
  --temp_location=gs://my-bucket/temp

効果: C4A の Arm プロセッサによる電力効率の向上とマシンタイプ非依存の料金体系により、同等の処理性能を維持しながらインフラコストを削減

🚀 UC2: リアルタイムストリーミングデータ処理

シナリオ: Pub/Sub からリアルタイムデータを受信し、変換・集計して BigQuery に書き込むストリーミングパイプラインを高コストパフォーマンスで運用したい

Bash 
java -jar streaming_pipeline.jar \
  --runner=DataflowRunner \
  --project=my-project \
  --region=us-west1 \
  --workerMachineType=c4a-highcpu-4 \
  --experiments=use_runner_v2 \
  --enableStreamingEngine \
  --diskSizeGb=30

効果: Streaming Engine との組み合わせにより、ワーカー VM のリソース消費を最小化しつつ、C4A の highcpu タイプでコンピュート集約型の処理を効率的に実行

🤖 UC3: ML データ前処理パイプライン

シナリオ: 画像やテキストデータの前処理を Dataflow で行い、ML トレーニング用のデータセットを準備。大量のメモリを必要とする変換処理がある

効果: c4a-highmem タイプ (8 GB/vCPU) を使用することで、メモリ集約型の前処理を効率的に実行。GPU が不要な前処理フェーズ (リサイズ、正規化、特徴抽出など) において、Arm の優れたコストパフォーマンスを活用できる

💰 料金情報

Dataflow の料金はマシンタイプファミリーに依存しない。C4A (Arm) を使用しても、x86 ベースのマシンと同じ料金体系が適用される。つまり、Arm プロセッサの効率性によるコストメリットを追加料金なしで享受できる。

📝 まとめ

Dataflow における C4A マシンシリーズ (Arm) サポートの GA は、データ処理パイプラインのコストパフォーマンス最適化において重要な選択肢を提供する。Google Axion プロセッサの電力効率と、Dataflow のマシンタイプ非依存の料金体系を組み合わせることで、既存パイプラインのコード変更を最小限に抑えながらインフラコストの削減が期待できる。

GPU や Dataflow Prime が不要なバッチ・ストリーミングワークロードを運用しているユーザーは、まず開発・テスト環境で C4A の性能とコストを評価し、段階的に本番環境への適用を検討することを推奨する。
