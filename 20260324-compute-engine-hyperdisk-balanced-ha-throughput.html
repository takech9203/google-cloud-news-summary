📋 概要

Hyperdisk Balanced High Availability (HA) は、リージョン内の 2 つのゾーン間で同期レプリケーション を行う高可用性ブロックストレージです。今回のアップデートで、単一ボリュームあたりの最大スループットが 1,200 MiB/s から 2,400 MiB/s へと 2 倍に引き上げられました (GA)。

大規模データベース、高性能コンピューティング、リアルタイム分析など、高スループットと高可用性の両方を必要とするワークロードに大きなメリットをもたらします。

❌ Before

✗最大 1,200 MiB/s のスループットしか得られなかった

✗アプリケーション側でストライピングなどの追加設計が必要
✗複数ボリューム管理でインフラ構成が複雑化

✓ボリューム管理が簡素化

✓インフラ構成がシンプルになり運用負荷が軽減

🏗️ アーキテクチャ図

Hyperdisk Balanced HA はリージョン内の 2 つのゾーンにデータを同期レプリケーションします。プライマリゾーンで障害が発生した場合、セカンダリゾーンのレプリカに force-attach することで 1 分以内にフェイルオーバーが可能です。

⚡ 主要機能

📈 最大スループットの倍増

→単一ボリュームで 1,200 MiB/s から 2,400 MiB/s に増加
→プロビジョニング時にスループット値を指定可能 (作成後も変更可)

🔁 同期レプリケーションの継続

→2 ゾーン間のデータ同期レプリケーションは変更なし
→高スループットでも RPO はゼロに近い値を維持

🔗 既存機能との互換性

→マルチライターモードによる複数 VM からの同時書き込み対応
→インスタントスナップショット / 標準スナップショット対応
→非同期レプリケーションによるリージョン間保護にも対応

項目 アップデート前 アップデート後

スループットの設定可能範囲

💡 計算式: プロビジョニングされた IOPS 値 P に対して
• 最小スループット: MAX(140, P/256) MiB/s
• 最大スループット: MIN(2,400, P/4) MiB/s

🛠️ 設定方法

前提条件

→対応マシンシリーズ: C3, C3D, C4, C4A, N4, N4A, N4D, Z3, A3, G4, M3
→2,400 MiB/s をサポートするマシンタイプを使用 (例: N4A 48 vCPU 以上, Z3 など)

ステップ 1: 高スループットディスクの作成

Bash Hyperdisk Balanced HA ボリューム作成 
gcloud compute disks create high-throughput-ha-disk \
    --type=hyperdisk-balanced-high-availability \
    --size=2000 \
    --provisioned-iops=100000 \
    --provisioned-throughput=2400 \
    --region=us-central1 \
    --replica-zones=us-central1-a,us-central1-b

最大スループットの 2,400 MiB/s を利用するには、十分な IOPS (9,600 以上) をプロビジョニングする必要があります。

ステップ 2: インスタンスへのアタッチ

Bash ディスクのアタッチ 
gcloud compute instances attach-disk my-instance \
    --disk=high-throughput-ha-disk \
    --disk-scope=regional \
    --zone=us-central1-a

ステップ 3: 既存ディスクのスループット変更

Bash スループットの更新 
gcloud compute disks update high-throughput-ha-disk \
    --provisioned-throughput=2400 \
    --region=us-central1
⚠️ 注意: パフォーマンスの変更は 4 時間に 1 回まで可能です。

✅ メリット

💰 ビジネス面

→コスト最適化: 複数ボリュームのストライピングが不要になり、管理コストと構成の複雑さが削減
→SLA の向上: 単一ボリュームで高スループットと高可用性を実現、ゾーン障害時も迅速なフェイルオーバーが可能

🔧 技術面

→パフォーマンス向上: 単一ボリュームで 2 倍のスループット、I/O 集約型ワークロードのボトルネック解消
→アーキテクチャ簡素化: 複数ボリュームの並列利用が不要に
→柔軟なプロビジョニング: 作成後もスループット値を動的に変更可能

⚠️ デメリット・制約事項

制限事項

✗2,400 MiB/s を実現するにはインスタンスのマシンタイプが対応している必要がある (小規模マシンではインスタンス側がボトルネック)
✗パフォーマンス変更は 4 時間に 1 回のみ可能
✗HA ボリュームからマシンイメージやイメージの作成は不可
✗Hyperdisk パフォーマンスはハーフデュプレックス (IOPS/スループットの上限は読み書き合計)

考慮すべき点

💡 注意点:
  • スループットを最大限活用するには 256 KB 以上の I/O サイズが必要
  • HA のストレージコストは Hyperdisk Balanced の 2 倍 (2 ゾーンレプリケーション)
  • Hyperdisk ボリュームはリソースベースの確約利用割引 (CUD) や継続利用割引 (SUD) の対象外

🎯 ユースケース

🗄️ 大規模データベースの高可用性運用

シナリオ: ミッションクリティカルなデータベース (SAP HANA, Oracle Database など) を Google Cloud 上で運用。高スループットと高可用性が必須。

Bash SAP HANA 用 HA ディスク作成例 
gcloud compute disks create sap-hana-data \
    --type=hyperdisk-balanced-high-availability \
    --size=4096 \
    --provisioned-iops=100000 \
    --provisioned-throughput=2400 \
    --region=asia-northeast1 \
    --replica-zones=asia-northeast1-a,asia-northeast1-b

効果: 単一ボリュームで 2,400 MiB/s を確保しつつ、ゾーン障害時も 1 分以内でフェイルオーバー。複数ボリュームのストライピングが不要になり運用が簡素化。

📊 リアルタイムデータ分析基盤

シナリオ: 大量のストリーミングデータをリアルタイムに書き込みながら、同時に分析クエリを実行。可用性を維持しつつ高い書き込みスループットが要求される。

効果: 2,400 MiB/s のスループットにより従来の 2 倍のデータ書き込み速度を単一ボリュームで実現。ゾーン障害時もデータロスなしに処理を継続可能。

💰 料金情報

Hyperdisk Balanced HA はプロビジョニングされたサイズ、IOPS、スループットに基づいて課金されます。ベースラインパフォーマンス (3,000 IOPS / 140 MiB/s) は無料です。

構成 月額料金構成要素
ストレージ (容量) プロビジョニングサイズ x GiB 単価 (Hyperdisk Balanced の 2 倍)
IOPS (ベースライン超過分) (プロビジョニング IOPS - 3,000) x IOPS 単価
スループット (ベースライン超過分) (プロビジョニングスループット - 140 MiB/s) x スループット単価
💰 ポイント: ストレージコストは 2 ゾーンレプリケーションのため Hyperdisk Balanced の 2 倍。具体的な単価は利用リージョンにより異なります。

🌍 利用可能リージョン

Hyperdisk Balanced High Availability は 全てのリージョン で利用可能です。ただし、AI ゾーンでは利用できません。

🔗 関連サービス・機能

💾 Hyperdisk Balanced

単一ゾーンの高性能ブロックストレージ。高可用性が不要な場合に利用。

🔄 Regional Persistent Disk

2 ゾーン間の同期レプリケーション。Hyperdisk Balanced HA より低いパフォーマンス上限。

🌐 非同期レプリケーション

リージョン間のデータ保護。Hyperdisk Balanced HA と組み合わせてリージョン障害にも対応可能。

📦 マネージドインスタンスグループ

リージョナル MIG と組み合わせ、自動フェイルオーバーを含む高可用性アーキテクチャを構築。

📝 まとめ

Hyperdisk Balanced HA の最大スループットが 2,400 MiB/s に倍増 したことにより、ミッションクリティカルな高スループットワークロードにおけるアーキテクチャが大幅に簡素化されます。既存ユーザーはディスクのプロビジョニングスループットを更新するだけでこの改善を活用可能です。対象ワークロードのパフォーマンス要件を見直し、スループット設定の引き上げを検討することを推奨します。
