☁️ BigQuery: Spanner データに対する非インクリメンタル Materialized View が GA

BigQuery Spanner GA

Google Cloud Release Notes

📋 概要

BigQuery において、Spanner 外部データセットに対する非インクリメンタル Materialized View の作成が一般提供 (GA) となりました。allow_non_incremental_definition オプションを使用することで、Spanner データに対するクエリ結果を定期的にキャッシュし、クエリパフォーマンスを大幅に向上させることが可能になります。

Spanner はトランザクション処理に優れた OLTP データベースですが、分析クエリの実行には最適化されていません。BigQuery の Materialized View を活用することで、Spanner に格納されたデータに対する集計・フィルタリング・結合などの分析クエリを高速化でき、ハイブリッドアーキテクチャが構築できます。

🔄 Before / After 比較

❌ アップデート前の課題

  • Materialized View は Preview 段階で本番利用に制限あり
  • 分析クエリは毎回 Data Boost 経由でデータ取得が必要
  • Spanner から BigQuery へのデータパイプラインを別途構築する必要があった

✅ アップデート後の改善

  • GA リリースにより本番環境での利用が正式サポート
  • max_staleness と refresh_interval_minutes で鮮度を柔軟に制御
  • 定期キャッシュ更新で分析クエリのコストとレイテンシを大幅削減

Spanner のデータは CLOUD_RESOURCE コネクション経由で BigQuery の外部データセットとして公開されます。Materialized View は定期的にリフレッシュされ、キャッシュされた結果を返却します。max_staleness を超過した場合は、ベーステーブル (Spanner) から直接データが読み取られます。

⚡ 主要機能

🔧 非インクリメンタル Materialized View の作成

allow_non_incremental_definition = true オプションを指定して、Spanner 外部データセットのテーブルに対する Materialized View を作成可能。リフレッシュ時にビュー全体が再構築されます。

🔄 自動リフレッシュ制御

enable_refresh = true で自動リフレッシュを有効化。refresh_interval_minutes でリフレッシュ間隔を分単位で指定可能。リフレッシュはバックグラウンドで実行され、クエリの可用性に影響しません。

📊 データ鮮度管理 (max_staleness)

max_staleness パラメータでキャッシュデータの許容鮮度を設定。範囲内であればキャッシュから返却、超過時は Spanner のベーステーブルから直接読み取ります。

📐 技術仕様

Materialized View の設定パラメータ

パラメータ 説明 例
allow_non_incremental_definition 非インクリメンタル定義を許可 (Spanner では必須) true
enable_refresh 自動リフレッシュの有効/無効 true
refresh_interval_minutes リフレッシュ間隔 (分) 60
max_staleness キャッシュデータの最大許容鮮度 INTERVAL "24" HOUR

🛠️ 設定方法

💡 前提条件: Spanner データベースが作成済み / CLOUD_RESOURCE コネクションが作成済み / サービスアカウントに読み取り権限付与済み / Spanner 外部データセットが作成済み

ステップ 1: CLOUD_RESOURCE コネクションの作成

Bash コネクション作成 
bq mk --connection --location=REGION --project_id=PROJECT_ID \
  --connection_type=CLOUD_RESOURCE CONNECTION_ID

# サービスアカウント ID の確認
bq show --connection PROJECT_ID.REGION.CONNECTION_ID

ステップ 2: Spanner 外部データセットの作成

SQL 外部データセット作成 
CREATE EXTERNAL SCHEMA spanner_external_dataset
  WITH CONNECTION PROJECT_ID.REGION.CONNECTION_ID
  OPTIONS (
    external_source = 'google-cloudspanner:/projects/PROJECT_ID/instances/INSTANCE/databases/DATABASE',
    location = 'REGION'
  );

ステップ 3: 非インクリメンタル Materialized View の作成

SQL Materialized View 作成 
CREATE MATERIALIZED VIEW my_dataset.my_spanner_mv
  OPTIONS (
    enable_refresh = true,
    refresh_interval_minutes = 60,
    max_staleness = INTERVAL "24" HOUR,
    allow_non_incremental_definition = true
  )
AS
  SELECT
    column1,
    COUNT(*) AS cnt,
    SUM(column2) AS total
  FROM
    spanner_external_dataset.my_table
  GROUP BY
    column1;

ステップ 4: Materialized View へのクエリ実行

SQL クエリ実行 
SELECT * FROM my_dataset.my_spanner_mv;

⚖️ メリット / デメリット

✅ メリット

  • 繰り返し分析クエリのコスト削減 (Data Boost 使用量 + BigQuery クエリコスト)
  • Spanner→BigQuery のデータパイプライン構築が不要になり運用負荷軽減
  • GA リリースにより SLA 適用、本番ワークロードを正式サポート
  • 事前計算された集計結果でクエリレイテンシが大幅改善
  • max_staleness + refresh_interval_minutes で鮮度とパフォーマンスのバランス調整可能
  • 分析クエリをキャッシュで処理し Spanner への負荷を軽減

⚠️ デメリット・制約

  • 差分リフレッシュ非対応 (全体再構築のため大量データ時にコスト増)
  • スマートチューニング (自動リライト) は非インクリメンタル MV に非対応
  • max_staleness 超過時は直接読み取りでレイテンシ増加の可能性
  • SQL 定義は作成後変更不可 (変更時は再作成が必要)
  • CLOUD_RESOURCE コネクション必須 (EUC 認証は使用不可)
  • 列レベルセキュリティ制約がある場合はクエリ実行時に制約検証

💡 ユースケース

🛒 ユースケース 1: EC サイトの売上ダッシュボード

シナリオ: Spanner でトランザクション処理を行う EC サイトにおいて、日次の売上集計ダッシュボードを Looker で表示。リアルタイム性は不要だが 1 時間以内のデータ鮮度を確保したい。

SQL EC サイト売上集計 
CREATE MATERIALIZED VIEW analytics.daily_sales_summary
  OPTIONS (
    enable_refresh = true,
    refresh_interval_minutes = 30,
    max_staleness = INTERVAL "1" HOUR,
    allow_non_incremental_definition = true
  )
AS
  SELECT
    DATE(order_timestamp) AS order_date,
    product_category,
    COUNT(*) AS order_count,
    SUM(total_amount) AS total_sales,
    AVG(total_amount) AS avg_order_value
  FROM
    spanner_ecommerce.orders
  GROUP BY
    order_date, product_category;

→ Looker ダッシュボードのロード時間が数十秒から数秒に短縮、Data Boost 使用量も大幅削減。

📡 ユースケース 2: IoT デバイスのセンサーデータ分析

シナリオ: Spanner に格納された IoT センサーデータに対して、デバイスごとの統計情報を定期的に分析。24 時間以内のデータ鮮度で十分。

→ 大量のセンサーデータに対する集計クエリが事前計算され、アドホッククエリの応答時間が劇的に改善。Data Boost コストも最適化。

💰 料金情報

Materialized View の料金は以下の 3 つの要素で構成されます。

コンポーネント オンデマンド料金 容量ベース料金
クエリ MV およびベーステーブルから処理されたバイト数に基づく クエリ実行中のスロット消費
メンテナンス (リフレッシュ) リフレッシュ時に処理されたバイト数に基づく リフレッシュ時のスロット消費
ストレージ MV に格納されたバイト数に基づく MV に格納されたバイト数に基づく
⚠️ 注意: Spanner 側では Data Boost の使用に対する料金が発生します。リフレッシュ時に Spanner からデータを取得する際は Data Boost が使用されるため、リフレッシュ頻度に応じた Data Boost コストを考慮してください。

🌍 利用可能リージョン

BigQuery の Materialized View は、BigQuery がサポートするすべてのリージョンおよびマルチリージョンで利用可能です。ただし、CLOUD_RESOURCE コネクションは Spanner データベースと同じロケーションに作成する必要があります。
