🔒 BigQuery: ルーティンに対するカスタム組織ポリシーのサポート

BigQuery Organization Policy Preview
2026-03-19

📋 概要

BigQuery において、ルーティン (Routines) リソースに対するカスタム組織ポリシーが新たにサポートされました。 Organization Policy Service のカスタム制約を使用して、UDF、UDAF、テーブル関数、リモート関数、ストアドプロシージャの作成・更新操作を 組織全体で統一的に制御できるようになります。

これまで BigQuery のカスタム組織ポリシーは Dataset や TransferConfig など限られたリソースのみをサポートしていましたが、 今回 bigquery.googleapis.com/Routine リソースタイプが追加され、より包括的なガバナンスが実現可能になりました。

🔄 Before / After 比較

❌ アップデート前の課題

  • ルーティンの作成・更新を組織ポリシーレベルで制御する手段がなく、IAM ロールのみに依存
  • 組織全体でルーティンの命名規則やプロパティを統一的に強制する仕組みがなかった
  • リモート関数やストアドプロシージャの作成を組織レベルで制限できなかった

✅ アップデート後の改善

  • カスタム組織ポリシーにより、ルーティン操作を CEL 条件式で柔軟に制御可能
  • 組織、フォルダ、プロジェクトの各レベルで階層的にポリシーを適用可能
  • IAM と組織ポリシーを組み合わせた多層防御が実現

⚡ 主要機能

🔧 ルーティンリソースに対するカスタム制約

bigquery.googleapis.com/Routine リソースタイプに対してカスタム制約を定義可能。 CEL (Common Expression Language) を使用して条件式を記述し、ALLOW (許可) または DENY (拒否) のアクションを設定できます。

🎯 対象操作の制御

→CREATE: ルーティンの新規作成を制御
→UPDATE: 既存ルーティンの更新を制御
→CREATE と UPDATE の両方、または片方のみを指定可能

🏛️ 階層的ポリシー継承

🏢 組織
→
📁 フォルダ

📂 プロジェクト

上位レベルのポリシーが下位に自動継承。階層評価ルールにより継承の無効化も可能。

🛠️ 設定方法

📋 前提条件: Google Cloud 組織が設定済み / 組織 ID を把握 / roles/orgpolicy.policyAdmin ロールが付与済み

ステップ 1: カスタム制約の YAML ファイルを作成

YAML constraint-restrict-routine.yaml 
# constraint-restrict-routine.yaml
name: organizations/ORGANIZATION_ID/customConstraints/custom.restrictRoutineCreation
resourceTypes:
  - bigquery.googleapis.com/Routine
methodTypes:
  - CREATE
condition: "resource.routineType == 'SCALAR_FUNCTION'"
actionType: DENY
displayName: Restrict scalar function creation
description: Deny creating scalar UDFs in BigQuery to enforce usage of approved routines only.

ステップ 2: カスタム制約を適用

Bash 
gcloud org-policies set-custom-constraint ~/constraint-restrict-routine.yaml

ステップ 3: 制約の存在を確認

Bash 
gcloud org-policies list-custom-constraints --organization=ORGANIZATION_ID

ステップ 4: 組織ポリシーを作成して適用

YAML policy-restrict-routine.yaml 
# policy-restrict-routine.yaml
name: projects/PROJECT_ID/policies/custom.restrictRoutineCreation
spec:
  rules:
    - enforce: true
Bash 
gcloud org-policies set-policy ~/policy-restrict-routine.yaml
💡 注意: ポリシー適用後、約 2 分で Google Cloud がポリシーの適用を開始します。

✅ メリット

💼 ビジネス面

→コンプライアンス強化: 規制要件に沿ったルーティン管理ポリシーを組織全体に強制でき、監査対応が容易に
→ガバナンスの一元化: 階層構造を活用し、部門ごとのポリシーを効率的に管理

🔧 技術面

→多層防御の実現: IAM + 組織ポリシーでセキュリティの多層化が可能
→柔軟な条件定義: CEL でルーティンのプロパティに基づく細かな条件を定義
→自動継承: 上位レベルのポリシーが下位に自動継承、設定漏れを防止

⚠️ デメリット・制約事項

⚠️ 制限事項:
  • Preview 段階 -- 本番環境での利用は「Pre-GA Offerings Terms」が適用される
  • Preview 機能はサポートが限定的であり、変更される可能性がある
  • 監査ログで PolicyViolationInfo が公開されない (エラーメッセージに constraintId は含まれる)
  • ポリシー変更は既存のリソースに対して遡及的に適用されない
💡 考慮すべき点:
  • 既存ルーティンはポリシー適用前のものであるため、新規の作成・更新操作のみが制御対象
  • ポリシーの適用には最大約 2 分のラグがある
  • 過度に厳格なポリシーは開発者の生産性に影響を与える可能性あり
  • カスタム制約は組織ごとにリソースタイプあたり最大 20 個まで作成可能

💡 ユースケース

🛡️ ユースケース 1: リモート関数の作成制限

シナリオ: セキュリティチームが、外部サービスと連携するリモート関数の作成を本番環境のプロジェクトで制限したい。

YAML リモート関数制限の例 
name: organizations/123456789/customConstraints/custom.denyRemoteFunctions
resourceTypes:
  - bigquery.googleapis.com/Routine
methodTypes:
  - CREATE
condition: "resource.routineType == 'SCALAR_FUNCTION' && resource.remoteFunctionOptions != null"
actionType: DENY
displayName: Deny remote function creation
description: Prevent creating remote functions that connect to external services.

効果: 本番環境での予期しない外部連携を防止し、データ流出リスクを低減。

📏 ユースケース 2: ルーティン管理の標準化

シナリオ: データガバナンスチームが、組織全体でルーティンの作成を特定のプロジェクトまたは命名規則に準拠させたい。

効果: ルーティンの乱立を防止し、承認されたルーティンのみが利用される環境を構築。管理対象のルーティンが明確になり、メンテナンスコストを削減。

💰 料金

Organization Policy Service 自体の追加料金は発生しません。BigQuery のルーティンに関する料金は、通常の BigQuery 料金体系に準じます。

詳細は BigQuery 料金ページ を参照してください。

🌍 利用可能リージョン

Organization Policy Service はグローバルサービスであり、BigQuery が利用可能なすべてのリージョンで使用可能です。

詳細は BigQuery のロケーションに関するドキュメント を参照してください。

🔗 関連サービス・機能

🏛️ Organization Policy Service

カスタム制約の管理基盤。BigQuery 以外のサービスにも同様の仕組みを適用可能。

📊 BigQuery カスタム組織ポリシー (Dataset)

既に GA の Dataset リソースに対するカスタム制約。今回のルーティン対応はこの拡張。

🔄 Data Transfer Service カスタム組織ポリシー

TransferConfig リソースに対するカスタム制約。

🔐 IAM

ルーティンに対する既存のアクセス制御。組織ポリシーと併用で多層防御を実現。

📝 Cloud Audit Logs

ポリシー違反の検出と監査に使用。

📌 まとめ

BigQuery のルーティンに対するカスタム組織ポリシーのサポートにより、UDF、ストアドプロシージャ、リモート関数などのルーティン操作を 組織レベルで統一的に制御できるようになりました。

セキュリティ・ガバナンスチームは、この Preview 機能を評価し、既存の IAM ベースのアクセス制御と組み合わせた 多層防御戦略を検討することを推奨します。GA への昇格を見据え、非本番環境での検証を開始するのが良いでしょう。
