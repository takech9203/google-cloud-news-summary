🔍 Cloud Trace: Telemetry API 使用時の Cloud Trace API 有効化要件

Cloud Trace Change
2026-03-09

☁️ 概要

Google Cloud は Cloud Trace におけるトレースデータ送信の API 要件を明確化しました。トレースデータを送信するために利用できる 2 つの API ── Cloud Trace API と Telemetry API ── について、Telemetry API エンドポイント経由でトレースデータを送信する場合、バックエンドでデータを保存するために Cloud Trace API の有効化が必須 であることが公式に文書化されました。

⚠️ 重要: Cloud Trace API が無効な状態で Telemetry API にデータを送信すると、トレースデータはエラーなく 破棄 されます。既に Telemetry API を使用している場合は、Cloud Trace API の有効化状態を直ちに確認してください。

🔄 Before / After

❌ Before (アップデート前)

  • Telemetry API と Cloud Trace API の依存関係が文書化されていなかった
  • Cloud Trace API 無効時にデータがエラーなく破棄される可能性
  • 組織ポリシーで API 無効化時のトラブルシューティングが困難

✅ After (アップデート後)

  • Cloud Trace API の有効化が必須であることが公式に文書化
  • データ破棄動作が明確に定義
  • トラブルシューティングの指針が明確化、データ欠損の原因特定が容易に

⚙️ 主要機能

🌐 2 つのトレースデータ送信 API

→Cloud Trace API (cloudtrace.googleapis.com): Google Cloud 独自の API。v1 (送信・取得) と v2 (送信のみ) をサポート
→Telemetry API (telemetry.googleapis.com): OpenTelemetry OTLP Protocol を実装した API。OpenTelemetry SDK/Collector との互換性が高い

🔗 API 間の依存関係

→Telemetry API と Cloud Trace API はそれぞれ独立して有効化・無効化できる
→Telemetry API でトレースデータを送信する場合、Cloud Trace API もプロジェクトで有効である必要がある
→Cloud Trace API が無効の場合、Telemetry API 経由のトレースデータは破棄される

⭐ 推奨構成

→Google Cloud は Telemetry API の使用を推奨
→Telemetry API は Cloud Trace API よりも緩やかな制限値を持つ
→Application Monitoring など一部の機能は Telemetry API 経由でのみ利用可能

🛠️ 設定方法

ステップ 1: Cloud Trace API と Telemetry API の両方を有効化

Bash 
# Cloud Trace API を有効化
gcloud services enable cloudtrace.googleapis.com --project=PROJECT_ID

# Telemetry API を有効化
gcloud services enable telemetry.googleapis.com --project=PROJECT_ID
💡 ヒント: 両方の API が有効であることを確認してください。特に組織ポリシーによる制約がないか事前に確認が必要です。

ステップ 2: API の有効化状態を確認

Bash 
# 有効な API の一覧を確認
gcloud services list --enabled \
  --filter="name:(cloudtrace.googleapis.com OR telemetry.googleapis.com)" \
  --project=PROJECT_ID

ステップ 3: サービスアカウントに適切なロールを付与

Bash 
# Telemetry API 用のロールを付与
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:SA_EMAIL" \
  --role="roles/telemetry.tracesWriter"

💡 ユースケース

📦 ユースケース 1: OpenTelemetry Collector を使用したトレース収集

マイクロサービスアーキテクチャで OpenTelemetry Collector を使用してトレースデータを収集し、Telemetry API エンドポイントに送信している環境。Cloud Trace API が有効であることを確認することで、送信されたトレースデータが確実に Observability Bucket に保存されます。

YAML 
# OpenTelemetry Collector の設定例
exporters:
  otlphttp:
    endpoint: "https://telemetry.googleapis.com"
    auth:
      authenticator: google_cloud_auth

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlphttp]

🏢 ユースケース 2: 組織ポリシーで API が制限されている環境

セキュリティ要件により組織ポリシーで一部の API が無効化されている企業環境で、トレースデータの収集を行う必要がある場合。Cloud Trace API の有効化が Telemetry API の前提条件であることを理解することで、組織のセキュリティチームと連携して必要な API を適切に有効化できます。

👍 メリット

💼 ビジネス面

✓データ損失リスクの低減: API の依存関係が明文化され、設定ミスによるトレースデータの意図しない破棄を防止
✓トラブルシューティング時間の短縮: データが表示されない場合の原因切り分けが明確化

💻 技術面

✓明確な API 依存関係: インフラ構成の設計時に考慮すべき点が明確に
✓OpenTelemetry エコシステムとの互換性: OTLP Protocol を実装、ベンダー固有エクスポーターに依存しない構成が可能

⚠️ デメリット・制約事項

🚫 制限事項
  • Cloud Trace API が組織ポリシーで無効化されている場合、Telemetry API 経由のトレースデータも保存されない
  • Assured Workloads (データ残留要件や IL4 要件) を使用している場合、いずれの API もトレーススパンの送信には使用できない
  • トレースデータの保持期間は 30 日間
📝 考慮すべき点
  • 既に Telemetry API のみを有効化してトレースデータを送信している場合、Cloud Trace API が無効であればデータが破棄されている可能性がある
  • VPC Service Controls の制約は Cloud Trace API と Telemetry API で独立して適用される
  • Cloud Trace API を有効化する際、組織ポリシー (constraints/serviceuser.services) の制約を事前に確認する必要がある
