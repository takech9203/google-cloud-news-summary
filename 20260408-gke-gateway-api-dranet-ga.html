Google Kubernetes Engine - 2026-04-08

📋 概要

Google Kubernetes Engine (GKE) において、2026 年 4 月 8 日付けで 3 つの重要なアップデート が発表されました。Kubernetes Gateway API v1.5 サポート、マネージド DRANET の GA (一般提供)、そしてログ処理高速化機能のロールバックです。

❌ Before

  • Gateway API の以前のバージョンでは一部の機能や適合性が限定的
  • DRANET はプレビュー段階、対応アクセラレーターは A4X Max のみ。SLA 適用外
  • ログ処理高速化機能の依存関係問題により予期しない動作の可能性

✅ After

  • Gateway API v1.5 で最新の Kubernetes ネットワーキング標準に準拠
  • DRANET GA: NVIDIA GPU (A3 Ultra / A4 / A4X / A4X Max) + Cloud TPU (v6e / v7x) 対応
  • ログ処理高速化のロールバックで依存関係の問題を解消

🌐 主要機能 1: Gateway API v1.5 サポート

🚀 Kubernetes Gateway API v1.5

GKE バージョン 1.35.2-gke.1842000 以降で利用可能。GKE Gateway コントローラーがコア適合性テスト (core conformance tests) に合格し、Gateway API 標準仕様に準拠したトラフィック管理が可能になりました。

  • シングルクラスタ / マルチクラスタの Gateway コントローラーをサポート
  • Cloud Load Balancing との統合で外部/内部ロードバランシングを Gateway API 標準で管理
  • 対応 GatewayClass: gke-l7-global-external-managed, gke-l7-regional-external-managed, gke-l7-rilb, gke-l7-gxlb など

📡 主要機能 2: GKE マネージド DRANET GA

⚡ DRANET - Dynamic Resource Allocation for Networking

Kubernetes の Dynamic Resource Allocation (DRA) API をネットワーキングリソースに対して実装。RDMA (Remote Direct Memory Access) 対応ネットワークインターフェースを Pod に専有割り当てできます。

  • networking-dra-driver DaemonSet としてノード上で動作
  • DeviceClass: mrdma.google.com (RDMA 対応) / netdev.google.com (非 RDMA)
  • DRA スケジューラーがデバイストポロジーを考慮して Pod を配置

🔄 主要機能 3: ログ処理高速化のロールバック

⚠️ Faster Log Processing のロールバック

2025 年 11 月 7 日に発表されたログ処理高速化機能 (faster log processing) が、基盤となる依存関係の問題によりロールバックされました。この機能を利用していた環境では、以前のログ処理動作に戻ります。

⚠️ 注意: ログ処理高速化機能のロールバックにより、ログ配信の遅延が発生する可能性があります。依存関係の問題が解消され次第、再リリースが期待されます。

🔧 設定方法

① 前提条件

  • GKE クラスタがバージョン 1.35.2-gke.1842000 以降であること
  • GKE Dataplane V2 が有効であること (DRANET 利用時)
  • Standard モードのクラスタであること (DRANET 利用時)

② DRANET ドライバーの有効化

GPU / TPU を搭載したノードプールに対して DRANET ドライバーを有効化します。

Bash DRANET ドライバーの有効化 
# ノードプールに DRANET ドライバーラベルを追加
gcloud container node-pools update NODE_POOL_NAME \
  --cluster=CLUSTER_NAME \
  --node-labels=cloud.google.com/gke-networking-dra-driver=true

GKE が自動的に networking-dra-driver DaemonSet をデプロイし、DeviceClass リソースを作成します。

③ ResourceClaimTemplate の作成

ワークロードが必要とするネットワークデバイスを定義します。

YAML ResourceClaimTemplate 
apiVersion: resource.k8s.io/v1beta1
kind: ResourceClaimTemplate
metadata:
  name: all-rdma-devices
spec:
  spec:
    devices:
      requests:
      - name: rdma
        deviceClassName: mrdma.google.com
        allocationMode: All

④ Pod での ResourceClaim の参照

Pod 仕様で ResourceClaimTemplate を参照して RDMA デバイスへのアクセスを要求します。

YAML Pod 仕様 
apiVersion: v1
kind: Pod
metadata:
  name: rdma-workload
spec:
  containers:
  - name: app
    image: my-app-image
    resources:
      claims:
      - name: rdma
  resourceClaims:
  - name: rdma
    resourceClaimTemplateName: all-rdma-devices

⑤ Gateway API v1.5 の利用

GKE バージョン 1.35.2-gke.1842000 以降では、Gateway API v1.5 が自動的に利用可能です。

YAML Gateway リソースの作成例 
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
spec:
  gatewayClassName: gke-l7-global-external-managed
  listeners:
  - name: https
    protocol: HTTPS
    port: 443
    allowedRoutes:
      kinds:
      - kind: HTTPRoute

🌟 メリット

💼 ビジネス面

  • AI/ML 本番運用強化: DRANET の GA 化で SLA に裏付けられた高性能ネットワーキングを利用可能
  • 幅広いアクセラレーター対応: GPU 4 種 + TPU 2 種で最適なハードウェアを選択可能
  • K8s 標準準拠: Gateway API v1.5 でベンダーロックインを軽減

🛠️ 技術面

  • 宣言的リソース管理: DRA API でネットワークインターフェースを K8s ネイティブに管理
  • RDMA 高帯域通信: Pod 専有の RDMA インターフェースでノード間通信を最大化
  • トポロジー認識: DRA スケジューラーが GPU/TPU と NIC の最適な組み合わせを実現

⚠️ 制限事項・注意点

🚫 DRANET の制限

  • Standard モードのクラスタでのみ利用可能 (Autopilot は未対応)
  • クラスタオートスケーリングは DRANET と併用できない
  • デフォルト NIC や仮想 NIC (veth) の割り当ては不可
  • GKE マルチネットワーク API (Device タイプの Network リソース) との併用不可
  • secondaryPodRange を設定したインターフェースでは使用不可
  • Pod に割り当てたネットワークインターフェースは専有 (同一ノード上の他 Pod と共有不可)
🔄 ログ処理高速化のロールバック: 2025 年 11 月 7 日の faster log processing 機能が依存関係の問題によりロールバックされ、ログ処理性能が以前の水準に戻ります。ログ配信の遅延が発生する可能性があります。
📝 考慮点: GKE Dataplane V2 が有効でない既存クラスタでは、DRANET を利用する前にデータプレーンの移行が必要です。

🎯 ユースケース

🤖 ユースケース 1: 大規模分散 AI トレーニング

シナリオ: NVIDIA A4X Max GPU を搭載した複数ノードで大規模言語モデル (LLM) の分散トレーニングを実施。ノード間の勾配同期に高帯域・低レイテンシ通信が必要。

効果: RDMA 対応インターフェースの専有割り当てにより、NCCL 等の集合通信ライブラリが GPUDirect RDMA を活用でき、分散トレーニングのスループットが向上。

YAML 分散トレーニング StatefulSet の例 
apiVersion: resource.k8s.io/v1beta1
kind: ResourceClaimTemplate
metadata:
  name: training-rdma
spec:
  spec:
    devices:
      requests:
      - name: all-rdma
        deviceClassName: mrdma.google.com
        allocationMode: All
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: distributed-training
spec:
  replicas: 8
  template:
    spec:
      containers:
      - name: trainer
        image: training-image:latest
        resources:
          claims:
          - name: all-rdma
          limits:
            nvidia.com/gpu: 8
      resourceClaims:
      - name: all-rdma
        resourceClaimTemplateName: training-rdma

🌐 ユースケース 2: Gateway API によるマルチサービストラフィック管理

シナリオ: 複数のマイクロサービスで構成されるアプリケーションにおいて、Gateway API v1.5 を使用して宣言的なトラフィックルーティングを実現。

効果: Kubernetes 標準の Gateway API で Cloud Load Balancing との統合を維持しつつ、ポータブルで宣言的なトラフィック管理が実現。

YAML Gateway + HTTPRoute の例 
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: app-gateway
spec:
  gatewayClassName: gke-l7-global-external-managed
  listeners:
  - name: https
    protocol: HTTPS
    port: 443
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: app-routes
spec:
  parentRefs:
  - name: app-gateway
  hostnames:
  - "app.example.com"
  rules:
  - matches:
    - path:
        value: /api
    backendRefs:
    - name: api-service
      port: 8080
  - matches:
    - path:
        value: /
    backendRefs:
    - name: frontend-service
      port: 80

💰 料金

GKE の料金体系は、クラスタ管理料金とノードに使用する Compute Engine リソースの料金で構成されます。

✨ Gateway API: シングルクラスタ Gateway コントローラーは GKE Standard / Autopilot の料金に含まれており、追加料金は発生しません。
✨ DRANET: DRANET 自体にも追加料金はなく、GPU/TPU ノードの Compute Engine インスタンス料金が適用されます。

詳細: GKE 料金 | GPU 料金 | TPU 料金

🌎 利用可能リージョン

DRANET の利用可能リージョンは、対応する GPU (A3 Ultra, A4, A4X, A4X Max) および TPU (v6e, v7x) が利用可能なリージョンに依存します。Gateway API v1.5 は GKE バージョン 1.35.2-gke.1842000 以降が利用可能な全リージョンで利用できます。

🔗 関連サービス・機能

⚙️ Kubernetes DRA

DRANET の基盤となる Kubernetes API。GPU やネットワークデバイスの宣言的な割り当てを提供。

🛰 GKE Dataplane V2

DRANET の前提条件。eBPF ベースのネットワーキングを提供。

⚖️ Cloud Load Balancing

Gateway API の GKE Gateway コントローラーが管理するロードバランサーの基盤。

💻 Cloud TPU

DRANET GA で新たに対応した v6e, v7x TPU。

⚡ NVIDIA GPUDirect RDMA

DRANET が提供する RDMA インターフェースと組み合わせて GPU 間の直接データ転送を実現。

📝 まとめ

今回の GKE アップデートでは、Gateway API v1.5 サポートとDRANET の GA 化という 2 つの重要な進展がありました。特に DRANET の GA 化は、NVIDIA A3 Ultra / A4 / A4X / A4X Max GPU および Cloud TPU v6e / v7x という幅広いアクセラレーターでの RDMA ネットワーキングを本番環境で利用可能にするものであり、大規模 AI/ML ワークロードを GKE 上で運用する組織にとって重要な一歩です。

一方、ログ処理高速化機能のロールバックについては、依存関係の問題が解消され次第再リリースされることが期待されますが、当面はログ処理が以前の動作に戻る点に注意してください。
