📋 概要

Spanner と BigQuery 間のデータフローを大幅に強化する 3 つの重要な機能が発表されました。自然言語によるデータ問い合わせ、分析クエリのキャッシュ、そして分析結果のアプリケーション層への書き戻しにより、完全な双方向データパイプラインが実現します。

🔄 Before / After 比較

❌ Before

  • Spanner データの分析には SQL の専門知識が必須で、ビジネスユーザーが直接アクセスできなかった
  • BigQuery の分析結果を Spanner に戻すにはカスタム ETL パイプラインの構築が必要だった
  • Spanner データに対する BigQuery マテリアライズドビューは Preview 段階で本番利用に制限があった

✅ After

  • Conversational Analytics API で自然言語による Spanner データの問い合わせが可能に
  • EXPORT DATA + CLOUD_RESOURCE 接続でリバース ETL が GA として利用可能に
  • BigQuery 非増分マテリアライズドビューが GA となり安定的にクエリパフォーマンス向上

💬 機能 1: 会話型分析 (Conversational Analytics)

Preview
🤖 Gemini 搭載: Conversational Analytics API (geminidataanalytics.googleapis.com) を使用して、自然言語で Spanner のオペレーショナルデータにクエリを実行できます。Preview 期間中は無料で利用可能です。

🗣️ 自然言語クエリ

QueryData メソッドにより GoogleSQL for Spanner のデータベースに対して自然言語で問い合わせが可能。Gemini を活用した推論エンジンがユーザーの質問を解釈し、SQL クエリを生成して実行します。

🤖 データエージェント

Spanner Studio でデータエージェントを作成し、ビジネスコンテキストや指示を提供してクエリの精度を向上可能。クエリテンプレートとクエリファセットによるエージェントコンテキストのカスタマイズに対応します。

📊 機能 2: BigQuery 非増分マテリアライズドビュー

GA

allow_non_incremental_definition オプションを使用して、Spanner 外部データセットテーブルを参照するマテリアライズドビューを作成できます。定期的に結果をキャッシュすることでクエリパフォーマンスを向上させます。

⚡ 主要パラメータ

→max_staleness: データの鮮度を制御し、キャッシュ結果と最新データのバランスを調整
→refresh_interval_minutes: 自動リフレッシュの間隔を設定
→CLOUD_RESOURCE 接続を使用して Spanner 外部データセットを事前作成が必要
項目詳細
タイプ非増分 (allow_non_incremental_definition)
前提条件CLOUD_RESOURCE 接続による Spanner 外部データセット
リフレッシュ全体リフレッシュ (増分リフレッシュ非対応)
SQL 方言GoogleSQL のみ

📝 マテリアライズドビューの作成例

SQL 
CREATE MATERIALIZED VIEW sample_dataset.sample_spanner_mv
OPTIONS (
  enable_refresh = true,
  refresh_interval_minutes = 60,
  max_staleness = INTERVAL "24" HOUR,
  allow_non_incremental_definition = true
)
AS SELECT COUNT(*) cnt
FROM spanner_external_dataset.spanner_table;

🔁 機能 3: リバース ETL (BigQuery to Spanner)

GA

EXPORT DATA ステートメントを使用して BigQuery テーブルから Spanner テーブルへデータをエクスポート。CLOUD_RESOURCE 接続により、ユーザーに Spanner への直接アクセス権を付与せずにエクスポートを実行できます。

🚀 主要特徴

→継続的エクスポートにも対応し、リアルタイムに近いデータ同期が可能
→BigQuery Iceberg テーブルからのエクスポートにも対応
→Enterprise 以上のリザベーションが必要 (オートスケーリングでコスト最適化可能)
項目詳細
メソッドEXPORT DATA ステートメント
接続タイプCLOUD_RESOURCE 接続 (推奨) または直接アクセス
出力形式CLOUD_SPANNER
必要な IAM ロールBigQuery Data Viewer, BigQuery User, Spanner Viewer, Spanner Database User

📝 リバース ETL の設定例

SQL 
EXPORT DATA
  WITH CONNECTION `PROJECT_ID.LOCATION.CONNECTION_NAME`
  OPTIONS (
    uri = "https://spanner.googleapis.com/projects/PROJECT_ID/instances/INSTANCE_ID/databases/DATABASE_ID",
    format = 'CLOUD_SPANNER',
    spanner_options = """{"table": "SPANNER_TABLE_NAME"}"""
  )
AS SELECT * FROM my_bq_dataset.table1;

⚙️ 設定方法

📌 前提条件: Google Cloud プロジェクトで Spanner API、BigQuery API、Vertex AI API が有効化されていること。Spanner Enterprise エディション以上のインスタンスが必要です。

ステップ 1: CLOUD_RESOURCE 接続の作成

Bash 
bq mk --connection \
  --location=REGION \
  --project_id=PROJECT_ID \
  --connection_type=CLOUD_RESOURCE \
  CONNECTION_ID

ステップ 2: サービスアカウントへの権限付与

Bash 
# サービスアカウント ID を取得
bq show --connection PROJECT_ID.REGION.CONNECTION_ID

# Cloud Spanner Database User ロールを付与
gcloud spanner databases add-iam-policy-binding DATABASE_ID \
  --instance=INSTANCE_ID \
  --member="serviceAccount:SERVICE_ACCOUNT_ID" \
  --role="roles/spanner.databaseUser"

✅ メリット

🏢 ビジネス面

📊データ民主化: SQL の専門知識がないビジネスユーザーでも、会話型分析を通じて Spanner のオペレーショナルデータから直接インサイトを取得可能
⚡リアルタイムアプリケーション提供: BigQuery での分析結果をリバース ETL で Spanner に書き戻し、低レイテンシでアプリケーションユーザーにデータを提供可能
💰運用コスト削減: カスタム ETL パイプラインの構築・保守が不要。マネージドサービスとして BigQuery-Spanner 間のデータ連携が実現

🔧 技術面

🚀クエリパフォーマンス向上: BigQuery のマテリアライズドビューにより、Spanner データに対する繰り返しクエリのパフォーマンスが大幅に向上
🔒権限管理の簡素化: CLOUD_RESOURCE 接続により、ユーザーに Spanner への直接アクセスを付与せずにリバース ETL を実行可能
🔄双方向データフロー: Spanner → BigQuery (フェデレーション) と BigQuery → Spanner (リバース ETL) で完全な双方向パイプラインが実現

⚠️ デメリット・制約事項

⚠️ 制限事項:
  • 会話型分析は Preview 段階 -- 本番環境や個人データの処理には推奨されない
  • 非増分マテリアライズドビューはスマートチューニング非対応のため、直接クエリが必要
  • リバース ETL には Enterprise 以上の BigQuery リザベーションが必要
  • 会話型分析の QueryData メソッドは GoogleSQL for Spanner のみ対応 (PostgreSQL 非対応)
  • 非増分マテリアライズドビューは全体リフレッシュが必要で、大規模データではコスト増の可能性
💡 考慮すべき点:
  • 会話型分析は AI ベースのため、生成された回答の正確性を検証することが推奨される
  • リバース ETL で同一行キーの複数行をエクスポートする場合、Spanner では単一行にマージされ、採用される行は保証されない
  • max_staleness 間隔を超えてリフレッシュが行われない場合、Spanner 外部データセットを直接読み取る
  • エージェントコンテキストはデータベースへの読み取りアクセスを持つ全ユーザーに表示される

💡 ユースケース

🛒 ユースケース 1: 小売業の在庫分析と補充最適化

Spanner で管理している在庫データを BigQuery で分析し、補充の最適化結果をリバース ETL で Spanner に書き戻してリアルタイムの在庫管理アプリケーションに反映。カスタム ETL パイプラインを構築することなく、分析結果をアプリケーション層にリアルタイムで反映可能。

SQL 
-- リバース ETL で Spanner に書き戻し
EXPORT DATA
  WITH CONNECTION `my_project.us-central1.spanner_conn`
  OPTIONS (
    uri = "https://spanner.googleapis.com/projects/my_project/instances/retail/databases/inventory",
    format = 'CLOUD_SPANNER',
    spanner_options = """{"table": "ReplenishmentOrders"}"""
  )
AS SELECT * FROM analytics_dataset.replenishment_recommendations;

👔 ユースケース 2: 経営層によるセルフサービス分析

経営層が SQL を書くことなく、「先月の売上トップ 10 の商品は?」「前年比で最も成長した地域は?」といった自然言語の質問を Spanner のオペレーショナルデータに対して実行。データチームへの分析依頼の待ち時間を削減し、意思決定の迅速化を実現。データエージェントにビジネスコンテキスト (用語定義や KPI 定義) を設定することで、精度の高い回答を取得可能。

📈 ユースケース 3: 定期的なレポート用キャッシュの構築

BI ダッシュボードが Spanner のトランザクションデータを頻繁にクエリしており、オペレーショナル負荷が懸念される場合に、BigQuery のマテリアライズドビューでクエリ結果をキャッシュ。Spanner への直接クエリ負荷を軽減しつつ、許容可能なデータ鮮度で高速なレポートクエリを実現。

SQL 
CREATE MATERIALIZED VIEW reporting_dataset.daily_sales_summary
OPTIONS (
  enable_refresh = true,
  refresh_interval_minutes = 30,
  max_staleness = INTERVAL "2" HOUR,
  allow_non_incremental_definition = true
)
AS SELECT
  DATE(order_timestamp) AS order_date,
  region,
  SUM(total_amount) AS daily_total,
  COUNT(*) AS order_count
FROM spanner_external_dataset.orders
GROUP BY order_date, region;
