☁️ Managed OpenTelemetry for GKE

GKE ワークロードのオブザーバビリティをフルマネージドで実現

Google Kubernetes Engine Preview 2026-03-10

📋 概要

GKE バージョン 1.34.1-gke.2178000 以降のクラスタで、OpenTelemetry Protocol (OTLP) のトレース、メトリクス、ログを完全マネージドかつ簡素化された方法で収集できる新機能が Preview として利用可能になりました。

Managed OpenTelemetry for GKE は、GKE 上でトレースデータを収集するための Google Cloud 唯一のマネージドソリューションです。従来必要だった OpenTelemetry Collector の自前デプロイ・運用が不要になります。

💡 ポイント: Autopilot / Standard の両クラスタモードに対応。OpenTelemetry SDK でインストルメント済みのワークロードが対象です。Agent Development Kit (ADK) などのフレームワークともネイティブ連携します。

🔄 Before / After 比較

Before (従来)

  • OpenTelemetry Collector を自前でデプロイ・管理が必要
  • 認証設定、構成管理、バージョンアップ、監視の運用負荷
  • 各ワークロードへの OTLP エクスポーター設定を手動構成
  • Collector の可用性・スケーラビリティを自分で確保

After (今回のアップデート)

  • Managed Collector がクラスタ内に自動デプロイ
  • Collector の運用管理が完全に不要に
  • Instrumentation CR で環境変数を自動注入
  • Cloud Trace / Monitoring / Logging に自動ルーティング

⚡ 主要機能

🔧 Managed Collection (マネージドコレクション)

→クラスタ内の gke-managed-otel Namespace に Collector が自動デプロイ
→インクラスタ OTLP エンドポイント提供: http://opentelemetry-collector.gke-managed-otel.svc.cluster.local:4318
→Cloud Telemetry API 経由で Google Cloud Observability に自動ルーティング

⚙️ Automatic Configuration (自動構成)

→instrumentations.telemetry.googleapis.com カスタムリソースでワークロードを自動構成
→Pod または Namespace 単位で環境変数の注入対象を指定可能
→収集するシグナルタイプ (トレース、メトリクス、ログ) を個別に制御

📊 シグナルタイプの選択的収集

→トレースサンプリングレート: OTEL_TRACES_SAMPLER_ARG で 0.0 - 1.0 設定
→メトリクスエクスポート間隔: 5,000ms - 300,000ms (デフォルト 60,000ms)
→各シグナルタイプのエクスポーターを個別に無効化可能

🛠️ 設定方法

前提条件

✓GKE クラスタが バージョン 1.34.1-gke.2178000 以降で稼働
✓gcloud CLI バージョン 551.0.0 以降がインストール済み
✓ワークロードが OpenTelemetry SDK でインストルメント済み

Step 1: 新規クラスタで有効化

Bash Autopilot / Standard クラスタ作成 
# Autopilot クラスタの場合
gcloud beta container clusters create-auto CLUSTER_NAME \
  --project=PROJECT_ID \
  --managed-otel-scope=COLLECTION_AND_INSTRUMENTATION_COMPONENTS \
  --location=LOCATION \
  --cluster-version=1.34.1-gke.2178000

# Standard クラスタの場合
gcloud beta container clusters create CLUSTER_NAME \
  --project=PROJECT_ID \
  --managed-otel-scope=COLLECTION_AND_INSTRUMENTATION_COMPONENTS \
  --location=LOCATION \
  --cluster-version=1.34.1-gke.2178000

Step 2: 既存クラスタで有効化

Bash 既存クラスタのアップデート 
gcloud beta container clusters update CLUSTER_NAME \
  --project=PROJECT_ID \
  --managed-otel-scope=COLLECTION_AND_INSTRUMENTATION_COMPONENTS \
  --location=LOCATION

Step 3: Instrumentation カスタムリソース作成

YAML Instrumentation CR 定義 
apiVersion: telemetry.googleapis.com/v1alpha1
kind: Instrumentation
metadata:
  name: my-instrumentation
  namespace: default
spec:
  selector: {}
  tracer_provider:
    sampler:
      type: parentbased_traceidratio
      argument: "0.5"
  meter_provider:
    export_interval_millis: 60000

Step 4: Terraform での設定 (オプション)

HCL Terraform リソース定義 
resource "google_container_cluster" "default" {
  provider = google-beta
  name     = "my-gke-cluster"
  location = "us-west1"

  release_channel {
    channel = "RAPID"
  }

  managed_opentelemetry_config {
    scope = "COLLECTION_AND_INSTRUMENTATION_COMPONENTS"
  }
}

⚠️ デメリット・制約事項

⚠️ Preview 段階: Pre-GA Offerings Terms が適用されます。本番環境での使用には注意が必要です。

制限事項

✗GKE バージョン 1.34.1-gke.2178000 以降が必須 (古いバージョンでは利用不可)
✗GKE Autopilot パートナーの特権ワークロードへの構成注入は非対応
✗Collector レベルのフィルタリング・カスタム制御が必要な場合は Google-Built OTel Collector を別途使用

考慮すべき点

!Instrumentation CR デプロイ後にワークロードの Pod を再起動する必要がある
!テレメトリデータの送信量に応じて Cloud Monitoring / Logging / Trace の料金が発生

🎯 ユースケース

🌐 マイクロサービスアーキテクチャの分散トレーシング

複数のマイクロサービスが GKE 上で稼働しており、サービス間のリクエストフローを追跡してレイテンシのボトルネックを特定したいケース。

YAML トレース専用の Instrumentation CR 
apiVersion: telemetry.googleapis.com/v1alpha1
kind: Instrumentation
metadata:
  name: microservices-tracing
  namespace: production
spec:
  selector: {}
  tracer_provider:
    sampler:
      type: parentbased_traceidratio
      argument: "0.1"
  meter_provider: null
  logger_provider: null

サンプリングレート 10% により、コストを抑えつつ十分なトレースデータを収集。Cloud Trace でサービス間のリクエストフローの可視化とレイテンシ分析が可能に。

🤖 AI エージェントワークロードのオブザーバビリティ

Agent Development Kit (ADK) を使用して構築した AI エージェントが GKE 上で稼働しており、エージェントの処理フローやパフォーマンスを監視したいケース。

ADK が OpenTelemetry をネイティブサポートしているため、Managed OpenTelemetry を有効化するだけで AI エージェントのトレース、メトリクス、ログが自動的に Google Cloud Observability に送信されます。各ステップの実行時間や成功率を Cloud Trace と Cloud Monitoring で可視化できます。

💰 料金情報

Managed OpenTelemetry for GKE 自体の追加料金は発生しません。送信されるテレメトリデータの量に応じて以下のサービスの料金が適用されます。

テレメトリタイプ課金基準
メトリクスGoogle Cloud Managed Service for Prometheus の料金体系に基づく
ログCloud Logging の料金体系に基づく
トレースCloud Trace の料金体系に基づく

🌍 利用可能リージョン

GKE バージョン 1.34.1-gke.2178000 以降が利用可能な全てのリージョンで使用できます。Preview 段階のため、利用可能なリージョンは今後変更される可能性があります。

🔗 関連サービス・機能

  • 📊 Cloud Trace - 分散トレースデータの保存・分析・可視化。OTLP トレースの表示先
  • 📈 Cloud Monitoring - メトリクスデータの保存・分析・アラート。OTLP メトリクスの表示先
  • 📝 Cloud Logging - ログデータの保存・分析。OTLP ログの表示先
  • 🔧 Google-Built OpenTelemetry Collector - Collector レベルの制御が必要な場合の代替ソリューション
  • 🤖 Agent Development Kit (ADK) - OpenTelemetry をネイティブサポートする AI エージェント開発フレームワーク

📌 まとめ

Managed OpenTelemetry for GKE は、GKE ワークロードのオブザーバビリティを大幅に簡素化するマネージドサービスです。OpenTelemetry Collector の運用負荷を解消し、Instrumentation カスタムリソースによる自動構成で迅速なテレメトリ収集を実現します。

💡 推奨: 現在 Preview 段階ですが、OpenTelemetry ベースのオブザーバビリティを GKE で導入する際には、まず開発・ステージング環境で本機能を試用し、GA リリースに備えることを推奨します。
