🌐 Network Intelligence Center: Connectivity Tests - GKE Pod 対応強化

Network Intelligence Center Feature GKE
2026-03-04
Google Cloud

📋 概要

Network Intelligence Center の Connectivity Tests に、GKE 環境のネットワーク診断を大幅に強化する 3 つの新機能 が追加されました。 GKE Pod をエンドポイントとして直接指定する機能、IP マスカレードの適用有無を自動評価する機能、そして GKE ネットワークポリシーを評価する機能です。

これにより、Pod レベルの精密なネットワーク診断が可能となり、kubectl exec による手動テストに頼ることなく、Google Cloud コンソールや gcloud CLI から統合的に GKE クラスタのネットワークトラブルシューティングが実行可能になりました。

主な対象ユーザーは GKE クラスタのネットワーク管理者、プラットフォームエンジニア、セキュリティエンジニアです。

🔄 Before / After 比較

❌ アップデート前

  • GKE Pod をエンドポイントとして直接指定できなかった
  • IP マスカレード (ip-masq-agent) の設定が分析に反映されなかった
  • GKE ネットワークポリシーが構成分析で評価されなかった
  • Pod レベルの診断には kubectl exec で手動テスト実行が必要だった

✅ アップデート後

  • GKE Pod をソース/宛先エンドポイントとして直接指定可能に
  • IP マスカレードの適用有無を自動判定、SNAT 後のアドレスでテスト実行
  • GKE ネットワークポリシーの評価が構成分析に含まれるように
  • Google Cloud コンソール / gcloud CLI から統合的に診断可能に

⚡ 主要機能

🎯 1. GKE Pod エンドポイント

Connectivity Tests のソースまたは宛先として GKE Pod を直接指定可能。VPC ネイティブクラスタの Pod に対応し、Pod IP アドレスに基づいた接続性の構成分析が実行されます。

✓Pod から外部サービスへの接続テスト
✓Pod 間の通信テスト
✓Pod から VM / Cloud SQL への接続テスト

🔀 2. IP マスカレード評価

GKE Pod からのトラフィックに対し、IP マスカレードの適用有無を自動判定。マスカレード適用時は変換後のノード IP を使用してテストが実行されます。

✓ip-masq-agent の ConfigMap 設定 (nonMasqueradeCIDRs) を考慮
✓Autopilot クラスタの EgressNATPolicy を考慮
✓マスカレード後のアドレスでファイアウォール評価

🛡️ 3. ネットワークポリシー評価

FQDN ネットワークポリシーが有効でない GKE クラスタにおいて、Kubernetes NetworkPolicy を評価。Ingress/Egress ルールが構成分析に反映されます。

✓Pod セレクタ / ネームスペースセレクタに基づく制御を反映
✓Ingress / Egress 両方のポリシーを評価
⚠️ 注意: FQDN ネットワークポリシーが有効なクラスタでは、この評価は適用されません。

IP マスカレードの評価ロジック

クラスタ構成SNAT 動作Connectivity Tests での扱い
ip-masq-agent あり + カスタム nonMasqueradeCIDRs nonMasqueradeCIDRs 以外の宛先で Pod IP をノード IP に変換 変換後のノード IP でテスト
ip-masq-agent あり + nonMasqueradeCIDRs 未設定 デフォルト非マスカレード宛先以外で変換 変換後のノード IP でテスト
ip-masq-agent なし + --disable-default-snat 未設定 デフォルト非マスカレード宛先以外で変換 変換後のノード IP でテスト
ip-masq-agent なし + --disable-default-snat 設定 全宛先で Pod IP を維持 Pod IP のままテスト

FQDN ネットワークポリシー有効評価されない

✅ メリット

💼 ビジネス面

→トラブルシューティング時間の大幅短縮: Pod レベルの接続性をコンソールから直接診断、kubectl exec 不要
→運用の標準化: VM、GKE Pod、サーバーレスなど全レイヤーの診断を統一ツールで実行
→セキュリティ監査の効率化: ネットワークポリシーの適用状況を構成分析レベルで確認可能

⚙️ 技術面

→Pod レベルの精密診断: IP マスカレード後の実際のソースアドレスを考慮したファイアウォール評価
→GKE ネットワーク構成の可視化: ネットワークポリシー、IP マスカレード、VPC ファイアウォールの各レイヤーを一元確認
→マイクロサービス間通信の検証: トラフィックを流さず構成レベルで Pod 間通信を検証

⚠️ デメリット・制約事項

制限事項
✗FQDN ネットワークポリシーが有効な GKE クラスタでは、ネットワークポリシー評価が適用されない
✗VPC ネイティブクラスタのみ対応 (ルートベースクラスタの Pod は非サポート)
✗構成分析はシミュレーションであり、データプレーンの実際の状態を保証するものではない
💡 考慮すべき点
📌IP マスカレードの設定が複雑な場合、ip-masq-agent の設定内容を事前に把握しておくことを推奨
📌ネットワークポリシー評価を活用するには、FQDN ネットワークポリシーの有効/無効状態を確認が必要
📌Windows Server ノードプール上の Pod は IP マスカレード評価の対象外

🎯 ユースケース

1. マイクロサービス間の通信障害の診断

📋 シナリオ: フロントエンド Pod からバックエンド API Pod への通信が失敗しており、ネットワークポリシーが原因かファイアウォールが原因かを切り分けたい。

✅ 効果: Connectivity Tests がネットワークポリシー、IP マスカレード、VPC ファイアウォールの各レイヤーを順番に評価し、どの段階でトラフィックがブロックされているかを明確に特定できます。

2. Pod から外部 API への通信の検証

📋 シナリオ: GKE Pod から外部 SaaS API への HTTPS 通信を許可しているが、IP マスカレード後のノード IP が正しくファイアウォールで許可されているか確認したい。

✅ 効果: Connectivity Tests が IP マスカレードの適用を自動検出し、変換後のノード IP アドレスを使用してファイアウォール評価を実行。Pod IP とノード IP の違いによる設定の不整合を事前に発見できます。

3. ネットワークポリシーの適用範囲の確認

📋 シナリオ: 新しい Kubernetes NetworkPolicy をデプロイする前に、既存の Pod 間通信への影響を事前に評価したい。

✅ 効果: Connectivity Tests でポリシー適用後の到達性を構成分析レベルで確認でき、本番環境に影響を与えることなくポリシー変更の影響範囲を検証可能です。

🔗 関連サービス・機能

🔍Network Analyzer (GKE IP マスカレード構成インサイト) - ip-masq-agent の設定と Pod CIDR 範囲の整合性を自動検出し、誤設定を警告
🌐GKE Dataplane V2 - GKE ネットワークポリシーや FQDN ネットワークポリシーの基盤。Connectivity Tests のネットワークポリシー評価対象に影響

🔥Firewall Insights - ファイアウォールルールの使用状況を分析。IP マスカレード後のノード IP に対するファイアウォール評価と合わせて活用
⚙️ip-masq-agent / EgressNATPolicy - GKE クラスタの IP マスカレード設定を管理するコンポーネント。Connectivity Tests はこれらの設定を自動的に読み取って評価に反映
