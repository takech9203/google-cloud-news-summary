📋 概要

Cloud Load Balancing の認可ポリシーにポリシープロファイル (Policy Profiles) 機能が Preview として追加されました。ロードバランサーで実行される認可の種類を定義するもので、従来の HTTP リクエストヘッダーベースの認可に加え、アプリケーションペイロードの深層検査によるコンテンツベースのセキュリティを実現します。

🤖 AI セキュリティの新たな防御ライン: CONTENT_AUTHZ プロファイルと Model Armor の統合により、アプリケーションコードを変更することなく、プロンプトインジェクション攻撃や機密データ漏洩をネットワーク層でブロックできるようになります。

🔄 Before / After

❌ アップデート前の課題
  • • HTTP リクエストヘッダーに基づくアクセス制御のみが可能で、リクエストボディの内容に基づく判定ができなかった
  • • プロンプトインジェクション攻撃や機密データ漏洩の防止にはアプリケーション層での個別実装が必要
  • • ロードバランサーレベルでのコンテンツ検査の標準的な仕組みがなく、セキュリティ対策が分散
✅ アップデート後の改善
  • • リクエスト認可とコンテンツ認可を明確に区別して構成可能に
  • • CONTENT_AUTHZ プロファイルでペイロード全体 (ヘッダー、ボディ、トレーラー) の深層検査が可能に
  • • Model Armor 統合でアプリケーションコード変更なしにプロンプトインジェクション防御を実現
  • • ext_proc フルデュプレックスストリーミングでリクエスト/レスポンス双方向検査

🏗️ アーキテクチャ図

ロードバランサーの認可パイプラインでは、REQUEST_AUTHZ によるヘッダー検査、DENY/ALLOW ポリシー評価、CONTENT_AUTHZ によるペイロード深層検査が順に実行されます。CONTENT_AUTHZ はレスポンスパスでも検査を実施できます。

🔢 ポリシー評価順序

ポリシープロファイル対応サービスでは、認可ポリシーの評価が以下の順序で実行されます。

🔒 CUSTOM
(REQUEST_AUTHZ)
→
⛔ DENY
→
✅ ALLOW
→
🔍 CONTENT_AUTHZ
💡 ポイント: REQUEST_AUTHZ のカスタムポリシーが拒否した場合、後続の DENY、ALLOW、CONTENT_AUTHZ は評価されません。ALLOW ポリシーが存在し一致しない場合もリクエストは拒否されます。CONTENT_AUTHZ は全ての前段チェックを通過したリクエストにのみ適用されます。

⚙️ 主要機能

📨 REQUEST_AUTHZ プロファイル
  • HTTP リクエストヘッダーに基づくアクセス判定 (デフォルト)
  • ALLOW / DENY / CUSTOM の 3 アクションをサポート
  • ALLOW / DENY はロードバランサーが直接判定
  • CUSTOM は外部の認可エクステンションに委任
  • ext_proc / ext_authz プロトコルで通信
  • IAP やカスタム認可エンジンへの委任が可能
🔍 CONTENT_AUTHZ プロファイル
  • ヘッダー、ボディ、トレーラーを含むフルペイロードの深層検査
  • CUSTOM アクションのみ (必ず外部委任)
  • ext_proc フルデュプレックスストリーミングモード
  • リクエストだけでなくレスポンスの検査にも対応
  • Model Armor またはカスタムサービスに委任
🛡️ Model Armor 統合
  • CONTENT_AUTHZ + Model Armor で AI ガードレールをネットワーク層に適用
  • プロンプトインジェクション攻撃の検出・ブロック
  • ジェイルブレイク検出
  • 機密データ漏洩防止
  • 有害コンテンツのフィルタリング
  • リクエスト/レスポンス両方にフィルタリングルールを設定可能
📊 評価順序の拡張
  • CUSTOM (REQUEST_AUTHZ) -> DENY -> ALLOW -> CONTENT_AUTHZ の順序で評価
  • 前段チェックで拒否されたリクエストは後段に進まない
  • 段階的なセキュリティチェックが確実に機能

📐 技術仕様: プロファイル比較

項目 REQUEST_AUTHZ CONTENT_AUTHZ

🛠️ 設定方法

ステップ 1: REQUEST_AUTHZ プロファイルで認可ポリシーを作成

YAML REQUEST_AUTHZ プロファイルの例 
# REQUEST_AUTHZ プロファイルの例
name: my-request-authz-policy
target:
  loadBalancingScheme: EXTERNAL_MANAGED
  resources:
    - "https://www.googleapis.com/compute/v1/projects/PROJECT_ID/regions/REGION/forwardingRules/LB_FORWARDING_RULE"
policyProfile: REQUEST_AUTHZ
httpRules:
  - from:
      sources:
        - ipBlocks:
            - prefix: "10.0.0.0"
              length: "24"
    to:
      operations:
        - paths:
            - exact: "/api/sensitive"
action: DENY
Bash 認可ポリシーのインポート 
gcloud beta network-security authz-policies import my-request-authz-policy \
  --source=authz-policy-request.yaml \
  --location=REGION

ステップ 2: Model Armor 認可エクステンションを作成

YAML Model Armor エクステンション定義 
name: my-model-armor-ext
loadBalancingScheme: INTERNAL_MANAGED
service: modelarmor.REGION.rep.googleapis.com
metadata:
  model_armor_settings: '[
    {
      "response_template_id": "projects/PROJECT_ID/locations/REGION/templates/RESPONSE_TEMPLATE_ID",
      "request_template_id": "projects/PROJECT_ID/locations/REGION/templates/REQUEST_TEMPLATE_ID"
    }
  ]'
failOpen: true
Bash 認可エクステンションの作成 
gcloud beta service-extensions authz-extensions import my-model-armor-ext \
  --source=authz-extension-model-armor.yaml \
  --location=REGION

ステップ 3: CONTENT_AUTHZ プロファイルで認可ポリシーを作成

YAML CONTENT_AUTHZ プロファイルの例 (Model Armor 委任) 
# CONTENT_AUTHZ プロファイルの例 (Model Armor 委任)
name: my-content-authz-policy
target:
  loadBalancingScheme: INTERNAL_MANAGED
  resources:
    - "https://www.googleapis.com/compute/v1/projects/PROJECT_ID/regions/REGION/forwardingRules/LB_FORWARDING_RULE"
policyProfile: CONTENT_AUTHZ
action: CUSTOM
customProvider:
  authzExtension:
    resources:
      - "projects/PROJECT_ID/locations/REGION/authzExtensions/my-model-armor-ext"
Bash CONTENT_AUTHZ ポリシーのインポート 
gcloud beta network-security authz-policies import my-content-authz-policy \
  --source=authz-policy-content.yaml \
  --location=REGION
⚠️ 前提条件: Regional external/internal Application Load Balancer が構成済みであること。gcloud beta コンポーネントがインストールされていること。CONTENT_AUTHZ を使用する場合は Model Armor テンプレートまたはカスタムサービスが準備済みであること。

💡 ユースケース

🤖 ユースケース 1: 生成 AI アプリのプロンプトインジェクション防御

シナリオ: 社内向け LLM チャットアプリケーションを Regional internal ALB の背後で運用し、ユーザーからのプロンプトインジェクション攻撃を防止したい。

YAML AI コンテンツセキュリティポリシー 
name: ai-content-security-policy
target:
  loadBalancingScheme: INTERNAL_MANAGED
  resources:
    - "https://www.googleapis.com/compute/v1/projects/my-project/regions/us-central1/forwardingRules/my-llm-app-rule"
policyProfile: CONTENT_AUTHZ
action: CUSTOM
customProvider:
  authzExtension:
    resources:
      - "projects/my-project/locations/us-central1/authzExtensions/model-armor-ext"
✅ 効果: アプリケーションコードを変更せずに、ロードバランサーレベルでプロンプトインジェクション攻撃を検出・ブロック。悪意のあるプロンプトはバックエンドに到達する前に拒否されます。
🌐 ユースケース 2: API ゲートウェイでの多層認可

シナリオ: マイクロサービスアーキテクチャで Regional external ALB を API ゲートウェイとして使用。IP ベースのアクセス制御に加え、リクエストボディの内容に基づく高度なセキュリティチェックを実施したい。

Bash 多層認可の設定 
# ステップ 1: REQUEST_AUTHZ で IP ベースのアクセス制御
gcloud beta network-security authz-policies import ip-deny-policy \
  --source=ip-deny-policy.yaml \
  --location=us-central1

# ステップ 2: CONTENT_AUTHZ でペイロード検査
gcloud beta network-security authz-policies import content-check-policy \
  --source=content-check-policy.yaml \
  --location=us-central1
✅ 効果: REQUEST_AUTHZ と CONTENT_AUTHZ の組み合わせによる多層防御。不正な IP からのアクセスをまずブロックし、通過したリクエストにペイロード検査を実施。評価順序が保証されるため段階的チェックが確実に機能します。
🔒 ユースケース 3: 機密データ漏洩防止

シナリオ: AI アシスタントが社内データベースの情報を参照して回答を生成するアプリケーションにおいて、レスポンスに個人情報や機密データが含まれていないかを検査し漏洩を防止したい。

✅ 効果: CONTENT_AUTHZ はレスポンスパスでも検査を実施できるため、LLM が生成したレスポンスに機密情報が含まれている場合に検出してブロックまたはマスク。Model Armor の Sensitive Data Protection フィルターと組み合わせ、アプリケーション層に依存しないデータ漏洩防止を実装可能。

⚖️ メリット / デメリット

✅ メリット

ビジネス面

  • AI アプリケーションのセキュリティ強化 -- アプリ開発チームの実装負担を軽減
  • コンプライアンス対応の効率化 -- 統一的なセキュリティポリシーを一元適用
  • 運用コスト削減 -- コード変更なしにセキュリティポリシーを適用・更新

技術面

  • レイヤードセキュリティ -- REQUEST_AUTHZ + CONTENT_AUTHZ の多層防御
  • フルデュプレックスストリーミング -- リクエスト/レスポンス双方向のリアルタイム検査
  • 柔軟な委任先選択 -- Model Armor (マネージド) / カスタムサービスから選択

⚠️ デメリット / 制約

制限事項

  • Preview ステータス -- Pre-GA Offerings Terms が適用。本番環境での利用には検証が必要
  • 1 つのポリシーに REQUEST_AUTHZ と CONTENT_AUTHZ の両方は設定不可
  • CONTENT_AUTHZ は CUSTOM アクションのみ。直接 ALLOW / DENY 不可
  • Global external ALB / Cross-region internal ALB は非対応

考慮すべき点

  • 大容量リクエストでのレイテンシ影響 (ペイロード全体転送のため)
  • Model Armor テンプレートの事前作成とリージョン選択が必要
  • gcloud beta コマンド使用 -- 将来的に構文変更の可能性
  • failOpen 設定の慎重な設計が必要
