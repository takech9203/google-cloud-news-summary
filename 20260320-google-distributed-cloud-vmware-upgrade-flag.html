⚙️ Google Distributed Cloud for VMware: --reuse-bootstrap-cluster フラグ必須化

Google Distributed Cloud Important
2026-03-20

📋 概要

Google Distributed Cloud (software only) for VMware のバージョン 1.32 以降において、管理クラスタ (admin cluster) のアップグレードが失敗した場合の再試行手順に重要な変更が加わりました。

失敗後に gkectl upgrade admin を再実行する際、--reuse-bootstrap-cluster フラグの指定が必須となります。このフラグを指定せずに再実行すると、ブートストラップクラスタに保持されている重要なステート情報が失われ、クリティカルなデータ損失が発生する可能性があります。

この変更は、Advanced Clusters アーキテクチャへの移行が進む中で、アップグレードプロセスの安全性を高めるための措置です。対象ユーザーは、Google Distributed Cloud for VMware を運用しているインフラストラクチャ管理者およびプラットフォームエンジニアです。

Before / After 比較

❌ Before (v1.32 未満)

✕ ブートストラップクラスタの扱いに関する明確なガイダンスが不足
✕ 再利用するかどうかがユーザーの判断に委ねられていた
✕ 誤って削除した場合にデータ損失のリスクがあった

✅ After (v1.32 以降)

✓ --reuse-bootstrap-cluster フラグで明示的に再利用を指定
✓ フラグ必須化により誤操作によるデータ損失リスクが軽減
✓ リカバリ手順がより明確かつ安全に

🔑 主要機能

⚠️ 1. --reuse-bootstrap-cluster フラグの必須化

→ バージョン 1.32 以降の管理クラスタアップグレード失敗時、gkectl upgrade admin の再実行に --reuse-bootstrap-cluster フラグが必須
→ フラグを指定しない場合、新しいブートストラップクラスタが作成され、既存のステート情報が失われる可能性がある
→ 対象コマンド: gkectl upgrade admin および gkectl update admin

🔒 2. ブートストラップクラスタのステート保護

→ ブートストラップクラスタには、アップグレードの再開に必要なすべてのステート情報が保持される
→ 失敗時にワークステーションから外部ブートストラップクラスタを削除してはならない
→ スナップショットは自動的に Cloud Storage にアップロードされる

🚀 3. Advanced Clusters へのアップグレード対応

→ バージョン 1.32 から 1.33 へのアップグレード時に自動的に Advanced Clusters に変換
→ 改善されたアーキテクチャにより、より高い柔軟性とスケーラビリティを提供
→ cert-manager が Advanced Clusters に自動インストールされる

📊 技術仕様

ブートストラップクラスタの役割

項目 詳細
実装方式 kind (Kubernetes in Docker) クラスタ
用途 管理クラスタのアップグレード時に Kubernetes コントローラを一時的にホスト
ステート保持 アップグレードの再開に必要なすべてのステート情報を保持
kubeconfig パス .kube/kind-config-gkectl
コンテナ名 gkectl-control-plane
自動スナップショット 失敗時に Cloud Storage バケットに自動アップロード

対象バージョンと要件

項目 詳細
対象バージョン 1.32 以降
影響を受けるコマンド gkectl upgrade admin, gkectl update admin
必須フラグ --reuse-bootstrap-cluster (失敗後の再試行時)
Advanced Clusters 最小バージョン 1.32.0 (cos/cos_cgv2 使用時)、1.31.100 (その他)

📝 設定方法

前提条件

✓ Google Distributed Cloud (software only) for VMware バージョン 1.32 以降がインストールされていること
✓ 管理者ワークステーションに gkectl コマンドラインツールがインストールされていること
✓ 管理クラスタの kubeconfig ファイルおよび構成ファイルにアクセスできること

ステップ 1: 通常のアップグレードの実行

Bash 通常のアップグレードコマンド 
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG_FILE

通常のアップグレードコマンドを実行します。成功した場合、ブートストラップクラスタは自動的にクリーンアップされます。

ステップ 2: アップグレード失敗時の再試行

⚠️ 重要: アップグレードが失敗した場合、ワークステーション上のブートストラップクラスタを絶対に削除しないでください。
Bash 失敗後の再試行コマンド (フラグ必須) 
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG_FILE \
  --reuse-bootstrap-cluster

--reuse-bootstrap-cluster フラグを付けて再実行します。同じバンドルとターゲットバージョンを使用してください。

ステップ 3: ブートストラップクラスタの状態確認 (必要に応じて)

Bash トラブルシューティング用コマンド 
# ブートストラップクラスタの Pod 状態を確認
kubectl --kubeconfig /home/ubuntu/.kube/kind-config-gkectl get pods -n kube-system

# ブートストラップクラスタのログを確認
docker exec -it gkectl-control-plane bash

🎯 メリット

💼 ビジネス面

→ ダウンタイムの最小化: アップグレード失敗時のリカバリが確実になることで、管理クラスタの復旧時間を短縮
→ 運用リスクの低減: データ損失の可能性が低下し、アップグレード作業に対する信頼性が向上

💻 技術面

→ ステート保護の強制: フラグの必須化により、ステート情報の誤った破棄を防止
→ リカバリの確実性: 中断した正確なポイントから再開可能
→ デバッグ情報の保持: スナップショットが Cloud Storage に自動保存

⚠️ 制約事項・注意点

🚨 制限事項
✕ --reuse-bootstrap-cluster フラグはバージョン 1.32 以降でのみ有効
✕ gcloud CLI、Google Cloud コンソール、Terraform からのアップグレードには適用されない (gkectl コマンドのみ)
✕ ブートストラップクラスタがワークステーションのリソース (CPU、メモリ、ディスク) を消費し続ける
🚫 禁止事項
✕ アップグレード失敗後にブートストラップクラスタを削除しないこと (リカバリが困難になる)
✕ gkectl repair admin-master を失敗したアップグレードの後に実行しないこと (クラスタが不正な状態になる)
💡 考慮すべき点
→ Advanced Clusters への自動変換 (1.32 -> 1.33) と組み合わせる場合、事前に cert-manager の既存設定を確認すること

📚 ユースケース

🔌 ユースケース 1: Advanced Clusters への移行中のアップグレード失敗対応

シナリオ: バージョン 1.32 の管理クラスタを 1.33 にアップグレードして Advanced Clusters に移行する際、ネットワーク障害によりアップグレードが中断された。

Bash Advanced Clusters 移行時のリカバリ例 
# 1. アップグレードの初回実行 (失敗)
gkectl upgrade admin \
  --kubeconfig /home/ubuntu/kubeconfig \
  --config /home/ubuntu/admin-cluster-config.yaml

# 2. 失敗後: ブートストラップクラスタは削除しない
# 3. ネットワーク問題を解決した後、フラグ付きで再実行
gkectl upgrade admin \
  --kubeconfig /home/ubuntu/kubeconfig \
  --config /home/ubuntu/admin-cluster-config.yaml \
  --reuse-bootstrap-cluster
✅ 効果: ブートストラップクラスタに保持されたステート情報を利用して、中断されたポイントからアップグレードを安全に再開できる。

🛡️ ユースケース 2: パッチリリースの適用失敗時のリカバリ

シナリオ: セキュリティパッチ適用のために 1.32.x から 1.32.900 へのアップグレードを実施したが、プリフライトチェックの失敗によりアップグレードが停止した。

✅ 効果: プリフライトチェックの問題を修正した後、--reuse-bootstrap-cluster フラグ付きで再実行することで、既存のステート情報を維持したまま安全にリカバリできる。

📋 まとめ

Google Distributed Cloud (software only) for VMware バージョン 1.32 以降で、管理クラスタのアップグレード失敗後の再試行時に --reuse-bootstrap-cluster フラグが必須となりました。

この変更により、ブートストラップクラスタに保持されたステート情報の誤った破棄によるデータ損失リスクが大幅に低減されます。

💡 推奨アクション: 対象バージョンを運用しているチームは、アップグレード手順書にこのフラグの使用を明記し、ブートストラップクラスタの削除禁止ルールを徹底することを推奨します。
