☁️ Cortex Framework: Airflow v3 対応と複数の修正

Cortex Framework Release 6.3.4 Airflow v3
2026-02-27
Fixed / Announcement

📋 概要

Google Cloud Cortex Framework Data Foundation v6.3.4 がリリースされた。本リリースの最大の変更点は、新規デプロイメントで生成される Cloud Composer DAG が Apache Airflow v3 に対応したことである。Airflow v2 は引き続きサポートされるが、End-of-Life が近づいているため、移行計画の策定が推奨されている。

SAP ワークロードにおいては、CDC スクリプトがスラッシュを含むカスタムテーブルに対応し、マーケティングワークロードでは Meta Marketing API が v21 から v25 にアップグレードされた。その他、gcloud storage CLI への移行やバグ修正も含まれている。

🔄 Before / After 比較

❌ アップデート前の課題

  • → Composer DAG は Airflow v2 のみ対応、v3 環境で互換性問題が発生
  • → CDC スクリプトがスラッシュ (/) を含むカスタムテーブルを処理不可
  • → Meta Marketing API が v21 で動作、新機能・フィールド変更に未対応
  • → 旧式の gsutil コマンドを使用
  • → DAG インポート時の改行問題
  • → バケットアクセス検証の不十分なエラーハンドリング

✅ アップデート後の改善

  • → Airflow v3 互換の DAG を生成 (v2 も引き続きサポート)
  • → スラッシュ含むカスタムテーブル名を正しく処理
  • → Meta Marketing API v25 に対応、最新仕様に追従
  • → gcloud storage CLI に移行、推奨ツールチェーンに準拠
  • → DAG パッケージインポートの改行問題を修正
  • → バケットアクセス検証のエラーハンドリング改善

⚡ 主要機能

1. Airflow v3 対応 DAG 生成

→新規デプロイメントで生成されるすべての Cloud Composer DAG が Airflow v3 と互換性を持つように更新
→Airflow v2 は引き続きサポートされるが、EOL が近づいているため移行計画の策定を推奨
→既存デプロイメントの DAG は自動更新されない。新規デプロイメントまたは再デプロイメントが必要

2. SAP CDC カスタムテーブル対応

→テーブル名にスラッシュ (/) を含むカスタムテーブル (例: /BIC/AZFIE0100, /SAPPO/POBJ) をサポート
→オプションの adrt テーブル (アドレス短縮テキスト) が SAP CDC テーブルリストに追加
→既存デプロイメントには影響なし。新規デプロイメントまたは再デプロイメント時に適用

3. Meta Marketing API v25 アップグレード

→Meta データソース用テンプレートが v21 から v25 にアップグレード
→一部のフィールド名が変更されている。詳細は Meta API Changelog を参照
→v21 から v25 の間に導入された新機能や改善が利用可能に

4. gcloud storage CLI への移行

→旧式 gsutil から gcloud storage CLI に移行
→パフォーマンス向上、Google Cloud 推奨ツール

5. DAG インポートの修正

→DAG パッケージインポート時の改行処理バグを修正
→DAG 読み込みエラーが解消

6. バケットアクセス検証の改善

→エラーハンドリングが改善
→権限不足やバケット不在がより明確に報告

🔧 設定方法

前提条件

✓Cortex Framework Data Foundation リポジトリのクローン
✓Cloud Composer (Airflow) 環境のセットアップ
✓BigQuery データセットの準備 (Raw, CDC, Reporting)
✓各データソースへの接続設定 (SAP, Meta 等)

Step 1: リポジトリの更新

Bash リポジトリを最新版に更新 
# Cortex Data Foundation リポジトリを最新版に更新
cd cortex-data-foundation
git pull origin main
git checkout v6.3.4

Step 2: デプロイメント設定の確認

JSON config.json - Meta ワークロード設定例 
{
  "marketing": {
    "deployMeta": true,
    "Meta": {
      "deployCDC": true,
      "datasets": {
        "cdc": "CDC_Meta",
        "raw": "RAW_Meta",
        "reporting": "REPORTING_Meta"
      }
    }
  }
}

Step 3: デプロイメントの実行

Bash Cloud Build でデプロイメント 
# Cloud Build でデプロイメントを実行
gcloud builds submit \
  --substitutions=_GCS_BUCKET=LOGS_BUCKET_NAME,_BUILD_ACCOUNT='projects/SOURCE_PROJECT/serviceAccounts/CLOUD_BUILD_SA@SOURCE_PROJECT.iam.gserviceaccount.com'

Step 4: DAG のコピー

Bash DAG を Cloud Composer バケットにコピー 
# 生成された DAG を Cloud Composer バケットにコピー
gcloud storage -m cp -r gs://OUTPUT_BUCKET/dags/ gs://COMPOSER_DAG_BUCKET/
gcloud storage -m cp -r gs://OUTPUT_BUCKET/data/ gs://COMPOSER_DAG_BUCKET/

✅ メリット

📈 ビジネス面

→将来への対応: Airflow v3 対応により Cloud Composer 最新バージョンへのスムーズな移行が可能、v2 の EOL に備えられる
→SAP データ統合の拡大: スラッシュを含むカスタムテーブル対応で、より多くの SAP データを取り込み可能に
→マーケティングデータの最新化: Meta API v25 対応で最新の広告分析機能とデータフィールドを活用可能

🔧 技術面

→互換性の向上: Airflow v3 と v2 の両方をサポートし段階的な移行が可能
→ツールチェーンの最新化: gcloud storage CLI への移行でパフォーマンス向上と推奨プラクティス準拠
→安定性の向上: DAG インポート修正とエラーハンドリング改善で運用信頼性向上

⚠️ デメリット・制約事項

⚠️ 制限事項:
  • Airflow v3 対応 DAG は新規デプロイメントでのみ生成される。既存 DAG を更新するには再デプロイメントが必要
  • SAP CDC のスラッシュ対応も同様に、既存デプロイメントには自動適用されない
  • Meta API v25 へのアップグレードに伴い、v21 からのフィールド名変更がある。既存のカスタムレポートやクエリの更新が必要になる場合がある
💡 考慮すべき点:
  • Airflow v2 の EOL が近づいているため、v3 への移行計画を早期に策定すること
  • Meta API Changelog を事前に確認し、フィールド名変更の影響範囲を特定すること
  • SAP カスタムテーブルの命名規則を確認し、スラッシュを含むテーブルが適切に設定されているか検証すること
  • Cloud Composer 2 から 3 への移行はサイドバイサイド移行が必要 (インプレースアップグレード不可)

🎯 ユースケース

🏭 ユースケース 1: SAP カスタムテーブルを含む CDC パイプラインの構築

シナリオ: SAP S/4HANA システムで /BIC/ プレフィックスの BW テーブルや /SAPPO/ プレフィックスのカスタムテーブルを使用しており、これらの変更データを BigQuery に連携したい。

YAML cdc_settings.yaml 
SAP:
  - base_table: "/BIC/AZFIE0100"
    raw_table: "bic_azfie0100"
    load_frequency: "@daily"
    row_identifiers: "MANDT,KEY_FIELD"
  - base_table: "adrt"
    raw_table: "adrt"
    load_frequency: "@daily"
    row_identifiers: "MANDT,ADDRNUMBER,DATE_FROM,NATION,ADDR_GROUP"

効果: これまで手動でスラッシュを含むテーブル名を変換する必要があったが、Cortex Framework が自動処理するようになり効率化される。

🚀 ユースケース 2: Airflow v3 環境での Cortex Framework デプロイメント

シナリオ: Cloud Composer 3 (Airflow v3) 環境を新規構築し、Cortex Framework を使用して SAP と Meta のデータを BigQuery に統合したい。

効果: Release 6.3.4 により、新規デプロイメントで生成される DAG が Airflow v3 と互換性を持つため、Cloud Composer 3 環境でそのまま利用できる。Airflow v2 用の DAG を手動で変換する手間が不要に。

📊 ユースケース 3: Meta マーケティングデータの最新 API 対応

シナリオ: Meta (Facebook/Instagram) の広告キャンペーンデータを Cortex Framework 経由で BigQuery に取り込み、Looker でマーケティングダッシュボードを構築している。API v25 の新機能を活用したい。

効果: Meta Marketing API v25 への対応により、最新のキャンペーン分析フィールドやインサイトデータが利用可能に。再デプロイで最新 API 仕様に自動対応。

📝 まとめ

Cortex Framework Release 6.3.4 は、Airflow v3 対応、SAP CDC カスタムテーブルサポート、Meta API v25 アップグレードなど複数の重要な改善を含むリリースである。

推奨アクション:
  • Cloud Composer 環境で Cortex Framework を運用しているユーザーは、Airflow v3 への移行計画を早期に策定し、新規デプロイメントまたは再デプロイメントにより v3 対応 DAG を取得すること
  • Meta マーケティングデータを利用しているユーザーは、API v25 へのアップグレードに伴うフィールド名変更を Meta API Changelog で確認し、カスタムクエリやレポートへの影響を事前に評価すること
