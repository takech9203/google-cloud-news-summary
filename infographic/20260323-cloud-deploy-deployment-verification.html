Cloud Deploy Preview
2026-03-23

📋 概要

Google Cloud Deploy に、デプロイ後のアプリケーションパフォーマンスを任意のモニタリングプラットフォームで分析し、 検証結果に基づいてロールバックなどのアクションを自動トリガーする機能が Preview として追加されました。

既存のデプロイ検証 (Deployment Verification) と自動化ルール (Automation Rules) を組み合わせ、 CI/CD パイプライン全体の安全性と信頼性を大幅に向上させます。DevOps エンジニアや SRE チームに特に有用な機能です。

🔄 Before / After 比較

❌ アップデート前

  • デプロイ後の検証は手動またはスクリプトが必要
  • ロールバック判断・実行が人手で対応遅延リスク
  • モニタリングとデプロイパイプラインが分断され MTTR が長期化

✅ アップデート後

  • 任意のモニタリングで自動パフォーマンス分析
  • 検証失敗時に自動ロールバックで迅速復旧
  • repairRolloutRule でリトライ+ロールバックの自動化

⚡ 主要機能

1. デプロイ検証 (Deployment Verification)

→ Cloud Deploy と Skaffold を構成し、ターゲットにデプロイしたアプリケーションが正常に動作しているかを自動検証
→ 任意のテストイメージを使用してデプロイ完了後にテストを実行
→ Cloud Deploy 実行環境またはアプリケーションクラスタ上 (GKE のみ) で検証コンテナを実行可能
→ 複数の検証コンテナを指定した場合は並列実行

2. 自動ロールバック (repairRolloutRule)

→ ロールアウト失敗時に指定回数のリトライを自動実行
→ 全リトライ失敗後、最後に成功したリリースへ自動ロールバック
→ リトライ間隔を LINEAR または EXPONENTIAL に設定可能
→ 特定フェーズやジョブのみをリトライ対象に指定可能

3. 自動化ルール (Automation Rules)

デリバリーパイプラインごとに最大 250 の自動化ルールを設定可能:

→ promoteReleaseRule: 成功後の自動プロモーション
→ advanceRolloutRule: カナリアデプロイメントのフェーズ自動進行
→ timedPromoteReleaseRule: cron スケジュールに基づく自動プロモーション
→ repairRolloutRule: 自動リトライとロールバック

🔧 技術仕様

デプロイ検証の設定

デリバリーパイプラインのステージ定義で verify: true を設定することで、対象ターゲットでのデプロイ検証が有効になります。

項目詳細
対応ターゲットGKE、GKE 接続クラスタ、Cloud Run
検証実行環境Cloud Deploy 実行環境 (デフォルト) またはアプリケーションクラスタ (GKE のみ)
検証コンテナ複数指定可能 (並列実行)
成功判定コンテナの終了コード (0: 成功、非 0: 失敗)
Skaffold バージョンskaffold/v3alpha1 以降が必要

repairRolloutRule 設定例

YAML Automation リソース定義 
apiVersion: deploy.cloud.google.com/v1
kind: Automation
metadata:
  name: auto-repair
  description: Automatically repair failed rollouts
suspended: false
serviceAccount: <SERVICE_ACCOUNT>
selector:
  targets:
    - id: <TARGET_ID>
rules:
  - repairRolloutRule:
      id: "repair-rollout"
      repairPhases:
        - retry:
            attempts: 3
            wait: 1m
            backoffMode: EXPONENTIAL
        - rollback:
            destinationPhase: "stable"
            disableRollbackIfRolloutPending: false

Skaffold 検証設定例

YAML skaffold.yaml - verify スタンザ 
apiVersion: skaffold/v4beta7
kind: Config
manifests:
  rawYaml:
    - kubernetes.yaml
deploy:
  kubectl: {}
verify:
  - name: verify-integration-test
    container:
      name: integration-test
      image: integration-test
      command: ["./test-systems.sh"]

📝 設定方法

📌 前提条件: Cloud Deploy API の有効化、デリバリーパイプライン/ターゲットの定義済み、Skaffold v3alpha1 以降の設定ファイル、検証用テストイメージの準備

ステップ 1: パイプラインで検証を有効化

YAML DeliveryPipeline - verify: true 
apiVersion: deploy.cloud.google.com/v1
kind: DeliveryPipeline
metadata:
  name: my-app-pipeline
serialPipeline:
  stages:
    - targetId: dev
      profiles: []
      strategy:
        standard:
          verify: true
    - targetId: staging
      profiles: []
      strategy:
        standard:
          verify: true
    - targetId: prod
      profiles: []
      strategy:
        standard:
          verify: true

ステップ 2: Skaffold に検証コンテナを設定

YAML skaffold.yaml - smoke test 
verify:
  - name: smoke-test
    container:
      name: smoke-test
      image: my-test-image:latest
      command: ["/bin/sh"]
      args: ["-c", "run-smoke-tests.sh"]

ステップ 3: 自動修復ルールを適用

Bash Automation リソースの適用 
# Automation リソースを適用
gcloud deploy apply --file=automation.yaml --region=<REGION>

ステップ 4: パイプラインを登録して検証

Bash リリース作成と検証フロー確認 
# デリバリーパイプラインを登録
gcloud deploy apply --file=clouddeploy.yaml --region=<REGION>

# リリースを作成して検証フローを確認
gcloud deploy releases create test-release-001 \
  --delivery-pipeline=my-app-pipeline \
  --region=<REGION> \
  --images=my-app-image=gcr.io/<PROJECT>/my-app:latest

✅ メリット

ビジネス面

→ MTTR の短縮: デプロイ失敗からの復旧が自動化、サービス停止時間を大幅削減
→ 運用コスト削減: 手動ロールバック判断・実行が不要、オンコール負荷の軽減

技術面

→ パイプラインの回復力向上: リトライ+ロールバック自動化で一時的障害に耐性
→ 柔軟な検証戦略: 任意のモニタリング・テストフレームワークと統合
→ 段階的導入: ターゲットごとに検証の有効/無効を設定可能

⚠️ デメリット・制約事項

制限事項
  • Preview 段階のため、本番環境での使用は慎重に検討が必要
  • Cloud Run では検証コンテナをアプリケーションクラスタ上で実行不可 (Cloud Deploy 実行環境のみ)
  • パイプラインあたりの自動化ルールは最大 250 個
💡 考慮すべき点
  • テストスクリプトの品質がパイプライン全体の信頼性に影響 (終了コードで判定)
  • リトライ設定 (回数・間隔・バックオフモード) は環境ごとに適切なチューニングが必要
  • 自動ロールバック時の通知設定 (Pub/Sub) を事前に構成しておくことを推奨

🎯 ユースケース

1. マイクロサービスのカナリアデプロイメントとヘルスチェック

シナリオ: GKE 上のマイクロサービスをカナリアデプロイメントで段階的にリリースし、各段階でエンドポイントのヘルスチェックとレイテンシテストを自動実行。

YAML skaffold.yaml - ヘルスチェック検証 
verify:
  - name: health-check
    container:
      name: health-check
      image: curlimages/curl:latest
      command: ["/bin/sh"]
      args: ["-c", "curl -f http://my-service:8080/health || exit 1"]
    executionMode:
      kubernetesCluster: {}

効果: デプロイ後にサービスの正常性が自動検証され、問題がある場合は自動リトライとロールバックにより安全に前バージョンに復元。

2. 本番環境への段階的リリースと自動保護

シナリオ: ステージング環境でのテスト後に本番環境へ自動プロモーションを行い、本番でもデプロイ検証を実施。検証失敗時は自動ロールバックで保護。

効果: promoteReleaseRule と repairRolloutRule を組み合わせることで、完全自動化されたデリバリーパイプラインを実現しつつ、安全性を担保。

Cloud Deploy のデプロイ検証と自動ロールバックアクション機能は、CI/CD パイプラインの安全性と自動化レベルを大幅に引き上げるアップデートです。 任意のモニタリングプラットフォームとの統合により、既存のテスト基盤を活かしながらデプロイの品質ゲートを自動化できます。
