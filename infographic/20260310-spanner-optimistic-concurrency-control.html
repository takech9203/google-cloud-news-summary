🗄️ Spanner: 楽観的同時実行制御

Spanner GA Database
2026-03-10

📋 概要

Spanner に楽観的同時実行制御 (Optimistic Concurrency Control) モードが追加されました。読み書きの競合が少ないトランザクションワークロードに最適化された同時実行制御方式で、トランザクション内の読み取りおよびクエリがロックを取得せずに実行されるため、同時実行性が大幅に向上します。

この機能は Repeatable Read 分離レベルを通じて実装されており、スナップショット分離 (Snapshot Isolation) の技術を活用しています。トランザクション間の競合がまれであることを前提として動作し、コミット時にのみ競合チェックが行われます。

💡 対象ユーザー: Spanner 上で高い同時実行性を持つアプリケーションを運用しており、特に読み書き競合が少ないワークロードを扱う開発者やデータベース管理者

🔄 Before / After

❌ Before: 悲観的ロック方式のみ

  • 読み取り操作が共有ロックを取得するため、高い同時実行性でロック競合が発生
  • ロック競合によりトランザクションのアボート率が上昇し、リトライ増加
  • デッドロック検出 (wound-wait) により、新しいトランザクションがアボートされるケース

✅ After: 楽観的同時実行制御

  • 読み取りがロックなしで実行、同時実行性が大幅に向上
  • ロック競合によるアボートが減少し、スループット改善
  • 読み取りヘビーなワークロードで読み書き / 読み取り専用トランザクションの性能差が縮小

📊 アーキテクチャ図: 悲観的 vs 楽観的同時実行制御

悲観的同時実行制御では読み取り時にロックを取得するため待機が発生しますが、楽観的同時実行制御ではロックなしで読み取りが進み、コミット時にのみ競合を検証します。

    C1->>S: WRITE row X
    C2->>S: WRITE row Y
    S-->>C1: コミット時に競合チェック - 成功
    S-->>C2: コミット時に競合チェック - 成功
    end

⚡ 主要機能

🔓 ロックフリーの読み取り

読み書きトランザクション内の読み取りおよびクエリがロックを取得せずに実行。トランザクション開始時点のデータベースの一貫したスナップショットに基づいて読み取りが行われる。

🔍 コミット時の競合検出

書き込みセットがトランザクションのスナップショットタイムスタンプからコミットタイムスタンプの間に変更されていないことを検証。競合検出時のみアボート。

🔗 Repeatable Read 統合

楽観的同時実行制御は Repeatable Read 分離レベルを指定することで有効化。クライアントレベルまたはトランザクションレベルで設定可能。

🛡️ 書き込みスキュー防止

SELECT...FOR UPDATE 句または lock_scanned_ranges=exclusive ヒントを使用して、Repeatable Read で発生しうる書き込みスキュー異常を防止。

📐 技術仕様: 分離レベルの比較

項目 シリアライザブル (悲観的) Repeatable Read (楽観的)
デフォルト はい いいえ
読み取り時のロック 共有ロックを取得 ロックなし
書き込み時のロック 排他ロックに昇格 コミット時に排他ロック取得
一貫性保証 外部一貫性 (最強) スナップショット分離
書き込みスキュー 発生しない 発生しうる (対策が必要)
ロック競合時の挙動 デッドロック検出 (wound-wait) コミット時に競合検出
推奨ワークロード 高い一貫性が必要なケース 読み書き競合が少ないケース

🔧 設定方法

前提条件

  • Google Cloud プロジェクトに Spanner インスタンスが作成済みであること
  • Spanner クライアントライブラリが最新バージョンに更新されていること
  • 適切な IAM 権限 (spanner.databases.beginOrRollbackReadWriteTransaction) が付与されていること

ステップ 1: クライアントライブラリの更新

Bash クライアントライブラリの更新 
# Python の場合
pip install --upgrade google-cloud-spanner

# Node.js の場合
npm install @google-cloud/spanner@latest

# Java の場合 (Maven)
# pom.xml の google-cloud-spanner 依存関係を最新バージョンに更新

ステップ 2: トランザクションの分離レベルを設定

Python クライアントレベル / トランザクションレベルの設定 
from google.cloud import spanner
from google.cloud.spanner_v1 import TransactionOptions

# クライアントレベルでの設定
spanner_client = spanner.Client(
    default_transaction_options=DefaultTransactionOptions(
        isolation_level=TransactionOptions.IsolationLevel.REPEATABLE_READ
    )
)

instance = spanner_client.instance("your-instance-id")
database = instance.database("your-database-id")

# トランザクションレベルでの設定
def update_data(transaction):
    results = transaction.execute_sql(
        "SELECT AlbumTitle FROM Albums WHERE SingerId = 1 AND AlbumId = 1"
    )
    for result in results:
        print("Current Title: {}".format(*result))

    row_ct = transaction.execute_update(
        "UPDATE Albums SET AlbumTitle = 'New Title' "
        "WHERE SingerId = 1 AND AlbumId = 1"
    )
    print("{} record(s) updated.".format(row_ct))

database.run_in_transaction(
    update_data,
    isolation_level=TransactionOptions.IsolationLevel.REPEATABLE_READ
)

ステップ 3: REST API での設定

JSON REST API トランザクションオプション 
{
  "options": {
    "readWrite": {},
    "isolationLevel": "REPEATABLE_READ"
  }
}

ステップ 4: 書き込みスキュー対策 (必要な場合)

SQL SELECT...FOR UPDATE による書き込みスキュー防止 
-- 書き込みが読み取りに依存する場合は SELECT...FOR UPDATE を使用
SELECT Balance FROM Accounts WHERE AccountId = @accountId FOR UPDATE;
⚠️ 注意: アプリケーションレベルの整合性制約に依存する書き込みがある場合は、SELECT...FOR UPDATE 句を使用して書き込みスキューを防止してください。

⚖️ メリット / デメリット

✅ メリット

ビジネス面

  • スループット向上: ロック競合削減で同時処理数が増加
  • レイテンシ改善: ロック待機時間削減とリトライ減少でエンドユーザー体感速度が改善

技術面

  • ロックフリー読み取り: 読み取りヘビーなワークロードで同時実行性が大幅向上
  • シンプルな設定: 分離レベル指定のみで有効化、既存コード変更は最小限
  • 柔軟なロック制御: SELECT...FOR UPDATE で楽観的と悲観的を組み合わせ可能

⚠️ デメリット / 制約事項

制限事項

  • 書き込みスキュー異常が発生する可能性あり (対策が必要)
  • パーティション DML トランザクションでは Repeatable Read を設定不可
  • チェック制約を持つスキーマで制約検証の既知問題あり
  • 同時スキーマ変更中に DML リトライエラーや DEADLINE_EXCEEDED の可能性

考慮すべき点

  • シリアライザブルの外部一貫性保証が不要であることを確認の上で採用
  • クリティカルセクションでは SELECT...FOR UPDATE を強く推奨

💡 ユースケース

🛒 ユースケース 1: 読み取りヘビーな在庫管理システム

シナリオ: EC サイトの在庫管理システムで、商品の在庫数を頻繁に参照しつつ、注文処理時にのみ在庫を更新するワークロード。多数の在庫参照リクエストと少数の在庫更新が同時に発生。

Python 在庫管理の実装例 
def process_order(transaction):
    # 在庫の読み取り (ロックなし)
    results = transaction.execute_sql(
        "SELECT stock_count FROM Products WHERE product_id = @pid",
        params={"pid": product_id},
        param_types={"pid": spanner.param_types.INT64}
    )
    stock = list(results)[0][0]

    if stock > 0:
        # 在庫の更新 (コミット時に競合チェック)
        transaction.execute_update(
            "UPDATE Products SET stock_count = stock_count - 1 "
            "WHERE product_id = @pid",
            params={"pid": product_id},
            param_types={"pid": spanner.param_types.INT64}
        )

database.run_in_transaction(
    process_order,
    isolation_level=TransactionOptions.IsolationLevel.REPEATABLE_READ
)

✅ 効果: 大量の在庫参照リクエストがロックなしで処理されるため、注文処理中でも在庫参照のレイテンシが悪化しない。

👤 ユースケース 2: ユーザープロファイルの更新

シナリオ: ソーシャルメディアアプリケーションで、ユーザーが自身のプロファイル情報を更新。各ユーザーが自身のプロファイルのみを更新するため、異なるユーザー間での書き込み競合はほぼ発生しない。

✅ 効果: ユーザー数の増加に伴い同時トランザクション数が増加しても、ロック競合によるパフォーマンス劣化を回避。読み取りと更新が同一トランザクション内で効率的に処理される。

📝 まとめ

Spanner の楽観的同時実行制御モードは、読み書き競合が少ないワークロードにおいてスループットとレイテンシを大幅に改善する重要なアップデートです。既存のトランザクションコードを大きく変更することなく、分離レベルの指定のみで有効化できるため、導入のハードルが低い点も魅力的です。ただし、書き込みスキュー異常のリスクを理解した上で、必要に応じて SELECT...FOR UPDATE を使用するなど、アプリケーション設計を見直すことを推奨します。
