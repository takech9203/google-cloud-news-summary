🔐 Cloud Run: IAP 直接設定が GA

Cloud Run Identity-Aware Proxy GA
2026-03-13

📋 概要

Cloud Run サービスに Identity-Aware Proxy (IAP) を直接設定する機能が General Availability (GA) となりました。これにより、ロードバランサーを構成することなく、Cloud Run サービスへのアクセスを IAP で保護できるようになります。

IAP は Google Cloud のゼロトラスト セキュリティの中核を担うサービスであり、ユーザーの ID とリクエストのコンテキストに基づいてアプリケーションへのアクセスを制御します。2025 年 4 月に Preview として公開されたこの機能が、本番環境での利用が正式にサポートされるステータスへ昇格しました。

💡 ポイント: 社内向け Web アプリケーションや管理ツールを Cloud Run で運用するすべてのユーザーにとって重要なアップデートです。特にロードバランサーの構成・管理コストを避けたい小規模チームや、迅速にゼロトラスト アクセス制御を導入したいエンタープライズ環境に大きな価値を提供します。

🔄 Before / After 比較

❌ Before (従来)

  • 外部 HTTPS ロードバランサーの構成が必須
  • SSL 証明書、静的 IP、バックエンドサービス等の管理が必要
  • デフォルトの run.app URL は IAP で保護されない
  • ロードバランサーの運用コスト ($18/月~) が発生
  • シンプルな社内ツールでも複雑なネットワーク構成が必要

✅ After (現在)

  • Cloud Run に直接 IAP を有効化、ロードバランサー不要
  • run.app URL を含むすべての Ingress パスが保護
  • Console / gcloud CLI / Terraform で簡単設定
  • ロードバランサー関連の追加コスト不要
  • GA により SLA 対象、本番ワークロードで安心利用

⚡ 主要機能

🛡️ ロードバランサー不要の IAP 保護

  • --iap フラグを指定するだけで IAP を有効化
  • 外部 HTTPS LB、SSL 証明書、静的 IP が不要
  • run.app URL を含むすべての Ingress パスを自動保護

🎯 コンテキストアウェア アクセスポリシー

  • ユーザー ID、グループメンバーシップ
  • デバイスセキュリティ、ロケーション、IP アドレス
  • Access Context Manager との連携

🔧 複数の設定方法に対応

  • Google Cloud Console: UI でワンクリック有効化
  • gcloud CLI: --iap フラグ
  • Terraform: iap_enabled = true

🔒 IAM との二重チェック

  • IAP チェックが先に実行
  • IAP サービスアカウントが IAM チェックを実行
  • 多層防御によるセキュリティ強化

🔧 技術仕様

必要な IAM ロール

ロール 用途
roles/run.admin IAP の有効化・Cloud Run サービスの管理
roles/iap.admin IAP アクセスポリシーの管理
roles/artifactregistry.reader コンテナイメージの読み取り (デプロイ時)
roles/iam.serviceAccountUser サービス ID の使用 (デプロイ時)
roles/run.invoker IAP サービスエージェントに自動付与

既知の制限事項

項目 詳細
組織要件 プロジェクトは Google Cloud 組織内に存在する必要がある
ID の制限 同一組織内の ID のみ利用可能 (外部ユーザーには OAuth カスタム設定が必要)
IAP の二重設定 LB と Cloud Run の両方で IAP を有効化することはできない
統合サービスへの影響 Pub/Sub など一部の統合が動作しなくなる可能性
Cloud CDN IAP は Cloud CDN と互換性がない

📝 設定方法

前提条件: Google Cloud プロジェクトが組織内に存在し、IAP API が有効化され、必要な IAM ロールが付与されていること。

Step 1: IAP API の有効化

Bash IAP API を有効化 
gcloud services enable iap.googleapis.com --project=PROJECT_ID

Step 2: IAP を有効化して Cloud Run サービスをデプロイ

Bash 新規サービスの場合 
# 新規サービスの場合
gcloud beta run deploy SERVICE_NAME \
    --region=REGION \
    --image=IMAGE_URL \
    --no-allow-unauthenticated \
    --iap
Bash 既存サービスの場合 
# 既存サービスの場合
gcloud beta run services update SERVICE_NAME \
    --region=REGION \
    --iap

Step 3: IAP サービスエージェントに Invoker ロールを付与

Bash IAP サービスエージェントへの権限付与 
gcloud run services add-iam-policy-binding SERVICE_NAME \
    --region=REGION \
    --member=serviceAccount:service-PROJECT_NUMBER@gcp-sa-iap.iam.gserviceaccount.com \
    --role=roles/run.invoker

Step 4: Terraform の場合

HCL Terraform 設定例 
resource "google_cloud_run_v2_service" "default" {
  provider     = google-beta
  name         = "cloudrun-iap-service"
  location     = "europe-west1"
  ingress      = "INGRESS_TRAFFIC_ALL"
  launch_stage = "BETA"
  iap_enabled  = true

  template {
    containers {
      image = "us-docker.pkg.dev/cloudrun/container/hello"
    }
  }
}

resource "google_cloud_run_v2_service_iam_member" "iap_invoker" {
  provider = google-beta
  project  = google_cloud_run_v2_service.default.project
  location = google_cloud_run_v2_service.default.location
  name     = google_cloud_run_v2_service.default.name
  role     = "roles/run.invoker"
  member   = "serviceAccount:service-PROJECT_NUMBER@gcp-sa-iap.iam.gserviceaccount.com"
}

✅ メリット

💰 ビジネス面

  • 大幅なコスト削減: 外部 HTTPS LB の転送ルール料金 (約 $18/月~) とデータ処理料金が不要
  • 運用負荷の軽減: LB、SSL 証明書、静的 IP、バックエンドサービスなどの管理が不要
  • 迅速なゼロトラスト導入: 数分で IAP を有効化しゼロトラストモデルへ移行
  • 本番環境での信頼性: GA により SLA 対象、エンタープライズ利用を正式サポート

🔧 技術面

  • アーキテクチャの簡素化: LB 層が不要でネットワーク構成がシンプルに
  • run.app URL の保護: 従来 LB 方式では保護されなかった URL も IAP で保護
  • 統一的なアクセス制御: LB 経由と run.app 経由の両方に同一 IAP ポリシー適用
  • コンテキストアウェア アクセス: デバイスセキュリティやロケーションに基づく高度な制御

⚠️ デメリット・制約事項

制限事項:
  • プロジェクトは Google Cloud 組織内に存在する必要があり、個人アカウントでは利用不可
  • 利用可能な ID は同一組織内のものに限定。外部ユーザーにはカスタム OAuth 設定が必要
  • LB と Cloud Run の両方で IAP を有効化することはできない
  • Pub/Sub など一部の Google Cloud サービスからの統合が正常に動作しなくなる可能性
考慮すべき点:
  • IAP は認証処理のためにレイテンシを増加させる (レイテンシに敏感なサービスは慎重に検討)
  • Cloud CDN との併用はサポートされていない
  • Pub/Sub などのサービスが IAP に対して正しく認証できない場合がある
  • X-Serverless-Authorization ヘッダーの扱いに注意が必要

💡 ユースケース

🏢 ユースケース 1: 社内管理ツールの保護

社内向けの管理ダッシュボードを Cloud Run でホストし、Google Workspace アカウントを持つ社員のみにアクセスを制限したい場合。

Bash 管理ダッシュボードを IAP 付きでデプロイ 
# 管理ダッシュボードを IAP 付きでデプロイ
gcloud beta run deploy admin-dashboard \
    --region=asia-northeast1 \
    --image=gcr.io/my-project/admin-dashboard:latest \
    --no-allow-unauthenticated \
    --iap

効果: LB 構成なしで、数分で社内ツールにゼロトラスト アクセス制御を導入。run.app URL も保護されるため、URL を知っているだけではアクセス不可。

🌍 ユースケース 2: マルチリージョン構成での使い分け

単一リージョンの Cloud Run サービスには直接 IAP を使用し、複数リージョンにまたがるサービスには LB 経由の IAP を使用するハイブリッド構成。

効果: サービスの規模やリージョン要件に応じて最適な IAP 構成を選択でき、コストとアーキテクチャの複雑さを最小限に抑えられる。

🧪 ユースケース 3: 開発・ステージング環境の保護

開発チームが使用するステージング環境の Cloud Run サービスを、チームメンバーのみがアクセスできるように制限したい場合。

効果: LB のコストをかけずにステージング環境に IAP を適用。Google グループを活用して、開発チームメンバーのみにアクセスを許可する設定が簡単に行える。

💰 料金情報

IAP 自体には追加料金は発生しません。直接設定機能の最大のメリットは、従来必須だった外部 HTTPS ロードバランサーの料金が不要になることです。

項目 Cloud Run 直接 IAP LB 経由 IAP
IAP 利用料 無料 無料
転送ルール料金 不要 約 $18.26/月~
データ処理料金 不要 処理データ量に応じて課金
SSL 証明書管理 不要 (run.app URL 使用時) 必要
静的 IP アドレス 不要 必要 (使用中は無料)

🌐 利用可能リージョン

Cloud Run が利用可能なすべてのリージョンで IAP の直接設定が利用可能です。ただし、プロジェクトが Google Cloud 組織内に存在することが前提条件となります。

📌 まとめ

Cloud Run への IAP 直接設定の GA は、サーバーレス アプリケーションのゼロトラスト セキュリティを大幅に簡素化する重要なアップデートです。

  • コスト削減: ロードバランサー関連の料金が不要に
  • 運用負荷の軽減: 複数リソースの管理が不要に
  • セットアップ時間の短縮: 数分で IAP を有効化

社内向け Web アプリケーションや管理ツールを Cloud Run で運用しているチームは、この機能の導入を強く推奨します。既に LB 経由で IAP を使用しているが、マルチリージョン要件がない場合は、直接 IAP への移行を検討すべきです。
