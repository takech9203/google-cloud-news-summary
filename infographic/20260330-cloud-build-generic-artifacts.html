🔧 Cloud Build: 汎用アーティファクトの
Artifact Registry 直接アップロード & 依存関係ダウンロード

Cloud Build Artifact Registry Feature
2026-03-30
Google Cloud Update

📋 概要

Cloud Build のビルド構成ファイルに genericArtifacts スタンザを追加することで、ビルド成果物を Artifact Registry の汎用 (generic) リポジトリに直接アップロードできるようになりました。また、dependencies スタンザで汎用リポジトリのアーティファクトをビルド依存関係として自動ダウンロードすることも可能になりました。

Docker イメージや言語固有のパッケージに限らず、あらゆる形式のファイル (バイナリ、設定ファイル、アーカイブ等) を Cloud Build の CI/CD パイプラインに宣言的に統合できます。

🔄 Before / After 比較

❌ Before

✗ビルドステップで gcloud artifacts generic upload を手動実行が必要
✗依存関係ダウンロードにカスタムスクリプトが必要
✗Docker/Maven/npm には専用フィールドがあるが、汎用フォーマットには宣言的サポートなし
✗ビルド構成が冗長で保守性が低い

✅ After

✓genericArtifacts フィールドで宣言的にアップロード指定
✓dependencies スタンザでビルド開始時に自動ダウンロード
✓Docker/Maven/npm/Python と同等の宣言的サポート
✓カスタムステップ不要で可読性・保守性が向上

⚡ 主要機能

📤 1. genericArtifacts によるアップロード

→ビルド構成ファイルのトップレベルに genericArtifacts スタンザを追加
→パッケージ名、バージョン、ファイルパスを宣言的に指定
→圧縮ファイル (tar, zip)、設定ファイル (YAML, TOML)、バイナリ、PDF など任意の形式に対応

📥 2. 汎用アーティファクトの依存関係指定

→dependencies スタンザで汎用リポジトリのアーティファクトを参照可能
→ビルド開始時に /workspace 配下に自動ダウンロード
→他のビルドで生成されたバイナリや設定ファイルをビルド間で共有可能

🔗 3. 既存のアーティファクト管理との統合

→Docker イメージ、Maven、npm、Python パッケージに加え、汎用フォーマットを追加
→Artifact Registry の IAM、VPC Service Controls、脆弱性スキャンをそのまま利用可能
→Cloud Build のサブスティテューション変数による動的設定にも対応

📐 技術仕様

項目詳細
設定場所ビルド構成ファイル (cloudbuild.yaml / cloudbuild.json) のトップレベル
リポジトリ形式Artifact Registry の generic フォーマット
パッケージ名制約英数字で始まり終わる、英数字・ハイフン・アンダースコア・ピリオドのみ、256 文字以内
バージョン制約英数字で始まり終わる、小文字英数字と -.+~: のみ、128 文字以内、latest は使用不可
対応ファイル形式任意 (圧縮ファイル、設定ファイル、バイナリ、PDF、メディアファイル等)

💻 設定例

アップロード設定 (genericArtifacts)

YAML cloudbuild.yaml - アーティファクトのアップロード 
steps:
  - name: 'gcr.io/cloud-builders/go'
    args: ['build', '-o', 'output/my-app', '.']

genericArtifacts:
  location: 'us-central1'
  repository: 'my-generic-repo'
  package: 'my-app'
  version: '$BUILD_ID'
  files:
    - 'output/my-app'

依存関係ダウンロード設定 (dependencies)

YAML cloudbuild.yaml - 依存関係として使用 
dependencies:
  genericArtifact:
    repository: 'projects/my-project/locations/us-central1/repositories/my-generic-repo'
    package: 'my-app'
    version: '1.0.0'
    destPath: 'deps/my-app'

steps:
  - name: 'ubuntu'
    entrypoint: 'bash'
    args:
      - '-c'
      - 'chmod +x /workspace/deps/my-app/my-app && /workspace/deps/my-app/my-app'

Step 1: 汎用リポジトリの作成

Bash gcloud CLI 
gcloud artifacts repositories create my-generic-repo \
  --repository-format=generic \
  --location=us-central1 \
  --description="Generic artifact repository for Cloud Build"

Step 2: ビルド構成ファイルに genericArtifacts を追加

YAML cloudbuild.yaml 
steps:
  - name: 'gcr.io/cloud-builders/go'
    args: ['build', '-o', 'output/my-binary', '.']

genericArtifacts:
  location: 'us-central1'
  repository: 'my-generic-repo'
  package: 'my-binary'
  version: '${SHORT_SHA}'
  files:
    - 'output/my-binary'

Step 3: ビルドの実行

Bash ビルド実行コマンド 
gcloud builds submit --config=cloudbuild.yaml .

Step 4: 別のビルドで依存関係として使用

YAML cloudbuild-deploy.yaml 
dependencies:
  genericArtifact:
    repository: 'projects/my-project/locations/us-central1/repositories/my-generic-repo'
    package: 'my-binary'
    version: '1.0.0'
    destPath: 'bin'

steps:
  - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
    entrypoint: 'bash'
    args:
      - '-c'
      - './bin/my-binary --deploy'

✅ メリット

💼 ビジネス面

✓CI/CD パイプラインの簡素化: 宣言的な記述だけでアーティファクト管理が完結し、構築・保守コストが削減
✓ビルド間のアーティファクト共有: 異なるパイプライン間でバイナリや設定ファイルを安全に共有、マイクロサービスや複雑なリリースフローに対応

🔧 技術面

✓宣言的な構成: カスタムビルドステップ不要、可読性と再現性が向上
✓Artifact Registry 統合: IAM、VPC Service Controls、脆弱性スキャンをそのまま活用
✓バージョン管理の一元化: パッケージ名 + バージョンによるトレーサビリティ向上

⚠️ デメリット・制約事項

⚠️ 制限事項:
  • 汎用リポジトリのアーティファクトは不変 (immutable) -- 同一パッケージ名・同一バージョンのファイルを上書き不可
  • バージョン名として latest は使用不可
  • パッケージ名は 256 文字以内、バージョン名は 128 文字以内
💡 考慮すべき点:
  • 汎用リポジトリはパッケージマネージャー (Docker, Maven, npm 等) によるネイティブ操作には非対応。必要な場合は専用形式のリポジトリを使用
  • 大量ファイルのアップロードはディレクトリ単位の指定を検討
  • クロスプロジェクト利用には Cloud Build サービスアカウントへの明示的な IAM ロール付与が必要

🎯 ユースケース

📦 ユースケース 1: Go バイナリのクロスコンパイルと配布

シナリオ: 複数プラットフォーム向けに Go バイナリをクロスコンパイルし、各環境のデプロイパイプラインで利用

YAML クロスコンパイル + アップロード 
steps:
  - name: 'golang'
    entrypoint: 'bash'
    args:
      - '-c'
      - |
        GOOS=linux GOARCH=amd64 go build -o output/my-app-linux-amd64 .
        GOOS=darwin GOARCH=arm64 go build -o output/my-app-darwin-arm64 .

genericArtifacts:
  location: 'us-central1'
  repository: 'release-binaries'
  package: 'my-app'
  version: '$TAG_NAME'
  files:
    - 'output/my-app-linux-amd64'
    - 'output/my-app-darwin-arm64'
✅ 効果: 複数プラットフォームのバイナリが自動的にバージョン管理され、デプロイパイプラインから依存関係として安全にダウンロード可能。

🏗️ ユースケース 2: Terraform プランファイルのビルド間共有

シナリオ: CI パイプラインで terraform plan の結果を保存し、承認後のデプロイパイプラインで terraform apply に使用

YAML Plan ステージ 
steps:
  - name: 'hashicorp/terraform'
    args: ['plan', '-out=tfplan']

genericArtifacts:
  location: 'us-central1'
  repository: 'terraform-plans'
  package: 'infra-plan'
  version: '$BUILD_ID'
  files:
    - 'tfplan'
YAML Apply ステージ 
dependencies:
  genericArtifact:
    repository: 'projects/.../terraform-plans'
    package: 'infra-plan'
    version: '<plan-build-id>'
    destPath: 'plan'

steps:
  - name: 'hashicorp/terraform'
    args: ['apply', '/workspace/plan/tfplan']
✅ 効果: プランファイルが Artifact Registry で不変かつバージョン管理された状態で保存され、承認プロセスを挟んだ安全なデプロイフローを実現。

🔗 関連サービス・機能

📦 Artifact Registry: 汎用アーティファクトの保存先。Docker、Maven、npm、Python、Go など複数パッケージ形式をサポートし、汎用フォーマットも同一プラットフォームで管理。
⚡ Cloud Build トリガー: ソースリポジトリの変更を検知して自動ビルドを実行。genericArtifacts と組み合わせてコード変更から成果物保存まで自動化。
🔍 Artifact Analysis: Artifact Registry に保存されたアーティファクトの脆弱性スキャンやメタデータ管理を提供。
🔒 VPC Service Controls: Artifact Registry リポジトリをセキュリティ境界内に配置し、データ流出を防止。

📝 まとめ

今回のアップデートにより、Cloud Build のビルド構成ファイルから Artifact Registry の汎用リポジトリへのアーティファクトのアップロードと、汎用リポジトリからの依存関係としてのダウンロードが宣言的に行えるようになりました。

Docker イメージや言語固有パッケージ以外のビルド成果物を扱う CI/CD パイプラインを構築している組織は、この機能を活用してパイプラインの簡素化とアーティファクト管理の一元化を進めることを推奨します。
