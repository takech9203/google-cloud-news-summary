📋 概要

Workforce Identity Federation の拡張属性 (Extended Attributes) が非推奨となりました。拡張属性は、外部 IdP からグループメンバーシップなどの追加情報を取得するために使用されていた機能です。今後は SCIM (System for Cross-domain Identity Management) への移行が推奨されます。

⚠️ 重要: 2027 年 4 月 3 日に拡張属性は完全廃止されます。移行期間中は引き続き動作しますが、新規設定は推奨されません。

🔄 Before / After 比較

❌ Before: 拡張属性方式

  • OAuth 2.0 クライアント ID/シークレットの設定が必要で構成が複雑
  • Microsoft Entra ID (Azure AD) に特化しており、IdP の種類に依存
  • トークンサイズ制限があり、大量のグループを扱う場合に制約
  • ネストされたグループメンバーシップの解決が困難

✅ After: SCIM 方式

  • IdP からグループ情報を自動プロビジョニング・同期
  • ネストされたグループメンバーシップを自動展開 (フラットニング)
  • --scim-usage=enabled-for-groups の設定のみで有効化、OAuth 不要
  • Gemini Enterprise 統合でグループ名での共有が可能

💡 補足: --extended-attributes-type で指定していた値の例: azure-ad-groups-id

⚡ SCIM による代替機能

👥 グループ自動プロビジョニング

IdP から Google Cloud へユーザー・グループ情報を自動的にプロビジョニング。手動管理が不要に。

🔗 グループフラットニング

ネストされたグループメンバーシップを自動展開し、IAM ポリシーチェックに反映。

📚 google.group 属性

従来の google.groups (トークンマッピング) から google.group (SCIM マッピング) に変更。

🌐 マルチ IdP サポート

Microsoft Entra ID だけでなく、Okta など SCIM 対応の IdP を幅広くサポート。

✅ 排他制御に注意: scim_usage と extended_attributes_oauth2_client は同一プロバイダーで同時に有効化できません。移行時はまず拡張属性を無効化してから SCIM を有効化してください。

Step 2: プロバイダーで SCIM を有効化

Bash OIDC プロバイダーの場合 
gcloud iam workforce-pools providers update-oidc PROVIDER_ID \
  --workforce-pool=WORKFORCE_POOL_ID \
  --location=global \
  --scim-usage=enabled-for-groups
Bash SAML プロバイダーの場合 
gcloud iam workforce-pools providers update-saml PROVIDER_ID \
  --workforce-pool=WORKFORCE_POOL_ID \
  --location=global \
  --scim-usage=enabled-for-groups
💡 Step 3 の注意: SCIM テナントの --claim-mapping とプロバイダーの --attribute-mapping で google.subject が同一アイデンティティを参照するよう一致させる必要があります。
⚠️ Step 4 の注意: グループ属性が google.groups から google.group に変更されます。IAM ポリシーや属性条件でグループを参照している箇所を全て更新してください。

👍 メリット

🌎 ビジネス面

→標準プロトコルへの統一: SCIM は業界標準のプロビジョニングプロトコルであり、IdP ベンダーへの依存度を低減
→運用負荷の軽減: グループ情報の自動同期により、手動でのグループ管理が不要に

⚙️ 技術面

→グループフラットニング: ネストされたグループを自動展開し、IAM ポリシーチェックの精度を向上
→トークンサイズ制限の回避: 大量のグループをサポート
→Gemini Enterprise 統合: グループ名による共有が可能に

⚠️ デメリット・制約事項

🔸各 Workforce Identity Pool に SCIM テナントは 1 つのみ設定可能
🔸SCIM テナントの削除には 30 日間のソフトデリート期間がある (ハードデリートオプションも利用可能)
🔸google.subject や google.group にマッピングされた SCIM 属性は不変として扱われ、変更するには IdP 側でユーザー/グループを削除・再作成が必要
🔸拡張属性と SCIM は同一プロバイダーで併用できないため、移行は切り替え方式
🔸グループ属性名が google.groups から google.group に変わるため、既存 IAM ポリシーの更新が必要
🔸Okta の場合は Import Users/Groups 機能は非対応

💡 ユースケース

🏢 ユースケース 1: Microsoft Entra ID からの移行

シナリオ: 拡張属性 (azure-ad-groups-id) を使用して Microsoft Entra ID のグループ情報を取得している組織が、SCIM に移行する場合。

効果: OAuth 2.0 クライアントの管理が不要になり、グループのネスト解決が自動化される。Gemini Enterprise での人間が読めるグループ名での共有も利用可能に。

💼 ユースケース 2: 大規模グループ環境での運用改善

シナリオ: 400 を超えるグループを持つ組織で、トークンサイズ制限により全グループのマッピングが困難だった場合。

効果: SCIM のグループフラットニングにより、トークンサイズの制約なくすべてのグループメンバーシップが IAM ポリシーチェックに反映される。
