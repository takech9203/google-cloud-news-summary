🔐 Apigee X: 新しい apigee.coreServiceAgent IAM ロールの導入

Apigee X IAM / Security Change
2026-04-06

📋 概要

Google Cloud は Apigee の新しい IAM ロール apigee.coreServiceAgent (Apigee Core Service Agent) を導入しました。従来の apigee.serviceAgent ロールから、コアランタイム機能に必要な権限のみに限定した新ロールへの移行により、最小権限の原則に沿ったアクセス制御を実現します。

対象ユーザーは、Apigee X を利用しているすべての Google Cloud プロジェクト管理者およびプラットフォームエンジニアです。新規セットアップでは apigee.coreServiceAgent ロールの使用が推奨されます。

🔄 Before / After 比較

❌ Before (従来)

  • apigee.serviceAgent は Apigee コア機能と API Hub 関連の権限を含む広範なロール
  • API Hub、モニタリング、ロギングなど多岐にわたる権限が単一ロールに集約
  • コアランタイム操作に必要な権限と補助サービス連携用の権限が未分離

✅ After (新ロール)

  • apigee.coreServiceAgent でコアランタイムに必要な権限のみを付与可能
  • サービスエージェントの権限がより細分化、最小権限の原則に準拠
  • セキュリティ監査やコンプライアンス対応で権限範囲が明確

⚡ 主要機能

🔑 新しい apigee.coreServiceAgent ロール

ロール ID: roles/apigee.coreServiceAgent
Apigee のコアサービスエージェント専用の IAM ロール。サービスエージェント (service-PROJECT_NUMBER@gcp-sa-apigee.iam.gserviceaccount.com) にのみ付与可能。

🧩 権限の細分化

コアランタイム操作に必要な権限 (Canary Evaluations、Ingress Configs、Operations、API Products など) を含む。従来の apigee.serviceAgent が持っていた広範な API Hub 関連権限を分離。

⏱️ 即時有効の変更

今回のリリースから即座に適用。新規の Apigee セットアップでは apigee.coreServiceAgent の使用が推奨される。

🛠️ 設定方法 (移行手順)

💡 前提条件: Google Cloud プロジェクトで Apigee API が有効化されていること / Apigee サービスエージェントが作成済みであること / プロジェクトに対する IAM 管理権限を持つアカウントでアクセスしていること

現在のサービスエージェントのロール確認

プロジェクトの IAM ポリシーを確認し、Apigee サービスエージェントのロールを表示

Bash 
gcloud projects get-iam-policy $PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:gcp-sa-apigee.iam.gserviceaccount.com" \
  --format="table(bindings.role)"

新しい coreServiceAgent ロールの付与

サービスエージェントに新しい coreServiceAgent ロールを付与

Bash 
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:service-${PROJECT_NUMBER}@gcp-sa-apigee.iam.gserviceaccount.com" \
  --role="roles/apigee.coreServiceAgent"

従来のロールの削除 (検証後)

動作確認後、従来の serviceAgent ロールを削除

Bash 
gcloud projects remove-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:service-${PROJECT_NUMBER}@gcp-sa-apigee.iam.gserviceaccount.com" \
  --role="roles/apigee.serviceAgent"
⚠️ 注意: 従来のロールを削除する前に、十分な動作検証を行ってください。

⚖️ メリット / 制約事項

✅ メリット

  • コンプライアンス強化: 最小権限の原則に沿った IAM 構成で監査要件への対応が容易に
  • リスク低減: 権限範囲が限定され、権限悪用リスクが低減
  • 権限の明確化: コアランタイムに必要な権限が明確に定義、IAM 管理が容易
  • 細粒度のアクセス制御: 機能単位で権限を管理でき、より精密な制御が可能

⚠️ 制約事項・考慮点

  • サービスエージェント以外のプリンシパルには付与不可
  • API Hub 関連機能に影響が出る可能性があるため事前検証が必要
  • 既存環境では段階的な移行を推奨
  • Terraform や Deployment Manager の IaC コード更新も必要

🆕 ユースケース 1: 新規 Apigee X 環境のセットアップ

  --role="roles/apigee.coreServiceAgent"

効果: 初期セットアップから最小権限の原則に沿った IAM 構成が実現でき、セキュリティベストプラクティスに準拠した環境を構築

🔒 ユースケース 2: 既存環境のセキュリティ強化

シナリオ: セキュリティ監査の指摘を受けて、既存の Apigee 環境でサービスエージェントの権限を見直す場合

効果: 不要な権限を削除し、コアランタイムに必要な最小限の権限のみに限定することで、監査要件を満たしつつ運用の安全性が向上
