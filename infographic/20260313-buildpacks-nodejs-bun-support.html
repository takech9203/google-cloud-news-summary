📦 Node.js ビルドパックが Bun をサポート

Google Cloud Buildpacks Preview
2026-03-13

☁️ 概要

Google Cloud の Node.js ビルドパックが、Bun パッケージマネージャーを Preview でサポートしました。Bun を使用して依存関係を管理している Node.js プロジェクトを、Cloud Run、Cloud Run functions、App Engine、GKE などの Google Cloud サービスへソースコードから直接デプロイできるようになります。

Bun は高速な JavaScript / TypeScript ランタイムおよびパッケージマネージャーとして急速に普及しており、npm と比較して大幅に高速なパッケージインストールを実現します。今回のサポートにより、NPM、Yarn、Pnpm に加えて 4 つ目のパッケージマネージャーとして Bun を利用できるようになりました。

🔄 Before / After 比較

❌ Before (アップデート前)

  • Bun プロジェクトのデプロイ時に npm/yarn の lock ファイルを再生成する必要があった
  • Bun の高速パッケージインストールをビルドパック環境で活用できなかった
  • デプロイ前にパッケージマネージャーの切り替え作業が発生

✅ After (アップデート後)

  • bun.lock をプロジェクトに含めるだけで自動的に Bun を使用
  • Bun の高速パッケージインストールをビルドプロセスで直接活用可能
  • 開発環境と本番デプロイで同一のパッケージマネージャーを使用可能

🏗️ アーキテクチャ図

ビルドパックはソースコード内の lock ファイルを検出してパッケージマネージャーを自動判定し、Bun で依存関係をインストールした後、OCI コンテナイメージを生成します。このイメージは Cloud Run をはじめとする複数の Google Cloud サービスにデプロイ可能です。

⚡ 主要機能

🔍 Bun パッケージマネージャーの自動検出

プロジェクトルートに Bun の lock ファイル (bun.lock または bun.lockb) が存在する場合、ビルドパックが自動的に Bun をパッケージマネージャーとして選択します。既存の NPM、Yarn、Pnpm と同様の検出メカニズムに従います。

🚀 高速な依存関係インストール

Bun のネイティブなパッケージインストール機能を活用し、ビルド時間の短縮が期待できます。Bun はパッケージの解決とインストールを並列処理するため、大規模プロジェクトで特に効果が高くなります。

🔧 既存ワークフローとの統合

gcloud run deploy --source . や pack build コマンドなど、既存のデプロイワークフローをそのまま利用可能。環境変数やカスタムビルドステップなどの既存機能との組み合わせもできます。

📋 技術仕様

パッケージマネージャー検出ロジック

Node.js ビルドパックは、プロジェクトルートに存在する lock ファイルに基づいてパッケージマネージャーを自動的に選択します。

パッケージマネージャー 検出ファイル サポート状況
NPM package-lock.json GA (デフォルト)
Yarn yarn.lock GA
Pnpm pnpm-lock.yaml GA
Bun bun.lock / bun.lockb Preview (今回追加)

対応ビルダー

ビルダー タグ OS
google-24 gcr.io/buildpacks/builder:google-24 Ubuntu 24
google-22 gcr.io/buildpacks/builder:google-22 Ubuntu 22
latest gcr.io/buildpacks/builder:latest Ubuntu 22 (デフォルト)

🛠️ 設定方法

💡 前提条件: Google Cloud プロジェクトが作成済み、gcloud CLI がインストール・認証済み、プロジェクトに Bun の lock ファイルが含まれていること。

ステップ 1: プロジェクトの準備

Bun を使用して Node.js プロジェクトの依存関係を管理します。bun install で生成される bun.lock をバージョン管理に含めてください。

Bash Bun で依存関係をインストールし lock ファイルを生成 
# Bun で依存関係をインストールし lock ファイルを生成
bun install

ステップ 2: Cloud Run へのデプロイ

ビルドパックが自動的に bun.lock を検出し、Bun を使用して依存関係をインストールします。

Bash ソースコードから直接 Cloud Run にデプロイ 
# ソースコードから直接 Cloud Run にデプロイ
gcloud run deploy my-service --source . --region asia-northeast1

ステップ 3: ローカルでのビルドテスト (オプション)

Bash pack CLI を使用してローカルでビルドをテスト 
# pack CLI を使用してローカルでビルドをテスト
pack build my-app --builder gcr.io/buildpacks/builder:google-24

📝 サンプルコード

Bun で管理された TypeScript ベースの Web API プロジェクトの package.json 例です。

JSON package.json の例 
{
  "name": "my-api",
  "scripts": {
    "start": "node dist/index.js",
    "build": "tsc"
  },
  "engines": {
    "node": "22.x.x"
  },
  "dependencies": {
    "express": "^4.18.0"
  }
}
Bash Bun で lock ファイルを生成してデプロイ 
# Bun で lock ファイルを生成
bun install

# Cloud Run にデプロイ
gcloud run deploy my-api --source . --region asia-northeast1

✅ メリット

💼 ビジネス面

→ビルド時間の短縮: Bun の高速パッケージインストールにより CI/CD パイプラインの実行時間を短縮
→開発者体験の向上: デプロイ時にパッケージマネージャーを切り替える必要がなくなる

🔧 技術面

→依存関係の一貫性: 開発・本番環境で同一の lock ファイルを使用し、環境差異によるバグを低減
→自動検出: lock ファイルの存在だけで自動選択、追加設定が不要

⚠️ デメリット・制約事項

⚠️ 制限事項:
  • 現在 Preview ステータスであり、本番ワークロードでの利用には注意が必要
  • Preview 機能には SLA が適用されない場合がある
  • 複数の lock ファイル (例: package-lock.json と bun.lock) が同時に存在する場合、デプロイが失敗する可能性がある
💡 考慮すべき点:
  • Bun 自体はまだ比較的新しいランタイムであり、一部の npm パッケージとの互換性に注意
  • Preview から GA への移行時に挙動が変更される可能性がある
  • チーム全体でパッケージマネージャーを統一する場合、Bun への移行コストを評価すること

🎯 ユースケース

🌐 ユースケース 1: Cloud Run での高速デプロイ

シナリオ: Bun で開発された TypeScript ベースの Web API を Cloud Run にデプロイ。開発チームは既に Bun をローカル開発環境で使用しており、依存関係の管理も Bun で行っている。

効果: パッケージマネージャーの切り替えが不要になり、開発からデプロイまでのワークフローが一貫する。

📦 ユースケース 2: モノレポでの大規模プロジェクト

シナリオ: 大量の依存関係を持つモノレポ構成のプロジェクトで、ビルド時間の最適化が課題となっている場合。

効果: Bun の並列パッケージインストール機能により、依存関係が多いプロジェクトでのビルド時間短縮が期待できる。

💰 料金

Buildpacks 自体の利用に追加料金は発生しません。ビルドプロセスで使用される Cloud Build のリソースに対して標準の Cloud Build 料金が適用されます。Bun パッケージマネージャーのサポートによる追加課金はありません。
