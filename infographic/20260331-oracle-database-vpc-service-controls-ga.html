🔒 Oracle Database@Google Cloud: VPC Service Controls サポートが GA に

Oracle Database@Google Cloud VPC Service Controls GA
2026-03-31

📋 概要

Oracle Database@Google Cloud が VPC Service Controls に正式対応し、GA (一般提供) となりました。 VPC Service Controls のサービス境界 (Service Perimeter) を設定することで、Oracle Database@Google Cloud のコントロールプレーン API (oracledatabase.googleapis.com) への不正アクセスやデータ流出のリスクを軽減できます。

Google Cloud 上で Oracle Exadata Database や Autonomous Database を運用するマネージドサービスにおいて、IAM に加えてネットワークベースの境界制御を重ねることで、金融機関、医療機関、政府機関など厳格なデータ保護要件を持つ組織にとって重要なセキュリティ強化が実現されます。

🔄 Before / After 比較

❌ Before

  • コントロールプレーン API に VPC Service Controls の境界保護を適用できず、IAM ポリシーのみに依存
  • Oracle Database@Google Cloud の管理操作を組織の VPC セキュリティ境界内に閉じ込めることが不可能
  • VPC Service Controls 非対応がセキュリティ監査上の懸念事項に

✅ After

  • コントロールプレーン API (oracledatabase.googleapis.com) を VPC Service Controls 境界で保護可能
  • 制限付きサービスとして追加し、不正 API アクセスやデータ流出リスクを低減
  • Access Context Manager と組み合わせた細かなアクセス制御が可能に

🏗️ アーキテクチャ図

VPC Service Controls のサービス境界が Oracle Database@Google Cloud のコントロールプレーン API を保護し、境界外からの不正アクセスを遮断します。認可済みユーザーは Access Context Manager のアクセスレベルを通じてアクセスが許可されます。

⚙️ 主要機能

🛡️ サービス境界によるコントロールプレーン API の保護

  • oracledatabase.googleapis.com を VPC Service Controls の制限付きサービスとして登録可能
  • サービス境界内のプロジェクトからのみ管理操作 (作成、削除、構成変更など) を実行可能
  • 境界外からの API リクエストは自動的にブロック

🔑 Access Context Manager との統合

  • 特定の IP 範囲やデバイスポリシーに基づくアクセス許可が可能
  • 境界外の信頼された環境からの管理アクセスを細かく制御
  • イングレスポリシーとエグレスポリシーによる双方向アクセス制御

🌐 Shared VPC 環境でのサポート

  • ホストプロジェクトとサービスプロジェクトの両方を同一サービス境界に含めることで対応
  • ODB Network と ODB Subnet を使用した組織横断的なネットワーク管理との併用が可能

📊 技術仕様

VPC Service Controls 対応状況

項目詳細
サービス名oracledatabase.googleapis.com
ステータスGA
境界保護対応 (コントロールプレーン API のみ)
データプレーン保護非対応 (OCI 環境内のデータベース接続は対象外)
OCI コントロールプレーン API非対応 (VPC SC 境界の制御対象外)
Oracle VCN の制御非対応 (Google Cloud 外のため対象外)

制限事項の詳細

制限事項説明
コントロールプレーンのみExadata / Autonomous Database へのデータプレーンアクセスは制御対象外
OCI API 非対応OCI コントロールプレーン API は VPC Service Controls の制御を受けない
Oracle VCN 非対応Google Cloud 外に存在するため VPC Service Controls で直接制限不可

🔧 設定方法

📋 前提条件:
  • Google Cloud プロジェクトが作成済みで課金が有効
  • Access Context Manager API が有効
  • roles/accesscontextmanager.policyAdmin IAM ロールが付与済み
  • Oracle Database@Google Cloud がプロビジョニング済み

Step 1: アクセスポリシーの確認

Bash 組織のアクセスポリシーを確認 
gcloud access-context-manager policies list \
    --organization=ORGANIZATION_ID

Step 2: サービス境界の作成

Bash Oracle Database 用のサービス境界を作成 
gcloud access-context-manager perimeters create oracle-db-perimeter \
    --title="Oracle Database Perimeter" \
    --resources=projects/PROJECT_NUMBER \
    --restricted-services=oracledatabase.googleapis.com \
    --policy=POLICY_ID

Step 3: 追加の制限付きサービスの設定 (推奨)

Bash 関連サービスも境界に追加 
gcloud access-context-manager perimeters update oracle-db-perimeter \
    --policy=POLICY_ID \
    --add-restricted-services=compute.googleapis.com,storage.googleapis.com

Step 4: アクセスレベルの設定 (オプション)

Bash 信頼された IP 範囲からのアクセスを許可 
gcloud access-context-manager levels create trusted-admin-access \
    --title="Trusted Admin Access" \
    --basic-level-spec=access-level-spec.yaml \
    --policy=POLICY_ID
YAML access-level-spec.yaml 
- ipSubnetworks:
    - 203.0.113.0/24
    - 198.51.100.0/24

✅ メリット

💼 ビジネス面

  • コンプライアンス対応の強化: PCI DSS、HIPAA、政府系規制などの要件を満たす追加セキュリティレイヤー
  • セキュリティ監査の簡素化: VPC SC のログで API アクセスを一元的に監視・監査可能
  • マルチクラウド戦略の安全な推進: Google Cloud 側のセキュリティガバナンスを確立

🔧 技術面

  • データ流出防止: コントロールプレーン API 経由の不正アクセスや構成変更を防止
  • ゼロトラスト親和性: IAM + ネットワーク境界制御で多層防御 (Defense in Depth) を実現
  • 既存 VPC SC 境界との統合: BigQuery、Cloud Storage など他サービスと同一境界で一元管理

⚠️ デメリット・制約事項

制限事項:
  • VPC Service Controls はコントロールプレーン API のみを保護し、データプレーンアクセス (SQL 接続など) は保護対象外
  • OCI 側のコントロールプレーン API やネットワーク設定は制御範囲外 (別途 OCI のセキュリティ機能を構成が必要)
  • Google Cloud と OCI 間のネットワーキングレイヤーはサービス境界の監視・制限対象外
考慮すべき点:
  • 境界設定後、境界外からの管理操作がブロックされるため、運用チームのアクセス要件を事前に整理
  • Shared VPC では、ホスト/サービスプロジェクトの両方を同一境界に含める必要あり (境界設計の複雑性増)
  • VPC SC 変更反映まで最大 30 分のタイムラグ
  • データプレーンセキュリティは OCI 側の Network Security Groups で別途対応が必要

🎯 ユースケース

🏦 金融機関の Oracle Database 基盤移行

シナリオ: 金融機関がオンプレミス Oracle Database を移行する際、管理 API へのアクセスを社内ネットワークからのみに制限する必要がある。

Bash 金融系プロジェクトの境界作成例 
gcloud access-context-manager perimeters create finance-oracle-perimeter \
    --title="Finance Oracle DB Perimeter" \
    --resources=projects/FINANCE_PROJECT_NUMBER \
    --restricted-services=oracledatabase.googleapis.com,compute.googleapis.com \
    --policy=POLICY_ID

効果: コントロールプレーン API への社外アクセスが自動ブロック。PCI DSS などの金融規制に準拠したデータベース管理環境を実現。

👥 マルチチーム環境での Oracle Database 管理の分離

シナリオ: 大規模組織で複数チームが異なる Oracle Database@Google Cloud インスタンスを利用しており、コントロールプレーン操作を分離したい。

効果: チームごとに異なるサービス境界を設定。各チームは自チームの Oracle Database のみを管理でき、他チームリソースへの誤操作・不正アクセスを防止。

💰 料金

VPC Service Controls 自体の利用に追加料金は発生しません。

項目料金
VPC Service Controls無料
Access Context Manager無料
Oracle Database@Google Cloud (Exadata)既存のサブスクリプション料金に従う
Oracle Database@Google Cloud (Autonomous Database)既存の OCPU / ストレージ料金に従う
VPC SC 監査ログ (Cloud Logging)ログ取り込み量に応じた Cloud Logging の料金

🌍 利用可能リージョン

Oracle Database@Google Cloud が利用可能な全リージョンで VPC Service Controls のサポートが提供されます。対応リージョンの詳細は Oracle Database@Google Cloud のリージョン を参照してください。

📝 まとめ

Oracle Database@Google Cloud の VPC Service Controls 対応 GA は、エンタープライズ環境でのセキュリティとコンプライアンス要件を満たすための重要なマイルストーンです。 コントロールプレーン API をサービス境界で保護することで、データ流出リスクの低減と規制対応の強化を実現します。

Oracle Database@Google Cloud を本番運用している組織は、既存の VPC Service Controls 境界に oracledatabase.googleapis.com を追加し、包括的なセキュリティ体制を構築することを推奨します。ただし、データプレーンのセキュリティは OCI 側の Network Security Groups で別途対応する必要がある点に留意してください。
