☁️ GKE: H4D マシンシリーズが一般提供開始

Google Kubernetes Engine GA HPC
2026-03-05

📋 概要

Google Kubernetes Engine (GKE) で、高性能コンピューティング (HPC) ワークロード向けに設計された H4D マシンシリーズ が一般提供 (GA) になりました。第 5 世代 AMD EPYC Turin プロセッサと Google Titanium オフロードプロセッサを搭載し、RDMA 対応 200 Gbps ネットワーキング で密結合アプリケーションのスケーラビリティを飛躍的に向上させます。

🔄 Before / After 比較

❌ Before

✗H3 マシンが最大 (88 vCPU / 352 GB メモリ) でコア数・メモリに制限
✗Cloud RDMA を活用した低レイテンシ通信の手段が限定的
✗密結合 HPC アプリケーションでオンプレミス同等性能の達成が困難

✅ After

✓192 コア / 最大 1,488 GB メモリで HPC 処理能力が大幅に向上
✓RDMA 200 Gbps でノード間レイテンシを大幅低減
✓GKE Standard / Autopilot 両方で HPC を柔軟に実行可能

🏗️ アーキテクチャ図

H4D ノードは gVNIC (通常通信) と IRDMA NIC (Cloud RDMA 通信) の 2 つのネットワークインターフェースを構成します。RDMA により、ノード間で 200 Gbps の高帯域幅・低レイテンシ通信が実現されます。

⚡ 主要機能

🔧 第 5 世代 AMD EPYC Turin プロセッサ

→ベース 2.7 GHz / 最大 4.1 GHz
→192 コア (SMT 無効) でホスト全体を専有
→オーバーコミットなしで一貫したパフォーマンス

🌐 Cloud RDMA 200 Gbps

→RDMA over Falcon トランスポートプロトコル
→ノード間で最大 200 Gbps の帯域幅
→Falcon VPC ネットワークプロファイルによる専用構成

💾 大容量メモリと Local SSD

→標準 720 GB / ハイメモリ最大 1,488 GB (DDR5)
→Titanium SSD 3,750 GiB Local SSD
→Hyperdisk Balanced ストレージもサポート

→GKE Standard で H4D ノードプール作成可能
→Autopilot Performance コンピュートクラス対応
→Cluster Toolkit による迅速なクラスタ構築

🔌 ネットワーク構成

H4D インスタンスで Cloud RDMA を使用する場合、2 つのネットワークインターフェース (vNIC) の構成が必要です。

インターフェース ドライバ 用途 インターネット接続
gVNIC gVNIC ドライバ 通常のネットワーク通信 可 (最大 1 Gbps)
IRDMA Intel iDPF/iRDMA ドライバ Cloud RDMA 通信 不可

🛠️ 設定方法

💡 前提条件: GKE API 有効化済み / H4D VM キャパシティ取得済み / Container-Optimized OS ノードイメージ使用 / H4D 利用可能リージョン/ゾーン

ステップ 1: RDMA 用 VPC ネットワークの作成

Bash 
# Falcon VPC ネットワークの作成
gcloud compute --project=PROJECT_ID \
  networks create RDMA_NETWORK_PREFIX-net \
  --network-profile=COMPUTE_ZONE-vpc-falcon \
  --subnet-mode=custom

# サブネットの作成
gcloud compute --project=PROJECT_ID \
  networks subnets create RDMA_NETWORK_PREFIX-sub-0 \
  --network=RDMA_NETWORK_PREFIX-net \
  --region=CONTROL_PLANE_LOCATION \
  --range=RDMA_SUBNET_CIDR

ステップ 2: マルチネットワーク対応の GKE クラスタ作成

Bash 
gcloud container clusters create CLUSTER_NAME \
  --project PROJECT_ID \
  --enable-dataplane-v2 \
  --enable-ip-alias \
  --location=CONTROL_PLANE_LOCATION \
  --enable-multi-networking

ステップ 3: GKE ネットワークオブジェクトの設定

YAML 
apiVersion: networking.gke.io/v1
kind: GKENetworkParamSet
metadata:
  name: rdma-0
spec:
  vpc: RDMA_NETWORK_PREFIX-net
  vpcSubnet: RDMA_NETWORK_PREFIX-sub-0
  deviceMode: RDMA
---
apiVersion: networking.gke.io/v1
kind: Network
metadata:
  name: rdma-0
spec:
  type: "Device"
  parametersRef:
    group: networking.gke.io
    kind: GKENetworkParamSet
    name: rdma-0

代替方法: Cluster Toolkit による構築

Bash 
# Cluster Toolkit を使用した H4D GKE クラスタのデプロイ
./gcluster deploy \
  -d examples/gke-h4d/gke-h4d-deployment.yaml \
  examples/gke-h4d/gke-h4d.yaml

✅ メリット

📈 ビジネス面

✓HPC のクラウド移行: オンプレミス同等の性能を GKE で実現、インフラ管理コスト削減
✓柔軟なキャパシティ: オンデマンド / Flex-start / 1年・3年 CUD から選択可能
✓Dynamic Workload Scheduler: HPC バーストワークロードの柔軟なスケジューリング

🔧 技術面

✓RDMA 200 Gbps: TCP/IP 比で大幅な低レイテンシ・高帯域幅
✓192 コア専有: SMT 無効 / オーバーコミットなしで一貫したパフォーマンス
✓K8s エコシステム統合: オートスケーリング / ノード自動プロビジョニングを HPC で活用

⚠️ 制限事項・注意点

✗カスタムマシンタイプは利用不可 (定義済みのみ)
✗GPU との併用は不可
✗アウトバウンドデータ転送は 1 Gbps に制限
✗ライブマイグレーション非対応 (メンテナンス時はインスタンス終了)
✗マシンイメージの作成不可
✗Hyperdisk Balanced は 15,000 IOPS / 240 MBps が上限
✗インスタンス間でのディスク共有不可
✗Container-Optimized OS ノードイメージのみサポート
⚠️ ホストメンテナンス: メンテナンス時にインスタンスが終了します (最低 30 日間隔、7 日前の事前通知あり)。ワークロードの中断対策設計が必要です。

🎯 ユースケース

🌦️ 気象予報シミュレーション

気象研究機関が大規模な気象モデルシミュレーションを実行。数百ノードにまたがる密結合計算で、H4D の 192 コアと RDMA 200 Gbps ネットワーキングによりメッシュ分割されたシミュレーション領域のデータ交換を高速化し、シミュレーション完了時間を大幅に短縮。

🔬 電子設計自動化 (EDA)

半導体設計企業が大規模な回路シミュレーションやタイミング解析を実行。192 コアの専有ホストにより単一ノードで大規模並列シミュレーションが可能。RDMA で複数ノードジョブの実行効率も向上。

🧬 ライフサイエンス・ゲノム解析

製薬企業が分子動力学シミュレーションやゲノムシーケンシングのデータ処理を GKE 上で実行。Kubernetes のジョブ管理と H4D の HPC 性能を組み合わせ、研究パイプラインを自動化。研究者がインフラ管理ではなく研究に集中できる環境を実現。
