Google Kubernetes Engine Gateway API HealthCheckPolicy Change GKE 1.34+

📝 概要

GKE バージョン 1.34 以降において、GKE Gateway API の HealthCheckPolicy カスタムリソースのバリデーションが強化されました。具体的には、HealthCheckPolicy の config セクション内の type フィールドと、実際に指定されたヘルスチェック設定フィールド (例: httpHealthCheck, tcpHealthCheck) の整合性が自動的に検証されるようになりました。

以前のバージョンでは、type フィールドと実際のヘルスチェック設定フィールドが一致しない場合でもリソースが受け入れられ、type フィールドの値が優先される暗黙的な動作がありました。GKE 1.34 以降では、このようなミスマッチが検出された場合に kubectl がポリシーを拒否し、設定ミスをデプロイ前に防止できます。

🔄 Before / After 比較

❌ Before (GKE 1.33 以前)

  • type: TCP と指定しながら httpHealthCheck を設定するなど、不一致でも受け入れられていた
  • 不一致の場合、type フィールドの値のみが使用され、設定したヘルスチェックフィールドの内容が無視されていた
  • 設定ミスが検出されないまま運用され、意図しないプロトコルでヘルスチェックが実行される可能性があった

✅ After (GKE 1.34 以降)

  • HealthCheckPolicy の作成・更新時に type フィールドとヘルスチェック設定フィールドの整合性が自動検証
  • ミスマッチが検出された場合、kubectl がポリシーを即座に拒否し、設定ミスをデプロイ前に防止
  • 既存のミスマッチを含むリソースは後方互換性のため免除され、引き続き動作

⚡ 主要機能

🔎 type フィールドとヘルスチェック設定の整合性検証

  • HealthCheckPolicy の config.type フィールドが、指定されたヘルスチェック設定フィールドと一致するかを検証
  • 例: type: TCP を指定した場合、tcpHealthCheck フィールドの設定が必要であり、httpHealthCheck を設定するとバリデーションエラー
  • GKE バージョン 1.34 以降で自動的に適用される

🔄 既存リソースの後方互換性

  • GKE 1.34 へのアップグレード前に作成された、ミスマッチを含む HealthCheckPolicy リソースは引き続き動作
  • これらの既存リソースは免除 (exempt) として扱われ、既存の動作が維持される
  • ただし、既存リソースを更新する際に新たなミスマッチを導入したり、既存のミスマッチフィールドを別の無効な値に変更することはできない

🚀 デプロイ時の即時フィードバック

  • kubectl apply や kubectl create の実行時にバリデーションエラーが即座に返される
  • CI/CD パイプラインでの設定ミス検出が容易になり、本番環境への不正な設定のデプロイを防止

💻 設定例

✅ 有効な HealthCheckPolicy の設定例

YAML type: TCP と tcpHealthCheck が一致 
apiVersion: networking.gke.io/v1
kind: HealthCheckPolicy
metadata:
  name: tcp-healthcheck
  namespace: my-app
spec:
  default:
    checkIntervalSec: 15
    timeoutSec: 5
    healthyThreshold: 2
    unhealthyThreshold: 3
    logConfig:
      enabled: true
    config:
      type: TCP
      tcpHealthCheck:
        port: 8080
  targetRef:
    group: ""
    kind: Service
    name: my-backend-service

❌ 無効な HealthCheckPolicy の設定例 (GKE 1.34 以降で拒否)

YAML type: TCP と httpHealthCheck が不一致 
apiVersion: networking.gke.io/v1
kind: HealthCheckPolicy
metadata:
  name: mismatched-healthcheck
  namespace: my-app
spec:
  default:
    config:
      type: TCP          # TCP を指定
      httpHealthCheck:    # しかし HTTP のヘルスチェック設定を記述 (不一致)
        port: 8080
        requestPath: /health
  targetRef:
    group: ""
    kind: Service
    name: my-backend-service

🛠 ミスマッチの修正例

パターン A: type に合わせて設定を変更

YAML type: TCP に合わせる 
spec:
  default:
    config:
      type: TCP
      tcpHealthCheck:   # type: TCP と一致
        port: 8080

パターン B: 設定に合わせて type を変更

YAML httpHealthCheck に合わせる 
spec:
  default:
    config:
      type: HTTP
      httpHealthCheck:  # type: HTTP と一致
        port: 8080
        requestPath: /health

🛠 設定方法

前提条件

✓GKE クラスタがバージョン 1.34 以降にアップグレードされていること
✓GKE Gateway API が有効化されていること (--gateway-api=standard)
✓HealthCheckPolicy CRD がクラスタにインストールされていること

手順

  1. 既存の HealthCheckPolicy リソースの確認

    クラスタ内の全ての HealthCheckPolicy リソースを一覧表示し、確認対象を把握します。

    Bash
    kubectl get healthcheckpolicies --all-namespaces
  2. 各ポリシーの type フィールドとヘルスチェック設定の整合性を確認

    出力された YAML の spec.default.config セクションで、type フィールドと実際のヘルスチェック設定フィールドが一致しているかを確認します。

    Bash
    kubectl get healthcheckpolicy <POLICY_NAME> -n <NAMESPACE> -o yaml
  3. ミスマッチの修正

    ミスマッチが見つかった場合、type フィールドまたはヘルスチェック設定フィールドのいずれかを修正して整合性を確保します。意図するヘルスチェックプロトコルに基づいて修正パターンを選択してください。

  4. 修正の適用と確認

    修正したポリシーを適用し、Conditions セクションで正常にアタッチされていることを確認します。

    Bash
    kubectl apply -f healthcheckpolicy.yaml
kubectl describe healthcheckpolicy <POLICY_NAME> -n <NAMESPACE>

💡 メリット

💼 ビジネス面

✓運用リスクの低減: 設定ミスが自動的に検出されるため、意図しないヘルスチェック動作による本番障害リスクが低減
✓デバッグ時間の短縮: ヘルスチェックの設定ミスがデプロイ時に即座にフィードバックされるため、問題の原因調査時間を大幅に短縮

⚙ 技術面

✓設定の明示性向上: type フィールドとヘルスチェック設定が必ず一致するため、ポリシーの意図が明確になり、コードレビューや監査が容易に
✓CI/CD パイプラインとの親和性: バリデーションエラーが kubectl レベルで検出されるため、GitOps ワークフローの品質ゲートとして機能
✓後方互換性の確保: 既存リソースが免除されるため、GKE 1.34 へのアップグレード時にワークロードへの即時影響なし

⚠ デメリット・制約事項

制限事項

✕GKE バージョン 1.34 以降でのみ有効であり、1.33 以前のクラスタでは従来の動作が継続される
✕既存のミスマッチリソースは免除されるため、アップグレード後に自動的に修正されるわけではない。手動での確認と修正が必要

考慮すべき点

  • GKE 1.34 へのアップグレード前に、既存の HealthCheckPolicy リソースに type フィールドとヘルスチェック設定のミスマッチがないか事前に確認することを推奨
  • 免除された既存リソースは動作し続けるが、type フィールドの値が実際のヘルスチェックプロトコルとして使用され、ヘルスチェック設定フィールドの内容は無視されるという従来の動作が維持される
  • CI/CD パイプラインでマニフェストの lint や dry-run テストを実行している場合、GKE 1.34 クラスタへの適用時に新たなバリデーションエラーが発生する可能性がある

🎯 ユースケース

🔌 ユースケース 1: TCP バックエンドサービスのヘルスチェック設定

シナリオ: GKE Gateway の背後で TCP ベースのバックエンドサービス (データベースプロキシなど) を運用しており、TCP ポートの疎通確認でヘルスチェックを行いたい。

YAMLTCP ヘルスチェック設定例
apiVersion: networking.gke.io/v1
kind: HealthCheckPolicy
metadata:
  name: tcp-backend-healthcheck
  namespace: database
spec:
  default:
    checkIntervalSec: 10
    timeoutSec: 5
    unhealthyThreshold: 3
    config:
      type: TCP
      tcpHealthCheck:
        port: 5432
  targetRef:
    group: ""
    kind: Service
    name: db-proxy-service
効果: type: TCP と tcpHealthCheck が一致しているため、GKE 1.34 でもバリデーションを通過。TCP ポート 5432 の疎通確認によるヘルスチェックが意図通りに動作。

🔍 ユースケース 2: GKE 1.34 アップグレード前の既存設定の監査

シナリオ: GKE 1.33 で運用中のクラスタを 1.34 にアップグレードする前に、既存の HealthCheckPolicy リソースにミスマッチがないか確認したい。

Bashミスマッチ確認コマンド
# 全 HealthCheckPolicy を YAML で出力して確認
kubectl get healthcheckpolicies --all-namespaces -o yaml | \
  grep -A 10 "config:" | grep -E "(type:|HealthCheck:)"
効果: アップグレード前にミスマッチを特定し、修正することで、アップグレード後のポリシー更新時にバリデーションエラーが発生するリスクを排除。

🔗 関連サービス・機能

⚓ GKE Gateway API

HealthCheckPolicy は GKE Gateway API のポリシーリソースの一つであり、ロードバランサのヘルスチェック動作を制御

⚖ Cloud Load Balancing

GKE Gateway の基盤となるロードバランシングサービス。HealthCheckPolicy の設定は Cloud Load Balancing のヘルスチェックリソースにマッピングされる
