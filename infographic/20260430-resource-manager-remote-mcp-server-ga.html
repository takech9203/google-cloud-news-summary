📋 概要

Resource Manager のリモート MCP (Model Context Protocol) サーバーが一般提供 (GA) となった。AI エージェントがユーザーの権限に基づいて Google Cloud プロジェクトを動的に検索・識別することが可能になる。

🤖 MCP とは: Anthropic が開発したオープンソースプロトコルで、LLM や AI アプリケーションが外部データソースに接続する方法を標準化する。Google Cloud のリモート MCP サーバーは、エンタープライズレベルのガバナンス、セキュリティ、アクセス制御を備えたマネージド HTTP エンドポイントとして提供される。

Claude、Gemini CLI、カスタム AI アプリケーションなどの MCP ホストが Resource Manager の search_projects ツールを本番環境で利用できるようになった。

❌ Before (アップデート前)

  • ユーザーが手動でプロジェクト ID を指定する必要があった
  • エージェントがプロジェクトを動的に発見する標準的な方法がなかった
  • Preview 段階で本番ワークロードの SLA が保証されていなかった
  • REST API を直接呼び出す実装が必要で統合が複雑だった

✅ After (アップデート後)

  • MCP 経由でプロジェクトを動的に検索・識別可能
  • エンタープライズ SLA が適用され本番利用推奨
  • 多様なフィルタでプロジェクト検索が可能
  • IAM deny ポリシーによる細粒度アクセス制御

⚡ 主要機能

🔍 search_projects ツール

→アクセス権限のある全プロジェクトを検索可能
→親リソース、プロジェクト ID、ラベル、表示名でフィルタリング
→ページネーション対応 (pageToken, pageSize)

🧠 暗黙的コンテキスト解決

→プロジェクト ID を明示しなくてもエージェントが自動特定
→自然言語での操作リクエストに対応
→プロジェクト ID の暗記やドキュメント参照が不要に

🔐 IAM deny ポリシーによるアクセス制御

→プリンシパル単位でのアクセス制御
→ツールプロパティ (read-only など) による制御
→OAuth クライアント ID によるアプリケーション単位の制御

🌐 グローバル/リージョナルエンドポイント

→グローバル: cloudresourcemanager.googleapis.com/mcp
→リージョナル: cloudresourcemanager.{REGION}.rep.googleapis.com/mcp
→データレジデンシー要件に対応可能

🛠️ 設定方法

💡 前提条件: Resource Manager API が有効化されていること / Service Usage Admin ロール (roles/serviceusage.serviceUsageAdmin) を持つこと / MCP クライアント対応 AI アプリケーション

Step 1: MCP サーバーの有効化

Bash MCP サーバーの有効化 
gcloud beta services mcp enable cloudresourcemanager.googleapis.com \
  --project=PROJECT_ID

※ 2026 年 3 月 17 日以降、Resource Manager API が有効なプロジェクトでは別途の有効化が不要になる (段階的リリース中)

Step 2: MCP クライアントの設定

JSON MCP クライアント設定 
{
  "mcpServers": {
    "resource-manager": {
      "url": "https://cloudresourcemanager.googleapis.com/mcp",
      "transport": "http",
      "auth": {
        "type": "oauth2",
        "scope": "https://www.googleapis.com/auth/cloudresourcemanager.read-only"
      }
    }
  }
}

Step 3: ツールの確認

Bash tools/list 呼び出し (認証不要) 
curl --location 'https://cloudresourcemanager.googleapis.com/mcp' \
  --header 'content-type: application/json' \
  --header 'accept: application/json, text/event-stream' \
  --data '{
    "method": "tools/list",
    "jsonrpc": "2.0",
    "id": 1
  }'

💡 ユースケース

📋 ユースケース 1: リソースインベントリ監査

シナリオ: 組織内のアクティブな Google Cloud プロジェクトの一覧と状態を AI エージェント経由で定期的に確認する。

実行例: 「アクティブな Google Cloud プロジェクトを全て一覧表示して」

効果: 手動でのコンソール確認やスクリプト実行が不要になり、対話形式で即座にプロジェクト情報を取得可能。

🔗 ユースケース 2: 暗黙的コンテキスト解決によるマルチサービス操作

シナリオ: ユーザーがプロジェクト ID を知らなくても、AI エージェントが適切なプロジェクトを特定してリソース操作を実行する。

実行例: 「staging 環境の Cloud Run サービス一覧を見せて」

効果: プロジェクト ID の暗記やドキュメント参照が不要になり、自然言語での操作が実現。

📂 ユースケース 3: フォルダ/組織ベースのプロジェクト管理

シナリオ: 特定の組織構造配下のプロジェクトをフィルタリングして管理タスクを実行する。

実行例: 「フォルダ 223 配下のプロジェクトで、production ラベルが付いているものを教えて」

効果: 組織階層に基づいた精密なプロジェクト検索が対話形式で可能。

⚠️ 制限事項・考慮点

制限事項:
  • 現時点で公開ツールは search_projects のみ (作成・削除などの書き込み操作は未提供)
  • MCP クライアント対応 AI アプリケーションが必要 (従来の REST API クライアントから直接利用不可)
  • リージョナルエンドポイントの利用可能リージョンは公式リファレンスを参照
💡 ベストプラクティス:
  • AI エージェント用に専用のサービスアカウントを作成し、アクセスを制御・監視する
  • 必要なサービスの MCP サーバーのみを有効化する
  • 異なるプロジェクトでクライアント認証情報とリソースをホスティングする場合、両方で有効化が必要

📝 まとめ

Resource Manager リモート MCP サーバーの GA は、Google Cloud 上の AI エージェント開発において重要なマイルストーンである。AI エージェントがプロジェクトを動的に発見・識別できるようになることで、エージェントワークフロー全体の基盤として機能する。
