🔒 Network Security Integration:
Zonal Affinity for In-band Integration

Network Security Integration GA
2026-05-04

📋 概要

Google Cloud の Network Security Integration において、In-band Integration デプロイメントでゾーンアフィニティ (Zonal Affinity) を有効化できるようになりました。リージョナルバックエンドを使用したゾーン単位のインターセプト (パケット傍受) を構成し、同一ゾーン内の検査用 VM にパケットを優先的にルーティングすることで、レイテンシの最適化とスループットの向上を実現します。

マルチゾーン構成のネットワークセキュリティアプライアンスを運用する組織にとって、パフォーマンスとコストの両面で大きな価値をもたらすアップデートです。

🔄 Before / After 比較

❌ Before (従来)

  • 各ゾーンに個別の内部パススルーネットワーク LB が必要
  • クロスゾーントラフィックによるレイテンシ増加
  • リージョナルバックエンドでのゾーン単位制御不可
  • 不要なネットワークホップが発生

✅ After (改善後)

  • リージョナル LB + ゾーンアフィニティで統合管理
  • 同一ゾーン内ルーティング優先でレイテンシ削減
  • リージョナルバックエンドでゾーン単位構成可能
  • クロスゾーントラフィックを制限

💡 ポイント: クロスゾーン通信が最小化され、レイテンシの低減とスループットの向上が実現されます。同一ゾーンに正常な VM がない場合のみ、他ゾーンへのスピルオーバーが発生します。

⚡ 主要機能

🎯 ゾーンアフィニティによる同一ゾーン優先ルーティング

→トラフィック発信元と同じゾーンの正常な検査用 VM にパケットを優先ルーティング
→GENEVE カプセル化により元パケット (送信元/宛先 IP) を保持したまま転送
→既存の確立済み接続 (コネクショントラッキング) は影響を受けない

🌐 リージョナルバックエンドでのゾーン単位インターセプト構成

→各ゾーンに個別の LB を配置する代わりに、リージョナル内部パススルーネットワーク LB を使用可能
→複数ゾーンにまたがるバックエンドインスタンスグループを単一 LB で管理
→インターセプトデプロイメントがリージョナル LB のフォワーディングルールを参照

🔧 フレキシブルなアフィニティオプション

→ZONAL_AFFINITY_STAY_WITHIN_ZONE: トラフィックを厳密に同一ゾーン内に維持
→ZONAL_AFFINITY_SPILL_CROSS_ZONE: 同一ゾーン優先 + スピルオーバー比率に基づく他ゾーンへの流出を許可
→ZONAL_AFFINITY_DISABLED: デフォルト動作 (ゾーンを考慮しない選択)

📐 技術仕様

ゾーンアフィニティオプション

オプション 動作 推奨ユースケース
ZONAL_AFFINITY_DISABLED デフォルト。ゾーンを考慮せずバックエンドを選択 レイテンシが問題にならない場合
ZONAL_AFFINITY_STAY_WITHIN_ZONE 同一ゾーン内に厳密にトラフィックを維持 データローカリティが重要な場合
ZONAL_AFFINITY_SPILL_CROSS_ZONE 同一ゾーン優先、スピルオーバー許可 可用性とレイテンシのバランスが必要な場合

🛠️ 設定方法

前提条件

  1. Network Security API が有効化されていること
  2. インターセプトデプロイメントグループが作成済みであること
  3. 内部パススルーネットワーク LB が構成済み (リージョナルバックエンド付き)
  4. パケット検査用 VM インスタンスがデプロイ済みであること
  5. networksecurity.interceptDeploymentAdmin ロールが付与されていること

Step 1: ロードバランサのバックエンドサービスにゾーンアフィニティを有効化

Bash ゾーンアフィニティ設定 
# ZONAL_AFFINITY_STAY_WITHIN_ZONE を設定する場合
gcloud beta compute backend-services update BACKEND_SERVICE_NAME \
    --zonal-affinity-spillover=ZONAL_AFFINITY_STAY_WITHIN_ZONE \
    --region=REGION

# ZONAL_AFFINITY_SPILL_CROSS_ZONE を設定する場合 (スピルオーバー比率 30%)
gcloud beta compute backend-services update BACKEND_SERVICE_NAME \
    --zonal-affinity-spillover=ZONAL_AFFINITY_SPILL_CROSS_ZONE \
    --zonal-affinity-spillover-ratio=0.3 \
    --region=REGION

Step 2: インターセプトデプロイメントの作成

Bash リージョナル LB を参照するデプロイメント作成 
gcloud network-security intercept-deployments create DEPLOYMENT_ID \
    --location=ZONE \
    --forwarding-rule=FORWARDING_RULE_NAME \
    --forwarding-rule-location=REGION \
    --no-async \
    --intercept-deployment-group=projects/PROJECT_ID/locations/global/interceptDeploymentGroups/DEPLOYMENT_GROUP_ID

Step 3: ファイアウォールポリシーにインターセプトルールを追加

Bash インターセプトルールの作成 
gcloud compute firewall-policies rules create PRIORITY \
    --firewall-policy=POLICY_NAME \
    --direction=INGRESS \
    --action=apply_security_profile_group \
    --security-profile-group=SECURITY_PROFILE_GROUP \
    --src-ip-ranges=0.0.0.0/0

Step 4: Terraform による構成例

HCL Terraform リソース定義 
resource "google_network_security_intercept_deployment" "default" {
  intercept_deployment_id   = "intercept-deployment-zone-a"
  location                  = "us-central1-a"
  forwarding_rule           = google_compute_forwarding_rule.regional.id
  intercept_deployment_group = google_network_security_intercept_deployment_group.default.id
}

💼 ビジネス面

🔧運用管理の簡素化: リージョナル LB で統合管理が可能
📈スケーラビリティ: マルチゾーンの利点を維持しつつゾーン単位で拡張容易

🔧 技術面

⚡レイテンシ削減: 同一ゾーン内検査でネットワークホップ最小化
🚀スループット向上: ゾーン内通信優先でボトルネック解消
🛡️高可用性: スピルオーバーによる自動フェイルオーバー
🔗互換性: 既存の In-band Integration 構成をそのまま活用

⚠️ 制約事項・注意点

制限事項:
  • 同一リージョン内の VM クライアントでのみ有効 (Cloud VPN / Cloud Interconnect 経由は非対応)
  • バックエンドサブセッティングとの併用不可
  • Packet Mirroring のコレクターデスティネーションとの併用不可
  • Private Service Connect パブリッシュドサービスとの併用不可
  • 対称ハッシュは、両方の内部パススルー LB でゾーンアフィニティ有効時のみサポート
💡 考慮すべき点:
  • STAY_WITHIN_ZONE 使用時、同一ゾーンの非正常バックエンドにもトラフィックが留まりパケットドロップの可能性あり
  • スピルオーバー比率の設定値によっては期待通りの効果が得られない場合あり
  • 設定変更は新規接続のみに影響 (既存接続は変更されない)
  • コンシューマーネットワークの MTU は 8500 バイト以下に制限

🎯 ユースケース

🛡️ ユースケース 1: マルチゾーン IDS/IPS デプロイメント

シナリオ: マルチゾーン構成で IDS/IPS をデプロイし、検査トラフィックのレイテンシを最小化したい場合

Bash 実装例 
gcloud beta compute backend-services update ids-backend-service \
    --zonal-affinity-spillover=ZONAL_AFFINITY_SPILL_CROSS_ZONE \
    --zonal-affinity-spillover-ratio=0.1 \
    --region=us-central1

効果: レイテンシが 50% 以上削減される可能性。10% スピルオーバーで障害時の可用性も確保。

📋 ユースケース 2: コンプライアンス要件に基づくゾーン内データ処理

シナリオ: データ主権やコンプライアンス要件により、特定のゾーン内でのデータ処理が求められる場合

Bash 実装例 
gcloud beta compute backend-services update compliance-backend \
    --zonal-affinity-spillover=ZONAL_AFFINITY_STAY_WITHIN_ZONE \
    --region=asia-northeast1

効果: パケット検査処理が必ず同一ゾーン内で完結し、データがゾーン境界を越えないことを保証。

🏢 ユースケース 3: 大規模マルチテナント環境でのセキュリティサービス提供

シナリオ: セキュリティサービスプロデューサーが複数のコンシューマーにパケット検査サービスを提供する場合

効果: リージョナル LB とゾーンアフィニティの組み合わせにより、運用効率を保ちながら各ゾーンのコンシューマーに低レイテンシのサービスを提供。

ゾーン間ネットワーク転送 ゾーン間通信が発生した場合 (スピルオーバー時)
Network Security Integration NSI リソースの使用量に基づく課金

🔗 関連サービス・機能

🛡️Cloud NGFW: ファイアウォールポリシーで apply_security_profile_group アクションを使用してインターセプトを構成
⚖️内部パススルーネットワーク LB: ゾーンアフィニティ設定を保持し、パケット検査 VM へのトラフィック分散を制御
📡Packet Intercept: ルーティングポリシーを変更せずにネットワークアプライアンスをトラフィックパスに挿入
📦GENEVE: パケットカプセル化プロトコル。元のパケット情報を保持して検査 VM に転送
🔐Security Profile Group: セキュリティプロファイルをグループ化し、ファイアウォールルールからインターセプトエンドポイントグループを参照
📋Out-of-band Integration: トラフィックのコピーをアプライアンスに送信して分析する方式 (元のトラフィックフローに影響しない)
