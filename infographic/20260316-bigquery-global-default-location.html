📊 BigQuery: グローバルデフォルトロケーション構成

BigQuery GA Location Data Governance
2026-03-16

☁️ 概要

BigQuery において、グローバルデフォルトロケーションを構成できる機能が General Availability (GA) としてリリースされました。 組織レベルまたはプロジェクトレベルでデフォルトのロケーションを設定でき、リクエストでロケーションが明示的に指定されていない場合や推測できない場合に、この設定が適用されます。

データガバナンスとコンプライアンス管理を大幅に簡素化し、複数リージョンにまたがるデータ基盤を管理する BigQuery 管理者、データガバナンス担当者、セキュリティアーキテクトを主な対象としています。

🔄 Before / After 比較

❌ Before (従来)

  • ロケーション未指定のクエリは US マルチリージョンがデフォルトとなり、意図しないリージョンで処理が発生する可能性
  • 各ユーザー・アプリケーションが個別にロケーションを指定する必要があり、管理コストが高い
  • Gemini in BigQuery のデータ処理ロケーション制御が困難

✅ After (改善後)

  • 組織/プロジェクトレベルで default_location を設定し、フォールバック先を一元管理
  • Gemini in BigQuery のデータ処理ロケーションもデフォルト設定に従い、AI 機能のガバナンスを強化
  • セッション/ジョブレベルのオーバーライドも可能で、柔軟性を維持

🏗️ アーキテクチャ図

BigQuery のロケーション決定は階層的に行われます。組織レベルのデフォルトをプロジェクトレベルで上書きでき、さらにセッションレベルやジョブレベルで個別に指定することも可能です。

⚡ 主要機能

🏢 組織レベルのデフォルトロケーション設定

ALTER ORGANIZATION SET OPTIONS DDL で組織全体のデフォルトロケーションを設定。組織配下の全プロジェクトに継承されます。

→ BigQuery Admin ロール (roles/bigquery.admin) が必要

📁 プロジェクトレベルのデフォルトロケーション設定

ALTER PROJECT SET OPTIONS DDL でプロジェクト単位のデフォルトを設定。組織レベルの設定をプロジェクトごとに上書き可能。

→ プロジェクト ID 省略時は現在のプロジェクトに適用

🤖 Gemini in BigQuery との統合

SQL エディタやデータキャンバスでの Gemini 処理ロケーション制御にも使用。デフォルトロケーション設定により Gemini のデータ処理が指定リージョンの管轄境界内で実行されます。

→ BigQuery Studio のクエリロケーション設定でオーバーライド可能

🔍 INFORMATION_SCHEMA による設定確認

設定の確認・監査を INFORMATION_SCHEMA ビューで実施可能。

→ ORGANIZATION_OPTIONS / PROJECT_OPTIONS / EFFECTIVE_PROJECT_OPTIONS

🔧 技術仕様

設定の優先順位

優先度 レベル 設定方法
最高 ジョブレベル API の location プロパティ、--location フラグ
高 セッションレベル @@location システム変数
中 プロジェクトレベル ALTER PROJECT SET OPTIONS
低 組織レベル ALTER ORGANIZATION SET OPTIONS

必要な IAM ロールと権限

操作 ロール 権限
デフォルトロケーションの設定 BigQuery Admin (roles/bigquery.admin) bigquery.config.update
設定の参照 BigQuery Job User (roles/bigquery.jobUser) bigquery.config.get

🛠️ 設定方法

📋 前提条件: BigQuery Admin ロール (roles/bigquery.admin) が付与されていること。Google Cloud コンソールまたは BigQuery SQL エディタへのアクセス権があること。

Step 1: 組織レベルのデフォルトロケーションを設定

SQL 組織全体のデフォルトを東京リージョンに設定 
ALTER ORGANIZATION SET OPTIONS (
  default_location = 'asia-northeast1'
);

Step 2: プロジェクトレベルのデフォルトロケーションを設定 (必要に応じて)

SQL 特定プロジェクトのデフォルトをオーバーライド 
ALTER PROJECT my_project SET OPTIONS (
  default_location = 'us-central1'
);

Step 3: 設定の確認

SQL 設定値の確認クエリ 
-- 組織レベルの設定を確認
SELECT * FROM INFORMATION_SCHEMA.ORGANIZATION_OPTIONS
WHERE option_name = 'default_location';

-- プロジェクトの実効設定を確認（組織設定の継承を含む）
SELECT * FROM INFORMATION_SCHEMA.EFFECTIVE_PROJECT_OPTIONS
WHERE option_name = 'default_location';

Step 4: 設定のクリア (必要に応じて)

SQL プロジェクトレベルの設定をクリアして組織設定にフォールバック 
-- プロジェクトレベルの設定をクリア
ALTER PROJECT SET OPTIONS (
  default_location = NULL
);

✅ メリット

💼 ビジネス面

✓ データレジデンシーコンプライアンスの強化 - 組織ポリシーとして特定リージョンをデフォルトに設定し、GDPR や各国のデータ保護規制への準拠を組織的に担保
✓ 運用管理の一元化 - 各ユーザーやアプリケーションが個別にロケーションを指定する必要がなくなり、管理オーバーヘッドを削減

⚙️ 技術面

✓ 意図しないリージョンでの処理を防止 - フォールバック先を明示的に制御し、予期しないデータ移動やコスト増を回避
✓ Gemini 処理ロケーションの一元管理 - AI 機能のデータ処理ロケーションもデフォルト設定に従い、セキュリティポリシーの一貫性を向上
✓ 階層的なオーバーライド - 組織、プロジェクト、セッション、ジョブの各レベルで柔軟に設定変更が可能

⚠️ デメリット・制約事項

制限事項:
  • 設定の反映には数分かかる場合があり、INFORMATION_SCHEMA ビューへの反映にもタイムラグがある
  • デフォルトロケーション設定は新しいリクエストに適用され、既存のデータセットやテーブルのロケーションは変更されない
  • デフォルトロケーション設定単体ではロケーション制限を強制するものではない (ユーザーがジョブレベルでオーバーライド可能)
💡 考慮すべき点:
  • 組織レベルの設定変更は全プロジェクトに影響するため、変更前に影響範囲を十分に確認する必要がある
  • constraints/gcp.resourceLocations 組織ポリシーと組み合わせることで、より強固なロケーション制御を実現できる
  • Gemini in BigQuery のロケーション制御を目的とする場合、ユーザーが BigQuery Studio で個別にクエリロケーションを設定するとオーバーライドされる

🎯 ユースケース

🏦 ユースケース 1: 日本国内のデータレジデンシー要件への対応

シナリオ: 金融機関が日本国内でのデータ処理を組織全体のデフォルトとして設定し、データレジデンシー要件を満たす。

SQL 実装例 
-- 組織レベルで東京リージョンをデフォルトに設定
ALTER ORGANIZATION SET OPTIONS (
  default_location = 'asia-northeast1'
);

-- 組織ポリシーと併用してロケーション制限を強制
-- (Resource Manager の組織ポリシーで設定)

効果: ロケーション未指定のクエリやジョブが自動的に東京リージョンで実行され、意図しない海外リージョンでのデータ処理を防止。

🌍 ユースケース 2: グローバル組織でのリージョン別プロジェクト管理

シナリオ: 多国籍企業がリージョンごとに異なるプロジェクトを運用し、各プロジェクトのデフォルトロケーションを個別に設定。

SQL 実装例 
-- EU 向けプロジェクト
ALTER PROJECT eu_analytics SET OPTIONS (
  default_location = 'EU'
);

-- アジア太平洋向けプロジェクト
ALTER PROJECT apac_analytics SET OPTIONS (
  default_location = 'asia-northeast1'
);

-- 米国向けプロジェクト
ALTER PROJECT us_analytics SET OPTIONS (
  default_location = 'US'
);

効果: 各リージョンのデータレジデンシー要件に合わせたデフォルトロケーションが自動適用され、クロスリージョンのデータ処理リスクを低減。

💰 料金

追加料金なし: デフォルトロケーション構成機能自体には追加料金は発生しません。ただし、BigQuery のコンピュート (クエリ処理) およびストレージの料金はロケーションによって異なる場合があるため、デフォルトロケーションの選択はコストに間接的に影響します。

🌐 利用可能リージョン

デフォルトロケーションとして設定可能なリージョンは、BigQuery がサポートするすべてのリージョンおよびマルチリージョンです。主要なロケーションは以下の通りです。

マルチリージョン US - 米国内のデータセンター
マルチリージョン EU - EU 加盟国内のデータセンター
シングル asia-northeast1 - 東京
シングル asia-northeast2 - 大阪
シングル us-central1 - アイオワ
シングル europe-west1 - ベルギー

全リージョンの一覧は BigQuery ロケーションのドキュメントを参照のこと。

🔗 関連サービス・機能

📋 BigQuery デフォルト構成管理

デフォルトロケーション以外にも、タイムゾーン、KMS キー、クエリタイムアウト、ストレージ課金モデルなど、組織/プロジェクトレベルで管理できる構成設定群

🤖 Gemini in BigQuery

SQL エディタやデータキャンバスにおける AI 支援機能のデータ処理ロケーションがデフォルトロケーション設定に連動

🌐 BigQuery グローバルクエリ

複数リージョンにまたがるクエリ実行機能。デフォルトロケーション設定と併用することで、グローバルクエリの起点リージョンを制御可能

🔒 VPC Service Controls

デフォルトロケーション設定と組み合わせることで、データの処理場所とネットワーク境界の両面からデータ保護を強化

📝 まとめ

BigQuery のグローバルデフォルトロケーション構成が GA となったことで、組織レベルおよびプロジェクトレベルでのデータ処理ロケーションの一元管理が可能になりました。

📌 推奨アクション:
  • データレジデンシー要件を持つ組織は、まず組織レベルでデフォルトロケーションを設定
  • 必要に応じてプロジェクト単位でオーバーライドする構成を推奨
  • 特に Gemini in BigQuery を利用している場合、AI 処理のデータロケーション制御にも直結するため、早期の設定を検討
