📋 概要

Cloud SQL Enterprise Plus エディションで提供されるニアゼロダウンタイム計画オペレーション機能の対象インスタンスが拡大されました。セキュリティ要件やネットワーク構成上の理由で特定の設定を使用していたユーザーも、計画メンテナンス時のダウンタイムを 1 秒未満に抑えることが可能になります。

今回追加された対象構成

🔐 コネクタ強制 (Connector Enforcement)

Cloud SQL Auth Proxy / Language Connectors の使用を必須とする設定が有効なインスタンス

🌐 非 RFC 1918 IP + PSA

非 RFC 1918 IP アドレスを使用するプライベートサービスアクセス構成のインスタンス

🔄 Before / After

❌ Before (アップデート前)

  • コネクタ強制有効インスタンスはニアゼロダウンタイム対象外
  • 非 RFC 1918 IP + PSA 構成もニアゼロダウンタイム非対象
  • メンテナンス時に通常のダウンタイム (数分) が発生
  • 構成変更が困難でダウンタイム低減の手段が限られていた

✅ After (アップデート後)

  • コネクタ強制有効でもニアゼロダウンタイムが利用可能
  • 非 RFC 1918 IP + PSA 構成でもニアゼロダウンタイム適用
  • 計画メンテナンス時のダウンタイムを 1 秒未満に削減
  • セキュリティポリシーやネットワーク構成の変更不要

🏗️ アーキテクチャ図

ニアゼロダウンタイム計画オペレーションでは、メンテナンス適用済みのシャドウ VM を事前に準備し、データ同期後にサブ秒でフェイルオーバーを実行します。

sequenceDiagram
    participant Client as クライアント
    participant Primary as プライマリ VM
    participant Shadow as シャドウ VM (新)
    participant Storage as 永続ストレージ

    Note over Primary,Shadow: 計画メンテナンス開始
    Primary->>Shadow: データ同期・レプリケーション
    Shadow->>Storage: ストレージ接続確立
    Note over Shadow: メンテナンスバージョン適用済み
    Primary->>Client: 接続維持中
    Note over Primary,Shadow: フェイルオーバー実行 (1秒未満)
    Client-->>Shadow: 接続切替 (サブ秒)
    Shadow->>Client: 新しい接続で応答
    Note over Shadow: メンテナンス完了
    Note over Primary: 旧 VM 停止
💡 ポイント: コネクタ強制や非 RFC 1918 IP 構成のインスタンスでも、このフェイルオーバープロセスが適用されるようになりました。フェイルオーバー時もコネクタ経由の接続が維持されます。

⚡ 主要機能

🔐 コネクタ強制有効インスタンスの対象化

  • Cloud SQL Auth Proxy や Cloud SQL Language Connectors の使用を強制する設定が有効なインスタンスでニアゼロダウンタイムが利用可能に
  • IAM 認証を必須とする設定であり、多くのエンタープライズ環境で採用されている
  • フェイルオーバー時もコネクタ経由の接続が維持される

🌐 非 RFC 1918 IP + プライベートサービスアクセス対応

  • RFC 1918 以外の IP アドレス範囲 (例: 100.64.0.0/10 の CGNAT 範囲、組織所有パブリック IP のプライベート利用) で PSA を使用するインスタンスが対象に
  • IP アドレス枯渇対策として非 RFC 1918 アドレスを使用している大規模環境に恩恵

🔧 設定方法

ステップ 1: ニアゼロダウンタイムのシミュレーション

実際のメンテナンス前にシミュレーションを実行して、ダウンタイムが 1 秒未満であることを確認できます。

Bash メンテナンスシミュレーション 
# メンテナンスシミュレーションの実行
gcloud sql instances patch INSTANCE_NAME --simulate-maintenance-event

ステップ 2: コネクタ強制の確認

Bash コネクタ強制設定の確認 
# インスタンスの設定確認
gcloud sql instances describe INSTANCE_NAME \
  --format="value(settings.connectorEnforcement)"

コネクタ強制が有効な場合、出力に REQUIRED と表示されます。

ステップ 3: 非 RFC 1918 IP 構成の確認

Bash ネットワークピアリングの確認と設定 
# ネットワークピアリングのルートエクスポート設定確認
gcloud compute networks peerings list --network=VPC_NETWORK_NAME

# 非 RFC 1918 ルートのエクスポート設定
gcloud compute networks peerings update PEERING_CONNECTION \
  --network=VPC_NETWORK_NAME \
  --export-subnet-routes-with-public-ip \
  --project=PROJECT_ID

⚠️ 制約事項・注意点

制限事項
  • ❌Cloud SQL Enterprise Plus エディションのみ対象 (Enterprise エディションでは利用不可)
  • ❌レプリカインスタンス (リードレプリカ含む) は非対象
  • ❌MySQL: バイナリログ有効 + 特定フラグがデフォルト値である必要あり
  • ❌スケールダウンは 3 時間に 1 回のみニアゼロダウンタイム適用
💡 考慮すべき点
  • 高負荷クエリ実行中はダウンタイムが 1 秒を超える可能性あり
  • DDL 実行中のメンテナンスでタイムスタンプにずれが発生する可能性
  • MySQL: メンテナンス後にサーバー UID が変更される
  • PostgreSQL: 未ログテーブル (unlogged tables) がメンテナンス後に空になる
  • クライアント側の接続設定 (タイムアウト、コネクションプーリング等) が体感に影響

💼 ユースケース

🏦 ユースケース 1: コネクタ強制を使用するエンタープライズ環境

シナリオ: 金融機関のデータベースでセキュリティポリシーにより Cloud SQL Auth Proxy 経由の接続のみを許可。メンテナンス時に数分のダウンタイムが発生し、深夜帯にメンテナンスウィンドウを設定する必要があった。

Bash シミュレーション実行例 
# コネクタ強制が有効なインスタンスでシミュレーション実行
gcloud sql instances patch my-finance-db --simulate-maintenance-event

効果: メンテナンスウィンドウを業務時間内に設定可能になり、運用チームの深夜対応が不要に

🏢 ユースケース 2: 大規模 VPC での非 RFC 1918 IP 利用

シナリオ: 大規模マルチプロジェクト環境で RFC 1918 アドレス空間が枯渇し、PSA に非 RFC 1918 アドレス (Privately Used Public IP) を割り当て。数百台の Cloud SQL インスタンスのメンテナンスを効率的に管理したい。

効果: ネットワーク構成を変更せずに全インスタンスでニアゼロダウンタイムメンテナンスが適用され、四半期メンテナンスの影響を最小限に抑制
