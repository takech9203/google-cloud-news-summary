🗄️ Cloud SQL: Knowledge Catalog 自動統合によるデータディスカバリー

2026-04-18 | Cloud SQL for PostgreSQL / Cloud SQL for SQL Server
Feature (新機能) Cloud SQL PostgreSQL SQL Server Knowledge Catalog

📋 概要

Cloud SQL for PostgreSQL および Cloud SQL for SQL Server の新規作成インスタンスが、Knowledge Catalog (旧 Dataplex Universal Catalog) と自動的に統合されるようになりました。この統合により、Cloud SQL インスタンスのメタデータが Knowledge Catalog に自動送信され、組織全体でのデータディスカバリーが容易になります。

PostgreSQL の場合、バージョン 14.0 以降のインスタンスでは、メタデータの更新がニアリアルタイムで Knowledge Catalog に反映されます。Knowledge Catalog コンソールの構成ペインで統合が有効かどうかを確認でき、不要な場合はオプトアウト (無効化) することも可能です。

🔄 Before / After 比較

❌ アップデート前

  • Knowledge Catalog への登録には手動設定が必要だった
  • 新規インスタンス作成時に統合を忘れるとデータサイロが発生
  • 各チームが個別にカタログ統合を設定する運用負荷があった

✅ アップデート後

  • 新規インスタンスは自動的に Knowledge Catalog と統合
  • PostgreSQL v14.0+ はニアリアルタイムでメタデータ反映
  • 組織のデータカタログ網羅性が向上し、ディスカバリーが容易に
  • 不要な場合はオプトアウト可能で柔軟な運用が可能

Cloud SQL インスタンスで作成・変更されたメタデータ (インスタンス、データベース、テーブル、カラム、ビュー) が自動的に Knowledge Catalog に送信され、データアナリストやデータスチュワード、AI エージェントが統合的にデータを検索・管理できます。

⚡ 主要機能

🔗 新規インスタンスの自動統合

  • 新規 Cloud SQL for PostgreSQL / SQL Server インスタンスはデフォルトで Knowledge Catalog 統合が有効化
  • 手動での有効化操作が不要
  • データカタログへの登録漏れを防止

⏱️ ニアリアルタイム同期 (PG v14.0+)

  • PostgreSQL v14.0 以降でメタデータ変更がニアリアルタイムで反映
  • テーブル作成・変更、スキーマ更新が即座にカタログに反映
  • Pub/Sub 経由でのメタデータ変更通知も可能

📊 収集メタデータの範囲

  • インスタンス名、ロケーション、作成日時、最終更新日時
  • データベース、テーブル、カラム、ビューの構造情報
  • スキーマ情報 (テーブルおよびビュー)
  • プライマリインスタンスのみ (リードレプリカ対象外)

🔧 オプトアウト機能

  • 統合が不要な場合はインスタンスごとに無効化が可能
  • gcloud CLI、REST API、Google Cloud コンソールから設定変更
  • フラグ: --no-enable-dataplex-integration

⚠️ 注意: cloudsql.schemas.view 権限はプロジェクトレベルで付与する必要があり、プロジェクト内の全 Cloud SQL インスタンスのメタデータへのアクセスを許可します。インスタンス単位での細かいアクセス制御は不可。

📝 設定方法

前提条件

  • Google Cloud プロジェクトで Dataplex API が有効化されていること
  • cloudsql.schemas.view 権限を含む IAM ロールが付与されていること
  • Cloud SQL for PostgreSQL (v14.0 以降推奨) または Cloud SQL for SQL Server のインスタンス

ステップ 1: 統合状態の確認

Bash インスタンスの Knowledge Catalog 統合状態を確認 
# インスタンスの Knowledge Catalog 統合状態を確認
gcloud sql instances describe INSTANCE_NAME

出力の settings セクションで enableDataplexIntegration: true を確認します。新規作成インスタンスではデフォルトで有効です。

ステップ 2: 既存インスタンスへの統合有効化 (任意)

Bash 既存インスタンスに統合を有効化 
# 既存インスタンスに Knowledge Catalog 統合を有効化
gcloud sql instances patch INSTANCE_NAME \
    --enable-dataplex-integration
Bash プロジェクト内の全インスタンスに一括有効化 
# プロジェクト内の全インスタンスに一括有効化
gcloud sql instances list --format="(NAME)" \
    | tail -n +2 | xargs -t -I % gcloud sql instances patch % --enable-dataplex-integration

ステップ 3: 統合の無効化 (オプトアウト)

Bash 統合の無効化 
# インスタンスの Knowledge Catalog 統合を無効化
gcloud sql instances patch INSTANCE_NAME \
    --no-enable-dataplex-integration
JSON REST API でのリクエストボディ 
{
  "settings": {
    "enableDataplexIntegration": false
  }
}

ステップ 4: Knowledge Catalog でのデータ確認

Knowledge Catalog コンソール (Google Cloud コンソール > Dataplex > Knowledge Catalog) で、Cloud SQL のメタデータが登録されていることを確認します。AI 検索やキーワード検索を使用して、Cloud SQL のテーブルやビューを発見できます。

⚖️ メリット・デメリット

✅ ビジネス面のメリット

  • データガバナンスの自動化: 新規インスタンスが自動的にカタログ登録され、管理者の運用負荷を軽減
  • データディスカバリー効率化: 自然言語検索で組織全体の Cloud SQL データ資産を迅速に発見
  • コンプライアンス強化: メタデータの一元管理で PII 追跡やデータリネージ可視化が容易に

✅ 技術面のメリット

  • ゼロコンフィギュレーション: 新規インスタンス作成時に追加設定不要
  • ニアリアルタイム同期: PG v14.0+ でスキーマ変更がほぼ即時反映
  • AI 検索統合: インテントベースの自然言語検索が可能
  • メタデータ変更フィード: Pub/Sub 経由で変更通知を受信し自動ワークフローをトリガー

⚠️ 制限事項

  • 自動統合は新規作成インスタンスのみ。既存インスタンスは手動有効化が必要
  • メタデータ取得はプライマリインスタンスのみ対象 (リードレプリカ除外)
  • cloudsql.schemas.view はプロジェクトレベルの権限 (インスタンス単位の制御不可)
  • PostgreSQL 14.0 未満ではニアリアルタイム同期が利用できない可能性

⚠️ 考慮すべき点

  • 自動有効化で意図せずメタデータが送信される場合がある。セキュリティポリシーに応じてオプトアウトを検討
  • Knowledge Catalog のメタデータストレージ SKU に基づく料金が発生する可能性
  • VPC Service Controls の境界設定により検索結果が制限される場合がある

💡 ユースケース

🏢 ユースケース 1: 組織横断的なデータディスカバリー

シナリオ: 大規模企業で複数の事業部門がそれぞれ Cloud SQL for PostgreSQL インスタンスを運用しており、データアナリストが特定のビジネスデータがどのインスタンスのどのテーブルに存在するかを把握できていない。

Bash 新規インスタンス作成 (自動統合) 
# 新規インスタンスは自動的に Knowledge Catalog に登録される
gcloud sql instances create sales-db \
    --database-version=POSTGRES_16 \
    --tier=db-custom-4-16384 \
    --region=asia-northeast1
効果: データアナリストが Knowledge Catalog の自然言語検索で、組織内の Cloud SQL インスタンスに分散したデータを横断的に発見でき、データアクセスまでの時間を大幅に短縮。

🏦 ユースケース 2: データスチュワードによるメタデータガバナンス

シナリオ: 金融機関でデータスチュワードが、Cloud SQL に格納されている個人情報 (PII) を含むテーブルを特定し、適切なメタデータラベル (アスペクト) を付与する必要がある。

効果: Knowledge Catalog に自動登録された Cloud SQL のスキーマ情報から PII カラムを特定し、アスペクトを使用してメタデータをエンリッチメント。コンプライアンス要件に対応したデータガバナンスを実現。

🤖 ユースケース 3: AI エージェントによるデータ活用

シナリオ: 生成 AI アプリケーションが、組織内のデータソースから適切なデータを自動選択してレポートを生成する必要がある。Knowledge Catalog の MCP インターフェースを通じて、AI エージェントがデータソースを検索。

効果: Cloud SQL のメタデータが自動登録されることで、AI エージェントが信頼性の高いメタデータに基づいてデータソースを発見・選択し、ハルシネーションリスクを低減したデータ活用が可能に。

💰 料金情報

Knowledge Catalog の利用料金はメタデータストレージ SKU に基づきます。Cloud SQL 側での統合有効化に追加料金は発生しません。

Cloud SQL for PostgreSQL および Cloud SQL for SQL Server が利用可能なすべてのリージョンで、Knowledge Catalog との統合が利用可能です。

🔗 関連サービス・機能

📚

Knowledge Catalog (Dataplex)

Cloud SQL のメタデータを統合管理するカタログサービス。AI 検索、メタデータエンリッチメント、データリネージ機能を提供。

📊

BigQuery

Knowledge Catalog と統合されたもう一つの主要データソース。Cloud SQL と BigQuery のメタデータを横断的に検索可能。

🐘

AlloyDB for PostgreSQL

Cloud SQL と同様に Knowledge Catalog との統合をサポート。クラスタレベルでの統合有効化が可能。

🪣

Cloud Storage

Knowledge Catalog でオブジェクトストレージのメタデータも管理可能。非構造化データのディスカバリーにも対応。

📨

Pub/Sub

Knowledge Catalog のメタデータ変更フィード機能と連携し、メタデータ変更の通知をリアルタイムで受信可能。

🔒

Sensitive Data Protection

データの機密性分類結果を Knowledge Catalog と連携させることで、PII データの検出・管理を強化。
