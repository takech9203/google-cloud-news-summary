📋 概要

Google Cloud IAM において、AI エージェントのセキュリティ管理に関する 2 つの重要なアップデートが発表されました。

✅ Agent Identity (GA)

エージェントごとに SPIFFE ベースの暗号学的 ID を自動プロビジョニング。サービスアカウントに依存しない安全な認証をプロダクションレベルで提供。Context-Aware Access (CAA) による mTLS バインディング、証明書バウンドトークン、IAM Allow/Deny ポリシーとの統合が本番 SLA で利用可能。

🔧 Agent Identity Auth Manager (Preview)

エージェントがサードパーティサービスに対して 3-legged OAuth、2-legged OAuth、API キーで安全に認証するための統合管理機能。Secret Manager と連携し、資格情報を暗号化された状態で管理。

🔄 Before / After 比較

❌ Before

  • エージェントの認証にサービスアカウントを使用。エージェント固有の ID がリソースのライフサイクルと紐づかない
  • サービスアカウントキーの漏洩リスクや過剰な権限付与が課題
  • サードパーティ認証 (OAuth, API キー) は開発者が個別に実装。認証ロジックの分散と不統一がセキュリティリスクに
  • アクセストークンの保護が不十分で、リプレイ攻撃の可能性あり

✅ After

  • Agent Identity (GA) により、SPIFFE ベースの暗号学的 ID が自動プロビジョニング。サービスアカウント不要
  • エージェントのライフサイクルに紐づいた ID 管理で最小権限の原則を実現
  • Auth Manager により 3 つの認証方式を統合管理。開発者の認証実装負担を大幅に軽減
  • CAA ポリシーによる証明書バウンドトークンで mTLS 保護。意図されたランタイム環境でのみ使用可能

⭐ 主要機能

🔒 Agent Identity (GA) - エージェント固有の暗号学的 ID
  • エージェントのデプロイ時に SPIFFE ベースの固有 ID が自動プロビジョニング
  • X.509 クライアント証明書の自動管理、mTLS による相互認証
  • IAM の Allow/Deny ポリシー、Principal Access Boundary ポリシーと統合
  • アクセストークンは証明書にバウンドされ、意図されたランタイム環境でのみ使用可能 (CAA 保護)
🌐 Auth Manager (Preview) - サードパーティ認証の統合管理
  • 3-legged OAuth: ユーザーの同意を得てエージェントがサードパーティサービスにアクセス (ユーザー委任認証)
  • 2-legged OAuth: サービス間のマシン対マシン認証。ユーザーの介入なし
  • API キー認証: Secret Manager に安全に保管された API キーをランタイムで取得
  • Secret Manager との統合でクライアント ID/シークレットおよび API キーを暗号化管理
🛡️ Context-Aware Access (CAA) によるトークン保護
  • デフォルトで有効化される Google 管理の CAA ポリシー
  • mTLS バインディングによる証明書バウンドトークンのリプレイ攻撃防止
  • RFC 8705 準拠の証明書バウンドトークンで資格情報窃取への耐性を実現
  • Cloud Run コンテナなど信頼されたランタイム環境からのみトークン使用可能

ステップ 1: Agent Identity を持つエージェントの作成

Python Agent Identity を有効にしてエージェントをデプロイ 
import vertexai
from vertexai import agent_engines
from vertexai import types

client = vertexai.Client(
    project="PROJECT_ID",
    location="LOCATION",
    http_options=dict(api_version="v1beta1")
)

# Agent Identity を有効にしてエージェントをデプロイ
remote_app = client.agent_engines.create(
    config={
        "display_name": "my-agent-with-identity",
        "identity_type": types.IdentityType.AGENT_IDENTITY,
    }
)

print(f"Agent Identity: {remote_app.api_resource.spec.effective_identity}")

ステップ 4: ADK エージェントでの OAuth 認証の実装

Python ADK エージェントで OAuth 認証を統合 
from google.adk.auth.auth_schemes import OpenIdConnectWithConfig
from google.adk.auth.auth_credential import AuthCredential, AuthCredentialTypes, OAuth2Auth
from google.adk.tools.openapi_tool.openapi_spec_parser.openapi_toolset import OpenAPIToolset
from google.adk.agents.llm_agent import LlmAgent
from google.cloud import secretmanager
from google.auth import default

def access_secret(project_id: str, secret_id: str, version_id: str = "latest") -> str:
    # Agent Identity の ADC で Secret Manager にアクセス
    credentials, _ = default()
    client = secretmanager.SecretManagerServiceClient(credentials=credentials)
    name = f"projects/{project_id}/secrets/{secret_id}/versions/{version_id}"
    response = client.access_secret_version(request={"name": name})
    return response.payload.data.decode("UTF-8")

# OAuth 認証スキームの定義
auth_scheme = OpenIdConnectWithConfig(
    authorization_endpoint="https://example.com/oauth2/authorize",
    token_endpoint="https://example.com/oauth2/token",
    scopes=['openid', 'profile', 'email']
)

# Auth Manager 経由で Secret Manager から資格情報を取得
auth_credential = AuthCredential(
    auth_type=AuthCredentialTypes.OPEN_ID_CONNECT,
    oauth2=OAuth2Auth(
        client_id=access_secret(project_id='my-project', secret_id='oauth_client_id'),
        client_secret=access_secret(project_id='my-project', secret_id='oauth_client_secret'),
    )
)

# ツールセットに認証を適用
toolset = OpenAPIToolset(
    spec_str=spec_content,
    spec_str_type='yaml',
    auth_scheme=auth_scheme,
    auth_credential=auth_credential,
)

# エージェントの作成
agent = LlmAgent(
    model='gemini-2.5-flash',
    name='authenticated_agent',
    instruction='サードパーティサービスと安全に連携するアシスタント',
    tools=[toolset],
)

⚖️ メリット・デメリット

✅ メリット

ビジネス面
  • コンプライアンス強化: エージェントごとの固有 ID により、アクセス追跡と監査対応が容易に
  • 運用コスト削減: Auth Manager でサードパーティ認証の実装・管理が統合化
  • セキュリティリスク低減: サービスアカウントキーの共有・管理が不要に
技術面
  • ゼロトラストセキュリティ: SPIFFE ID + mTLS + 証明書バウンドトークンの多層防御
  • 最小権限の原則: エージェント単位の IAM ポリシー設定が可能
  • ライフサイクル管理の自動化: エージェント削除時に ID も自動無効化
  • ADC との統合: 既存の Google Cloud SDK 認証ライブラリがそのまま利用可能

⚠️ デメリット・制約事項

制限事項
  • Auth Manager は Preview 段階。Pre-GA Offerings Terms が適用
  • Cloud Storage の Legacy Bucket ロールは付与不可
  • Vertex AI Agent Engine Runtime 上のエージェントに限定 (他環境は Managed Workload Identity を使用)
  • CAA オプトアウト時はセキュリティレベルが低下
考慮すべき点
  • 既存サービスアカウントからの移行計画が必要
  • Secret Manager の料金が別途発生
  • 3-legged OAuth ではフロントエンド側の認可フロー実装が必要
  • トークン有効期限はデフォルト 1 時間。長時間タスクでは自動更新を考慮

💡 ユースケース

💬 Slack 連携エージェントのセキュアな認証

シナリオ: 企業の IT チームが、社内問い合わせに自動応答する AI エージェントを構築。Slack API でメッセージの送受信を行い、ユーザーごとの権限レベルに応じたアクセス制御が必要。

Python Slack OAuth 認証の例 
from google.adk.auth.auth_schemes import OpenIdConnectWithConfig
from google.adk.auth.auth_credential import AuthCredential, AuthCredentialTypes, OAuth2Auth

# Agent Identity Auth Manager 経由で Slack の OAuth 資格情報を取得
auth_scheme = OpenIdConnectWithConfig(
    authorization_endpoint="https://slack.com/oauth/v2/authorize",
    token_endpoint="https://slack.com/api/oauth.v2.access",
    scopes=['chat:write', 'channels:read']
)

auth_credential = AuthCredential(
    auth_type=AuthCredentialTypes.OPEN_ID_CONNECT,
    oauth2=OAuth2Auth(
        client_id=access_secret('my-project', 'slack_client_id'),
        client_secret=access_secret('my-project', 'slack_client_secret'),
    )
)
✔️ 効果: 3-legged OAuth によりユーザーごとの権限でエージェントが Slack にアクセスし、Agent Identity により Google Cloud リソースへのアクセスも最小権限で管理。資格情報は Secret Manager で一元管理されキーローテーションも容易。

🤖 マルチエージェントシステムでの相互認証

シナリオ: 複数の AI エージェントが協調して業務を遂行するマルチエージェントシステムにおいて、エージェント間の通信を安全に行い、各エージェントのアクセス権限を個別に管理。

Bash エージェントごとの IAM ポリシー設定 
# エージェント A: データ分析エージェント (BigQuery アクセス権限)
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="principal://agents.global.org-ORG_ID.system.id.goog/resources/aiplatform/projects/PROJECT_NUM/locations/us-central1/reasoningEngines/agent-a" \
  --role="roles/bigquery.dataViewer"

# エージェント B: レポート生成エージェント (Cloud Storage アクセス権限)
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="principal://agents.global.org-ORG_ID.system.id.goog/resources/aiplatform/projects/PROJECT_NUM/locations/us-central1/reasoningEngines/agent-b" \
  --role="roles/storage.objectCreator"
✔️ 効果: 各エージェントが固有の Agent Identity を持ち、IAM ポリシーにより最小限のリソースにのみアクセス可能。A2A プロトコルとの連携でエージェント間通信も mTLS で保護。Cloud Logging でエージェントごとのアクセスログを記録し監査が容易。
