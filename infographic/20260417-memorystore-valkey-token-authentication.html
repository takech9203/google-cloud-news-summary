🔑 Memorystore for Valkey: Basic Token-Based Authentication

2026-04-17 | Memorystore for Valkey
Memorystore Valkey Security Preview

📋 概要

Memorystore for Valkey に基本トークンベース認証 (Basic Token-Based Authentication) が Preview として追加されました。既存の IAM 認証に加えて、軽量なトークンベースの認証方式を使用してインスタンスへのアクセスを保護できます。特に Memorystore for Redis やオンプレミス環境からの移行をスムーズに行うことができます。

❌ Before

  • IAM 認証のみが利用可能
  • アクセストークンの有効期限が短く (1 時間)、自動更新ロジックが必要
  • Redis からの移行時に AUTH ベースの認証を引き継げない
  • オンプレミスのシンプルなトークン認証を再現できない

✅ After

  • 軽量なトークンベース認証が選択可能に
  • Redis / オンプレミスからの後方互換性を確保
  • ゼロダウンタイムのトークンローテーション
  • 最大 1,000 ローカルユーザーできめ細かなアクセス制御

⚡ 主要機能

🔑 シンプル認証

デフォルトユーザーとしてトークンのみで認証。インスタンス作成時または有効化時にデフォルトユーザーとトークンが自動生成されます。単一ユーザーのアクセス制御に最適です。

👥 マルチユーザー認証

ユーザー名とトークンの組み合わせで認証。複数のユーザーを作成し、それぞれに個別のトークンを割り当てて、きめ細かなアクセス制御を実現します。

🔄 ゼロダウンタイムトークンローテーション

各ユーザーは最大 2 つの認証トークンを保持可能。新しいトークンを作成→アプリ更新→古いトークン削除で、ダウンタイムなしのローテーションが可能です。

📝 監査ログ連携

認証トークンとユーザーに関連する操作について Admin Activity および Data Access 監査ログが生成され、Cloud Audit Logs との統合によりアクセスの監視と監査が可能です。

🔄 トークンローテーションフロー

ゼロダウンタイムでのトークン更新手順。各ユーザーが最大 2 つのトークンを同時保持できることを利用します。

接続方法

Bash URI 文字列を使用した接続 
valkey-cli -u redis://USERNAME:TOKEN@IP_ADDRESS:PORT
Bash フラグを使用した接続 
valkey-cli --user USERNAME -a TOKEN -h IP_ADDRESS -p PORT

🛠️ 設定方法

ℹ️ 前提条件: Google Cloud プロジェクトで Memorystore for Valkey API が有効化されていること、gcloud CLI がインストールされていること、適切な IAM 権限が付与されていること。

ステップ 1: 新規インスタンスでトークンベース認証を有効化

Bash 新規インスタンス作成 
gcloud beta memorystore instances create INSTANCE_ID \
  --location=REGION \
  --authorization-mode=token-auth

ステップ 2: 既存インスタンスで有効化

Bash 既存インスタンスへの適用 
gcloud beta memorystore instances update INSTANCE_ID \
  --location=REGION \
  --authorization-mode=token-auth

ステップ 3: 追加ユーザーの作成

Bash ユーザー作成 
gcloud beta memorystore instances create-token-auth-user INSTANCE_ID \
  --location=REGION \
  --token-auth-user=USERNAME

ステップ 4: トークンローテーション (ゼロダウンタイム)

Bash ゼロダウンタイムトークンローテーション 
# 1. 新しいトークンを作成 (両方のトークンが有効)
gcloud beta memorystore instances token-auth-users create-auth-token USERNAME \
  --instance=INSTANCE_ID \
  --location=REGION

# 2. アプリケーションを新しいトークンに更新

# 3. 古いトークンを削除
gcloud beta memorystore instances token-auth-users auth-tokens delete AUTH_TOKEN \
  --instance=INSTANCE_ID \
  --location=REGION \
  --token-auth-user=USERNAME

💼 ビジネス面

✓ 移行コストの削減 - Redis / オンプレミスからの移行時に既存の認証方式との後方互換性でアプリ側の変更を最小限に
✓ 運用負荷の軽減 - ゼロダウンタイムトークンローテーションで計画的なメンテナンスウィンドウが不要

⚙️ 技術面

✓ 軽量な認証 - リソース要件が最小限、IAM 認証よりオーバーヘッドが低い
✓ 柔軟なユーザー管理 - 最大 1,000 ユーザーでマルチテナント環境に対応
✓ 監査対応 - Cloud Audit Logs との統合でコンプライアンス要件に対応

⚠️ デメリット・制約事項

⚠️ 制限事項
  • ✗ 現在 Preview ステータス——本番環境では Pre-GA 利用規約が適用
  • ✗ 有効化後は無効化できない——設定変更は慎重に
  • ✗ デフォルトユーザーは削除不可
  • ✗ 各ユーザーのトークンは最大 2 つまで
  • ✗ gcloud beta コマンドの使用が必要 (GA 版未対応)
💡 考慮すべき点
  • ● 有効化後は新しいクライアント接続に認証が必須となるため、既存アプリの更新が必要
  • ● TLS (In-Transit Encryption) と組み合わせて使用することを推奨 (トークンの平文送信防止)
  • ● トークンはアプリコードにハードコードせず、Secret Manager に保管することを推奨

💡 ユースケース

🔄 ユースケース 1: Memorystore for Redis からの移行

シナリオ: 既存の Memorystore for Redis インスタンスで AUTH パスワードベースの認証を使用しており、Memorystore for Valkey へ移行したい場合。

Bash 実装例 
# Valkey インスタンスをトークンベース認証で作成
gcloud beta memorystore instances create my-valkey-instance \
  --location=asia-northeast1 \
  --authorization-mode=token-auth

# アプリケーションの接続設定を更新
valkey-cli -u redis://default:TOKEN@INSTANCE_IP:6379

効果: 既存の AUTH コマンド互換の認証方式を維持しつつ、Valkey の新機能を活用可能。アプリ側の認証ロジックの変更が最小限で済みます。

👥 ユースケース 2: マルチテナント環境でのアクセス制御

シナリオ: 複数のマイクロサービスが同一の Memorystore for Valkey インスタンスにアクセスする環境で、サービスごとに異なる認証情報を割り当てたい場合。

効果: 各サービスに固有のユーザーとトークンを割り当てることで、アクセスの追跡と制御が容易に。不正アクセス発生時にも、該当ユーザーのトークンのみを無効化して影響範囲を限定できます。

💰 料金

✅ 追加料金なし

基本トークンベース認証の利用に追加料金は発生しません。Memorystore for Valkey の通常のインスタンス料金が適用されます。

料金はノードタイプ、シャード数、レプリカ数に基づいて計算されます。Committed Use Discounts (CUD) で 1 年契約 20%、3 年契約 40% の割引が適用されます。

📝 まとめ

Memorystore for Valkey に基本トークンベース認証が Preview として追加されたことで、IAM 認証に加えて軽量で広くサポートされた認証方式が利用可能になりました。特に Memorystore for Redis やオンプレミス環境からの移行を検討している場合、後方互換性のある認証方式によりスムーズな移行が実現できます。

📌 推奨アクション: Preview ステータスであるため本番環境への導入は慎重に検討し、開発・検証環境での評価を開始して GA 昇格に備えることを推奨します。
