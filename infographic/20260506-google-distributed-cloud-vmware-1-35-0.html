📄 概要

Google Distributed Cloud (software only) for VMware 1.35.0-gke.525 がリリースされました。Kubernetes 1.35 への大規模プラットフォームアップデートを含み、コンテナランタイム、OS イメージ、cgroups の要件に関する複数の破壊的変更を伴うメジャーリリースです。

⚠️ 重要な破壊的変更: cgroupsv2 が必須要件となり、cgroupsv1 のサポートが完全に廃止されました。プリフライトチェックにより cgroupsv1 が検出された場合、クラスタの作成やアップグレードが自動的にブロックされます。

❌ Before (1.34.x)

  • レガシー OS イメージタイプ (ubuntu, ubuntu_containerd, cos) が利用可能

  • vsphere-csi-controller がコントロールプレーンに移動
  • ノードプール更新がデフォルトで並列実行

⚡ 主要機能

⚙️ Kubernetes 1.35 プラットフォームアップデート

Kubernetes v1.35.2-gke.300 をベースとして動作。最新の Kubernetes API および機能を利用可能です。

🚨 cgroupsv2 必須化 (破壊的変更)

  • cgroupsv1 のサポートが完全に廃止
  • クラスタ作成・アップグレード時にプリフライトチェックが自動実行
  • cgroupsv1 検出時はオペレーションが自動ブロック
  • 事前に全ノードの cgroups バージョン確認・移行が必要

📦 コンテナランタイムアップグレード

  • containerd: 2.0 → 2.1 にアップグレード
  • Ubuntu イメージ: 全ノードタイプで 24.04 に統一

🚫 レガシー OS イメージタイプの廃止

ubuntu ubuntu_containerd cos

これらの OSImageType を使用しているクラスタはアップグレード前に設定変更が必要です。

🚀 Advanced Cluster 向けの改善

  • ノードプール更新ポリシー: シーケンシャル → 並列実行 (デフォルト)
  • Docker ブリッジ IP: 172.17.0/16 → 169.254.123.1/24
  • vsphere-csi-controller: ワーカーノード → コントロールプレーンノード

🛠️ オペレーション改善

gkectl がクラスタ操作後に Operation ID と Operation Type をコンソールに出力。トラブルシューティングや監査ログとの連携が容易になりました。

🔧 主要な修正 (9件)

✅ 非 Advanced → Advanced アップグレードのスタック問題
✅ gke-nodepool ラベルの不正な -np サフィックス
✅ stackdriver.enableVPC がアップグレードをブロック
✅ node-problem-detector の誤デプロイ
✅ プロキシ設定のホワイトスペース問題
✅ gkectl upgrade admin リトライの AlreadyExists エラー
✅ レジストリミラー CA 証明書のシステム証明書プール無視
✅ vSphere VM 作成の無期限ハング (1時間タイムアウト追加)
✅ Advanced Cluster アップグレード後の OIDC ログイン失敗

📝 設定方法

📌 前提条件:
  • 全ノードが cgroupsv2 で動作していること
  • レガシー OS イメージタイプ (ubuntu, ubuntu_containerd, cos) を使用していないこと
  • GDC for VMware 1.34.x が稼働していること

Step 1: cgroupsv2 の確認

Bash 
# 各ノードで cgroups バージョンを確認
stat -fc %T /sys/fs/cgroup/
# "cgroup2fs" と表示されれば cgroupsv2

Step 2: クラスタのアップグレード

Bash 
# アップグレードの実行
gkectl upgrade cluster \
  --kubeconfig [ADMIN_CLUSTER_KUBECONFIG] \
  --config [USER_CLUSTER_CONFIG]
⚠️ 注意: アップグレード時にプリフライトチェックが自動的に実行され、cgroupsv1 が検出された場合はオペレーションがブロックされます。

🌟 メリット

💼 ビジネス面

📈 運用効率の向上: ノードプールの並列更新により、大規模クラスタのアップグレード時間を大幅に短縮
💪 安定性の向上: 多数のバグ修正により、アップグレード失敗のリスクを低減

💻 技術面

🛡️ 最新プラットフォーム基盤: Kubernetes 1.35、Ubuntu 24.04、containerd 2.1
🔗 ネットワーク競合の排除: Docker ブリッジ IP のリンクローカルアドレスへの変更
⚡ リソース効率: vsphere-csi-controller のコントロールプレーン移動でワーカーリソース解放
🔍 デバッグ容易性: gkectl の Operation ID/Type 出力で問題追跡が容易に

⚠️ デメリット・制約事項

制限事項:
  • cgroupsv1 環境ではアップグレード不可 (プリフライトチェックによるブロック)
  • レガシー OS イメージタイプ使用クラスタは事前に設定変更が必要
  • Docker ブリッジ IP の変更が既存ワークロードに影響する可能性 (Advanced Cluster のみ)
考慮すべき点:
  • cgroupsv2 への移行は古いコンテナイメージに互換性問題を引き起こす可能性
  • containerd 2.0 → 2.1 の設定ファイル形式変更の可能性
  • GKE On-Prem API クライアント (Console, gcloud, Terraform) で利用可能になるまで約 7-14 日

🎯 ユースケース

🏢 大規模オンプレミス環境のモダナイゼーション

シナリオ: 数十台のノードを持つ VMware 上の GDC クラスタを運用しており、OS とランタイムの最新化が必要な環境。

効果: Ubuntu 24.04 と containerd 2.1 への自動移行により、セキュリティパッチの適用範囲が広がり、最新の CVE に対する保護が強化されます。

🔗 ネットワーク競合の解消

シナリオ: 企業ネットワークで 172.17.0.0/16 帯を使用しており、Docker ブリッジ IP との競合で問題が発生していた環境。

効果: デフォルト Docker ブリッジ IP が 169.254.123.1/24 (リンクローカル) に変更され、企業ネットワーク帯との競合が解消されます。
