🛡️ Google Cloud Armor: 事前構成 WAF ルールが ModSecurity CRS 4.22 に対応

2026-04-06
Google Cloud Armor Security Preview

📋 概要

Google Cloud Armor の事前構成 WAF (Web Application Firewall) ルールが、ModSecurity Core Rule Set (CRS) 4.22 をルールソースとして新たにサポートしました。これまでの CRS 3.3.2 / 3.0 に加え、最新の CRS 4.22 が選択可能になります。

CRS 4.x 系はアーキテクチャの大幅な刷新を含み、誤検知の低減、検出精度の向上、プラグインアーキテクチャの導入など多くの改善が盛り込まれています。

🔄 Before / After

❌ アップデート前の課題

  • CRS 3.3.2 / 3.0 のみサポートで、最新の攻撃パターンへの対応が限定的
  • CRS 3.x 系では誤検知 (false positive) が発生しやすく、チューニングに多くの工数が必要
  • モノリシックなルール構成で、必要なルールだけを選択的に適用する柔軟性が不足

✅ アップデート後の改善

  • CRS 4.22 で最新の攻撃ベクターや脆弱性パターンの検出カバレッジが向上
  • 改善された検出ロジックにより誤検知が低減し、チューニング工数を削減
  • より細かい粒度でのルール制御が可能になり、柔軟な WAF 設定を実現

⭐ 主要機能

🛡️ CRS 4.22 ベースの事前構成 WAF ルール

ModSecurity Core Rule Set 4.22 をルールソースとした新しい事前構成ルールセットが利用可能。SQL インジェクション、XSS、RCE、LFI、RFI など主要な攻撃カテゴリをカバー。stable / canary の 2 チャネルで提供される見込みです。

🧠 CRS 4.x 系の改善された検出ロジック

誤検知の低減を重視した検出ロジックの再設計。プラグインアーキテクチャによりルールセットの拡張性が向上。感度レベル (sensitivity) に基づくより精緻な制御が可能になっています。

🤝 既存 CRS バージョンとの共存

CRS 3.0 および CRS 3.3.2 ベースのルールは引き続きサポート。テスト環境で CRS 4.22 を評価しながら本番環境は CRS 3.3 を維持するといった段階的な移行が可能です。

⚙️ 設定方法

前提条件: Google Cloud プロジェクト、Cloud Armor セキュリティポリシー、外部アプリケーションロードバランサーが構成済みであること。

Step 1: セキュリティポリシーの作成

Bash セキュリティポリシー新規作成 
gcloud compute security-policies create my-waf-policy \
  --description "WAF policy with CRS 4.22 rules"

Step 2: CRS 4.22 ベースの WAF ルールを追加

Bash SQL インジェクション対策 + XSS 対策 
# SQL インジェクション対策ルール
gcloud compute security-policies rules create 1000 \
  --security-policy my-waf-policy \
  --expression "evaluatePreconfiguredWaf('sqli-v422-stable', {'sensitivity': 1})" \
  --action deny-403

# XSS 対策ルール
gcloud compute security-policies rules create 2000 \
  --security-policy my-waf-policy \
  --expression "evaluatePreconfiguredWaf('xss-v422-stable', {'sensitivity': 1})" \
  --action deny-403

Step 3: バックエンドサービスにアタッチ

Bash ポリシーの関連付け 
gcloud compute backend-services update my-backend-service \
  --security-policy my-waf-policy \
  --global
💡 ヒント: 本番適用前にプレビューモード (--preview フラグ) でテストすることを推奨します。

⚠️ 制限事項・注意点

Preview 段階の制約:
  • 本番環境での利用には SLA が適用されない可能性がある
  • GA までにルール名や設定方法が変更される可能性がある

🔄 移行時の注意: CRS 4.22 固有のルール名やシグネチャ ID は CRS 3.x 系と異なるため、既存のチューニング設定 (除外設定など) の移行作業が必要です。まずプレビューモードでの評価が推奨されます。

🚫 XML ボディ未対応: XML ボディのパースは Cloud Armor の事前構成 WAF ルールではサポートされていません (CRS バージョンに関係なく)。

📦 リクエストボディの検査上限: 設定された検査上限が適用されるため、大きなリクエストボディを持つアプリケーションでは追加の対策を検討する必要があります。

💡 ユースケース

🛒 E コマースサイトの WAF ルール最新化

大規模な E コマースサイトで CRS 3.3.2 ベースのルールによる誤検知が課題。CRS 4.22 の改善された検出ロジックにより誤検知が低減され、正規ユーザーのトランザクションがブロックされるリスクが減少。チューニングに費やす運用工数も削減されます。

Bash プレビューモードでの適用例 
# テスト環境で CRS 4.22 ルールをプレビューモードで適用
gcloud compute security-policies rules create 1000 \
  --security-policy test-policy \
  --expression "evaluatePreconfiguredWaf('sqli-v422-stable', {'sensitivity': 2})" \
  --action deny-403 \
  --preview

🏦 金融サービスのコンプライアンス要件対応

金融サービスを提供する企業で、PCI DSS 準拠のために最新の WAF ルールセットの採用が求められるケース。CRS 4.22 は最新の攻撃パターンに対応しており、監査時に最新のセキュリティ標準に基づいた WAF 保護を実施していることを証明できます。
