🗄️ AlloyDB for PostgreSQL: Knowledge Catalog 統合

AlloyDB for PostgreSQL Preview デフォルト有効化
2026-04-21

📋 概要

AlloyDB for PostgreSQL と Knowledge Catalog (旧 Dataplex Universal Catalog) の統合が、すべての新規 AlloyDB クラスターにおいてデフォルトで有効化されました。新しいクラスターを作成するだけで、データベース、スキーマ、テーブル、ビューなどのメタデータが自動的に Knowledge Catalog に同期され、データガバナンスと分析を簡素化する統一的なメタデータビューが利用可能になります。

ニアリアルタイム同期が実現されており、メタデータ変更が数分以内に反映されます。さらに、同期対象メタデータが拡張され、Primary Key / Foreign Key 情報も含まれるようになりました。

🔄 Before / After 比較

❌ Before (アップデート前)

  • Knowledge Catalog 統合を手動で有効化する必要があり、設定漏れが発生
  • メタデータの同期間隔が数時間おきで、反映まで最大 48 時間
  • Primary Key / Foreign Key 情報が含まれず、リレーションシップを確認不可

✅ After (アップデート後)

  • 新規クラスターで統合がデフォルト有効、追加設定なしで即座に同期開始
  • ニアリアルタイム同期で最短数秒、最大 5 分以内に反映
  • PK / FK (参照先テーブル、カラムマッピング) がメタデータとして同期

⚡ 主要機能

🔧 デフォルト有効化

  • すべての新規 AlloyDB クラスターで Knowledge Catalog 統合がデフォルトで有効
  • gcloud alloydb clusters create で --enable-dataplex-integration フラグの省略時に自動的に有効化
  • REST API でも dataplexConfig 省略でデフォルト有効
  • 無効化したい場合は明示的に --no-enable-dataplex-integration を指定

⏱️ ニアリアルタイム同期

  • 2026 年 4 月 3 日以降に作成・復元されたクラスターでメタデータ変更が最大 5 分以内に反映
  • 抽出プロセス自体は通常数秒で完了
  • Cloud Pub/Sub トピックを活用したイベントドリブンな同期メカニズム
  • 2026 年 2 月 26 日 ~ 4 月 3 日に作成されたクラスターは数時間ごとの同期 (ニアリアルタイムへのアップグレードには Google Cloud サポートへの連絡が必要)

🔗 拡張メタデータ (Primary Key / Foreign Key)

  • テーブルの Primary Key 情報が自動的に同期
  • Foreign Key の参照先テーブルとカラムマッピング情報が同期
  • Foreign Key の説明 (Description) も同期対象
  • Knowledge Catalog 上でテーブル間のリレーションシップを直接確認可能

制約事項 (リソースレベルのメタデータ)

  • クラスターおよびインスタンスのリソースレベルメタデータは常に有効であり、無効化できない
  • 統合の有効/無効設定が影響するのは、データベース、スキーマ、テーブル、ビューのメタデータのみ
  • プライマリクラスターにのみ適用。クロスリージョンレプリケーション用セカンダリクラスターでは非対応

🛠️ 設定方法

前提条件

✓Google Cloud プロジェクトで AlloyDB API が有効化されていること
✓適切な IAM ロール (alloydb.admin または alloydb.clusters.create 権限) が付与されていること
✓Knowledge Catalog (Dataplex API) が有効化されていること

ステップ 1: 新規クラスターの作成 (デフォルトで統合有効)

Bash Knowledge Catalog 統合はデフォルトで有効 
# Knowledge Catalog 統合はデフォルトで有効
# --enable-dataplex-integration フラグの指定は不要
gcloud alloydb clusters create CLUSTER_ID \
    --password=PASSWORD \
    --region=REGION

ステップ 2: 既存クラスターで統合を有効化する場合

Bash 既存クラスターへの統合有効化 
gcloud alloydb clusters update CLUSTER_ID \
    --region=REGION \
    --enable-dataplex-integration

ステップ 3: 統合を無効化する場合

Bash 統合を明示的に無効化 
# 新規クラスター作成時に統合を無効化
gcloud alloydb clusters create CLUSTER_ID \
    --password=PASSWORD \
    --region=REGION \
    --no-enable-dataplex-integration

REST API での設定例

JSON 統合を無効化してクラスターを作成 
{
  "databaseVersion": "POSTGRES_16",
  "initialUser": {
    "user": "postgres",
    "password": "YOUR_PASSWORD"
  },
  "dataplexConfig": {
    "enabled": false
  }
}
Bash REST API での作成リクエスト 
curl -X POST \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json; charset=utf-8" \
  -d @request.json \
  "https://alloydb.googleapis.com/v1/projects/PROJECT_ID/locations/REGION/clusters?cluster_id=CLUSTER_ID"

✅ メリット

ビジネス面

→データガバナンスの自動化

新規クラスター作成と同時にメタデータがカタログ化されるため、データガバナンスポリシーの適用漏れを防止できる

→データ発見性の向上

Knowledge Catalog の統合検索により、組織内の AlloyDB データ資産を迅速に発見・理解でき、データ活用までの時間を短縮

→コンプライアンス対応の効率化

メタデータ変更追跡がニアリアルタイムで行われるため、監査やコンプライアンスレポートの作成が容易に

技術面

→設定不要 (Zero Configuration)

デフォルト有効化により、インフラチームの設定作業が不要。ヒューマンエラーによる設定漏れを排除

→ニアリアルタイムのメタデータ整合性

スキーマ変更が最大 5 分以内にカタログに反映され、ダウンストリームシステムが常に最新のスキーマ情報を参照可能

→リレーションシップの可視化

PK / FK 情報の同期により、テーブル間の依存関係を Knowledge Catalog 上で直接確認でき、影響分析やマイグレーション計画が効率化

⚠️ デメリット・制約事項

制限事項

  • 本機能は Preview ステータスであり、「Pre-GA Offerings Terms」が適用される。サポートが限定的である可能性がある
  • セカンダリクラスター (クロスリージョンレプリケーション用) では、データベース・スキーマ・テーブル・ビューのメタデータ同期は非対応
  • 2026 年 2 月 26 日 ~ 4 月 3 日に作成されたクラスターのニアリアルタイム同期へのアップグレードには Google Cloud サポートへの連絡が必要
  • クラスターおよびインスタンスのリソースレベルメタデータの同期は無効化不可

考慮すべき点

  • 意図しないメタデータエクスポート: デフォルト有効化により、意図せずメタデータが Knowledge Catalog にエクスポートされる可能性がある。セキュリティポリシー上メタデータの外部共有を制限している場合は、明示的に無効化する設定を運用フローに組み込む必要がある
  • Knowledge Catalog のストレージ課金: metadata storage SKU に基づく課金が発生する。大量のテーブル・カラムを持つクラスターではコストを考慮すること
  • VPC Service Controls: 利用している環境では、Dataplex API (dataplex.googleapis.com) をサービス境界内に含める設定が必要

💡 ユースケース

🏢 ユースケース 1: マルチチームでのデータガバナンス統一

シナリオ: 大規模組織において、複数のチームがそれぞれ AlloyDB クラスターを運用しており、データカタログの整備が属人化していた。新規クラスター作成時にカタログ登録を忘れるケースが頻発し、組織全体のデータ資産の可視性が低下していた。

Bash チームが新規クラスターを作成するだけで自動カタログ化 
# 各チームが新規クラスターを作成するだけで自動的にカタログ化
gcloud alloydb clusters create team-analytics-cluster \
    --password=SECURE_PASSWORD \
    --region=asia-northeast1

# Knowledge Catalog で組織横断的にデータ資産を検索
# Google Cloud Console > Dataplex > Knowledge Catalog > 検索
効果: 全クラスターのメタデータが自動的に Knowledge Catalog に集約されるため、組織全体のデータ資産の一元管理が実現し、データガバナンスの属人化を解消できる。

🔍 ユースケース 2: スキーマ変更の影響分析

シナリオ: マイクロサービスアーキテクチャにおいて、共有データベースのテーブルスキーマを変更する際に、どのサービスが影響を受けるか把握するのが困難だった。Foreign Key 情報がカタログに含まれていなかったため、手動でリレーションシップを追跡する必要があった。

効果: Knowledge Catalog に同期された PK / FK 情報を活用することで、テーブル間の依存関係をカタログ上で即座に確認でき、スキーマ変更の影響範囲を正確に特定できる。ニアリアルタイム同期により、変更後の状態も数分以内にカタログに反映される。

🤖 ユースケース 3: AI エージェントによるデータ検索の基盤構築

シナリオ: 生成 AI アプリケーションが組織内のデータ資産を検索し、適切なデータソースを自動的に選択する必要がある。Knowledge Catalog は MCP (Model Context Protocol) を通じた AI エージェントのデータ検索基盤として機能する。

効果: AlloyDB のメタデータが自動的に Knowledge Catalog に登録されることで、AI エージェントが組織内のデータ資産を正確に把握し、ハルシネーションを抑制した信頼性の高いデータアクセスを実現できる。
