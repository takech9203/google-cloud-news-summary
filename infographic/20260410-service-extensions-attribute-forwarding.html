⚙️ Service Extensions: 属性転送機能

プラグイン/コールアウト設定時にリクエスト属性・接続属性をバックエンドサービスに転送
Networking Service Extensions Cloud Load Balancing Feature
2026-04-10

📋 概要

Service Extensions において、プラグインまたはコールアウトを使用して拡張を設定する際に、一部のリクエスト属性および接続属性をバックエンドサービスに転送 (フォワード) できるようになりました。これにより、拡張サービスはロードバランサーが抽出したリクエストや接続のコンテキスト情報を直接受け取り、より高度なカスタム処理を実現できます。

ソース IP アドレス、TLS 接続情報、バックエンドサービス名といったロードバランサーレベルの属性にアクセスし、セキュリティ判断やルーティングロジック、ログ記録などのユースケースがより柔軟に実装可能となります。Application Load Balancer 上で Service Extensions を使用するプラットフォームエンジニア、セキュリティエンジニア、アプリケーション開発者に特に有用です。

🔄 Before / After 比較

❌ Before

  • 拡張サービスがソース IP やクライアントポートなどの接続レベル情報を直接取得することが困難
  • TLS バージョンやクライアント証明書ダイジェストを拡張サービスに渡す標準的な方法がなかった
  • バックエンドサービス名やプロジェクト番号の利用にはカスタムヘッダーによる間接的な方法が必要
  • CEL マッチング条件で使用可能な属性データは拡張サービスには送信されなかった

✅ After

  • リクエスト属性 (HTTP メソッド、ホスト、パス、クエリ、スキーム、バックエンドサービス名など) を明示的に転送可能
  • 接続属性 (ソース IP、ソースポート、SNI、TLS バージョン、クライアント証明書ダイジェスト) を転送可能
  • 拡張サービス側でリッチなコンテキスト情報に基づいた判断やロジックを実装可能

⭐ 主要機能

📨 リクエスト属性の転送

HTTP リクエストから抽出される属性 (メソッド、ホスト、パス、クエリ、スキームなど) を拡張サービスに転送可能。request.backend_service_name でリクエストの転送先バックエンドサービス名を通知し、request.backend_service_project_number で共有 VPC 環境でのプロジェクト番号を転送できます。

🔐 接続属性の転送

source.ip と source.port でクライアントのソース IP アドレスとポート情報を転送。connection.sni で TLS 接続の SNI 情報、connection.tls_version で TLS バージョン、connection.sha256_peer_certificate_digest でクライアント証明書のハッシュダイジェスト (mTLS 環境) を転送できます。

🔌 プラグインとコールアウトの両方に対応

Route Extensions と Traffic Extensions の両方で属性転送が設定可能。プラグイン (Wasm) とコールアウト (ext_proc gRPC) の両方の拡張タイプで利用でき、同一の Extension Chain 内でプラグインとコールアウトを組み合わせた場合にも属性転送が機能します。

⚙️ 技術仕様: 転送可能なリクエスト属性

属性名 型 説明
request.headers map{string, string} HTTP リクエストヘッダーの文字列マップ。複数値はカンマ区切り、キーは小文字
request.method string HTTP リクエストメソッド (GET, POST など)
request.host string リクエストのホスト名 (request.headers['host'] と同等)
request.path string リクエストされた HTTP URL パス
request.query string HTTP URL クエリ文字列 (デコードなし)
request.scheme string HTTP URL スキーム (http, https。小文字)
request.backend_service_name string 転送先バックエンドサービス名 (Edge Extensions では利用不可)
request.backend_service_project_number int 共有 VPC 使用時のバックエンドサービスのプロジェクト番号

🔍 技術仕様: 転送可能な接続属性

属性名 型 説明
source.ip string リクエストのソース IP アドレス
source.port int ダウンストリームクライアントの接続ポート
connection.sni string ダウンストリーム TLS 接続の要求サーバー名 (SNI)
connection.tls_version string TLS バージョン (TLSv1, TLSv1.1, TLSv1.2, TLSv1.3)
connection.sha256_peer_certificate_digest string ピア証明書の SHA256 ハッシュ (16 進エンコード、mTLS 環境)

🛠️ 設定方法

📋 前提条件:
  • Google Cloud プロジェクトで Service Extensions API が有効化されていること
  • サポート対象の Application Load Balancer が設定済みであること
  • プラグインまたはコールアウトバックエンドサービスが準備済みであること

Step 1: 属性転送を含む拡張の設定 (gcloud CLI)

Route Extension でコールアウトを設定し、属性転送を指定する例です。

Bash Route Extension の作成 
gcloud beta service-extensions route-extensions create my-route-extension \
    --location=REGION \
    --load-balancing-scheme=EXTERNAL_MANAGED \
    --forwarding-rules=FORWARDING_RULE \
    --extension-chains='[{
        "name": "chain1",
        "matchCondition": {
            "celExpression": "true"
        },
        "extensions": [{
            "name": "ext1",
            "authority": "myext.example.com",
            "service": "projects/PROJECT_ID/locations/REGION/backendServices/CALLOUT_BACKEND",
            "timeout": "1s",
            "failOpen": false
        }]
    }]'

Step 2: Google Cloud Console での設定

1.Google Cloud Console で Service Extensions に移動
2.拡張 (Route Extension / Traffic Extension) を作成または編集
3.Extension Chain 内の Extension 設定で、転送する属性を指定
4.Forward headers セクションで転送するヘッダーを追加
5.Metadata セクションで追加のメタデータを設定 (必要に応じて)
6.設定を保存

Step 3: 拡張サービス側での属性受信

コールアウトバックエンドサービス (ext_proc gRPC サーバー) は、ProcessingRequest メッセージ内で転送された属性を受信します。

Python ext_proc サーバーでの属性受信例 
# ext_proc サーバーでの属性受信例 (概念コード)
def process(request_stream):
    for processing_request in request_stream:
        if processing_request.HasField("request_headers"):
            headers = processing_request.request_headers
            # 転送された属性にアクセス
            # ヘッダーやメタデータから属性情報を取得
            for header in headers.headers.headers:
                print(f"Header: {header.key} = {header.value}")

⚖️ メリットとデメリット

✅ メリット

ビジネス面

  • 🔒セキュリティポリシーの高度化 -- ソース IP や TLS 情報に基づくきめ細かなアクセス制御で、ゼロトラストセキュリティモデルの実装が容易に
  • 📝コンプライアンス対応の強化 -- クライアント証明書情報や TLS バージョンを拡張サービスで検証でき、規制要件への準拠を支援
  • ⚙️運用効率の向上 -- ロードバランサーレベルの属性を直接渡すことで、カスタムヘッダーの手動管理が不要に

技術面

  • 📦リッチなコンテキスト情報 -- ヘッダーやボディだけでなく、接続レベルのメタデータにもアクセス可能に
  • 🔐mTLS 連携の強化 -- クライアント証明書の SHA256 ダイジェスト転送で、証明書ベースの認証/認可ロジック実装が容易に
  • 🏢マルチテナント対応の改善 -- backend_service_name と project_number により、共有 VPC 環境でのテナント識別が可能に

⚠️ デメリット・制約事項

制限事項

  • ❌Edge Extensions では backend_service_name / backend_service_project_number は利用不可
  • ❌sha256_peer_certificate_digest はピア証明書が存在する場合のみ利用可能 (mTLS 設定が必要)
  • ⚠️属性はコンテキストに依存するため、一部の属性は特定のタイミングや設定でのみ利用可能
  • ⚠️ヘッダー操作に関する既存の制限 (X-user-IP、CDN-Loop、X-Forwarded-* 等) は引き続き適用

考慮すべき点

  • 📝転送する属性の数や種類が増えると、メタデータサイズ増加によるわずかなレイテンシ影響の可能性
  • 📝メタデータ合計サイズは 1 KiB 未満、キーの数は最大 16 (Traffic Extensions では最大 20) に制限
  • 📝拡張サービス側で属性を適切に処理するロジックの実装が必要

💡 ユースケース

🛡️ ユースケース 1: IP ベースのアクセス制御

シナリオ: 拡張サービスでクライアントの実際のソース IP アドレスに基づいてアクセス制御を実施。地理的な制限やレート制限をカスタムロジックで実装する場合。

実装: 拡張設定で source.ip 属性を転送するよう指定し、ext_proc サーバー側でソース IP を受け取り、カスタムの IP 許可/拒否リストと照合して ImmediateResponse で 403 を返すか、リクエストを通過させる。

効果: Cloud Armor のルールだけでは対応しきれない複雑な IP ベースのアクセス制御ロジックを、拡張サービスで柔軟に実装できる。

🔐 ユースケース 2: mTLS クライアント証明書の検証強化

シナリオ: mTLS 環境でクライアント証明書のダイジェストを拡張サービスに渡し、証明書のホワイトリストに基づくカスタム認可を行いたい場合。

実装: 拡張設定で connection.sha256_peer_certificate_digest 属性を転送するよう指定し、ext_proc サーバー側で証明書ダイジェストを受け取り、内部の証明書管理システムと照合して認可判断を行う。

効果: TLS 終端後にも証明書情報を拡張サービスで利用でき、より厳密なクライアント認証が実現可能。

🏢 ユースケース 3: マルチテナント環境でのバックエンドサービス識別

シナリオ: 共有 VPC 環境で複数のバックエンドサービスにトラフィックが分散される構成において、拡張サービスがどのバックエンドサービスにリクエストが転送されるかを把握してログ記録やメトリクス収集を行いたい場合。

実装: 拡張設定で request.backend_service_name と request.backend_service_project_number 属性を転送するよう指定し、ext_proc サーバー側でバックエンドサービスの識別情報を取得して Cloud Logging やカスタムメトリクスシステムに記録。

効果: リクエストのルーティング先を拡張サービスで可視化でき、トラフィック分析やデバッグが容易になる。

💰 料金情報

Service Extensions の属性転送機能自体に追加料金は発生しませんが、Service Extensions の利用には通常の料金が適用されます。コールアウトやプラグインの使用に応じた課金に加え、Cloud Load Balancing の基本料金が発生します。

詳細は Service Extensions の料金ページ をご確認ください。

📌 まとめ

Service Extensions でプラグインやコールアウトを設定する際にリクエスト属性および接続属性をバックエンドサービスに転送できるようになったことで、拡張サービスはロードバランサーレベルのリッチなコンテキスト情報を活用した高度なカスタム処理を実装できるようになりました。

ソース IP アドレス、TLS バージョン、クライアント証明書ダイジェスト、バックエンドサービス名などの属性を直接受け取れるため、セキュリティ判断の精度向上やトラフィック分析の強化が期待できます。Service Extensions を利用している場合は、属性転送を活用して既存のカスタムヘッダーベースの実装をよりシンプルかつ信頼性の高い方式に移行することを検討してください。
