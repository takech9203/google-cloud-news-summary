☵ GKE バージョンアップデート (2026-R11)

Google Kubernetes Engine GA 2026-R11
2026-03-18

📋 概要

GKE の全リリースチャネル (Rapid, Regular, Stable, Extended) において、クラスタバージョンが更新されました。各チャネルのデフォルトバージョンが変更され、新しいバージョンが利用可能になるとともに、一部の古いパッチバージョンが非推奨 (deprecated) となりました。

自動アップグレードを有効にしているクラスタでは、各チャネルの新しいデフォルトバージョンへのアップグレードが順次実行されます。

📚 リリースチャネルの特性

チャネル 特性 推奨用途
⚡ Rapid 最新バージョンを最も早く提供。GKE SLA の対象外となる場合がある プリプロダクション環境、新機能の早期検証
⚙ Regular Rapid で 2-3 か月の安定性確認後に提供 大多数のワークロード (推奨)
🛡 Stable Regular で 2-3 か月の追加安定性確認後に提供 安定性を最優先するワークロード
🕒 Extended Regular チャネルと同期。最大 24 か月のサポート 長期サポートが必要なワークロード

🚨 非推奨バージョン一覧

チャネル 非推奨バージョン
Rapid 1.35.2-gke.1269000
Regular 1.35.0-gke.2745005, 1.35.0-gke.3047002

⚠ 非推奨ポリシー: 非推奨となったパッチバージョンは、90 日間または当該マイナーバージョンのサポート終了日まで引き続き利用可能です。ただし、非推奨バージョンはコントロールプレーンの利用可能バージョンとしてリストされなくなるため、早期アップグレードが推奨されます。

✨ メリット

💼 ビジネス面

✓ セキュリティの強化 - 最新パッチバージョンにはセキュリティ修正が含まれ、クラスタの安全性が向上
✓ 安定性の向上 - 各チャネルでの安定性確認プロセスを経たバージョンが提供される

🔧 技術面

✓ 最新機能へのアクセス - Kubernetes 1.35 系の最新パッチが利用可能に
✓ バグ修正の適用 - 既知のバグが新しいパッチバージョンで修正される
✓ バージョンスキューの防止 - コントロールプレーンとノード間のバージョンスキューが適切に管理される

⚠ 制約事項・考慮点

🚫 制限事項:
  • バージョンのロールアウトはリリース日から 4 営業日以上かけて全ゾーンに展開
  • コントロールプレーンのマイナーバージョンスキップは不可 (例: 1.33 → 1.35 は不可)
  • Rapid チャネルのバージョンは GKE SLA の対象外となる場合がある
💡 考慮すべき点:
  • 非推奨バージョンを使用中のクラスタは 90 日以内に新パッチバージョンへのアップグレードを計画
  • メンテナンスウィンドウとメンテナンス除外の適切な設定を推奨
  • 本番環境へのアップグレード前にプリプロダクション環境での動作検証を実施
  • ノードのバージョンはコントロールプレーンの 2 マイナーバージョン以内が必要

🛠 設定方法

前提条件: GKE クラスタが作成済みであること、適切な IAM 権限 (container.clusters.update) を持つこと

Step 1: 現在のクラスタバージョンとチャネルの確認

Bash 
# クラスタの現在のバージョンとリリースチャネルを確認
gcloud container clusters describe CLUSTER_NAME \
  --location=LOCATION \
  --format="table(name, currentMasterVersion, releaseChannel.channel)"

Step 2: 利用可能なアップグレードバージョンの確認

Bash 
# クラスタで利用可能なアップグレードバージョンを確認
gcloud container get-server-config \
  --location=LOCATION \
  --format="yaml(channels)"

Step 3: 手動アップグレード (必要な場合)

Bash 
# コントロールプレーンを特定のバージョンにアップグレード
gcloud container clusters upgrade CLUSTER_NAME \
  --location=LOCATION \
  --master \
  --cluster-version=TARGET_VERSION
💡 自動アップグレード: 自動アップグレードが有効な場合、新しいデフォルトバージョンへのアップグレードはリリース日から 4 営業日以上かけて全ゾーンに順次展開されます。

Step 4: ノードプールのアップグレード (Standard クラスタ)

Bash 
# ノードプールを特定のバージョンにアップグレード
gcloud container clusters upgrade CLUSTER_NAME \
  --location=LOCATION \
  --node-pool=NODE_POOL_NAME \
  --cluster-version=TARGET_VERSION
✨ Autopilot: Autopilot クラスタの場合、ノードはコントロールプレーンのバージョンに自動的にアップグレードされます。

Rapid チャネルの開発環境で新バージョンを検証した後、段階的に本番環境へ展開することでリスクを最小化しながら最新機能を活用できます。

🕒 長期サポートが必要なワークロード

シナリオ: コンプライアンス要件により、クラスタバージョンの頻繁な変更を避けたい場合

Extended チャネルを使用して、マイナーバージョンを最大 24 か月間維持。同一マイナーバージョン内でのパッチアップグレードのみが自動適用され、セキュリティ修正を受けながら長期間維持できます。

🔗 関連サービス・機能

→GKE リリースチャネル - クラスタのバージョン管理を安定性レベルに基づいて自動化する機能
→GKE 自動アップグレード - コントロールプレーンとノードを自動的に新しいバージョンにアップグレードする機能
→メンテナンスウィンドウ/除外 - 自動アップグレードのタイミングを制御し、アップグレードの影響を最小限に抑える機能
→ロールアウトシーケンシング - リリースチャネルに登録されたクラスタ間でアップグレードの順序を管理する機能

📝 まとめ

GKE 2026-R11 バージョンアップデートにより、全リリースチャネルのデフォルトバージョンと自動アップグレードターゲットが更新されました。非推奨バージョンを使用中のクラスタ運用者は、以下のアクションを推奨します。

✓新しいパッチバージョンへのアップグレード計画を確認
✓メンテナンスウィンドウの設定を見直し
✓プリプロダクション環境でのテストを実施
✓計画的なバージョン移行を推進
