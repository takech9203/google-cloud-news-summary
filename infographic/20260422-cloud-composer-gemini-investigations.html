🤖 Cloud Composer: Gemini Cloud Assist Investigations

Managed Airflow における AI 駆動トラブルシューティング

Cloud Composer Gemini Cloud Assist Private Preview
2026-04-22

☁️ 概要

Managed Service for Apache Airflow (旧 Cloud Composer) が Gemini Cloud Assist Investigations をサポート。 失敗した Airflow タスクや DAG ランを AI ベースのトラブルシューティングエージェントが自動的に調査・診断できるようになります。

💡 ポイント: DAG UI から直接「Investigate」ボタンをクリックするだけで、ログ・メトリクス・設定を横断的に分析し、根本原因の仮説と修正アクションを自動提示します。Airflow 2.7.3 以降の環境で利用可能です。

🔄 Before / After 比較

❌ Before (従来)

  • タスク失敗時にログ、メトリクス、設定を個別に確認する必要があり、原因特定に時間がかかっていた
  • ゾンビタスク、OOM エラー、スケジューラの問題など複数コンポーネントにまたがる障害の RCA が困難
  • ログが出力されないタスク失敗ケースでは原因調査の手がかりが限定的
  • トラブルシューティングの知見が個人依存で、体系的な障害対応が難しかった

✅ After (改善後)

  • DAG UI から「Investigate」ボタンをクリックするだけで AI 根本原因分析を開始
  • Gemini がログ、設定、メトリクスを横断分析し、関連 Observations を自動生成
  • ログが出力されないタスク失敗でもスケジューラログ等から原因特定が可能
  • 根本原因の仮説と修正推奨アクションが自動提示される

⭐ 主要機能

🖱️ 1. DAG UI からのワンクリック調査開始

→失敗した Airflow タスクの State 列に表示される「Investigate」ボタンから直接調査を開始
→環境詳細情報 (問題の説明、時間範囲、関連リソース) が自動ポピュレート
→既存調査の「View investigation」確認、または「New investigation」で新規調査

🧠 2. AI ベースの根本原因分析 (RCA)

→ログ、設定、メトリクスなど複数のデータソースを横断的に分析
→Observations (観測結果) をランク付け・フィルタリングして提示
→各観測結果にはソースデータへのリンクが含まれ、ファクトチェックが可能
→不確実性がある場合は複数の仮説を提示し、反復的な深掘り調査が可能

✅ 3. 修正推奨アクションの提示

→根本原因に基づく具体的な修正手順や次のトラブルシューティングステップを提示
→特定のリソースや時間帯への参照情報を含むため、手動収集よりも効率的
→Premium Support 利用時は調査結果をそのままサポートケースに引き継ぎ可能

🛠️ 設定方法

前提条件

📝 前提:
  • Managed Airflow 環境が Airflow 2.7.3 以降で稼働していること
  • Premium Support 契約を有している、またはアカウントチーム経由で Private Preview へのアクセスをリクエスト済みであること
  • 必要な API が有効化されていること

ステップ 1: 必要な API の有効化

Bash 必須 API の有効化 
# 必須 API の有効化
gcloud services enable cloudaicompanion.googleapis.com \
  cloudasset.googleapis.com \
  cloudresourcemanager.googleapis.com \
  geminicloudassist.googleapis.com \
  --project=PROJECT_ID

# 推奨 API の有効化
gcloud services enable logging.googleapis.com \
  monitoring.googleapis.com \
  --project=PROJECT_ID

ステップ 2: IAM ロールの付与

Bash IAM ロール付与コマンド 
# Investigation Creator ロールの付与
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="user:USER_EMAIL" \
  --role="roles/geminicloudassist.investigationCreator"

# Gemini Cloud Assist User ロールの付与
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="user:USER_EMAIL" \
  --role="roles/geminicloudassist.user"

ステップ 3: DAG UI から調査を開始

①Google Cloud Console で「Environments」ページに移動
②対象の環境を選択し、環境詳細ページの「DAGs」タブを開く
③DAG 名をクリックし、「Run history」タブで失敗したタスクを含む DAG ランを選択
④失敗した Airflow タスクの State 列で「Investigate」をクリック
⑤調査パラメータを確認・編集し、「Create」をクリックして調査を開始

📈 メリット

💼 ビジネス面

✅インシデント解決時間の短縮: AI による自動分析で障害の根本原因特定にかかる時間を大幅に削減。手動でのログ調査やメトリクス確認の工数が不要に
✅運用コストの削減: トラブルシューティングの専門知識がなくても障害対応が可能に。SRE やデータエンジニアの負荷を軽減
✅サポートケースの効率化: 調査結果をサポートケースにシームレスに引き継げるため、サポートエンジニアとのやり取りの往復が減少

💻 技術面

✅横断的なデータ分析: ログ、メトリクス、設定情報を横断的に分析するため、単一データソースでは発見できない原因を特定可能
✅コンテキスト自動入力: DAG UI から起動することで環境情報が自動ポピュレートされ、調査の精度が向上
✅反復的な調査: 調査結果に基づいて新しいリビジョンを作成し、特定の領域をさらに深掘り可能

⚠️ デメリット・制約事項

制限事項

  • Private Preview のため、Premium Support 契約またはアカウントチーム経由でのアクセスリクエストが必要
  • 調査スコープは単一の Google Cloud プロジェクトまたは単一の App Hub アプリケーションに限定
  • Airflow 2.7.3 未満のバージョンでは利用できない
  • AI 技術の一般的な制限事項が適用 (調査結果の再実行で異なる結果が生じる可能性)

考慮すべき点

⚠️調査実行時に OAuth 2.0 トークンが使用される。アクセス範囲はユーザーのアクセス権限に限定
⚠️調査リソース (アノテーション、観測結果) は任意の Google Cloud データセンターに保存される可能性があるため、データレジデンシーや管轄に関する規制コンプライアンスの対象データに対しては調査を実施すべきでない
⚠️調査の精度を高めるため、時間範囲が正確であることを確認することが重要

💡 ユースケース

💻 ユースケース 1: OOM によるタスク失敗の調査

シナリオ: データ処理 DAG のタスクが断続的に失敗し、タスクログにエラーメッセージが出力されていない状況。原因の特定が困難。

手順:

1.DAG UI で失敗した DAG ランを選択
2.失敗タスクの「Investigate」ボタンをクリック
3.Gemini がスケジューラログを分析し、Worker Pod の OOM による再起動とゾンビタスクへの終了を検出
4.推奨アクション: Worker のリソース制限の引き上げが提示

→ ログが出力されないケースでも、スケジューラログや Pod メトリクスから根本原因を特定し解決策を取得

📦 ユースケース 2: 依存関係エラーによる DAG 失敗の調査

シナリオ: 新しい PyPI パッケージの追加後、複数の DAG が連鎖的に失敗している状況。

→ Gemini が環境の設定変更履歴とタスクログを関連付けて分析し、パッケージの依存関係の競合を根本原因として特定。バージョン固定やパッケージ削除などの具体的な修正手順が提示される
