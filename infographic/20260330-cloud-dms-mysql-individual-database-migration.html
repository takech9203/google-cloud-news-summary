🗄️ Cloud Database Migration Service:
MySQL 同種移行で個別データベースの選択的移行が可能に

Cloud Database Migration Service GA (一般提供) MySQL Homogeneous Migration
2026-03-30

📋 概要

Database Migration Service (DMS) の MySQL 同種移行 において、ソースインスタンスから個別のデータベースを選択して移行できる機能が追加されました。これまで全データベースが一括移行されていましたが、移行ジョブ作成時に対象を個別選択できるようになりました。

PostgreSQL (Cloud SQL / AlloyDB) や SQL Server では既に提供されていた機能であり、今回の対応で MySQL でも同等の選択的移行が実現しました。

🎯 対象ユーザー: オンプレミスや他クラウド (AWS RDS, Amazon Aurora, Azure Database for MySQL) から Cloud SQL for MySQL への移行を計画しているデータベース管理者やクラウドアーキテクト

🔄 Before / After 比較

❌ アップデート前

  • 全データベースが一括移行、個別選択不可
  • 不要なDBも含め移行時間・帯域を余分に消費
  • 特定DBのみ移行するには手動ダンプ・リストアが必要
  • PostgreSQL/SQL Serverでは可能だがMySQLは非対応

✅ アップデート後

  • 移行対象データベースを個別に選択可能
  • 必要なDBのみ移行し、時間・リソースを最適化
  • コンソールおよび gcloud CLI の両方から設定可能
  • MySQL でも PostgreSQL/SQL Server と同等の柔軟性

⭐ 主要機能

☑️
個別データベース選択
移行ジョブ作成ウィザードの「Configure migration objects」でチェックボックスにより個別選択。全DB一括移行も引き続きサポート。
💻
gcloud CLI サポート
--databases-filter フラグで移行対象DBを指定可能。--all-databases で従来の一括移行も対応。
🔄
継続移行との互換性
選択したDBに対してフルダンプ後のCDCが適用。移行対象外DBの変更は無視され帯域を節約。

⚙️ 設定方法

💡 前提条件:
  • ソース MySQL インスタンスへの接続プロファイルが作成済み
  • Database Migration Service API が有効化済み
  • 適切な IAM ロールが付与済み

🌐 Google Cloud コンソールでの設定

  1. Migration jobs ページに移動
  2. 「Create migration job」をクリック
  3. ソースデータベースエンジンとして「MySQL」を選択
  4. ソース接続プロファイルを指定
  5. 「Configure migration objects」ステップで移行対象のデータベースを選択
  6. テストを実行し、移行ジョブを作成

💻 gcloud CLI での設定

Bash 
# 個別データベースを指定して移行ジョブを作成
gcloud database-migration migration-jobs create my-mysql-migration \
  --region=us-central1 \
  --type=CONTINUOUS \
  --source=mysql-source-profile \
  --destination=mysql-dest-profile \
  --databases-filter=app_db,user_db,analytics_db
Bash 
# 全データベースを移行する場合 (従来の動作)
gcloud database-migration migration-jobs create my-mysql-migration \
  --region=us-central1 \
  --type=CONTINUOUS \
  --source=mysql-source-profile \
  --destination=mysql-dest-profile \
  --all-databases

⚖️ メリット・デメリット

✅ メリット

ビジネス面
→移行コスト削減: 必要なDBのみ移行し時間・リソースを削減
→段階的移行: アプリごとに段階移行しリスク分散
→ダウンタイム最小化: データ量削減で切替時間を短縮
技術面
→帯域効率化: 不要DBレプリケーションを排除
→移行ジョブ簡素化: 1ジョブで選択的移行が可能
→自動化対応: --databases-filter で CI/CD 統合容易

⚠️ 制限事項・注意点

制限事項
•MySQL 8.4 の移行先では物理バックアップによる移行が非サポート
•リージョナルサービスのため、ソース・移行先・接続プロファイルは同一リージョン必須
•system DB (mysql, information_schema, performance_schema, sys) は自動除外
考慮すべき点
•DB間に外部キー等の依存がある場合、関連DBを全て含める必要あり
•選択後にソースで新規DB作成しても自動追加されない
•ストアドプロシージャのDB横断参照は事前に依存関係を確認

💡 ユースケース

🏢 ユースケース 1: マルチテナントアプリケーションの段階的移行

シナリオ: 1つの MySQL インスタンスに複数テナントのDBが格納されている SaaS アプリケーションで、テナントごとに段階的に Cloud SQL へ移行したい場合。

Bash 
# 第1バッチ: テナントA とテナントB のデータベースを移行
gcloud database-migration migration-jobs create batch1-migration \
  --region=asia-northeast1 \
  --type=CONTINUOUS \
  --source=saas-mysql-source \
  --destination=csql-mysql-dest-1 \
  --databases-filter=tenant_a_db,tenant_b_db

→ テナントごとにリスクを分散し、問題発生時の影響範囲を限定。アプリケーション切り替えも段階的に実施可能。

🛠️ ユースケース 2: 開発環境の部分移行

シナリオ: 本番環境のソースインスタンスには多数のDBが存在するが、開発・テスト用に特定のDBのみを Cloud SQL に移行したい場合。

→ 開発環境に必要なDBのみを移行することで、コストとストレージ使用量を抑え、移行時間を大幅に短縮。

📝 まとめ

今回のアップデートにより、Database Migration Service の MySQL 同種移行で個別データベースの選択的移行が可能になりました。PostgreSQL や SQL Server では既に提供されていた機能の MySQL への展開であり、段階的な移行戦略やコスト最適化を実現する上で重要な改善です。

MySQL から Cloud SQL への移行を計画している場合は、移行ジョブ作成時にデータベース選択オプションを活用し、効率的な移行計画を策定することを推奨します。
