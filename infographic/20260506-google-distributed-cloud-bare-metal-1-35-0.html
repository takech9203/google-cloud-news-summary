⚙️ Google Distributed Cloud (ベアメタル) 1.35.0

GDC Bare Metal Available Breaking Change
2026-05-06

📋 概要

Google Distributed Cloud (software only) for bare metal 1.35.0-gke.525 がリリースされました。Kubernetes v1.35.2-gke.300 をベースとし、プラットフォーム全体の大幅なアップデートを含みます。

⚠️ Breaking Change: 本リリースから cgroupsv2 が必須要件 となり、cgroupsv1 のサポートが完全に廃止されました。cgroupsv1 が検出された場合、プリフライトチェックによりクラスタの作成やアップグレードがブロックされます。

🔄 Before / After 比較

❌ Before (1.34.x)

  • cgroupsv1 環境でも動作していたが、最新 Kubernetes 機能やリソース管理に制限
  • containerd 2.0 で一部の互換性やパフォーマンスに制約
  • レジストリミラー設定が分散し管理が煩雑
  • Secret/ConfigMap マウント不整合の自動検出なし

✅ After (1.35.0)

  • cgroupsv2 統一でリソース管理の一貫性と最新カーネル機能を活用
  • containerd 2.1 でランタイムのパフォーマンスと安定性向上
  • レジストリミラーをヘッダーセクションで一元管理
  • 定期ヘルスチェックで不整合を 5 分以内に検出

⚡ 主要機能

🚨 1. cgroupsv2 必須化 (Breaking Change)

→cgroupsv1 のサポートが完全に廃止、cgroupsv2 が必須要件に
→プリフライトチェックで cgroupsv1 検出時にクラスタ作成/アップグレードを自動ブロック
→RHEL 7/8 では手動で OS 設定変更が必要

⚡ 2. Kubernetes 1.35 プラットフォームアップデート

→Kubernetes v1.35.2-gke.300 に更新
→最新の Kubernetes API および機能が利用可能に

📦 3. containerd 2.1 へのアップグレード

→コンテナランタイムが 2.0 から 2.1 に更新
→パフォーマンス改善とセキュリティ強化

💚 4. 定期ヘルスチェック (Secret/ConfigMap マウント監視)

→Secret/ConfigMap マウントの整合性を定期的にチェック
→不一致が 5 分以上続いた場合にレポート
→Stale マウントの早期検出が可能に

🔧 5. Ansible 2.18 へのアップグレード

→ターゲットノードに Python 3.9 が必要
→RHEL 8.10 以降が必要

📚 6. レジストリミラーのヘッダーセクション設定

→クラスタ構成ファイルのヘッダーセクションでレジストリミラーを一元管理
→構成管理の簡素化

🌐 7. EgressDSCP タギングサポート (Preview)

→Egress トラフィックに DSCP タグを付与可能
→ネットワーク QoS ポリシーの実装を支援

🛠️ 設定方法

前提条件

✅現行クラスタが 1.33.x または 1.34.x であること
✅全ノードで cgroupsv2 が有効であること
✅ターゲットノードに Python 3.9 がインストール済み
✅RHEL 使用時は 8.10 以降であること
✅サードパーティストレージベンダーの 1.35.0 認定を確認

Step 1: cgroupsv2 の確認と有効化

Bash 
# cgroupsv2 が有効か確認
if [ -f /sys/fs/cgroup/cgroup.controllers ]; then
    echo "cgroupsv2 is enabled"
else
    echo "cgroupsv1 detected - must enable cgroupsv2 before upgrade"
fi

Step 2: RHEL 8 での cgroupsv2 有効化

Bash 
# RHEL 8 での cgroupsv2 有効化
sudo grubby --update-kernel=ALL --args="systemd.unified_cgroup_hierarchy=1"

# ノードを再起動
sudo reboot

# cgroupsv2 が有効化されたことを確認
ls /sys/fs/cgroup/cgroup.controllers

Step 3: クラスタのアップグレード

Bash 
# アップグレードの実行
bmctl upgrade cluster -c CLUSTER_NAME --kubeconfig ADMIN_KUBECONFIG
⚠️ 注意: リリース後、GKE On-Prem API クライアント (Google Cloud コンソール、gcloud CLI、Terraform) で利用可能になるまで約 7~14 日かかります。

💼 ビジネス面

✓ネットワーク QoS 管理: EgressDSCP タギングでトラフィック優先度制御が可能に

💻 技術面

✓最新プラットフォーム: K8s 1.35 + containerd 2.1 で最新機能とパフォーマンス向上
✓一貫したリソース管理: cgroupsv2 統一でリソース制御の予測可能性向上
✓構成管理の簡素化: レジストリミラーのヘッダー設定でクラスタ構成がシンプルに

⚠️ 制約事項・注意点

🚫 制限事項

❌cgroupsv1 環境ではクラスタ作成/アップグレードが不可
❌RHEL 7/8 環境では手動での OS 設定変更と再起動が必要
❌Ansible 2.18 により Python 3.9 必須 (古い OS バージョンでは対応不可)
❌RHEL 8.10 未満のバージョンはサポート対象外

💡 考慮すべき点

⚠️cgroupsv2 移行に伴い、既存モニタリングツールや cgroup ベースのカスタムスクリプトの動作確認が必要
⚠️containerd 2.1 へのアップグレードで一部コンテナイメージや設定の互換性を事前検証
⚠️サードパーティストレージベンダーの 1.35.0 認定完了を事前確認
⚠️リリース後 7~14 日で GKE On-Prem API クライアント経由の操作が可能に

💡 ユースケース

💻 ユースケース 1: RHEL 環境のアップグレード

シナリオ: RHEL 8.6 で稼働中のオンプレミスクラスタを 1.35.0 にアップグレードしたい

Bash 
# 1. RHEL バージョンを 8.10 以上にアップデート
sudo dnf update

# 2. cgroupsv2 を有効化
sudo grubby --update-kernel=ALL --args="systemd.unified_cgroup_hierarchy=1"
sudo reboot

# 3. Python 3.9 を確認
python3 --version

# 4. アップグレード実行
bmctl upgrade cluster -c my-cluster --kubeconfig admin-kubeconfig

効果: 最新の Kubernetes 1.35 機能と containerd 2.1 のパフォーマンス向上を活用可能

🔌 ユースケース 2: マウント不整合の早期検出

シナリオ: 本番環境で Secret ローテーション後に Pod のマウントが更新されないケースを検出したい

効果: 定期ヘルスチェックにより、Secret/ConfigMap の変更が Pod に反映されない状態を 5 分以内 に自動検出し、運用チームに通知

📝 まとめ

Google Distributed Cloud for bare metal 1.35.0 は、cgroupsv2 必須化という重要な Breaking Change を含むメジャーアップデートです。RHEL 7/8 環境では事前の OS 設定変更が不可欠であり、アップグレード計画時に十分なテストと準備期間を確保することを強く推奨します。

🚀 ポイント: 新機能の定期ヘルスチェックや EgressDSCP タギングは運用の安定性とネットワーク管理の柔軟性を向上させる重要な追加です。
