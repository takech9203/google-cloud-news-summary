🔒 Google SecOps: VPC Service Controls サポート

Preview Google SecOps Security
2026-02-25

📋 概要

Google SecOps (Google Security Operations) が VPC Service Controls との統合をサポートしました。VPC Service Controls は Google Cloud リソースに対するサービス境界を設定し、意図しないデータ流出や標的型のデータ漏洩から保護するセキュリティ機能です。

今回のアップデートにより、Google SecOps の SIEM および SOAR 環境を VPC Service Controls のサービス境界内に配置し、データの入出力を厳密に制御できるようになりました。セキュリティオペレーションで扱う機密性の高いテレメトリデータやインシデント情報が、許可されたネットワーク境界の外部に流出するリスクを大幅に低減できます。

この機能は、規制対応が求められる金融機関、医療機関、政府機関などの組織、およびゼロトラストセキュリティを推進する企業のセキュリティチームを主な対象としています。

⚠️ 注意: 現在 Preview ステージのため、VPC-SC の Pre-GA Offerings Terms が適用されます。本番環境での完全なサポートは今後の GA リリースで提供予定です。

⭐ 主要機能

🛡️ 1. VPC Service Controls によるサービス境界保護

✔ Google SecOps の API (chronicle.googleapis.com, chronicleservicemanager.googleapis.com) をサービス境界で保護
✔ サービス境界内のリソースと外部との通信を Ingress/Egress ルールで制御
✔ Dry-run モードでの事前検証により、本番適用前に違反を検出可能

🔍 2. SIEM 向け VPC-SC 構成

✔ ログデータ取り込み用の Egress ルール設定 (Pub/Sub 経由)
✔ Google 内部プロジェクト (malachite-oob) へのデータ転送をルールで制御
✔ BigQuery へのセルフマネージド/Advanced BigQuery Export によるデータエクスポート対応

⚙️ 3. SOAR 向け VPC-SC 構成

✔ Secret Manager、Binary Authorization、Monitoring サービスへの Ingress ルール制御
✔ SOAR プロビジョニングサービスアカウントによる Egress 通信を制御
✔ Security Command Center 連携時の Pub/Sub および SecurityCenter API アクセスルール設定

🛠️ 設定方法

前提条件

✔組織レベルの IAM ロール (roles/accesscontextmanager.policyAdmin など) を保有
✔Google SecOps の Feature RBAC が有効化済み
✔Google Cloud Identity / BYOID / Workforce Identity Federation のいずれかを使用
1

サービス境界の作成

Google SecOps プロジェクトを含むサービス境界を作成し、制限対象サービスを指定します。

Bash サービス境界の作成 
gcloud access-context-manager perimeters create PERIMETER_NAME \
    --title="SecOps Perimeter" \
    --resources="projects/PROJECT_NUMBER" \
    --restricted-services="chronicle.googleapis.com,chronicleservicemanager.googleapis.com" \
    --policy=POLICY_ID
2

Ingress/Egress ルールの設定 (Dry-run 推奨)

Dry-run モードで事前検証してから Enforced モードに移行することが推奨されています。

Bash Dry-run モードでの検証 
gcloud access-context-manager perimeters dry-run create PERIMETER_NAME \
    --resources="projects/PROJECT_NUMBER" \
    --restricted-services="chronicle.googleapis.com,chronicleservicemanager.googleapis.com" \
    --ingress-policies=ingress.yaml \
    --egress-policies=egress.yaml \
    --policy=POLICY_ID
3

違反の確認と Enforced モードへの移行

Cloud Audit Logs で違反がないことを数週間確認してから、Enforced モードに移行します。

Bash VPC-SC 違反ログの確認 
gcloud logging read \
    'protoPayload.metadata.@type="type.googleapis.com/google.cloud.audit.VpcServiceControlAuditMetadata"' \
    --project=PROJECT_ID \
    --limit=50

📝 Ingress/Egress ルール設定例

SOAR 向け Ingress ルール

YAML Secret Manager / SOAR プロビジョニング用 Ingress ルール 
# Secret Manager へのアクセスを許可
- ingressFrom:
    identityType: ANY_SERVICE_ACCOUNT
    sources:
    - accessLevel: "*"
  ingressTo:
    operations:
    - serviceName: secretmanager.googleapis.com
      methodSelectors:
      - method: "*"
    resources:
    - projects/PROJECT_NUMBER

# SOAR プロビジョニングサービス用
- ingressFrom:
    identities:
    - serviceAccount: chronicle-soar-provisioning-service@system.gserviceaccount.com
    sources:
    - accessLevel: "*"
  ingressTo:
    operations:
    - serviceName: binaryauthorization.googleapis.com
      methodSelectors:
      - method: "*"
    - serviceName: monitoring.googleapis.com
      methodSelectors:
      - method: "*"
    resources:
    - projects/PROJECT_NUMBER

SIEM 向け Egress ルール

YAML Pub/Sub 経由のログデータ転送用 Egress ルール 
- egressTo:
    operations:
    - serviceName: pubsub.googleapis.com
      methodSelectors:
      - method: "*"
    resources:
    - projects/389186463911
  egressFrom:
    identities:
    - user: "*"
    sources:
    - resource: PROJECT_NUMBER

📈 メリット

💼 ビジネス面

✔ コンプライアンス強化: PCI DSS、SOX、HIPAA などの規制で求められるデータ境界要件を Google SecOps 環境で充足可能に
✔ データガバナンス向上: 機密データの流出リスクを大幅に低減し、組織のデータガバナンス戦略を強化
✔ セキュリティ投資の最大化: 既存の VPC-SC インフラを Google SecOps に拡張でき、追加ツール不要

⚙️ 技術面

✔ ネットワークレベルの防御: API レベルのアクセス制御 + ネットワーク境界によるデータ流出防止
✔ きめ細かなアクセス制御: サービスアカウント、ユーザー、プロジェクト単位で通信を制御可能
✔ Dry-run モードによる安全な導入: 本番環境に影響を与えずに設定を検証、段階的な導入が可能

⚠️ 制限事項・注意点

制限事項

✘ VPC-SC で保護されるのは chronicle.googleapis.com と chronicleservicemanager.googleapis.com の 2 API のみ
✘ Cloud Monitoring による SecOps ヘルスメトリクス監視は VPC-SC 保護対象外
✘ Looker ダッシュボードは非対応 (ネイティブダッシュボードのみ対応)
✘ Xenon フィードは非対応 (GOOGLE_CLOUD_STORAGE_V2 ソースタイプの使用が必要)
✘ Security Validation (攻撃シミュレーション) は VPC-SC 非対応
✘ DataTap は VPC-SC 非対応

考慮すべき点

💡Preview ステージのため、本番環境での完全サポートは未提供。Pre-GA Offerings Terms が適用
💡CMEK を異なるプロジェクトで管理している場合、追加の Egress ルール設定が必要
💡既存環境への適用時は Dry-run モードで数週間のテストを推奨
💡Feature RBAC が未有効化の場合、まず RBAC の設定完了が必要

🎯 ユースケース

🏦 ユースケース 1: 金融機関のセキュリティオペレーション保護

シナリオ: 金融機関が Google SecOps でセキュリティ監視を行い、PCI DSS や SOX 規制準拠のためにデータ境界設定が求められている。

YAML 社内ネットワークからのみアクセスを許可 
# 金融機関向け: 社内ネットワークからのみ SecOps へのアクセスを許可
- ingressFrom:
    identityType: ANY_IDENTITY
    sources:
    - accessLevel: accessPolicies/POLICY_ID/accessLevels/corp-network
  ingressTo:
    operations:
    - serviceName: chronicle.googleapis.com
      methodSelectors:
      - method: "*"
    resources:
    - projects/PROJECT_NUMBER
✅ 効果: セキュリティテレメトリデータが境界内に隔離され、許可ネットワークからのみアクセス可能に。規制監査時にデータ境界の証跡を提示可能。

🌐 ユースケース 2: マルチクラウド環境でのデータ流出防止

シナリオ: マルチクラウド環境で Google SecOps を運用し、オンプレミスや他クラウドからのログ収集時のデータ流出を防止したい。

✅ 効果: Ingress/Egress ルールにより、ログソースからのデータ取り込みと BigQuery へのエクスポートのみが許可され、それ以外の経路でのデータ流出を防止。SOAR による自動対応プレイブック実行時の外部サービス連携も制御下に。

💰 料金情報

Google SecOps はサブスクリプションベースの課金モデルを採用しています。パッケージは Standard、Enterprise、Enterprise Plus の 3 種類で、データ取り込み量に基づくクレジットベースの課金が行われます。

💡 ポイント: VPC Service Controls 自体の利用に追加料金はかかりません。ただし、関連する Google Cloud サービス (BigQuery, Pub/Sub, Secret Manager 等) の利用料金は別途発生する場合があります。

🌍 利用可能リージョン

Google SecOps はリージョン固有の SKU を使用しており、データ所在地要件に対応しています。VPC Service Controls の利用可能リージョンについては、VPC Service Controls のドキュメントを参照してください。

🔗 関連サービス

🛡️ VPC Service Controls

Google Cloud リソースに対するサービス境界を設定し、データ流出を防止。本アップデートの基盤となるサービス。

🔍 Security Command Center

SOAR と連携して脅威検出結果を取り込み。VPC-SC 環境下では専用 Ingress/Egress ルール設定が必要。

🔑 Cloud KMS (CMEK)

顧客管理暗号鍵による暗号化。VPC-SC と組み合わせて暗号化+ネットワーク境界の両面から保護。

📋 Access Context Manager

アクセスポリシーとアクセスレベルを管理し、VPC Service Controls の制御を補完。

📄 Cloud Audit Logs

Dry-run / Enforced モードでの VPC-SC 違反ログを記録し、設定の検証に使用。

📝 まとめ

Google SecOps の VPC Service Controls サポートは、セキュリティオペレーション環境におけるデータ流出防止の重要な強化です。特に規制産業の顧客にとって、セキュリティ監視基盤そのものをネットワーク境界で保護できることは、コンプライアンス要件の充足に直結します。
