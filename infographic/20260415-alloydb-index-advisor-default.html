🗄️ AlloyDB for PostgreSQL: インデックスアドバイザーのデフォルト有効化と ScaNN インデックス推奨

AlloyDB for PostgreSQL GA Database
2026-04-15

📋 概要

AlloyDB for PostgreSQL のインデックスアドバイザーがデフォルトで有効化されました。新規および既存の AlloyDB インスタンスにおいて、追加の設定作業なしにインデックス推奨機能が利用可能になります。さらに、ベクトル検索用の ScaNN (Scalable Nearest Neighbors) インデックスの推奨もデフォルトで提供されるようになりました。

インデックスアドバイザーは、データベースが定期的に処理するクエリを追跡・分析し、クエリパフォーマンスを向上させる新しいインデックスを自動的に推奨する機能です。従来の B-tree インデックスの推奨に加え、AI/ML ワークロードで重要なベクトル検索のための ScaNN インデックスも推奨対象に含まれるようになりました。

🤖 AI/ML への影響: 生成 AI やセマンティック検索アプリケーションの最適化が大幅に容易になります。RAG パイプラインやセマンティック検索を構築しているチームにとって、パフォーマンスチューニングの負担を軽減する重要な改善です。

Before / After 比較

❌ Before (アップデート前)
✗インデックスアドバイザーの利用には手動での有効化設定が必要
✗ScaNN インデックス推奨には enable_vector_index_advisor フラグを個別に有効化する必要
✗新規インスタンスで有効化を忘れると最適化の機会を逃す
✗ScaNN のパラメータ (num_leaves, quantizer) を手動で計算・設定する専門知識が必要
✅ After (アップデート後)
✓インスタンス作成直後からインデックスアドバイザーが自動有効化
✓ScaNN ベクトルインデックス推奨がデフォルトで有効
✓Query Insights ダッシュボードからワンクリックで推奨確認・適用
✓AI/ML ワークロードのインデックス最適化が自動化

⚡ 主要機能

🔧 1. インデックスアドバイザーのデフォルト有効化

→google_db_advisor.enabled フラグがデフォルトで on に設定
→24 時間ごとに自動分析が実行され、パフォーマンス向上インデックスの候補を推奨
→遅いクエリの上位 100 件を対象に分析。推奨インデックスの推定ストレージサイズや影響クエリ数も提示

🧠 2. ScaNN ベクトルインデックス推奨

→google_db_advisor.enable_vector_index_advisor フラグがデフォルトで on
→ScaNN は HNSW と比較してインデックス作成 10x、検索 4x、フィルタ付き検索 10x 高速
→alloydb_scann 拡張機能が有効な場合に ScaNN インデックスの推奨を自動生成

📈 3. Query Insights との統合

→Query Insights ダッシュボードの「Top dimensions by database load」テーブルの Recommendations 列にインデックス推奨を表示
→ScaNN ベクトルインデックスの推奨も同じインターフェースで確認可能
→推奨される CREATE INDEX コマンドをコピーして直接適用

インデックスアドバイザー データベースフラグ

フラグ名 型 デフォルト 説明

⚙️ 設定方法

前提条件

✓AlloyDB for PostgreSQL クラスタおよびインスタンスが作成済みであること
✓ScaNN インデックスの推奨を利用する場合: alloydb_scann および vector 拡張機能がインストール済みであること

ステップ 1: 拡張機能のインストール (ScaNN 推奨を利用する場合)

SQL pgvector と alloydb_scann 拡張機能のインストール 
-- pgvector 拡張機能のインストール
CREATE EXTENSION IF NOT EXISTS vector;

-- alloydb_scann 拡張機能のインストール
CREATE EXTENSION IF NOT EXISTS alloydb_scann;
💡 注意: B-tree インデックスの推奨のみであれば、追加の拡張機能は不要です。ScaNN ベクトルインデックスの推奨を受けるには上記の拡張機能が必要です。

ステップ 2: インデックスアドバイザーの動作確認

SQL インデックスアドバイザーの有効化確認 
-- インデックスアドバイザーが有効であることを確認
SHOW google_db_advisor.enabled;
-- 結果: on

-- ベクトルインデックスアドバイザーが有効であることを確認
SHOW google_db_advisor.enable_vector_index_advisor;
-- 結果: on

-- 自動分析スケジュールの確認
SHOW google_db_advisor.auto_advisor_schedule;
-- 結果: EVERY 24 HOURS

ステップ 3: 推奨インデックスの確認と適用

SQL 推奨インデックスの確認と適用 
-- 推奨インデックスの一覧を取得
SELECT * FROM google_db_advisor_recommended_indexes;

-- 推奨インデックスと関連クエリの詳細を取得
SELECT DISTINCT recommended_indexes, query
FROM google_db_advisor_workload_report r,
     google_db_advisor_workload_statements s
WHERE r.query_id = s.query_id;

-- 即時分析を実行して推奨を取得
SELECT * FROM google_db_advisor_recommend_indexes();

🎯 ユースケース

🤖 ユースケース 1: RAG パイプラインのベクトル検索最適化

シナリオ: 企業のナレッジベースを AlloyDB に格納し、Vertex AI のエンベディングモデルでベクトル化したドキュメントに対して RAG パイプラインからベクトル検索を実行。クエリ数の増加に伴いレスポンスが遅延。

SQL RAG パイプライン実装例 
-- ベクトルカラムを含むテーブルの作成
CREATE TABLE knowledge_base (
    id SERIAL PRIMARY KEY,
    title TEXT,
    content TEXT,
    embedding vector(768),
    category VARCHAR(50),
    updated_at TIMESTAMP
);

-- インデックスアドバイザーの推奨を確認
SELECT * FROM google_db_advisor_recommend_indexes();

-- 推奨に従い ScaNN インデックスを作成 (推奨例)
CREATE INDEX ON knowledge_base
USING scann (embedding cosine)
WITH (num_leaves = 100, quantizer = 'SQ8');

効果: インデックスアドバイザーがワークロードに基づいて最適な ScaNN インデックス設定を推奨するため、手動でのパラメータチューニングが不要になり、ベクトル検索のレイテンシが大幅に改善。

🛒 ユースケース 2: E コマースのセマンティック商品検索

シナリオ: E コマースサイトで商品のテキスト説明と画像をベクトル化し、ユーザーの自然言語クエリに対してセマンティック検索を実行。カテゴリフィルタとの組み合わせクエリが多いが、パフォーマンスが安定しない。

SQL フィルタ付きベクトル検索クエリ 
-- フィルタ付きベクトル検索クエリ (アプリケーション側)
SELECT product_id, name, price
FROM products
WHERE category = 'electronics'
ORDER BY embedding <=> $1::vector
LIMIT 20;

効果: ScaNN のアダプティブフィルタリング機能によりカテゴリフィルタとベクトル検索が同時に最適化。フィルタ付きベクトル検索が HNSW 比で最大 10 倍高速化。

🚀 ユースケース 3: 新規プロジェクトのデータベース立ち上げ

シナリオ: 新規 AlloyDB インスタンスを作成し、アプリケーションの初期開発を開始。クエリパターンが固まっていない段階から、インデックス最適化を意識したい。

効果: デフォルトでインデックスアドバイザーが有効なため、開発段階からクエリパターンが追跡され、本番リリース前にインデックス推奨に基づいた最適化を実施可能。「本番稼働後にパフォーマンス問題が発覚してから対応する」というリアクティブなアプローチを回避。

⚠️ デメリット・制約事項

制限事項

✗ScaNN インデックスの推奨を受けるには、alloydb_scann および vector 拡張機能を事前にインストールしておく必要がある
✗自動分析には最大 1800 秒/日の CPU リソースが消費される (デフォルト設定)
✗推奨インデックスは自動作成されない。手動で CREATE INDEX コマンドを実行する必要がある
✗alloydbsuperuser ロール以外のユーザーで即時分析を実行した場合、そのユーザーが発行したクエリに基づく推奨のみが表示される

考慮すべき点

→推奨インデックスの適用はストレージを消費するため、推定ストレージサイズと影響クエリ数を考慮して判断すること
→極めてリソースに制約のあるインスタンスでは auto_advisor_max_time_in_seconds の調整を検討
→ScaNN インデックスの auto_maintenance を有効にすることで、データセットの成長に合わせてセントロイドが自動更新される
