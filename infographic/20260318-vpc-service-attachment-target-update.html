🌐 VPC Service Attachment: Target Service In-Place Update

Virtual Private Cloud Private Service Connect GA
2026-03-18

📋 Overview

Private Service Connect のService Attachmentにおいて、 ターゲットサービスをインプレース (in-place) で更新できる機能が GA (一般提供) となりました。

サービスプロデューサーは、Service Attachment を削除・再作成することなくターゲットサービスを別のサービスへ変更可能に。 コンシューマーの接続ステータスは ACCEPTED のまま維持され、再接続の手動調整が不要になります。

✅ 0
Service Attachment 再作成
🔗 ACCEPTED
接続ステータス維持
⚡ Short
トラフィック中断時間

🔄 Before / After

❌ Before

  • ターゲット変更には Service Attachment の削除と再作成が必要
  • コンシューマー接続が切断され、再接続の手動調整が必要
  • LB 種類変更やバックエンド移行時にダウンタイムが長期化するリスク

✅ After

  • 再作成なしでターゲットサービスを直接更新可能
  • コンシューマー接続ステータスが維持され再接続不要
  • トラフィック中断が短時間に限定されサービス可用性が向上

🏗️ Architecture

サービスプロデューサーがターゲットサービスを更新すると、Service Attachment は新しいターゲットへの切り替えを実施。短時間のトラフィック中断後、コンシューマーの接続は ACCEPTED のまま維持されます。

🔧 Key Features

🔄 Target Service In-Place Update
  • Service Attachment の削除・再作成なしでターゲットサービスを変更可能
  • Google Cloud Console、gcloud CLI、REST API の全インターフェースから操作可能
  • 内部プロトコル転送ターゲットへの変更は gcloud CLI または API の使用が必要
🔗 Consumer Connection Maintenance
  • 更新時にコンシューマーの接続ステータスは変更されない
  • コンシューマーへの自動通知は行われないが、接続は保持される
  • 必要に応じてダウンタイムをコンシューマーに通知する手順も用意
🔔 Downtime Notification Options
  • 接続再調整 (connection reconciliation) を有効化し、コンシューマーを拒否リストに追加してダウンタイムを明示的に通知
  • テスト用 VPC ネットワークとインスタンスを作成して接続復旧を検証するオプションも提供

⚙️ Technical Specifications

Item Details
Supported Target Services Internal Passthrough NLB, Regional Internal ALB, Cross-region Internal ALB, Internal Protocol Forwarding, Regional Internal Proxy NLB, Secure Web Proxy
Operations Google Cloud Console, gcloud CLI, REST API
Traffic Interruption Brief interruption during switchover
Connection Status Impact None (remains ACCEPTED)

💻 Configuration

gcloud CLI

Bash Update target service 
gcloud compute service-attachments update ATTACHMENT_NAME \
  --region=REGION \
  --target-service=projects/PROJECT_ID/regions/RULE_REGION/forwardingRules/RULE_NAME

REST API

Bash 1. Get fingerprint 
GET https://compute.googleapis.com/compute/v1/projects/PROJECT_ID/regions/REGION/serviceAttachments/ATTACHMENT_NAME
JSON 2. PATCH target service 
{
  "targetService": "projects/PROJECT_ID/regions/RULE_REGION/forwardingRules/RULE_NAME",
  "fingerprint": "FINGERPRINT"
}

📋 Procedure

1
(Optional) Notify consumers of downtime

接続再調整が有効であることを確認し、コンシューマーを拒否リストに追加

Bash 
gcloud compute service-attachments update ATTACHMENT_NAME \
  --region=REGION \
  --consumer-reject-list=CONSUMER_PROJECT_1,CONSUMER_PROJECT_2
2
Update target service

Service Attachment のターゲットを新しいサービスに切り替え

Bash 
gcloud compute service-attachments update ATTACHMENT_NAME \
  --region=REGION \
  --target-service=projects/PROJECT_ID/regions/RULE_REGION/forwardingRules/NEW_RULE_NAME
3
Verify and restore access

コンシューマーを拒否リストから削除して接続を復元

Bash 
gcloud compute service-attachments update ATTACHMENT_NAME \
  --region=REGION \
  --consumer-reject-list=

📈 Benefits

💼 Business

→Downtime Minimization: Service Attachment 再作成不要でサービス中断時間を大幅短縮
→Operational Cost Reduction: コンシューマーとの再接続調整が不要になり運用負荷を軽減

⚙️ Technical

→Connection Continuity: エンドポイントの再作成・再構成が不要
→Flexible Infra Management: LB 種類変更やバックエンド移行をコンシューマー影響なしで実施
→Staged Migration: ダウンタイム通知やテストコンシューマー準備手順を提供

⚠️ Limitations / Considerations

⚠️ Limitations
  • トラフィックが短時間中断されるため完全にゼロダウンタイムではない
  • コンシューマーへのサービス中断は自動通知されないため手動通知が必要
  • サポートされる構成の組み合わせに制限あり (公式ドキュメントの「Service mutability」参照)
💡 Considerations
  • 内部プロトコル転送ターゲットへの変更は Console 非対応。gcloud CLI / API を使用
  • Terraform 使用時は Google Provider v4.82.0 以降へ更新推奨 (意図しない再作成を防止)
  • REST API 使用時は更新前にフィンガープリント取得が必要

🎯 Use Cases

🔄 Use Case 1: Load Balancer Type Migration

Scenario: 内部パススルー Network Load Balancer からリージョン内部 Application Load Balancer への移行。L7 ルーティングや SSL 終端機能が必要になった場合。

Bash NLB to ALB migration 
gcloud compute service-attachments update my-service-attachment \
  --region=us-central1 \
  --target-service=projects/my-project/regions/us-central1/forwardingRules/new-alb-rule

Effect: コンシューマーのエンドポイント再作成なしで、バックエンドの LB を L4 から L7 に移行可能。

🚀 Use Case 2: Multi-tenant Service Version Upgrade

Scenario: SaaS プロバイダーが複数テナント (コンシューマー) にサービス提供中。新バージョンのバックエンドは新しい LB の背後にデプロイ済み。

Effect: 各テナントの PSC エンドポイントを維持したまま、バックエンドを新バージョンに切り替え可能。テナント側の設定変更不要。
