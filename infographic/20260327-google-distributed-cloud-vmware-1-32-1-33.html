📋 概要

Google Distributed Cloud (software only) for VMware の2つの新バージョンがリリースされました。クラスタのアップグレードや運用に影響を与える重要なバグ修正が含まれており、該当バージョンを利用中のユーザーは早急なアップグレードが推奨されます。

⚠️ 重要: バージョン 1.32 から 1.33 へのアップグレード時にはクラスタが自動的に Advanced クラスタ に変換されます。今回の修正により、この移行プロセスの信頼性が大幅に向上しました。

🔄 Before / After 比較

❌ アップデート前の課題

  • node-problem-detector が非 Advanced クラスタに誤ってデプロイされ、containerd ランタイムが継続的に再起動
  • stackdriver.enableVPC が true の場合、Advanced クラスタへのアップグレードがブロック
  • カスタム CA 証明書設定時にシステム証明書プールが無視
  • gkectl upgrade admin の再試行時に「AlreadyExists」エラー
  • proxy/noProxy 設定の空白でクラスタ作成/アップグレードが失敗
  • アップグレード失敗時にブートストラップクラスタ削除でデータ損失

✅ アップデート後の改善

  • node-problem-detector が Advanced クラスタのみに正しくデプロイ
  • stackdriver.enableVPC がバリデーション中に無視され、移行がスムーズに
  • カスタム CA とシステム証明書プールの両方が正しく参照
  • gkectl upgrade admin の再試行が正常動作
  • proxy/noProxy フィールドの空白が適切にトリミング
  • アップグレード失敗時のデータ保護が強化

🔧 主要機能 (バグ修正)

🚨 1. node-problem-detector デプロイ対象の修正

v1.32v1.33

node-problem-detector が非 Advanced (V1) VMware クラスタに誤ってデプロイされ、containerd ランタイムが継続的に再起動し、ETCD/CRI 障害やクラスタアップグレードの失敗が発生していた問題を修正。修正後は Advanced クラスタにのみ正しくデプロイされます。

🚨 2. アップグレード失敗時のデータ損失防止

v1.32v1.33

Advanced 管理クラスタのアップデート/アップグレードが失敗し、外部ブートストラップクラスタが削除された場合に重要なデータが失われる問題を修正。--reuse-bootstrap-cluster (または -r) フラグの使用が引き続き推奨されます。

🛠️ 3. stackdriver.enableVPC フィールドの処理改善

v1.32

非推奨の stackdriver.enableVPC フィールドが true に設定されている場合、Advanced クラスタへのアップグレードバリデーションがブロックされていた問題を修正。アップグレード時にこの設定が無視されるようになりました。

🔒 4. カスタム CA 証明書処理の修正

v1.32

レジストリミラー用にカスタム CA 証明書が設定されている場合、システム証明書プールが無視される問題を修正。カスタム CA とシステム証明書プールの両方が正しく参照されるようになりました。

🔄 5. gkectl upgrade admin リトライ時の AlreadyExists エラー修正

v1.32

以前の失敗後に gkectl upgrade admin コマンドを再実行すると、ブートストラップクラスタで「AlreadyExists」エラーが発生する問題を修正。

␡️ 6. proxy/noProxy 設定の空白処理修正

v1.32

proxy/noProxy 設定フィールドに余分な空白が含まれていると、内部コマンドライン引数のパースに干渉し、コントロールプレーンのロードバランサー初期化が失敗する問題を修正。

📝 アップグレード手順

前提条件

✅管理ワークステーションが適切なバージョンにアップグレード済み
✅管理クラスタ/ユーザークラスタの現行バックアップを取得済み
✅gkectl バージョンがターゲットバージョンと互換

ステップ 1: OS イメージの準備

BASH 
gkectl prepare \
  --bundle-path /var/lib/gke/bundles/gke-onprem-vsphere-TARGET_VERSION.tgz \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG

ステップ 2: Advanced クラスタへのアップグレード準備 (該当する場合)

BASH 
gkectl prepare \
  --bundle-path BUNDLE_PATH \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --advanced-cluster

ステップ 3: 管理クラスタのアップグレード

BASH 
gkectl upgrade admin \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config ADMIN_CLUSTER_CONFIG \
  --reuse-bootstrap-cluster

ステップ 4: ユーザークラスタのアップグレード

BASH 
gkectl upgrade cluster \
  --kubeconfig ADMIN_CLUSTER_KUBECONFIG \
  --config USER_CLUSTER_CONFIG

🎯 ユースケース

🔬 ユースケース 1: Advanced クラスタへの移行

シナリオ: 現在バージョン 1.32 の非 Advanced クラスタを運用しており、1.33 の Advanced クラスタに移行したい。以前は stackdriver.enableVPC の設定や node-problem-detector の問題によりアップグレードが失敗していた。

BASH 
# バックアップの取得
gkectl backup admin --kubeconfig ADMIN_KUBECONFIG

# Advanced クラスタ用の準備
gkectl prepare \
  --bundle-path /var/lib/gke/bundles/gke-onprem-vsphere-1.33.600-gke.40.tgz \
  --kubeconfig ADMIN_KUBECONFIG \
  --advanced-cluster

# 管理クラスタのアップグレード
gkectl upgrade admin \
  --kubeconfig ADMIN_KUBECONFIG \
  --config ADMIN_CONFIG \
  --reuse-bootstrap-cluster
💡 効果: 非推奨フィールドや node-problem-detector の問題に阻まれることなく、安全に Advanced クラスタへ移行できる

🌐 ユースケース 2: プロキシ環境でのクラスタ運用

シナリオ: 企業のプロキシ環境下で Google Distributed Cloud を運用しており、proxy/noProxy 設定の空白が原因でクラスタの作成やアップグレードが失敗していた。

💡 効果: proxy/noProxy 設定フィールドの空白が適切にトリミングされるようになり、プロキシ環境下でのクラスタ運用が安定化する
