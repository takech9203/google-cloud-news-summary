Bigtable Preview

📋 概要

Bigtable の階層型ストレージ (Tiered Storage) の設定を Google Cloud コンソールから直接管理できるようになりました。 これまで gcloud CLI でのみ可能だった階層型ストレージの有効化、年齢しきい値の変更、無効化といった操作が GUI 上で実行可能になりました。

さらに、System Insights で階層型ストレージに関するメトリクスを可視化して確認できるようになり、 SSD ティアと低頻度アクセスティア間のデータ分布やパフォーマンスをコンソール上で一元的に監視できます。

🔄 Before / After

❌ Before

  • 階層型ストレージの設定には gcloud beta コマンドが必要
  • 年齢しきい値の変更・無効化も CLI 必須
  • メトリクス確認には Cloud Monitoring を別途参照

🛠️ 設定方法

📌 前提条件: Bigtable SSD インスタンスが作成済みであること (HDD インスタンスでは利用不可)。 bigtable.tables.update 権限を持つ IAM ロール (例: roles/bigtable.admin) が必要。

コンソールから設定

1

テーブルの階層型ストレージ設定

Google Cloud コンソールで Bigtable インスタンス一覧を開き、対象インスタンスの「Tables」から階層型ストレージを構成

2

System Insights でメトリクスを確認

対象の Bigtable インスタンスの System Insights セクションで、各ティアのデータ使用量、レイテンシ、リクエスト数を監視

CLI での設定 (参考)

Bash 階層型ストレージの設定コマンド 
# テーブル作成時に階層型ストレージを有効化
gcloud beta bigtable instances tables create TABLE_ID \
  --instance=INSTANCE_ID \
  --project=PROJECT_ID \
  --tiered-storage-infrequent-access-older-than=60d

# 年齢しきい値の変更
gcloud beta bigtable instances tables update TABLE_ID \
  --instance=INSTANCE_ID \
  --project=PROJECT_ID \
  --tiered-storage-infrequent-access-older-than=90d

# 階層型ストレージの無効化
gcloud beta bigtable instances tables update TABLE_ID \
  --instance=INSTANCE_ID \
  --project=PROJECT_ID \
  --clear-tiered-storage-config

✅ メリット

💼 ビジネス面

✓ ストレージコスト削減: アクセス頻度の低い古いデータを低コストの低頻度アクセスティアに自動移動
✓ 運用負荷の軽減: CLI 不要で運用チームの学習コストと操作ミスのリスクが低下

🔬 技術面

✓ 可視性の向上: System Insights でティアごとのメトリクスを一元監視
✓ 透過的なデータアクセス: 既存の読み取り・書き込みコードの変更不要
✓ 双方向のデータ移動: しきい値変更や無効化でデータを SSD に戻すことも可能

⚠️ デメリット・制約事項

制限事項

✗ Data Boost はサポートされていない
✗ ホットバックアップはサポートされていない
✗ HDD インスタンスでは階層型ストレージを利用できない
✗ 年齢しきい値の最小値は 30 日

考慮すべき点

→ 低頻度アクセスティアの読み取りは SSD と比較してスループットが低下
→ タイムスタンプ範囲フィルタ未使用時は両ティアを確認するためレイテンシ増加の可能性
→ 頻繁なデータ移動でコンパクションコスト発生・処理遅延の可能性
→ 現在 Preview 段階のため GA 時に仕様が変更される可能性あり

🎯 ユースケース

🏦 ユースケース 1: 規制コンプライアンスのためのデータ長期保存

シナリオ: 金融機関が取引データを規制上の理由で数年間保存する必要がある。直近 60 日間のデータは頻繁にアクセスされるが、それ以前のデータはほとんどアクセスされない。

効果: 年齢しきい値を 60 日に設定することで、60 日以上経過したデータは自動的に低コストの低頻度アクセスティアに移動される。直近のデータは SSD の高パフォーマンスを維持しつつ、全体のストレージコストを削減できる。

📡 ユースケース 2: IoT 時系列データの階層化管理

シナリオ: IoT デバイスから大量のセンサーデータが Bigtable に書き込まれる。リアルタイム分析は直近 30 日分で十分だが、ML モデルのトレーニングのために過去データも保持しておきたい。

効果: 年齢しきい値を 30 日に設定し、古いデータを低頻度アクセスティアに移動。モデルトレーニング時にはレイテンシが多少増加するが、日常のリアルタイム分析は SSD ティアで高パフォーマンスを維持できる。
