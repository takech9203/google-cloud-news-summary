⚙️ Cloud Workstations: running_timeout 到達前の通知機能を追加

Cloud Workstations GA

📋 概要

Cloud Workstations の事前構成済みベースイメージに、ワークステーションの running_timeout に近づいた際にユーザーへ通知する機能が追加されました。これにより、開発者はワークステーションが自動シャットダウンされる前に、作業内容の保存やタイムアウト設定の見直しといった適切な対応を取ることが可能になります。

Cloud Workstations では、ワークステーション構成 (WorkstationConfig) に running_timeout を設定することで、一定時間後にワークステーションを自動停止できます。デフォルト値は 43,200 秒 (12 時間) であり、コスト最適化とセキュリティアップデート適用のために日次でのシャットダウンが推奨されています。

❌ Before (アップデート前)

  • running_timeout に到達するとワークステーションが予告なく自動シャットダウン
  • 未保存の作業が失われるリスク
  • 残り稼働時間を把握する手段がない
  • 独自のタイマーやスクリプトでの運用が必要

✅ After (アップデート後)

  • running_timeout に近づくと事前通知が表示
  • シャットダウン前に作業内容を保存する時間的余裕
  • 追加設定やカスタムスクリプトは不要
  • 全ベースイメージで自動的に通知が有効

⭐ 主要機能

🔔 running_timeout 到達前の通知

ベースイメージに組み込まれた通知機能により、ワークステーションの running_timeout に近づくとユーザーに警告が表示されます。事前構成済みベースイメージ (Code OSS、JetBrains IDE、Base イメージなど) すべてに含まれます。

✅ 自動有効化

通知機能はベースイメージの更新に含まれるため、追加設定なしで自動的に利用可能になります。既存のワークステーションもベースイメージを更新することで通知機能を利用できます。

🔗 既存のタイムアウト管理との連携

running_timeout (最大稼働時間) と idle_timeout (アイドルタイムアウト) は独立して動作します。既存の /google/scripts/keep_alive.sh スクリプトはアイドルタイムアウトの防止に使用されますが、running_timeout による停止は防止しません。

running_timeout の設定値

項目 詳細
デフォルト値 "43200s" (12 時間)
最小値 "0s" (タイムアウトなし、非推奨)
最大値 (暗号化キー使用時) "86400s" (24 時間) 未満
推奨設定 日次でのシャットダウンを推奨
形式 秒単位の Duration ("54000s" = 15 時間)

🛠️ 設定方法

💡 前提条件: Cloud Workstations API が有効化されていること、ワークステーションクラスタとワークステーション構成が作成済みであること。

ステップ 1: running_timeout を確認・設定する

Bash running_timeout の確認と設定 
# 既存のワークステーション構成の詳細を確認
gcloud workstations configs describe CONFIG_NAME \
  --cluster=CLUSTER_NAME \
  --region=REGION

# 新しいワークステーション構成を作成 (running_timeout を 15 時間に設定)
gcloud workstations configs create CONFIG_NAME \
  --cluster=CLUSTER_NAME \
  --region=REGION \
  --running-timeout=54000

running_timeout のデフォルト値は 43,200 秒 (12 時間) です。業務時間に合わせて適切な値を設定することを推奨します。

ステップ 2: ベースイメージが最新であることを確認する

Bash ベースイメージを指定してワークステーション構成を作成 
# ワークステーション構成でベースイメージを指定して作成
gcloud workstations configs create CONFIG_NAME \
  --cluster=CLUSTER_NAME \
  --region=REGION \
  --container-predefined-image=codeoss \
  --running-timeout=54000

事前構成済みベースイメージを使用している場合、ワークステーション再起動時に最新のイメージが自動的に適用されます。通知機能はベースイメージに含まれているため、追加の設定は不要です。

📈 メリットと制約事項

✅ メリット

💼 ビジネス面

✓ 作業データ損失の防止 -- 自動シャットダウン前に通知が行われるため、未保存の作業が失われるリスクが大幅に軽減
✓ 開発者の生産性向上 -- 予期しないセッション切断による作業の中断と再開のコストが削減

💻 技術面

✓ ゼロ設定での利用 -- ベースイメージに組み込まれているため、追加の設定やカスタムスクリプトの開発が不要
✓ 全ベースイメージへの対応 -- Code OSS、JetBrains IDE、Base イメージなど、すべての事前構成済みベースイメージで利用可能

⚠️ 制約事項・注意点

🚫 制限事項

• カスタムコンテナイメージを使用している場合、通知機能は自動的には含まれない (ベースイメージを拡張している場合のみ利用可能)

• running_timeout を "0s" に設定している場合 (タイムアウトなし)、通知は発生しない

💡 考慮すべき点

• 既存のワークステーションでは次回起動時にイメージが更新されるまで通知機能は利用できない

• running_timeout による停止は /google/scripts/keep_alive.sh では防止できない (keep_alive.sh は idle_timeout にのみ影響)

💡 ユースケース

💻 ユースケース 1: 長時間のコーディングセッション

シナリオ: 開発者が朝からワークステーションを起動し、終日にわたってコーディング作業を行っている。running_timeout が 12 時間に設定されている場合、夕方頃にタイムアウトが到達する。

➤ 効果: タイムアウト到達前に通知が表示されるため、開発者は変更をコミットし、作業状態を保存してから、必要に応じてワークステーションを再起動できる。

🔧 ユースケース 2: 長時間のビルド・テスト実行中

シナリオ: 開発者が大規模なビルドやテストスイートを実行中に running_timeout に近づく。ビルドプロセスは未完了の状態。

➤ 効果: 事前通知により、開発者はビルドの完了を待つか、タイムアウトを延長した構成への変更を管理者に依頼するなど、適切な判断を下す時間を得られる。
