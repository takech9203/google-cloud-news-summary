📊 BigQuery Agent Analytics が ADK で GA

BigQuery Agent Developer Kit GA Open Source
2026-04-15
Google Cloud

📋 概要

BigQuery agent analytics が Google Agent Developer Kit (ADK) において 一般提供 (GA) となりました。マルチモーダルなエージェントインタラクションデータを大規模にキャプチャ、分析、可視化するためのオープンソースソリューションです。

ADK のプラグインアーキテクチャと BigQuery Storage Write API を活用し、エージェントの運用イベントを BigQuery テーブルに直接ストリーミングすることで、デバッグ、リアルタイムモニタリング、包括的なオフライン性能評価を実現します。

✅ 1 行のコード追加で導入可能: App オブジェクトの plugins パラメータにプラグインを追加するだけで、エージェントの全イベントが自動キャプチャされます。

⚡ 主要機能

🎨 マルチモーダルキャプチャ

テキスト、画像、音声、動画などのマルチモーダルデータを BigQuery に記録。大容量コンテンツは GCS に自動オフロード (閾値 500KB)。

📡 包括的イベントサポート

LLM インタラクション、ツール使用、エージェントライフサイクル、HITL イベント、状態管理まで全てキャプチャ。

LLM_REQUEST TOOL_COMPLETED HITL_INPUT_REQUEST STATE_DELTA

🔍 ツールプロバナンス追跡

各ツール呼び出しの出自を自動分類。ツールの依存関係とパフォーマンスの可視化に活用可能。

LOCAL MCP SUB_AGENT A2A TRANSFER_AGENT

🔗 分散トレーシング

OpenTelemetry 互換の trace_id, span_id, parent_span_id による実行フロー追跡。TracerProvider なしでも内部 UUID で相関維持。

📊 自動ビュー生成 (v1.27.0+)

イベントタイプごとのフラットなビュー (v_llm_request, v_tool_completed 等) を自動生成。JSON をアンネストしたクエリしやすい形式。

🔒 セキュリティ機能

OAuth トークン、API キー、クライアントシークレットの自動マスク。temp: / secret: プレフィックスの状態キー自動除去。カスタムフォーマッタ対応。

🛠️ 設定方法

ステップ 1: ADK と依存パッケージのインストール

Bash パッケージインストール 
pip install "google-adk[bigquery]>=1.26.0" \
    google-cloud-bigquery-storage \
    pyarrow \
    opentelemetry-api \
    opentelemetry-sdk

ステップ 2: プラグインの設定とエージェントへの登録

Python プラグイン設定とエージェント登録 
import os
from google.adk.agents import Agent
from google.adk.apps import App
from google.adk.models.google_llm import Gemini
from google.adk.plugins.bigquery_agent_analytics_plugin import (
    BigQueryAgentAnalyticsPlugin,
    BigQueryLoggerConfig,
)

os.environ["GOOGLE_GENAI_USE_VERTEXAI"] = "True"

# プラグインの初期化
bq_analytics_plugin = BigQueryAgentAnalyticsPlugin(
    project_id="my-project-id",
    dataset_id="agent_analytics",
    location="US",
    config=BigQueryLoggerConfig(
        batch_size=1,
        batch_flush_interval=0.5,
        log_session_metadata=True,
    ),
)

# エージェントの定義
root_agent = Agent(
    model=Gemini(model="gemini-flash-latest"),
    name="my_agent",
    instruction="You are a helpful assistant.",
)

# App にプラグインを登録
app = App(
    name="my_agent_app",
    root_agent=root_agent,
    plugins=[bq_analytics_plugin],
)

⚙️ プラグイン設定パラメータ

Python BigQueryLoggerConfig の全オプション 
bq_config = BigQueryLoggerConfig(
    enabled=True,                       # プラグインの有効化
    gcs_bucket_name="my-gcs-bucket",    # GCS オフロード先バケット
    log_multi_modal_content=True,       # マルチモーダルコンテンツのログ
    max_content_length=500 * 1024,      # インラインテキスト上限 (500KB)
    batch_size=1,                       # バッチサイズ (低レイテンシ向け)
    batch_flush_interval=0.5,           # フラッシュ間隔 (秒)
    shutdown_timeout=10.0,              # シャットダウンタイムアウト
    log_session_metadata=True,          # セッションメタデータのログ
)

💡 ユースケース

📈 ユースケース 1: エージェント品質ダッシュボードの構築

カスタマーサポート向け AI エージェントの応答品質をリアルタイムで監視し、改善ポイントを特定。エラー率の上昇やレイテンシの悪化を早期に検知し、プロンプト改善やモデル切り替えの対応を迅速に行える。

SQL エージェントのレイテンシとエラー率を日次集計 
SELECT
    DATE(timestamp) AS date,
    agent,
    COUNT(*) AS total_events,
    COUNTIF(status = 'ERROR') AS error_count,
    ROUND(COUNTIF(status = 'ERROR') / COUNT(*) * 100, 2) AS error_rate_pct,
    AVG(CAST(JSON_VALUE(latency_ms, '$.total_ms') AS FLOAT64)) AS avg_latency_ms
FROM `project.agent_analytics.agent_events`
WHERE event_type IN ('LLM_RESPONSE', 'TOOL_COMPLETED')
GROUP BY date, agent
ORDER BY date DESC;

🔧 ユースケース 2: マルチモーダルエージェントの行動分析

画像認識やドキュメント処理エージェントにおけるツール呼び出しパターンを分析。ボトルネックの特定や不要なツール呼び出しの削減に活用。

SQL ツール呼び出しの来歴と使用頻度を分析 
SELECT
    JSON_VALUE(content, '$.tool') AS tool_name,
    JSON_VALUE(content, '$.tool_origin') AS tool_origin,
    COUNT(*) AS call_count,
    AVG(CAST(JSON_VALUE(latency_ms, '$.total_ms') AS FLOAT64)) AS avg_latency_ms
FROM `project.agent_analytics.agent_events`
WHERE event_type = 'TOOL_COMPLETED'
GROUP BY tool_name, tool_origin
ORDER BY call_count DESC;

💰 ユースケース 3: トークン消費量の最適化

大規模エージェント運用環境でトークン消費量を最適化し、LLM コストを削減。モデルごとのトークン使用量を可視化し、プロンプト簡潔化やコンテキストウィンドウ最適化を推進。

SQL モデル別トークン消費量を集計 
SELECT
    JSON_VALUE(attributes, '$.model_version') AS model,
    SUM(CAST(JSON_VALUE(attributes, '$.usage_metadata.prompt_token_count') AS INT64)) AS total_prompt_tokens,
    SUM(CAST(JSON_VALUE(attributes, '$.usage_metadata.candidates_token_count') AS INT64)) AS total_response_tokens,
    SUM(CAST(JSON_VALUE(attributes, '$.usage_metadata.total_token_count') AS INT64)) AS total_tokens
FROM `project.agent_analytics.agent_events`
WHERE event_type = 'LLM_RESPONSE'
GROUP BY model
ORDER BY total_tokens DESC;

✅ メリット

📊 ビジネス面

→品質の継続的改善: インタラクションデータから応答品質を定量評価し改善サイクルを構築
→運用コスト最適化: トークン消費量やレイテンシの可視化でコスト削減を実現
→コンプライアンス対応: 全インタラクションの監査ログを BigQuery に一元保存
→迅速な障害対応: リアルタイムモニタリングで異常動作を即座に検知

🔧 技術面

→1 行インテグレーション: プラグイン追加だけで包括的ロギング有効化
→非同期ストリーミング: Storage Write API による非同期書き込みでパフォーマンス影響なし
→オープンソース: カスタマイズやコミュニティ貢献が可能
→マルチフレームワーク: ADK + LangGraph (Preview) をサポート

⚠️ デメリット・制約事項

制限事項
  • BigQuery Storage Write API は有料 (月 2 TiB まで無料、以降 $0.025/GB)
  • ADK Python v1.24.0 未満ではサーバーレスランタイム上で非同期ログライターが正常フラッシュされない問題あり
  • LangGraph サポートはまだ Preview 段階 (GA は ADK プラグインのみ)
  • 自動スキーマアップグレード・ツールプロバナンス追跡には ADK v1.26.0 以上が必要
💡 考慮すべき点
  • 秘匿情報のログ混入に注意 - built-in redaction の動作確認とカスタムマスク処理の実装を推奨
  • 高スループット環境では batch_size と batch_flush_interval の適切なチューニングが必要
  • GCS オフロード使用時は追加の GCS ストレージコストとバケット管理が必要
  • BigQuery テーブルへのアクセス権限を IAM で適切に制限すること

📝 まとめ

BigQuery agent analytics の GA リリースは、AI エージェントの可観測性における重要なマイルストーンです。オープンソースの ADK プラグインとして 1 行のコード追加で導入でき、マルチモーダルデータの記録、分散トレーシング、ツールプロバナンス追跡、HITL イベント追跡といった包括的な分析機能を BigQuery 上で実現します。

AI エージェントを本番運用している、または運用予定のチームは、まず開発環境でプラグインを有効化し、エージェントのインタラクションデータの可視化から始めることを推奨します。
