⚙️ Service Extensions: GKE Gateway でのコールアウトが GA に

Service Extensions GKE Gateway GA

ロードバランシング処理パスへのカスタムロジック挿入が本番環境で正式サポート

📅 2026-04-09

📌 概要

GKE Gateway で Service Extensions のコールアウト機能が一般提供 (GA) になりました。 GCPRoutingExtension および GCPTrafficExtension リソースを使用して、Cloud Load Balancing のデータパスにカスタムコードを組み込むことが本番環境で正式にサポートされます。

コールアウトは、ロードバランサーからユーザー管理のバックエンドサービスや Google サービスへ Envoy gRPC 呼び出しを行う仕組みで、トラフィックルーティングのカスタマイズ、リクエスト/レスポンスの変更、外部サービスとの統合が可能になります。

❌ Before (アップデート前)

  • プレビュー段階で本番環境利用に制限あり
  • カスタムロジック挿入にサイドカーや独自プロキシが必要
  • 高度なトラフィックカスタマイズが Kubernetes ネイティブに困難
  • SLA 非適用でミッションクリティカル採用が難しい

✅ After (アップデート後)

  • GA で SLA 適用、本番環境利用が正式サポート
  • 2 種類のリソースで Kubernetes ネイティブに設定可能
  • 3 つのバックエンド参照方式を要件に応じて選択
  • Autopilot / Standard 両クラスタで利用可能

⭐ 主要機能

🔌 GCPRoutingExtension (ルーティング拡張)

Cloud Load Balancing にカスタムロジックを挿入してトラフィックルーティングを制御します。リクエスト属性 (ヘッダー、URL など) に基づいて、元のルーティング先とは異なるバックエンドサービスへリクエストを転送可能。

✓各 ExtensionChain に 1 つの Extension のみ設定可能
✓Regional external / Regional internal ALB で利用可能

🔧 GCPTrafficExtension (トラフィック拡張)

サービス選択後のトラフィックにカスタムロジックを適用し、リクエスト/レスポンスのヘッダーやペイロードを変更します。サービス選択やセキュリティポリシーには影響を与えません。

✓各 ExtensionChain に 最大 3 つの Extension を設定可能
✓Regional external / Regional internal / Global external ALB で利用可能

🔗 3 つのバックエンド参照方式

⚙️ Kubernetes Service 参照

カスタムロジックを別の gRPC バックエンドアプリケーションとしてデプロイし、ロードバランサーからコールアウト。

  • GCPRoutingExtension と GCPTrafficExtension の両方で利用可能
  • HTTP/2 プロトコル必須
  • ext_proc プロトコル対応

⚡ GCPWasmPlugin 参照

WebAssembly (Wasm) モジュールをロードバランサーのデータパスに直接注入する高性能方式。

  • GCPTrafficExtension でのみ利用可能
  • Global external ALB でのみ利用可能
  • Artifact Registry からイメージを取得

💪 Google API サービス参照

googleAPIServiceName フィールドで Model Armor 等の Google サービスを直接呼び出し。

  • GCPTrafficExtension でのみ利用可能
  • AI セキュリティ統合に最適
  • プロンプトインジェクション防止など

Step 2: GCPTrafficExtension の定義

YAML gcp-traffic-extension.yaml 
kind: GCPTrafficExtension
apiVersion: networking.gke.io/v1
metadata:
  name: my-traffic-extension
  namespace: default
spec:
  targetRefs:
    - group: "gateway.networking.k8s.io"
      kind: Gateway
      name: GATEWAY_NAME
  extensionChains:
    - name: chain1
      matchCondition:
        celExpressions:
          - celMatcher: request.path.contains("serviceextensions")
      extensions:
        - name: ext1
          authority: "myext.com"
          timeout: 1s
          backendRef:
            group: ""
            kind: Service
            name: extension-service
            port: 443
          supportedEvents:
            - REQUEST_HEADERS
            - REQUEST_BODY
            - RESPONSE_HEADERS
            - RESPONSE_BODY

👍 メリットと ⚠️ 考慮事項

👍 メリット

🏢 ビジネス面

✓本番環境への正式対応: GA により SLA が適用され、ミッションクリティカルなワークロードで利用可能
✓開発コストの削減: カスタムプロキシやサイドカーの独自構築・運用が不要に
✓パートナー統合: API ゲートウェイセキュリティ、BOT 管理、WAF などのサードパーティ製品との統合が容易に

💻 技術面

✓Kubernetes ネイティブ: CRD として定義、kubectl で管理、GitOps 親和性が高い
✓柔軟なバックエンド参照: 3 つの方式から要件に応じて選択可能
✓CEL 式マッチング: 柔軟なリクエストマッチングで特定トラフィックのみにロジック適用
✓Envoy ext_proc 対応: 業界標準プロトコルで既存 gRPC 実装を活用可能

⚠️ 制限事項・考慮点

🚫 制限事項

✗GCPRoutingExtension は Global external ALB で非対応
✗GCPWasmPlugin は GCPTrafficExtension + Global external ALB のみ
✗コールアウトバックエンドで Cloud Armor / IAP / Cloud CDN 使用不可
✗HTTP/2 プロトコル必須
✗gRPC レスポンス最大 128KB (超過で RESOURCE_EXHAUSTED)
✗一部ヘッダー (X-user-IP, CDN-Loop, X-Forwarded-* 等) の操作制限
✗Extension と Gateway は同一名前空間にデプロイ必須

⚠️ パフォーマンス考慮

→コールアウト追加でレイテンシが増加 (データ種別ごとに加算)
→コールアウトサービスはバックエンドと同じゾーンにデプロイ推奨
→スケーラビリティと可用性はユーザー自身が管理
→supportedEvents を必要最小限に設定してレイテンシを最適化

🎯 ユースケース

🔌 ユースケース 1: カスタムトラフィックルーティング

シナリオ: マルチテナント SaaS アプリケーションで、リクエストヘッダー内のテナント ID に基づいて異なるバックエンドサービスにトラフィックをルーティング。

YAML GCPRoutingExtension - テナントルーティング例 
kind: GCPRoutingExtension
apiVersion: networking.gke.io/v1
metadata:
  name: tenant-routing
spec:
  targetRefs:
    - group: "gateway.networking.k8s.io"
      kind: Gateway
      name: my-gateway
  extensionChains:
    - name: tenant-chain
      matchCondition:
        celExpressions:
          - celMatcher: "has(request.headers['x-tenant-id'])"
      extensions:
        - name: tenant-router
          authority: "tenant-router.internal"
          timeout: 500ms
          backendRef:
            kind: Service
            name: tenant-routing-service
            port: 443

✨ 効果: テナントごとに異なるバックエンドバージョンやリージョンへの動的ルーティングが実現でき、テナント分離とカスタマイズが容易に。

🔒 ユースケース 2: カスタム認証・認可

シナリオ: 社内の認証基盤と連携し、ロードバランサーレベルでカスタム認証トークンの検証やヘッダーの付与を実施。

YAML GCPTrafficExtension - カスタム認証例 
kind: GCPTrafficExtension
apiVersion: networking.gke.io/v1
metadata:
  name: custom-auth
spec:
  targetRefs:
    - group: "gateway.networking.k8s.io"
      kind: Gateway
      name: my-gateway
  extensionChains:
    - name: auth-chain
      matchCondition:
        celExpressions:
          - celMatcher: "request.path.startsWith('/api/')"
      extensions:
        - name: auth-service
          authority: "auth.internal"
          timeout: 1s
          backendRef:
            kind: Service
            name: auth-callout-service
            port: 443
          supportedEvents:
            - REQUEST_HEADERS

✨ 効果: アプリケーションコードに認証ロジックを埋め込む必要がなくなり、認証の一元管理とセキュリティの強化を実現。

🤖 ユースケース 3: AI 推論トラフィックのセキュリティ (Model Armor 統合)

シナリオ: GKE Inference Gateway を使用した AI 推論ワークロードで、プロンプトとレスポンスに対してセキュリティポリシーを適用。

YAML GCPTrafficExtension - Model Armor 統合例 
kind: GCPTrafficExtension
apiVersion: networking.gke.io/v1
metadata:
  name: model-armor-extension
spec:
  targetRefs:
    - group: "gateway.networking.k8s.io"
      kind: Gateway
      name: inference-gateway
  extensionChains:
    - name: model-armor-chain
      matchCondition:
        celExpressions:
          - celMatcher: "request.path.startsWith('/')"
      extensions:
        - name: model-armor-service
          supportedEvents:
            - RequestHeaders
            - RequestBody
            - ResponseHeaders
            - ResponseBody
          timeout: 1s
          googleAPIServiceName: "modelarmor.us-central1.rep.googleapis.com"

✨ 効果: AI 推論トラフィックに対して、プロンプトインジェクション防止や不適切なコンテンツフィルタリングなどのセキュリティポリシーを一元的に適用可能。

📝 まとめ

Service Extensions の GKE Gateway 統合が GA となり、GKE ユーザーはロードバランシング処理パスにカスタムロジックを本番環境で安心して挿入できるようになりました。

✓2 つのリソースタイプ: GCPRoutingExtension (ルーティング制御) と GCPTrafficExtension (トラフィック変更)
✓3 つのバックエンド参照方式: Kubernetes Service / Wasm プラグイン / Google API サービス
✓幅広いユースケース: カスタムルーティング、認証、AI セキュリティなど
💡 推奨アクション: GKE Gateway を使用している場合は、既存のカスタムプロキシやサイドカーの処理を Service Extensions に移行できるか検討することをお勧めします。
