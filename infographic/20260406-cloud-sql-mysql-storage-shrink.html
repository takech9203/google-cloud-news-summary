🗄️ Cloud SQL for MySQL: Storage Shrink

Cloud SQL for MySQL Feature
2026-04-06

📋 概要

Cloud SQL for MySQL において、インスタンスのストレージ容量を手動で縮小 (shrink) できる機能が利用可能になりました。過剰にプロビジョニングされたストレージを削減し、不要なストレージコストを直接削減できます。

プライマリインスタンスおよびリードレプリカの両方で利用可能。すべての Cloud SQL エディション (Enterprise / Enterprise Plus) でサポートされています。

⚠️ 注意: ストレージ縮小操作にはインスタンスのダウンタイムが伴います。ダウンタイムを最小限に抑えたい場合は Database Migration Service を使用して新しいインスタンスへデータを移行する方法が推奨されます。

🔄 Before / After

❌ アップデート前

  • ストレージ容量は増加のみ可能で、一度拡張すると縮小できなかった
  • 過剰にプロビジョニングされたストレージに不要なコストが継続的に発生
  • ストレージ削減には新しいインスタンスを作成してデータ移行が必要

✅ アップデート後

  • gcloud CLI や REST API でストレージ容量を直接縮小可能
  • 不要なストレージコストを削減、リソース最適化が実現
  • プライマリ・リードレプリカの両方でサポート

🏗️ アーキテクチャ図

ストレージ縮小操作の全体フローです。ユーザーは最小許容サイズを確認した上でバックアップを作成し、縮小リクエストを実行します。操作中はインスタンスが停止し、ディスクサイズが変更された後に再起動されます。

    A -->|2. バックアップ作成| C[Cloud SQL バックアップ]
    A -->|3. ストレージ縮小リクエスト| D[Cloud SQL Admin API performDiskShrink]
    D -->|インスタンス停止| E[Cloud SQL MySQL インスタンス]
    E -->|ディスクサイズ変更| F[Hyperdisk Balanced 永続ディスク]
    F -->|完了後再起動| E
    E -->|4. 操作完了通知| A

⚡ 主要機能

📉 ストレージ容量の手動縮小

  • プロビジョニングされたストレージ容量を指定したターゲットサイズまで縮小可能
  • ターゲットサイズはインスタンスに対して安全と判断される最小許容容量より大きい必要がある
  • パフォーマンスの一貫性のため、現在の使用量の 20% または 100 GB のいずれか大きい方をバッファとして確保することを推奨

🔗 プライマリ & リードレプリカ サポート

  • プライマリインスタンスとリードレプリカの両方でストレージ縮小が可能
  • リードレプリカを縮小する場合は、先にプライマリインスタンスの縮小を完了する必要がある
  • リードレプリカはプライマリインスタンスより小さいストレージ容量を持つことはできない

🔙 操作のキャンセル機能

  • プライマリインスタンスまたはスタンドアロンインスタンスでは縮小操作のキャンセルが可能
  • gcloud sql operations cancel コマンドまたは REST API でキャンセルを実行

📝 設定手順

前提条件

  • gcloud CLI バージョン 563.0.0 以降がインストールされていること
  • 対象インスタンスが RUNNABLE 状態であること
  • 必要な IAM 権限が付与されていること
  • 外部接続を使用する拡張機能やフィーチャーが無効化されていること

ステップ 1: インスタンスの状態を確認

Bash インスタンス状態の確認 
gcloud sql instances describe INSTANCE_NAME

ステップ 2: バックアップを作成

Bash オンデマンドバックアップの作成 
gcloud sql backups create --instance=INSTANCE_NAME

ステップ 3: 最小許容ストレージ容量を確認

Bash 縮小可能な最小サイズの確認 
gcloud sql instances describe INSTANCE_NAME \
  --format="value(diskShrinkContext)"

ステップ 4: ストレージ縮小を実行

Bash ストレージ縮小の実行 (gcloud CLI) 
gcloud sql instances perform-storage-shrink INSTANCE_NAME \
  --storage-size=TARGET_STORAGE_SIZE \
  --async

ステップ 5 (REST API の場合)

Bash REST API での実行 
curl -X POST \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/json; charset=utf-8" \
  -d '{"targetSizeGb": TARGET_STORAGE_SIZE}' \
  "https://sqladmin.googleapis.com/v1/projects/PROJECT_ID/instances/INSTANCE_ID/performDiskShrink"

✅ メリット

💼 ビジネス面

  • ストレージコストの最適化: 過剰プロビジョニングを削減し、月額ストレージ費用を直接削減
  • リソース管理の柔軟性向上: ビジネス要件の変化に応じてストレージ容量を増減可能に

🛠 技術面

  • シンプルな操作: gcloud CLI / REST API を使用した直感的なコマンド
  • 安全性の確保: 最小許容容量のチェック機能によりデータ損失リスクを最小化
  • キャンセル対応: プライマリインスタンスでは実行中の縮小操作をキャンセル可能

⚠️ デメリット・制約事項

制限事項
  • 共有コアインスタンスでは非サポート
  • MySQL レガシー高可用性構成では利用不可
  • 外部サーバーからのレプリケーション構成では非サポート
  • カスケードレプリカ・リードプールへの適用不可
  • Terraform でのサポートなし
考慮すべき点
  • 縮小操作中はインスタンスにダウンタイムが発生 (ディスクサイズに依存して相当な時間になる可能性)
  • 操作中は他のすべての操作 (バックアップ、インポートなど) が利用不可
  • 操作が 10 日を超えるとタイムアウトする場合がある (CPU アップグレードで短縮可能)
  • リードレプリカの最小ストレージ容量を直接確認する手段がない
  • ダウンタイムを許容できない場合は Database Migration Service の利用を検討

💡 ユースケース

💻 開発/テスト環境のコスト最適化

シナリオ: 本番環境と同じ構成で作成された開発環境のインスタンスが 500 GB のストレージを持つが、実際のデータ使用量は 50 GB 程度。
Bash 開発環境のストレージ縮小例 
# 最小許容サイズを確認
gcloud sql instances describe dev-mysql-instance

# ストレージを 100 GB に縮小 (50 GB の使用量 + バッファ)
gcloud sql instances perform-storage-shrink dev-mysql-instance \
  --storage-size=100 \
  --async
📈 効果: 月額ストレージコストを最大 80% 削減しつつ、十分なバッファを確保。

🗃 データクリーンアップ後のストレージ最適化

シナリオ: 大量の履歴データを削除またはアーカイブした後、実際のデータ使用量に対してプロビジョニングされたストレージが大幅に過剰になっている。
📈 効果: データクリーンアップの効果をストレージコストの削減として直接反映でき、継続的なコスト最適化が実現。

💰 料金

Cloud SQL のストレージ料金はプロビジョニングされた容量 (GiB/月) に基づいて課金されます。ストレージ縮小操作自体に追加料金は発生しませんが、縮小後は新しい (小さい) ストレージ容量に基づいた料金が適用されます。

詳細な料金はリージョンやエディションにより異なります。

🌎 利用可能リージョン

すべての Cloud SQL エディション (Enterprise / Enterprise Plus) で利用可能です。Cloud SQL が利用可能なすべてのリージョンでサポートされています。
