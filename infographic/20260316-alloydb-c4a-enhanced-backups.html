📋 概要

AlloyDB for PostgreSQL に 2 つの重要なアップデートが同時にリリースされました。C4A 2 vCPU マシンタイプの追加と、Enhanced Backups の GAです。

Google Axion プロセッサベースの C4A マシンシリーズに新たに c4a-highmem-2-lssd (2 vCPU / 16 GB RAM) が追加され、小規模な本番ワークロードでも Axion の高い価格性能比を活用できるようになりました。また、Backup and DR Service と統合された Enhanced Backups が GA となり、エンタープライズグレードのバックアップ管理が正式に利用可能になりました。

🔄 Before / After

❌ Before (アップデート前)

  • C4A は 1 vCPU (開発用) の次が 4 vCPU で、小規模本番向けの選択肢がなかった
  • 1 vCPU はアップタイム SLA なし、本番環境に不向き
  • Enhanced Backups は Preview 段階で本番利用が制限
  • イミュータブルストレージによる保護が GA で利用不可

✅ After (アップデート後)

  • c4a-highmem-2-lssd (2 vCPU / 16 GB) が利用可能に
  • Enhanced Backups が GA、クラスタ作成時にティア選択可能
  • プロジェクトレベルでティア別タブ管理
  • Enhanced バックアップの個別削除が可能に

⚡ 主要機能

🖥️ C4A 2 vCPU マシンタイプ

c4a-highmem-2-lssd

  • Google Axion (Arm) プロセッサ搭載
  • 2 vCPU / 16 GB RAM
  • ローカル Titanium SSD 付き
  • Ultra-fast キャッシュをサポート
  • 本番ワークロード向け最小 Axion 構成
  • N2 からは machine_type 変更のみで切替可能

🔒 Enhanced Backups GA

Backup and DR Service 統合

  • クラスタ作成時に Standard / Enhanced を選択
  • イミュータブル + 削除不可な Backup Vault
  • 強制保持期間による不正削除防止
  • エンタープライズグレードのバックアップ管理

📊 バックアップ管理の改善

  • プロジェクトレベルでティア別タブ管理
  • Enhanced バックアップの個別削除
  • バックアッププランの関連付け・変更・解除
  • コンソール / gcloud CLI から操作可能

🔧 設定方法

💡 前提条件: AlloyDB クラスタが作成済み / Enhanced Backups には Backup and DR API の有効化と alloydb.backupDrAdmin ロールが必要

ステップ 1: C4A 2 vCPU マシンタイプへの変更

Bash 
# 既存インスタンスのマシンタイプを C4A 2 vCPU に変更
gcloud alloydb instances update INSTANCE_ID \
  --cluster=CLUSTER_ID \
  --region=REGION \
  --machine-type=c4a-highmem-2-lssd
⚠️ 注意: N2 から C4A への切り替えは machine_type の変更で行い、インスタンスの削除・再作成は推奨されません。

ステップ 2: Enhanced Backups の有効化

Bash 
# Backup and DR API の有効化
gcloud services enable backupdr.googleapis.com

# バックアッププランとクラスタの関連付け
gcloud backup-dr backup-plan-associations create BPA_ID \
  --project=PROJECT_ID \
  --location=REGION \
  --resource-type=alloydb.googleapis.com/Cluster \
  --resource=projects/PROJECT_ID/locations/REGION/clusters/CLUSTER_ID \
  --backup-plan=projects/VAULT_PROJECT_ID/locations/REGION/backupPlans/BP_ID

Google Cloud コンソールからは、クラスタ作成時に「Show advanced options」 > 「Data Protection」 > 「Configure Backup Tier」から Enhanced backup tier を選択できます。

必要な権限 (Enhanced Backups)

alloydb.backupDrAdmin ロール、または以下の個別権限:

  • backupdr.backupPlans.list
  • backupdr.backupPlanAssociations.createForAlloydbCluster
  • backupdr.backupPlanAssociations.fetchForAlloydbCluster
  • backupdr.backupPlanAssociations.list
  • backupdr.backupPlanAssociations.getForAlloydbCluster
  • backupdr.backupPlanAssociations.triggerBackupForAlloydbCluster
  • backupdr.backupPlanAssociations.deleteForAlloydbCluster
  • backupdr.backupPlans.useForAlloydbCluster
  • backupdr.bvdataSources.get
  • backupdr.bvdataSources.list

✅ メリット

💼 ビジネス面

→コスト最適化

C4A 2 vCPU により、小規模ワークロードでも Axion ベースの高い価格性能比を活用。本番環境コストを抑制。

→コンプライアンス対応

Enhanced Backups の GA で、金融・医療など規制産業のイミュータブルバックアップ要件を GA 品質で充足。

→スケーラビリティ

2 vCPU から始めて C4A シリーズ内 (最大 72 vCPU) でスケールアップ可能。

🔩 技術面

→Axion プロセッサ

Google 独自の Arm ベースプロセッサ。キャッシュ適合ワークロードで最高のトランザクションスループット。

→Ultra-fast キャッシュ

lssd (ローカル Titanium SSD) 付きで、2 vCPU でも Ultra-fast キャッシュが利用可能。

→集中バックアップ管理

Backup and DR Service で AlloyDB、Cloud SQL、Compute Engine のバックアップを統一管理。

⚠️ デメリット・制約事項

制限事項
  • C4A は限定リージョンのみ利用可能
  • asia-southeast1 は 1 vCPU のみ
  • Enhanced Backups はクロスリージョンリストア非対応
  • Enhanced 有効化で既存バックアップ設定が上書きされる
考慮すべき点
  • N2 → C4A は machine_type 変更のみ (削除・再作成不可)
  • バックアッププランはクラスタと同じリージョンに配置
  • 異なるプロジェクトの Vault 使用時は追加ロール付与が必要

💡 ユースケース

🚀 ユースケース 1: スタートアップの小規模本番データベース

シナリオ: スタートアップ企業が AlloyDB で本番 DB を運用したいが、初期段階では小規模ワークロードでコストを抑えたい。

Bash 
# C4A 2 vCPU で AlloyDB クラスタを作成
gcloud alloydb clusters create my-startup-cluster \
  --region=us-central1 \
  --password=MY_PASSWORD

gcloud alloydb instances create my-primary \
  --cluster=my-startup-cluster \
  --region=us-central1 \
  --instance-type=PRIMARY \
  --machine-type=c4a-highmem-2-lssd

効果: 4 vCPU の半分のコストで Axion ベースの高性能 DB を本番運用開始。成長に合わせてスケールアップ可能。

🏦 ユースケース 2: 規制産業向けのバックアップコンプライアンス

シナリオ: 金融機関が AlloyDB に顧客データを格納。バックアップの不正削除防止と長期保持が規制で必要。

効果: Enhanced Backups の Backup Vault による強制保持期間とイミュータブルストレージで規制要件を充足。Backup and DR Service で他の Google Cloud ワークロードと統一的に監視・報告が可能。
