📋 概要

Cloud Spanner で 2 つの重要な機能が一般提供 (GA) となった。Repeatable Read Isolation は読み取りが多く書き込みが少ないワークロードのレイテンシを低減し、Columnar Engine は運用中のデータに対する分析クエリのスキャン性能を最大 200 倍高速化する。さらに Columnar Engine の PostgreSQL インターフェース対応 も追加された。

これらの GA により、Spanner は OLTP と OLAP の両方に対応するデータベースとしてさらに強化され、他データベースからの移行容易化と ETL パイプラインの簡素化を推進する。

🔄 Before / After 比較

❌ アップデート前の課題

  • デフォルトの Serializable 分離レベルでは読み書き競合が多いワークロードでトランザクションのアボートが頻発
  • MySQL/PostgreSQL からの移行時にアプリケーション設計の変更が必要
  • 分析クエリには ETL パイプラインで BigQuery 等にデータ転送が必要で、リアルタイム性が損失
  • Columnar Engine は GoogleSQL インターフェースのみ対応

✅ アップデート後の改善

  • Repeatable Read で読み取りがロックを取得せず、トランザクション失敗率が低下しレイテンシ改善
  • MySQL/PostgreSQL と同等のデフォルト分離レベルに設定可能で、移行時のアプリ変更を最小化
  • Columnar Engine により ETL なしで最大 200 倍高速なリアルタイム分析を実現
  • PostgreSQL インターフェースでも Columnar Engine が利用可能に

⭐ 主要機能

🔒 Repeatable Read Isolation (GA)

  • スナップショット分離に基づく実装で、トランザクション開始時点の一貫したスナップショットを読み取り
  • デフォルトの楽観的同時実行制御では、読み取りがロックを取得せず並行書き込みをブロックしない
  • 悲観的同時実行制御では SELECT ... FOR UPDATE や lock_scanned_ranges=exclusive ヒントによる排他ロックをサポート
  • クライアントレベルまたはトランザクションレベルで分離レベルを設定可能
  • Go, Java, Python, Node.js, C++, C# のクライアントライブラリおよび REST API で対応

📊 Columnar Engine (GA)

  • 列指向ストレージ形式により、分析クエリのスキャン性能を最大 200 倍高速化
  • バックグラウンドのコンパクション処理で列指向フォーマットを自動生成し、クエリ時に最新の更新とマージして強整合性を維持
  • データベース、テーブル、インデックス単位で有効化/無効化が可能
  • 運用ワークロード (トランザクション処理) への影響なし

🐘 PostgreSQL インターフェースでの Columnar Engine サポート (NEW)

  • ALTER DATABASE db_name SET spanner.columnar_policy TO enabled で有効化
  • テーブルやインデックス単位での COLUMNAR POLICY 指定に対応
  • GoogleSQL インターフェースと同等の機能を提供

💻 設定例・コードサンプル

Repeatable Read の設定 (Python)

Python クライアント/トランザクションレベルの分離レベル設定 
from google.cloud.spanner_v1 import TransactionOptions

# クライアントレベルで Serializable を設定
spanner_client = spanner.Client(
    default_transaction_options=DefaultTransactionOptions(
        isolation_level=TransactionOptions.IsolationLevel.SERIALIZABLE
    )
)

# トランザクションレベルで Repeatable Read にオーバーライド
database.run_in_transaction(
    update_function,
    isolation_level=TransactionOptions.IsolationLevel.REPEATABLE_READ
)

Columnar Engine の設定 (PostgreSQL)

SQL PostgreSQL インターフェースでの Columnar Engine 有効化 
-- データベース全体で Columnar Engine を有効化
ALTER DATABASE "Music" SET spanner.columnar_policy TO enabled;

-- 特定テーブルで無効化
CREATE TABLE Concerts(
    VenueId bigint NOT NULL,
    SingerId bigint NOT NULL,
    ConcertDate date NOT NULL,
    PRIMARY KEY(VenueId, SingerId, ConcertDate)
) COLUMNAR POLICY disabled;

-- 既存テーブルで Columnar Engine を無効化
ALTER TABLE Singers SET COLUMNAR POLICY disabled;

Columnar Engine の設定 (GoogleSQL)

SQL GoogleSQL インターフェースでの Columnar Engine 有効化 
-- データベース全体で Columnar Engine を有効化
ALTER DATABASE Music SET OPTIONS (columnar_policy = 'enabled');

-- 特定テーブルで無効化
CREATE TABLE Concerts(
    VenueId INT64 NOT NULL,
    SingerId INT64 NOT NULL,
    ConcertDate DATE NOT NULL,
) PRIMARY KEY(VenueId, SingerId, ConcertDate),
OPTIONS (columnar_policy = 'disabled');

✅ メリット

💼 ビジネス面

✔ 移行コスト削減 - Repeatable Read が MySQL/PostgreSQL のデフォルト分離レベルと同等の動作を提供し、アプリケーション再設計が不要に
✔ ETL パイプライン簡素化 - Columnar Engine で運用 DB 上で直接分析が可能となり、ETL 構築・運用コストを削減
✔ リアルタイム BI - 最新の運用データに対してインタラクティブなレイテンシで分析クエリを実行し、意思決定スピードが向上

⚙️ 技術面

✔ レイテンシ改善 - Repeatable Read のロックフリー読み取りにより高い読み書き競合シナリオでのレイテンシが低下
✔ トランザクション失敗率低減 - スナップショット分離で Serializable 比のアボート率が低下
✔ 最大 200 倍のスキャン高速化 - 列指向ストレージにより分析クエリが劇的に高速化
✔ 強整合性の維持 - 列指向フォーマット生成後もクエリ時に最新データとマージ

⚠️ デメリット・制約事項

⚠️ 制限事項
  • Repeatable Read では Write Skew 異常が発生する可能性がある (SELECT FOR UPDATE で回避が必要)
  • Columnar Engine は Enterprise / Enterprise Plus エディションのみ利用可能 (Standard エディションでは使用不可)
  • Columnar Engine を有効化するとストレージ使用量が約 60% 増加するため、十分なストレージ容量が必要
  • カラムナーフォーマットはバックアップに含まれないため、リストア後にコンパクション完了まで恩恵なし
💡 考慮すべき点
  • Repeatable Read と Serializable の選択は、ワークロード特性 (読み書き競合の程度、データ異常の許容度) に応じて慎重に判断
  • Columnar Engine のコンパクション処理は数日かかる場合があり、有効化直後にフル性能は発揮されない
  • 高頻度の更新やランダム挿入がある場合、Columnar Engine のパフォーマンスに影響する可能性 (追記型ワークロードが最適)
  • SELECT * ではなく、必要なカラムのみを指定するクエリ設計が効果を最大化

💡 ユースケース

🔀 ユースケース 1: 他データベースからの移行

シナリオ: MySQL や PostgreSQL で動作しているアプリケーションを Spanner に移行する場合。既存アプリケーションはデフォルトの分離レベル (MySQL: REPEATABLE READ, PostgreSQL: READ COMMITTED) を前提として設計されている。

Python Repeatable Read をデフォルトに設定 
# クライアント作成時に Repeatable Read をデフォルトに設定
from google.cloud.spanner_v1 import TransactionOptions, DefaultTransactionOptions

spanner_client = spanner.Client(
    default_transaction_options=DefaultTransactionOptions(
        isolation_level=TransactionOptions.IsolationLevel.REPEATABLE_READ
    )
)

効果: アプリケーションコードの変更を最小限に抑えつつ、Spanner のスケーラビリティと高可用性の恩恵を享受。

📈 ユースケース 2: リアルタイム運用レポーティング

シナリオ: EC サイトの運用データベースとして Spanner を使用し、売上ダッシュボードやリアルタイム在庫レポートを運用データに対して直接実行したい。

SQL PostgreSQL インターフェースでの分析クエリ 
-- PostgreSQL インターフェースで Columnar Engine を有効化
ALTER DATABASE "ecommerce" SET spanner.columnar_policy TO enabled;

-- 大量データスキャンを伴う分析クエリ
SELECT category, SUM(amount) AS total_sales, COUNT(*) AS order_count
FROM orders
WHERE order_date >= CURRENT_DATE - INTERVAL '30 days'
GROUP BY category
ORDER BY total_sales DESC;

効果: ETL パイプラインなしで最新の運用データに対して高速な分析が可能。ビジネスの意思決定スピードが向上。

📦 ユースケース 3: 読み取りが多いマイクロサービス

シナリオ: 商品カタログや在庫情報を頻繁に読み取るマイクロサービスが多数あり、少数のサービスがデータを更新する。Serializable ではリードヘビーなサービスのトランザクション失敗が頻発していた。

効果: Repeatable Read に切り替えることで、読み取りトランザクションのアボート率が低下し、全体的なシステムスループットが向上。

💰 料金情報

✅ Repeatable Read Isolation

全エディション (Standard, Enterprise, Enterprise Plus) で追加料金なしで利用可能。

💠 Columnar Engine

Enterprise / Enterprise Plus エディションで利用可能。追加のストレージ使用量 (約 60% 増) に対してストレージ料金が発生。

詳細な料金は Spanner 料金ページ を参照。

🌍 利用可能リージョン

Repeatable Read Isolation および Columnar Engine は、Spanner が利用可能なすべてのリージョン、デュアルリージョン、マルチリージョン構成で使用可能。ただし Columnar Engine は Enterprise / Enterprise Plus エディションのインスタンスでのみ利用可能。
