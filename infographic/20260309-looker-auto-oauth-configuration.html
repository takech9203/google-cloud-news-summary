📋 概要

Looker (Google Cloud core) において、パブリックセキュア接続のみを使用し、カスタムドメインを設定しないインスタンスでは、 OAuth 資格情報の手動構成が不要になりました。Looker が OAuth クライアントとシークレット情報を自動的に管理し、セットアッププロセスが大幅に簡素化されます。

なお、プライベート接続、ハイブリッド接続、またはカスタムドメインを使用するインスタンスでは、引き続き手動での OAuth 構成が必要です。

🔄 Before / After 比較

❌ Before (従来)

  • 1. OAuth コンセント画面を構成 (ユーザータイプ、スコープ、テストユーザー)
  • 2. Google Cloud コンソールで OAuth クライアント ID を作成
  • 3. クライアント ID とシークレットを取得

✅ After (新機能)

  • 1. Google Cloud コンソールからインスタンスを作成
  • 2. インスタンス利用可能 (OAuth は自動構成)
💡 ポイント: OAuth クライアント ID、シークレット、リダイレクト URI の設定がすべて自動化

⚙️ 主要機能

🔐 OAuth 自動管理

パブリックセキュア接続かつカスタムドメインなしのインスタンスに対して、Looker (Google Cloud core) が OAuth クライアントとシークレット情報を自動的に生成・管理します。管理者は OAuth 資格情報を意識することなくインスタンスを作成できます。

✅ 適用条件

→パブリックセキュア接続: 外部からインターネット経由でアクセス可能な IP アドレスを持つ接続
→カスタムドメインなし: デフォルトの Looker ドメインを使用
→上記の両方の条件を満たす場合にのみ自動構成が適用

⚠️ 手動構成が引き続き必要なケース

→プライベート接続 (Private Service Connect / private services access) を使用するインスタンス
→ハイブリッド接続 (パブリック URL + PSC エンドポイント) を使用するインスタンス
→カスタムドメインを設定するインスタンス

🎯 メリット

💼 ビジネス面

🚀導入時間の短縮: セットアップ時間が短縮され、Looker 利用開始までのリードタイムが削減
📉運用負荷の軽減: OAuth 資格情報のライフサイクル管理が自動化

🔧 技術面

✅セットアップエラーの削減: リダイレクト URI の設定漏れなどのミスがなくなる
🔒セキュリティ向上: クライアントシークレットの自動管理で漏洩リスク低減
⚡IaC の簡素化: Terraform / gcloud CLI で OAuth パラメータ指定不要

⚠️ 制限事項・考慮点

制限事項
  • ❌ パブリックセキュア接続かつカスタムドメインなしの場合にのみ適用
  • ❌ プライベート接続やハイブリッド接続の Enterprise/Embed エディションでは利用不可
  • ❌ カスタムドメインを後から追加する場合、手動 OAuth 構成への移行が必要になる可能性
考慮すべき点
  • 💡 自動管理された OAuth 資格情報のカスタマイズ可否は公式ドキュメントで確認が必要
  • 💡 既存インスタンス (手動 OAuth 構成済み) への影響は公式ドキュメントを参照

💡 ユースケース

🏢 ユースケース 1: 小規模チームでの Looker 導入

シナリオ: 50 名以下のチームが Standard エディションの Looker (Google Cloud core) を初めて導入。パブリックセキュア接続で十分であり、カスタムドメインも不要なケース。

効果: OAuth の事前設定が不要になり、Google Cloud コンソールからインスタンスを作成するだけで、すぐにチームメンバーがアクセス可能に。初期セットアップの手順が大幅に簡素化されます。

🧪 ユースケース 2: 開発・テスト環境の迅速な構築

シナリオ: 既存の本番 Looker インスタンスに加えて、開発やテスト用の非本番インスタンスを素早くプロビジョニングしたい場合。

効果: OAuth 構成のオーバーヘッドなしに非本番インスタンスを作成でき、CI/CD パイプラインや自動テスト環境の構築が容易になります。
