🛡️ Cloud NGFW: Network Contexts による効率的なファイアウォールポリシー管理

Cloud NGFW VPC GA
2026-02-25
Network Contexts

📋 概要

Cloud NGFW (Cloud Next Generation Firewall) に Network Contexts (ネットワークコンテキスト) 機能が GA (一般提供) として追加された。トラフィックの送信元や宛先のネットワークコンテキスト (INTERNET、NON_INTERNET、VPC_NETWORKS、INTRA_VPC) に基づいてファイアウォールルールを定義でき、少ないルール数でより効率的にセキュリティ目標を達成できる。

階層型ファイアウォールポリシー、グローバルネットワークファイアウォールポリシー、リージョナルネットワークファイアウォールポリシーのルールにおいて、ソースコンビネーションまたはデスティネーションコンビネーションの一部として使用可能。2024 年 12 月に Preview (当時の名称は「Network Scopes」) として公開され、GA に昇格した。

❌ Before (従来)

→インターネットと VPC 間トラフィックの区別に、送信元 IP レンジを明示的に指定した複数ルールが必要
→VPC ピアリングや NCC 経由トラフィックの論理的分類手段がなく、IP レンジ管理が煩雑
→ルール数の増大により、管理コストと誤設定リスクが増加

✅ After (改善後)

→INTERNET / NON_INTERNET で 1 つのパラメータでトラフィック区別が可能
→VPC_NETWORKS で最大 250 VPC を対象とした送信元識別が可能
→INTRA_VPC で同一 VPC 内の East-West トラフィックを簡潔に識別

⚙️ 主要機能 - 4 種類のネットワークコンテキスト

🌐 INTERNET コンテキスト

インターネット経由で到着/送出されるトラフィックを識別。

→Ingress: Google Maglev 経由でルーティングされた VM ネットワークインターフェースへのパケット
→Egress: デフォルトインターネットゲートウェイを next hop とする静的ルート経由のパケット (Google API/サービス向けは除外)
→Ingress ソース / Egress デスティネーション両方で使用可能

🔒 NON_INTERNET コンテキスト

インターネットを経由しないすべてのトラフィックを識別 (INTERNET と相互排他)。

→サブネットルート、静的ルート、ポリシーベースルート、特殊ルーティングパス経由のトラフィック
→Google API/サービスへのレスポンスパケットも含まれる
→Ingress / Egress 両方向で使用可能

🔗 VPC_NETWORKS コンテキスト

特定の送信元 VPC ネットワークからのトラフィックを識別 (NON_INTERNET のサブセット)。

→Ingress ルールのソースコンビネーションでのみ使用可能
→送信元 VPC リストに最大 250 VPC を指定可能
→VPC ピアリング / NCC ハブ経由で接続された VPC も対象
→VM、Cloud VPN、Cloud Interconnect、ルーターアプライアンス、PSC エンドポイント等が対象リソース

🏠 INTRA_VPC コンテキスト

同一 VPC ネットワーク内のリソースからのトラフィックを識別 (NON_INTERNET のサブセット)。

→Ingress ルールのソースコンビネーションでのみ使用可能
→East-West トラフィック (VPC 内部通信) のマイクロセグメンテーションに最適

📐 技術仕様

ネットワークコンテキストの対応表

ネットワークコンテキスト ターゲットタイプ Ingress ソース Egress デスティネーション
INTERNET INSTANCES, INTERNAL_MANAGED_LB ✅ 対応 ✅ 対応
NON_INTERNET INSTANCES, INTERNAL_MANAGED_LB ✅ 対応 ✅ 対応
VPC_NETWORKS INSTANCES, INTERNAL_MANAGED_LB ✅ 対応 ❌ 非対応
INTRA_VPC INSTANCES, INTERNAL_MANAGED_LB ✅ 対応 ❌ 非対応

コンテキスト間の関係

関係 詳細
INTERNET と NON_INTERNET 相互排他 (同時指定不可)
VPC_NETWORKS NON_INTERNET のサブセット
INTRA_VPC NON_INTERNET のサブセット

制約事項

項目 制限
VPC_NETWORKS の最大 VPC 数 250 VPC
INTERNET コンテキスト ソースセキュアタグとの併用不可
NON_INTERNET / VPC_NETWORKS / INTRA_VPC Google Threat Intelligence リストおよびジオロケーションソースとの併用不可
VPC_NETWORKS / INTRA_VPC Egress ルールでは使用不可

💻 設定方法・gcloud CLI コマンド例

前提条件

💡 必要な準備:
→Compute Engine API が有効であること
→IAM ロール: compute.securityAdmin または compute.networkAdmin
→階層型/グローバル/リージョナルネットワークファイアウォールポリシーが作成済みであること

ステップ 1: ファイアウォールポリシーの確認または作成

Bash ポリシー作成と VPC への関連付け 
# グローバルネットワークファイアウォールポリシーの作成
gcloud compute network-firewall-policies create my-policy \
    --global \
    --description="Policy with network contexts"

# VPC ネットワークへの関連付け
gcloud compute network-firewall-policies associations create \
    --firewall-policy=my-policy \
    --network=my-vpc \
    --global-firewall-policy \
    --name=my-association

ステップ 2: ネットワークコンテキストを使用したルールの作成

Bash INTERNET コンテキスト - Ingress HTTPS 許可 
# Ingress ルール: インターネットからの HTTPS トラフィックを許可
gcloud compute network-firewall-policies rules create 100 \
    --firewall-policy=my-policy \
    --action=allow \
    --direction=INGRESS \
    --src-network-context=INTERNET \
    --src-ip-ranges=0.0.0.0/0 \
    --layer4-configs=tcp:443 \
    --global-firewall-policy \
    --description="Allow HTTPS from internet"
Bash VPC_NETWORKS コンテキスト - 特定 VPC からのトラフィック許可 
# Ingress ルール: 特定の VPC ネットワークからのトラフィックを許可
gcloud compute network-firewall-policies rules create 200 \
    --firewall-policy=my-policy \
    --action=allow \
    --direction=INGRESS \
    --src-network-context=VPC_NETWORKS \
    --src-networks=projects/my-project/global/networks/vpc-a,projects/my-project/global/networks/vpc-b \
    --src-ip-ranges=10.0.0.0/8 \
    --layer4-configs=tcp:8080 \
    --global-firewall-policy \
    --description="Allow traffic from specific VPCs"
Bash INTERNET コンテキスト - Egress SMTP ブロック 
# Egress ルール: インターネット向けトラフィックを制限
gcloud compute network-firewall-policies rules create 300 \
    --firewall-policy=my-policy \
    --action=deny \
    --direction=EGRESS \
    --dest-network-context=INTERNET \
    --dest-ip-ranges=0.0.0.0/0 \
    --layer4-configs=tcp:25 \
    --global-firewall-policy \
    --description="Block outbound SMTP to internet"
Bash INTRA_VPC コンテキスト - 同一 VPC 内 MySQL 許可 
# Ingress ルール: 同一 VPC 内トラフィックのみ許可
gcloud compute network-firewall-policies rules create 400 \
    --firewall-policy=my-policy \
    --action=allow \
    --direction=INGRESS \
    --src-network-context=INTRA_VPC \
    --src-ip-ranges=10.0.0.0/8 \
    --layer4-configs=tcp:3306 \
    --global-firewall-policy \
    --description="Allow MySQL only from same VPC"
Bash 階層型ファイアウォールポリシーでの使用 
# 階層型ファイアウォールポリシーでの使用
gcloud compute firewall-policies rules create 100 \
    --firewall-policy=123456789 \
    --action=allow \
    --direction=INGRESS \
    --src-network-context=NON_INTERNET \
    --src-ip-ranges=10.0.0.0/8 \
    --layer4-configs=tcp:443 \
    --description="Allow internal HTTPS"

ステップ 3: ルールの確認

Bash ポリシー内ルール一覧表示 
# ポリシー内のルールを一覧表示
gcloud compute network-firewall-policies rules list \
    --firewall-policy=my-policy \
    --global-firewall-policy

✅ メリット

💼 ビジネス面

→運用コスト削減: ルール数を大幅に削減でき、ポリシー管理の工数が減少
→セキュリティガバナンスの強化: インターネット/内部トラフィック分類が明確になり、セキュリティ監査対応が容易に
→コンプライアンス対応: インターネット境界の明確な定義で、規制要件への適合を証明しやすくなる

🔧 技術面

→ルール数の効率化: 従来の IP レンジベース複数ルールをネットワークコンテキスト 1 ルールに集約可能
→マイクロセグメンテーションの簡素化: INTRA_VPC で East-West トラフィック制御が容易に
→マルチ VPC 管理の改善: VPC_NETWORKS で最大 250 VPC を一括制御
→3 種類のポリシーで利用可能: 階層型、グローバル、リージョナルの全ファイアウォールポリシーで使用可能

⚠️ デメリット・制約事項

制限事項

❌VPC_NETWORKS と INTRA_VPC は Ingress ルールのソースコンビネーションでのみ使用可能。Egress ルールでは使用不可
❌INTERNET コンテキストはソースセキュアタグとの併用不可
❌NON_INTERNET / VPC_NETWORKS / INTRA_VPC は Google Threat Intelligence リストやジオロケーションソースと併用不可
❌VPC_NETWORKS で指定可能な VPC 数は最大 250 に制限
❌VPC_NETWORKS で指定した VPC が削除された場合、参照は残るがルールは無効になる

考慮すべき点

⚠️既存の IP レンジベースのルールからの移行計画が必要
⚠️ネットワークコンテキストは他のソース/デスティネーションパラメータと組み合わせて使用する必要があり、単独では使用不可
⚠️VPC ピアリングや NCC ハブ経由の接続関係を正確に把握した上でルール設計が必要

🎯 ユースケース

🌐 ユースケース 1: インターネット境界のセキュリティ強化

シナリオ: Web アプリケーションを公開しており、インターネットからの Ingress トラフィックと内部サービス間のトラフィックに異なるセキュリティポリシーを適用したい場合。

Bash インターネット境界セキュリティ設定 
# インターネットからの HTTPS のみ許可
gcloud compute network-firewall-policies rules create 100 \
    --firewall-policy=web-policy \
    --action=allow --direction=INGRESS \
    --src-network-context=INTERNET \
    --src-ip-ranges=0.0.0.0/0 \
    --layer4-configs=tcp:443 --global-firewall-policy

# 内部トラフィックは全ポート許可
gcloud compute network-firewall-policies rules create 200 \
    --firewall-policy=web-policy \
    --action=allow --direction=INGRESS \
    --src-network-context=NON_INTERNET \
    --src-ip-ranges=10.0.0.0/8 \
    --layer4-configs=all --global-firewall-policy

効果: 2 つのルールでインターネット境界のセキュリティポリシーを完結でき、IP レンジの詳細管理が不要になる。

🔗 ユースケース 2: マルチ VPC 環境でのアクセス制御

シナリオ: 複数の VPC ネットワークを運用しており、特定の VPC からのアクセスのみを許可する共有サービス VPC を構築したい場合。

Bash マルチ VPC アクセス制御 
# 本番 VPC と開発 VPC からのアクセスのみ許可
gcloud compute network-firewall-policies rules create 100 \
    --firewall-policy=shared-services-policy \
    --action=allow --direction=INGRESS \
    --src-network-context=VPC_NETWORKS \
    --src-networks=projects/prod/global/networks/prod-vpc,projects/dev/global/networks/dev-vpc \
    --src-ip-ranges=10.0.0.0/8 \
    --layer4-configs=tcp:8080,tcp:8443 \
    --global-firewall-policy

効果: VPC_NETWORKS コンテキストにより、VPC ピアリングや NCC ハブ経由で接続された特定の VPC からのトラフィックのみを許可でき、ゼロトラストアーキテクチャの実現に貢献する。

🗄️ ユースケース 3: データベースのマイクロセグメンテーション

シナリオ: データベース VM への接続を同一 VPC 内のアプリケーション VM からのみに制限したい場合。

Bash データベースマイクロセグメンテーション 
# 同一 VPC 内からの MySQL 接続のみ許可
gcloud compute network-firewall-policies rules create 100 \
    --firewall-policy=db-policy \
    --action=allow --direction=INGRESS \
    --src-network-context=INTRA_VPC \
    --src-ip-ranges=10.0.0.0/8 \
    --layer4-configs=tcp:3306 \
    --target-secure-tags=db-servers \
    --global-firewall-policy

効果: INTRA_VPC コンテキストにより、VPC ピアリング経由や他の VPC からのデータベースアクセスを確実にブロックし、データベース層のセキュリティを強化できる。

💰 料金

Network Contexts は Cloud NGFW のファイアウォールポリシールールの機能であり、Cloud NGFW のティア (Essentials / Standard / Enterprise) に応じた料金体系の中で提供される。Network Contexts 自体の追加料金はない。

詳細は Cloud NGFW 料金ページ を参照。

🌍 利用可能リージョン

Network Contexts はグローバルネットワークファイアウォールポリシーおよびリージョナルネットワークファイアウォールポリシーの両方で利用可能。

→グローバルポリシー: 全リージョンに適用
→リージョナルポリシー: 指定されたリージョンに適用
→階層型ファイアウォールポリシー: 組織またはフォルダレベルで適用可能

🔗 関連サービス・機能

🌐 VPC (Virtual Private Cloud)

VPC ネットワーク間のトラフィック分類に直接関連。VPC ピアリングや NCC ハブ経由の接続を識別可能。

🛡️ Cloud NGFW Enterprise

侵入防止サービス (IDS/IPS) や TLS インスペクションと組み合わせて、ネットワークコンテキストに基づいた高度な脅威検出が可能。

🔍 Cloud NGFW Standard

FQDN オブジェクト、ジオロケーション、Google Threat Intelligence とネットワークコンテキストを組み合わせた多層防御が可能 (一部制約あり)。

🔗 Network Connectivity Center

NCC ハブ経由で接続された VPC 間のトラフィックも VPC_NETWORKS コンテキストで識別可能。

📊 Cloud Logging / Cloud Monitoring

ファイアウォールルールのログ記録を有効にすることで、ネットワークコンテキストに基づくトラフィックパターンの可視化が可能。

📝 まとめ

Cloud NGFW の Network Contexts は、ファイアウォールポリシー管理の効率化とセキュリティ強化を両立する重要なアップデートである。4 種類のネットワークコンテキスト (INTERNET / NON_INTERNET / VPC_NETWORKS / INTRA_VPC) を活用することで、従来は多数の IP レンジベースのルールで実現していたトラフィック制御を、論理的で管理しやすい少数のルールに集約できる。

特にマルチ VPC 環境やハイブリッドクラウド環境を運用している組織は、既存のファイアウォールポリシーを見直し、Network Contexts を活用したルール最適化を検討することを推奨する。
