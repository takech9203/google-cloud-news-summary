🌐 VPC Hybrid Subnets が General Availability に昇格

Virtual Private Cloud Networking GA
2026-04-03
Hybrid Subnets

📋 概要

VPC の Hybrid Subnets 機能が GA に昇格しました。 オンプレミスネットワークと VPC ネットワーク間で同一の CIDR ブロックを共有し、 IP アドレスを変更することなくワークロードを Google Cloud に段階的に移行できる機能です。 2023 年 3 月に Preview として提供が開始され、約 3 年の検証期間を経て GA に到達しました。

🔄 Before / After 比較

❌ Before (従来)

✖移行時に IP アドレスの変更が必要
✖DNS、ファイアウォール、アプリ設定の大規模書き換え
✖同一サブネット内の内部通信の維持が困難
✖一括移行が必要で段階的移行が複雑
✖Preview のため SLA・サポートに制限

✅ After (GA 後)

✔IP アドレス変更不要で段階的に移行可能
✔GA 昇格により SLA が適用
✔移行期間中も内部通信を維持
✔Layer 3 ルーティングでパフォーマンス維持
✔VPC Network Peering との統合をサポート

🏗️ アーキテクチャ図

⭐ 主要機能

🔗 単一論理サブネットの構成

オンプレミスのネットワークセグメントと VPC サブネットを結合し、単一の論理サブネットとして動作。VPC サブネットのプライマリ IPv4 アドレス範囲がオンプレミスの CIDR ブロックと一致する必要があります。セカンダリ IPv4 アドレス範囲もサポート。

📦 段階的ワークロード移行

個別の VM やワークロードを 1 つずつ移行可能。移行済み VM の IP を Cloud Router のカスタムアドバタイズドルートとして /32 で個別にアドバタイズ。移行完了後にハイブリッドサブネットルーティングを無効化して通常運用に復帰。

⚡ Layer 3 ルーティング

OSI モデルの Layer 3 でパケットをルーティングし、Layer 2 トンネリングを回避。ネットワークトロンボーニングを防止し、Hybrid Subnets を使用しない場合と同等のパフォーマンスを実現。

🌐 VPC Network Peering 連携

ハイブリッドサブネットルーティングが有効な VPC を VPC Network Peering で他の VPC に接続可能。ピアリングネットワークから共有 CIDR ブロック内のリソースにアクセスできます。

🛠️ 設定方法

前提条件

  • VPC ネットワークの作成または既存ネットワークの特定
  • Compute Engine API の有効化
  • Network Connectivity API の有効化
  • HA VPN または Cloud Interconnect によるオンプレミスとの接続確立
  • オンプレミスルーターでの Proxy ARP 設定

Step 1: ハイブリッドサブネットルーティングの有効化

Bash サブネットの CIDR オーバーラップ許可 
gcloud compute networks subnets update SUBNET_NAME \
    --region=REGION \
    --allow-cidr-routes-overlap

Step 2: Cloud Router の BGP ピア設定

Bash カスタムルートアドバタイズメント設定 
gcloud compute routers update-bgp-peer ROUTER_NAME \
    --peer-name=PEER_NAME \
    --region=REGION \
    --advertisement-mode=CUSTOM

Step 3: VM 移行時のルートアドバタイズ

Bash 移行済み VM の /32 ルート追加 
gcloud compute routers update-bgp-peer ROUTER_NAME \
    --peer-name=PEER_NAME \
    --region=REGION \
    --add-advertisement-ranges=VM_IP/32

Step 4: 移行完了後の無効化

Bash ハイブリッドサブネットルーティングの無効化 
gcloud compute networks subnets update SUBNET_NAME \
    --region=REGION \
    --no-allow-cidr-routes-overlap

💪 メリット

💼 ビジネス面

→移行リスクの最小化: IP アドレス変更不要で DNS やファイアウォール設定の大規模変更を回避
→段階的移行によるコスト最適化: 一括移行のダウンタイムウィンドウや大規模体制構築が不要
→GA 昇格の安心感: SLA が適用され、エンタープライズの本番ワークロードでの採用が可能に

⚙️ 技術面

→Layer 3 ルーティング: レイテンシやスループットの劣化なく通信品質を維持
→既存ファイアウォールとの互換性: Layer 2 オーバーレイ不使用のため既存ルールがそのまま適用可能
→VPC Network Peering との統合: ピアリングネットワークからハイブリッドサブネット内リソースにアクセス可能

⚠️ デメリット・制約事項

🚫 制限事項

✖IPv6 トラフィックは非対応
✖ブロードキャスト / マルチキャストトラフィックは非対応
✖IP アドレスの競合検出機能がないため、手動での IP アドレス管理が必要
✖VPC サブネットのプライマリ IPv4 範囲の最初の 2 つと最後の 2 つの IP アドレスは Google Cloud リソースで使用不可
✖Network Connectivity Center (NCC) との併用は非推奨
✖Hybrid NAT との併用は非対応
✖ハイブリッド接続 NEG のセントラライズドヘルスチェックとの併用は非対応

🔍 考慮すべき点

⚠️オンプレミスネットワークが Proxy ARP をサポートしている必要あり (AWS や Azure からの移行は非対応)
⚠️オンプレミスネットワークが /32 ルートアドバタイズメントを受け入れる必要あり
⚠️VPN トンネル / VLAN / Cloud Router / ハイブリッドサブネットはすべて同一リージョン
⚠️Google Cloud VMware Engine を移行先とするシナリオは非対応 (VMware HCX 推奨)
⚠️ソースネットワークから Private Google Access や Private Service Connect へのアクセスは不可

💡 ユースケース

🏢 データセンターからの段階的リフトアンドシフト移行

シナリオ: 企業がオンプレミスデータセンターの 100 台以上の VM を Google Cloud に移行する計画があるが、全 VM の IP アドレス変更は DNS、ファイアウォール、アプリ設定の大規模書き換えが必要。

効果: Hybrid Subnets で IP アドレスを保持したまま VM を 1 台ずつ移行可能。移行中もオンプレミス / VPC 間の内部通信が維持され、段階的なスケジュールで移行を進められます。

🖥️ VMware 環境から Compute Engine への移行

シナリオ: オンプレミスの VMware 環境から Compute Engine へ移行する際、一部の VM が相互依存しており移行中の通信断を許容できない。

効果: Migrate to Virtual Machines と組み合わせ、移行済み VM と未移行 VM 間の内部通信を IP アドレス変更なしで維持。ビジネス継続性を確保しながら移行を進められます。

🔄 VPC 間のワークロード移行

シナリオ: 組織再編やプロジェクト統合に伴い、ある VPC ネットワークから別の VPC ネットワークへワークロードを移行する必要がある。

効果: Hybrid Subnets はオンプレミスだけでなく VPC 間の移行もサポート。2 つの VPC 間で同一 CIDR ブロックを共有し段階的に移行可能です。

💰 料金

Hybrid Subnets の料金情報については Virtual Private Cloud の料金ページ を参照してください。基盤となる接続サービス (HA VPN または Cloud Interconnect) の料金が別途発生します。

🔗 関連サービス

🔐 Cloud VPN (HA VPN)

Hybrid Subnets のオンプレミス接続に使用する VPN サービス。暗号化されたトンネルを提供。

🔌 Cloud Interconnect

専用線 / パートナー接続サービス。高帯域・低レイテンシの接続を提供。

📡 Cloud Router

BGP によるカスタムアドバタイズドルートの管理。移行済み VM の /32 ルートを動的にアドバタイズ。

📦 Migrate to Virtual Machines

VMware 環境からの VM 移行ツール。Hybrid Subnets と組み合わせて IP 保持移行を実現。

🌐 VPC Network Peering

ハイブリッドサブネット有効な VPC を他の VPC と接続し、共有 CIDR 内リソースへのアクセスを拡張。

📝 まとめ

Hybrid Subnets の GA 昇格は、オンプレミスから Google Cloud への大規模移行を計画している組織にとって重要なマイルストーンです。 IP アドレスの変更なしに段階的な移行を実現する本機能は、移行リスクの低減とビジネス継続性の確保に直結します。 オンプレミスからの移行プロジェクトを検討中の場合は、Hybrid Subnets の導入を含めたネットワーク設計の見直しを推奨します。
