📋 概要

Cluster Toolkit は Google Cloud が提供するオープンソースソフトウェアであり、HPC、AI、ML ワークロードの Google Cloud 上へのデプロイを簡素化するツールです。v1.87.0 では、GKE A4X-Max マシンタイプにおける Kueue ジョブスケジューリングのサポート追加、Managed Lustre における顧客管理暗号鍵 (CMEK) のサポート導入、install_asapd_lite モジュールの Helm への移行、および GKE A3 High ブループリントの更新が含まれます。

🎯 対象者: 大規模 AI/ML クラスタを運用するインフラストラクチャエンジニア、プラットフォームエンジニア、HPC ワークロードを管理するクラスタ管理者

🔄 Before / After

❌ アップデート前の課題

  • GKE A4X-Max で Kueue によるジョブキューイングとリソースクォータ管理が利用不可
  • Managed Lustre は Google デフォルト暗号化のみ、鍵のライフサイクル管理が不可
  • install_asapd_lite は静的マニフェストベース、Helm によるバージョン管理・ロールバック不可
  • A3 High ブループリントが最新ベストプラクティスに非対応

✅ アップデート後の改善

  • GKE A4X-Max で Kueue による TAS、リソースクォータ管理、コホートによるリソース共有が可能に
  • Cloud KMS を使用した CMEK で鍵のローテーション・無効化・破棄などが可能に
  • install_asapd_lite が Helm チャートに移行し管理性・再現性が向上
  • A3 High ブループリントが最新構成 (Jobset + Kueue サポート含む) に更新

⭐ 主要機能

🤖 1. GKE A4X-Max における Kueue サポート

A4X-Max マシンタイプを使用する GKE クラスタで Kueue によるジョブキューイングが利用可能になりました。

  • Kubernetes ネイティブのジョブスケジューラによるリソースクォータ制御
  • ClusterQueue、LocalQueue、ResourceFlavor でチーム間リソース公平共有
  • Topology Aware Scheduling (TAS) による A4X-Max の物理トポロジー (ブロック/サブブロック/ホスト) を考慮したワークロード配置
  • A4X-Max は NVIDIA GB300 GPU + Multi-Node NVLink (MNNVL) 搭載、最大 18 ノード (1x72 GPU)

🔐 2. Managed Lustre の CMEK サポート

Google Cloud Managed Lustre のファイルデータおよびメタデータを 顧客管理の Cloud KMS 鍵 で暗号化可能になりました。

  • 保護レベル: SOFTWARE、HSM、EXTERNAL の 3 種類をサポート
  • 鍵のローテーション、無効化、再有効化、破棄などのライフサイクル管理
  • Cluster Toolkit ブループリントから CMEK 保護された Managed Lustre インスタンスを構成可能

📦 3. install_asapd_lite の Helm 移行

install_asapd_lite モジュールが静的 Kubernetes マニフェストから Helm チャート へ移行されました。

  • Helm によるバージョン管理、ロールバック、カスタマイズが容易に
  • v1.70.0 で開始された Jobset の Helm 移行に続く、モジュール管理の近代化の一環

🚀 4. GKE A3 High ブループリントの更新

A3 High (H100 GPU 搭載) 向けの GKE ブループリントが最新構成に更新されました。

  • Jobset および Kueue のサポートを含む

📝 Kueue リソース定義例

GKE A4X-Max クラスタで Kueue のトポロジーアウェアスケジューリングを構成する Topology と ResourceFlavor の定義例です。

YAML Topology & ResourceFlavor 定義 (GKE A4X-Max) 
apiVersion: kueue.x-k8s.io/v1alpha1
kind: Topology
metadata:
  name: "gke-default"
spec:
  levels:
    - nodeLabel: "cloud.google.com/gce-topology-block"
    - nodeLabel: "cloud.google.com/gce-topology-subblock"
    - nodeLabel: "cloud.google.com/gce-topology-host"
    - nodeLabel: "kubernetes.io/hostname"
---
kind: ResourceFlavor
apiVersion: kueue.x-k8s.io/v1beta1
metadata:
  name: "a4x-max-flavor"
spec:
  nodeLabels:
    cloud.google.com/gke-nodepool: "a4x-max-pool"
  topologyName: "gke-default"
  tolerations:
    - key: "nvidia.com/gpu"
      operator: "Exists"
      effect: NoSchedule

ステップ 3: CMEK 保護された Managed Lustre の設定

Bash Cloud KMS キー作成 & Managed Lustre CMEK 設定 
# Cloud KMS キーリングとキーの作成
gcloud kms keyrings create my-keyring \
  --location=REGION

gcloud kms keys create my-lustre-key \
  --keyring=my-keyring \
  --location=REGION \
  --purpose=encryption

# Managed Lustre サービスエージェントへの権限付与
gcloud kms keys add-iam-policy-binding my-lustre-key \
  --keyring=my-keyring \
  --location=REGION \
  --member=serviceAccount:service-PROJECT_NUMBER@gcp-sa-lustre.iam.gserviceaccount.com \
  --role=roles/cloudkms.cryptoKeyEncrypterDecrypter

# CMEK 保護された Managed Lustre インスタンスの作成
gcloud lustre instance create my-instance \
  --filesystem=my-fs \
  --per-unit-storage-throughput=250 \
  --capacity-gib=18000 \
  --location=ZONE \
  --network=my-network \
  --kms-key-name=projects/KMS_PROJECT/locations/REGION/keyRings/my-keyring/cryptoKeys/my-lustre-key

👍 メリット

💼 ビジネス面

➤ GPU リソース利用率の最大化: Kueue のクォータ管理とコホートによるリソース共有で、高価な A4X-Max GPU の利用率が向上し ROI が改善
➤ コンプライアンス要件への対応: CMEK により暗号鍵の管理を顧客側で制御、金融・医療・公共セクターの規制要件に対応
➤ 運用コストの削減: Helm ベースのモジュール管理で、クラスタのデプロイ・メンテナンスの運用工数を削減

💻 技術面

➤ トポロジーアウェアなワークロード配置: Kueue の TAS 機能で MNNVL トポロジーを考慮した最適配置を実現、分散学習のパフォーマンス向上
➤ 暗号鍵のライフサイクル管理: Cloud KMS 統合で鍵のローテーション、監査ログ、アクセス制御を一元管理
➤ 宣言的なインフラ管理: Helm チャートへの移行でバージョン管理、差分確認、ロールバックが容易に

⚠️ デメリット・制約事項

制限事項

  • Managed Lustre の CMEK: GKE での動的プロビジョニングはサポートされておらず、静的プロビジョニングのみ対応
  • CMEK + VPC Service Controls: Cloud KMS キーが同一境界内にあるか、エグレスルールでアクセス可能である必要あり
  • A4X-Max ノードプールの最大ノード数は 18 ノード (1 サブブロック) に制限
  • A4X-Max は GKE バージョン 1.34.3 以上 または 1.35.0 以上 が必要

考慮すべき点

  • CMEK で SUSPENDED 状態になった Managed Lustre インスタンスも課金は継続される
  • Helm 移行に伴い、既存の install_asapd_lite デプロイメントではアップグレード手順の確認が必要
  • 鍵バージョンが破棄された場合、Managed Lustre インスタンスは永久に SUSPENDED となり削除のみ可能

💡 ユースケース

🎯 ユースケース 1: 大規模分散学習クラスタでのジョブスケジューリング

シナリオ: 複数の AI 研究チームが A4X-Max クラスタを共有し、分散学習ジョブを並行して実行する環境。各チームに GPU クォータを割り当て、未使用リソースをチーム間で共有したい。

YAML チーム間リソース共有のための ClusterQueue 設定 
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: team-a-queue
spec:
  cohort: "shared-a4x-max"
  resourceGroups:
    - coveredResources: ["nvidia.com/gpu"]
      flavors:
        - name: "a4x-max-flavor"
          resources:
            - name: "nvidia.com/gpu"
              nominalQuota: 36  # 9 ノード分
              borrowingLimit: 36
---
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: team-b-queue
spec:
  cohort: "shared-a4x-max"
  resourceGroups:
    - coveredResources: ["nvidia.com/gpu"]
      flavors:
        - name: "a4x-max-flavor"
          resources:
            - name: "nvidia.com/gpu"
              nominalQuota: 36
              borrowingLimit: 36
✅ 効果: チーム A がリソースを使用していない時間帯に、チーム B がコホート内の未使用 GPU を借用でき、クラスタ全体の GPU 利用率が向上します。

🎯 ユースケース 2: 規制対応が必要な AI ワークロード向けストレージ暗号化

シナリオ: 金融機関の AI 部門が、顧客データを含む学習データセットを Managed Lustre に配置し、暗号鍵の管理を自社で行う必要がある。
