📋 概要

Looker (Google Cloud core) において、パブリックセキュア接続のみを使用し、カスタムドメインを設定しないインスタンスでは、OAuth 認証情報の手動設定が不要になりました。Looker が OAuth クライアントとシークレット情報を自動的に管理するようになり、インスタンスのセットアッププロセスが大幅に簡素化されます。

なお、プライベート接続、ハイブリッド接続、またはカスタムドメインを使用するインスタンスでは、引き続き手動での OAuth 設定が必要です。

⭐ 主要機能

🤖 Looker managed OAuth 認証情報

パブリックセキュア接続のみのインスタンスに対して、Looker が自動的に OAuth クライアントとシークレットを割り当てます。Google Cloud コンソールでは「Looker managed」と表示され、クライアント ID / シークレットは外部に露出しません。

🔧 手動設定への切り替えオプション

自動管理されている OAuth 認証情報を「Manual」に切り替えることが可能です。インスタンスの編集画面から「Manual」を選択し、カスタムの OAuth クライアント ID とシークレットを入力できます。一度「Manual」に変更した場合は手動での管理が必要になります。

🔀 接続タイプによる動作の分岐

  • パブリックセキュア接続のみ (カスタムドメインなし): OAuth 自動管理
  • プライベート接続 (PSC / PSA): 手動設定が必須
  • ハイブリッド接続: 手動設定が必須
  • カスタムドメイン使用時: 手動設定が必須

⚙️ 設定方法

前提条件

✓ Google Cloud プロジェクトで Looker (Google Cloud core) API が有効化されていること
✓ インスタンス作成に必要な IAM ロール (roles/looker.admin など) を持っていること

ステップ 1: インスタンスの作成

パブリックセキュア接続でカスタムドメインを使用しない場合、OAuth 認証情報の指定は不要です。

Bash 
# gcloud CLI でインスタンスを作成する場合
gcloud looker instances create INSTANCE_NAME \
  --project=PROJECT_ID \
  --region=REGION \
  --edition=EDITION_TYPE \
  --public-ip-enabled

ステップ 2: OAuth 設定の確認

インスタンス作成後、Google Cloud コンソールでインスタンスの「Edit」画面を開き、OAuth application credentials セクションが「Looker managed」になっていることを確認します。

💼 ビジネス面

✓ 初期セットアップ時間の短縮: OAuth の事前設定が不要に
✓ 運用負荷の軽減: 認証情報のローテーションやメンテナンスが自動化
✓ ヒューマンエラーの防止: 設定ミスやリダイレクト URI 漏れを防止

💻 技術面

✓ セットアップの簡素化: 5 ステップ → 実質 0 ステップに削減
✓ セキュリティの向上: 認証情報が外部に露出しない
✓ 柔軟性の維持: 必要に応じて Manual に切り替え可能

⚠️ デメリット・制約事項

制限事項:
  • パブリックセキュア接続かつカスタムドメインなしのインスタンスのみが対象
  • Looker managed の状態では OAuth クライアント ID やシークレットを直接確認できない
  • 「Looker managed」から「Manual」に切り替えた後、再度「Looker managed」に戻す手順は要確認
💡 考慮すべき点:
  • 既存の手動設定済みインスタンスについては、Looker managed への移行可否を確認する必要あり
  • 将来カスタムドメインの追加を予定している場合、その時点で手動 OAuth 設定への移行が必要

🎯 ユースケース

🚀 小規模チームでの迅速な BI 環境構築

シナリオ: スタートアップ企業が 10 名程度のチーム向けに Looker Standard エディションを導入。パブリックインターネット経由のアクセスで十分であり、カスタムドメインも不要。

✅ 効果: OAuth の事前設定なしで即座にインスタンスを作成でき、短時間で BI ダッシュボードの構築を開始できる。Google Cloud の管理経験が少ないチームでもスムーズにセットアップが完了。

🔄 開発・テスト環境の迅速なプロビジョニング

シナリオ: Enterprise エディションを利用している組織が、本番環境とは別にステージング用のインスタンスを頻繁に作成・削除する。ステージング環境はパブリック接続で十分。

✅ 効果: 開発・テスト用インスタンスの作成が簡素化され、CI/CD パイプラインの一環としてインスタンスのプロビジョニングを自動化しやすくなる。環境構築のサイクルタイムが短縮。
