📊 BigQuery: レプリケーション モニタリング & Continuous Queries to Spanner

BigQuery GA Cloud Monitoring Spanner
2026-03-04

☁️ 概要

BigQuery の 2 つの機能が GA (一般提供) になりました。クロスリージョン レプリケーションの運用監視と Spanner へのリアルタイム データ連携の両方が本番環境で安心して利用できるようになりました。

対象ユーザー: クロスリージョン レプリケーションを利用しているデータエンジニア、DR 計画を策定するインフラストラクチャ チーム、BigQuery と Spanner 間のリアルタイム データ同期を必要とするアプリケーション開発者

🔄 Before / After 比較

❌ Before (アップデート前)

  • CRR/MDR のレイテンシやエグレス バイトを Cloud Monitoring で直接監視する手段がなかった
  • BigQuery から Spanner へのデータ同期はバッチ ジョブによるスケジュール実行が必要
  • DR 環境のレプリケーション遅延を検知するためのカスタム監視基盤を別途構築する必要があった

✅ After (アップデート後)

  • Cloud Monitoring で staleness と network_egress_bytes_count を直接監視可能
  • Continuous Queries により BigQuery から Spanner にリアルタイム ストリーミング
  • Metrics Explorer やカスタム ダッシュボードでレプリケーション状態を可視化 + アラート設定

📊 機能 1: レプリケーション モニタリング

クロスリージョン レプリケーション (CRR) およびマネージド ディザスタ リカバリ (MDR) で有効なデータセットのレプリケーション状態を Cloud Monitoring で監視可能になりました。

📈 storage/replication/staleness

セカンダリ リージョンで利用可能なデータの鮮度 (レイテンシ) を測定。60 秒ごとにサンプリングされ、最大 420 秒の遅延で表示されます。

→ 型: GAUGE, INT64 / 単位: ms

📡 storage/replication/network_egress_bytes_count

レプリケーションによる課金対象のエグレス バイト数を測定。60 秒ごとにサンプリングされ、最大 420 秒の遅延で表示されます。

→ 型: DELTA, INT64 / 単位: By

メトリクスのラベル

ラベル説明
locationレプリケーション先のロケーション
replica_idレプリカ ID
replication_errorレプリケーション エラー (staleness メトリクスのみ)
💡 ポイント: Metrics Explorer でのリアルタイム表示、カスタム ダッシュボードの作成、アラート ポリシーの設定が可能です。

⚡ 機能 2: Continuous Queries to Spanner

BigQuery の EXPORT DATA ステートメントと APPENDS 関数を組み合わせ、BigQuery テーブルに追加された新規行を自動的に検出して Spanner にストリーミングします。

🔄 リアルタイム リバース ETL

✓ format = 'CLOUD_SPANNER' オプションで Spanner テーブルへ直接エクスポート
✓ change_timestamp_column でデータの書き込み順序を制御可能
✓ ニアリアルタイムのデータ同期を実現、定期バッチ ジョブが不要に

🎛️ CONTINUOUS ジョブタイプ

✓ CONTINUOUS ジョブタイプのスロット予約割り当てが必要
✓ BigQuery Studio UI または BQ CLI (--continuous=true) から実行可能
✓ サービス アカウントを使用した長時間実行に対応

設定パラメータ

パラメータ説明
format'CLOUD_SPANNER' を指定
uriSpanner データベースの URI
spanner_options.table書き込み先の Spanner テーブル名
spanner_options.priorityリクエスト優先度 (HIGH, MEDIUM, LOW)
spanner_options.change_timestamp_columnタイムスタンプ列の指定 (書き込み順序の制御)

💻 設定方法 / コードサンプル

ステップ 1: レプリケーション モニタリングの設定

Google Cloud コンソール > Monitoring > Metrics Explorer でリソースタイプ「BigQuery Dataset」を選択し、レプリケーション関連メトリクスを追加します。アラート ポリシーを作成して閾値を超えた場合に通知を受け取れます。

Bash Metrics Explorer での確認 
# Cloud Monitoring の Metrics Explorer で確認
# Google Cloud コンソール > Monitoring > Metrics Explorer
# リソースタイプ: BigQuery Dataset
# メトリクス: storage/replication/staleness
#          または storage/replication/network_egress_bytes_count

ステップ 2: Continuous Query to Spanner の設定

SQL Continuous Query の SQL 例 
EXPORT DATA
OPTIONS (
  uri = 'https://spanner.googleapis.com/projects/PROJECT_ID/instances/INSTANCE_ID/databases/DATABASE_ID',
  format = 'CLOUD_SPANNER',
  spanner_options = '{"table": "my_table", "priority": "HIGH", "change_timestamp_column": "update_time"}'
)
AS
SELECT
  id,
  name,
  value,
  _CHANGE_TIMESTAMP AS update_time
FROM APPENDS(
  TABLE `project.dataset.source_table`,
  CURRENT_TIMESTAMP() - INTERVAL 10 MINUTE
);
Bash bq CLI での実行 
# bq CLI を使用した Continuous Query の実行
bq query --continuous=true --use_legacy_sql=false '
EXPORT DATA
OPTIONS (
  uri = "https://spanner.googleapis.com/projects/my-project/instances/my-instance/databases/my-db",
  format = "CLOUD_SPANNER",
  spanner_options = "{\"table\": \"target_table\", \"priority\": \"HIGH\"}"
)
AS
SELECT * FROM APPENDS(
  TABLE `my-project.my_dataset.source_table`,
  CURRENT_TIMESTAMP() - INTERVAL 10 MINUTE
);
'
💡 ヒント: BigQuery Studio の UI から実行する場合は、クエリ エディタで SQL を入力し、「その他」メニューから「クエリモードの選択」で「Continuous query」を選択します。

📋 前提条件

📊 レプリケーション モニタリング

✓ BigQuery のクロスリージョン レプリケーション (CRR) またはマネージド ディザスタ リカバリ (MDR) が有効なデータセット

⚡ Continuous Queries

✓ Enterprise エディション以上の予約
✓ CONTINUOUS ジョブタイプのスロット予約割り当て
✓ Spanner データベースおよび書き込み先テーブルが作成済み

✅ メリット

ビジネス面

🎯 DR 運用の信頼性向上

RPO (目標復旧時点) の遵守状況をリアルタイムで把握し、DR 計画の実効性を確保

⚡ リアルタイム アプリケーション サービング

BigQuery の分析結果を Spanner 経由で低レイテンシに提供し、ユーザー体験を向上

💰 運用コストの削減

バッチ ジョブのスケジューリングと管理が不要になり、パイプラインの運用負荷が軽減

技術面

📊 統合監視

Cloud Monitoring の既存ダッシュボードやアラート基盤にレプリケーション メトリクスを統合

🔄 イベント駆動アーキテクチャ

APPENDS 関数により新規行の追加をイベントとして捕捉し、即座に Spanner に反映

📝 SQL ベースのパイプライン定義

標準 SQL で記述でき、専用 ETL ツール不要でリバース ETL を実現

⚠️ デメリット・制約事項

制限事項
  • Continuous Queries to Spanner は Enterprise エディション以上が必要 (Standard / オンデマンドは不可)
  • APPENDS 関数は INSERT のみを捕捉 (UPDATE / DELETE は検出しない)
  • Spanner の自動生成主キーを持つテーブルへの Continuous Query エクスポートは不可
  • PostgreSQL 方言の Spanner データベースへのエクスポートは不可
  • ユーザー アカウントのアクセス トークン TTL は 2 日間 (長時間実行にはサービス アカウントが必要)
  • レプリケーション モニタリング メトリクスはサンプリング後最大 420 秒の遅延
考慮すべき点
  • データベース エラー (重複主キー、参照整合性違反など) が発生するとパイプラインがキャンセルされるため、エラー ハンドリングの設計が重要
  • Spanner テーブルの主キーに単調増加する整数を使用するとパフォーマンスの問題が発生する可能性あり
  • レプリケーション モニタリングのメトリクスは CRR/MDR が有効なデータセットでのみ利用可能

💡 ユースケース

🏦 ユースケース 1: DR レプリケーション SLA 監視

シナリオ: 金融サービス企業が BigQuery のマネージド ディザスタ リカバリを利用し、東京 (asia-northeast1) から大阪 (asia-northeast2) へのレプリケーションを監視する。

Config Cloud Monitoring アラート ポリシー 
# Cloud Monitoring アラート ポリシーの設定
# 条件: storage/replication/staleness > 300000 ms (5 分) が 10 分間継続
# 通知チャンネル: PagerDuty, Slack

効果: レプリケーション レイテンシが閾値を超えた場合に即座にアラートが発報され、RPO 違反を未然に防止できる。

🛒 ユースケース 2: リアルタイム リバース ETL パイプライン

シナリオ: EC サイトが BigQuery で集約したユーザー行動分析データを Spanner に同期し、パーソナライゼーション エンジンからリアルタイムに参照する。

SQL EC サイト パーソナライゼーション 
EXPORT DATA
OPTIONS (
  format = 'CLOUD_SPANNER',
  uri = 'https://spanner.googleapis.com/projects/ecommerce-prod/instances/serving/databases/personalization',
  spanner_options = '{"table": "user_recommendations", "priority": "HIGH", "change_timestamp_column": "updated_at"}'
)
AS
SELECT
  user_id,
  recommendation_scores,
  _CHANGE_TIMESTAMP AS updated_at
FROM APPENDS(
  TABLE `ecommerce-prod.analytics.user_recommendations`,
  CURRENT_TIMESTAMP() - INTERVAL 10 MINUTE
);

効果: BigQuery での ML 推論結果が数秒以内に Spanner を経由してアプリケーションに反映され、リアルタイムなパーソナライゼーションが可能になる。

💰 料金

📊 レプリケーション モニタリング

Cloud Monitoring のメトリクス利用に関する追加費用なし。カスタム ダッシュボードやアラート設定数が無料枠を超える場合は Cloud Monitoring の料金が適用。

🔄 クロスリージョン レプリケーション

→ ストレージ: セカンダリ リージョンのストレージ バイトが別途課金
→ データ レプリケーション: データ レプリケーション料金が別途課金 (スロット リソースは消費しない)

⚡ Continuous Queries

→ スロット予約: CONTINUOUS ジョブタイプの予約割り当てに基づくスロット消費
→ Enterprise エディション以上が必須
