🔐 Secret Manager: Kubernetes Secrets への統合シークレット同期機能が GA

Secret Manager GKE GA
2026-04-16
Google Cloud Update

📋 概要

Secret Manager の統合シークレット同期機能が GA (一般提供) となりました。この機能により、Secret Manager に保存されたシークレットを GKE クラスタ内の Kubernetes Secret オブジェクトへ自動的に同期できるようになります。

アプリケーションは環境変数やボリュームマウントなど標準的な Kubernetes の手法を使って Secret Manager のシークレットにアクセスできます。コード変更なしで Secret Manager の集中管理、IAM によるアクセス制御、監査ログ、CMEK 暗号化といったエンタープライズグレードのセキュリティ機能を活用できます。

🔄 Before / After 比較

❌ Before (アップデート前)

  • Secret Manager API を直接呼び出すコード変更が必要
  • K8s Secret と Secret Manager のシークレットを二重管理
  • ローテーション時に両方を手動で更新する運用負荷
  • 既存 K8s アプリの移行に大幅なリファクタリングが必要

✅ After (アップデート後)

  • Secret Manager から K8s Secret へ自動同期、コード変更不要
  • SecretProviderClass / SecretSync による宣言的な同期設定
  • 自動ローテーション (最短 1 分間隔で自動反映)
  • CSI Driver からの段階的な移行パスを提供

📝 設定方法

ステップ 1: GKE クラスタでシークレット同期を有効化

Bash 既存クラスタの場合 
# シークレット同期を有効化
gcloud container clusters update CLUSTER_NAME \
    --location=CONTROL_PLANE_LOCATION \
    --enable-secret-sync

# 自動ローテーション付きで有効化 (カスタム間隔)
gcloud container clusters update CLUSTER_NAME \
    --location=CONTROL_PLANE_LOCATION \
    --enable-secret-sync-rotation \
    --secret-sync-rotation-interval=300s
Bash 新規クラスタ作成時 
# Standard クラスタの場合
gcloud container clusters create CLUSTER_NAME \
    --location=CONTROL_PLANE_LOCATION \
    --workload-pool=PROJECT_ID.svc.id.goog \
    --enable-secret-sync \
    --enable-secret-sync-rotation

# Autopilot クラスタの場合
gcloud container clusters create-auto CLUSTER_NAME \
    --location=CONTROL_PLANE_LOCATION \
    --enable-secret-sync \
    --enable-secret-sync-rotation

ステップ 2: Kubernetes ServiceAccount と IAM 設定

Bash ServiceAccount 作成と IAM バインディング 
# Kubernetes ServiceAccount を作成
kubectl create serviceaccount KSA_NAME --namespace NAMESPACE

# IAM ポリシーバインディングを設定
gcloud secrets add-iam-policy-binding SECRET_NAME \
    --role=roles/secretmanager.secretAccessor \
    --member=principal://iam.googleapis.com/projects/PROJECT_NUMBER/locations/global/workloadIdentityPools/PROJECT_ID.svc.id.goog/subject/ns/NAMESPACE/sa/KSA_NAME

ステップ 3: SecretProviderClass の作成

YAML SecretProviderClass 設定例 
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: my-app-secrets
  namespace: my-namespace
spec:
  provider: gke
  parameters:
    secrets: |
      - resourceName: "projects/my-project/secrets/api-key/versions/latest"
        path: "api-key.txt"
      - resourceName: "projects/my-project/secrets/db-password/versions/latest"
        path: "db-password.txt"

ステップ 4: SecretSync の作成

YAML SecretSync 設定例 
apiVersion: secret-sync.gke.io/v1
kind: SecretSync
metadata:
  name: my-kube-secret
  namespace: my-namespace
spec:
  serviceAccountName: my-ksa
  secretProviderClassName: my-app-secrets
  secretObject:
    type: Opaque
    data:
      - sourcePath: "api-key.txt"
        targetKey: "API_KEY"
      - sourcePath: "db-password.txt"
        targetKey: "DB_PASSWORD"

ステップ 5: 同期ステータスの確認

Bash 確認コマンド 
# 同期された Kubernetes Secret を確認
kubectl get secret KUBERNETES_SECRET_NAME -n NAMESPACE -o yaml

# 同期ステータスのトラブルシューティング
kubectl describe secretsync KUBERNETES_SECRET_NAME -n NAMESPACE

👍 メリット

ビジネス面

  • 運用コストの削減: シークレットの二重管理が不要。管理対象が Secret Manager に一元化され、運用工数とヒューマンエラーのリスクが削減
  • コンプライアンス対応の強化: Secret Manager の監査ログ、CMEK 暗号化、IAM 制御を活用し、規制要件への対応が容易に
  • 移行コストの最小化: 既存 K8s アプリのコード変更が不要。段階的かつ低コストでの Secret Manager 導入が可能

技術面

  • シームレスな統合: 環境変数やボリュームマウントなど、既存の K8s シークレット消費パターンをそのまま利用可能
  • 自動ローテーション: Secret Manager でのシークレット更新が GKE クラスタに自動反映。手動同期が不要に
  • 宣言的管理: SecretProviderClass と SecretSync のカスタムリソースにより、GitOps ワークフローとの統合が容易
  • マルチテナント対応: Namespace 単位の分離と K8s ServiceAccount ごとの IAM バインディングによるきめ細かいアクセス制御

⚠️ デメリット・制約事項

制限事項

  • GKE バージョン 1.33 以降でのみ利用可能。古いバージョンではアップグレードが必要
  • Workload Identity Federation for GKE が有効である必要がある (Standard では手動有効化が必要)
  • シークレットのペイロードサイズは 64 KiB まで
  • ポーリングベースの同期方式。リアルタイム同期ではなく最短 1 分間隔

考慮すべき点

  • 💡 K8s Secret は etcd に保存されるため、Secret Manager 内と比較してセキュリティレベルが異なる。Cloud KMS によるアプリケーション層暗号化を推奨
  • 💡 環境変数でのシークレット公開はログやプロセスリストでの露出リスクあり。ボリュームマウントの使用を推奨
  • 💡 リージョナルシークレットはクラスタと同一リージョンである必要がある
  • 💡 同期無効化時、既存の K8s Secret は自動削除されない (手動削除が必要)
  • 💡 VPC Service Controls 使用環境ではサービスペリメータ設定に注意
  • 💡 公式ドキュメントでは、アプリが対応する場合は Secret Manager アドオン (CSI ドライバー) の方がよりセキュアとして推奨

🎯 ユースケース

🔌 ユースケース 1: 既存マイクロサービスの Secret Manager 統合

シナリオ: 多数のマイクロサービスが K8s Secret から環境変数として DB 認証情報や API キーを読み取っている。Secret Manager を導入したいが、全サービスのコード変更の余裕がない。

効果: アプリケーションのコードやデプロイメント設定を変更することなく、Secret Manager による集中管理、監査ログ、自動ローテーションを導入可能。

🌐 ユースケース 2: マルチクラスタ環境でのシークレット一元管理

シナリオ: 複数の GKE クラスタ (開発、ステージング、本番) にまたがるアプリケーションで、各環境のシークレットを一元的に管理したい。

効果: Secret Manager を単一のシークレットソースとして使用し、各 GKE クラスタでシークレット同期を有効化。IAM ポリシーにより環境間のアクセス分離も維持。

🔀 ユースケース 3: Secrets Store CSI Driver からの移行

シナリオ: 既にオープンソースの Secrets Store CSI Driver を使用しているが、GKE ネイティブの統合シークレット同期機能に移行したい。

効果: SecretProviderClass の provider フィールドを gcp から gke に変更するだけで移行開始。GKE マネージドの同期コントローラーに移行し、CSI Driver の個別管理やアップデート作業が不要に。

💡 料金の考慮事項:
  • -> 同期コントローラーが定期的に Secret Manager API を呼び出すため、ローテーション間隔設定によりアクセスオペレーション数が増加する可能性あり
  • -> 例: 100 個のシークレットを 2 分間隔で同期 = 月間約 216 万回のオペレーション = 約 $6.48 の追加コスト
  • -> ハッシュ比較により変更のないシークレットの書き込みは最小化されるが、チェック用の読み取りは発生

🌏 利用可能リージョン

シークレット同期機能は GKE 1.33 以降が利用可能な全リージョンで使用できます。Secret Manager 自体はグローバルサービスですが、リージョナルシークレットを使用する場合は GKE クラスタと同一リージョンのシークレットバージョンを指定する必要があります。

🔗 関連サービス・機能

→ Secret Manager アドオン (CSI コンポーネント): シークレットをインメモリファイルとして Pod に直接マウントする、よりセキュアな代替手法。アプリが対応する場合はこちらが推奨
→ Workload Identity Federation for GKE: GKE ワークロードから Google Cloud サービスへの認証に使用。シークレット同期機能の前提条件
→ Cloud KMS: K8s Secret の etcd ストレージをアプリケーション層で暗号化。CMEK によるシークレット暗号化にも対応
→ Cloud Audit Logs: Secret Manager のシークレットアクセスと管理操作の監査ログを提供
📌 まとめ: Secret Manager の K8s Secrets 統合シークレット同期機能の GA リリースにより、GKE 上のアプリケーションはコード変更なしで Secret Manager のエンタープライズグレードのシークレット管理機能を活用できるようになりました。特に、既に K8s Secret を使用している多数のマイクロサービスを運用する組織にとって、段階的かつ低リスクな Secret Manager 導入を可能にする重要なマイルストーンです。GKE 1.33 以降を利用している場合は、まず非本番環境でのシークレット同期有効化から検証を推奨します。
