☁️ Overview

Google Cloud Telemetry (OTLP) API において、メトリクスおよびトレースのインジェストクォータがグローバルクォータからリージョナルクォータへ移行されました。

OpenTelemetry OTLP 仕様に基づく Telemetry API を利用してメトリクスやトレースデータを送信しているすべてのユーザーに影響します。リージョナルクォータへの移行により、特定のリージョンに集中するワークロードが他のリージョンのクォータを消費することがなくなり、マルチリージョン環境での公平なリソース配分が実現されます。

🔄 Before / After

❌ Before (グローバルクォータ)

  • 特定リージョンからの大量リクエストが他リージョンのインジェスト能力に影響
  • リージョンごとのインジェスト容量を予測・管理することが困難
  • トレースインジェストがリクエスト数ベースで、実際のデータ量に基づく制御が不可

✅ After (リージョナルクォータ)

  • リージョンごとに独立したクォータで安定したインジェストが可能
  • Cloud Trace がバイトベースに変更され、直感的なクォータ管理を実現
  • 主要 11 リージョンで毎分 2.4 GB の大容量トレースインジェスト

🏗️ Architecture

各リージョンのアプリケーションは対応するリージョンの Telemetry API エンドポイントにデータを送信し、リージョンごとに独立したクォータが適用されます。

⚡ Key Features

📈 Metrics Ingest Regional Quota (Cloud Monitoring)

→リージョンあたり毎分 60,000 メトリクスインジェストリクエスト
→リクエストあたり最大 200 データポイント (実効デフォルト: 毎秒 200,000 サンプル)
→従来のグローバルクォータを置き換える形で導入

🔍 Trace Ingest Regional Quota (Cloud Trace)

→主要 11 リージョン: 毎分 2.4 GB のトレースインジェスト
→その他リージョン: 毎分 300 MB のトレースインジェスト
→リクエスト数ベースからバイトベースのクォータへ移行

🔧 Unified Quota Overhaul

→メトリクスとトレースの両方で一貫したリージョナルクォータモデル
→Telemetry API 固有のクォータ (Cloud Monitoring API のクォータとは独立)
→データインジェスト後は既存の Cloud Monitoring / Cloud Trace のクォータが引き続き適用

🛠️ Setup

Step 1: Check Current Quota Usage

Bash 
# Google Cloud Console: Quotas Dashboard
# https://console.cloud.google.com/iam-admin/quotas

# Or via gcloud CLI
gcloud alpha services quota list \
  --service=monitoring.googleapis.com \
  --project=<PROJECT_ID>

Step 2: Verify OTel Collector Endpoint Configuration

YAML 
# otel-collector-config.yaml
exporters:
  otlp:
    endpoint: "telemetry.googleapis.com:443"
    headers:
      x-goog-user-project: "<PROJECT_ID>"
    compression: gzip
💡 Note: Regional quotas are applied automatically. No endpoint changes are required for existing Telemetry API configurations.

Step 3: Request Quota Increase (if needed)

Bash 
# Request quota increase via Google Cloud Console
# IAM & Admin > Quotas > Select quota > Edit Quotas

✅ Benefits

🏢 Business

✓Multi-region stability: Independent quotas per region prevent cross-region impact
✓Predictable capacity planning: Region-level quota management enables accurate workload distribution

⚙️ Technical

✓Byte-based trace quotas: Consistent control regardless of span size
✓High-capacity regions: 2.4 GB/min trace ingest across 11 primary regions
✓Backward compatible: No changes needed to existing Telemetry API settings

⚠️ Limitations

Constraints:
  • Telemetry API is Pre-GA; support may be limited
  • Non-primary regions have a smaller trace ingest quota (300 MB/min)
  • Metrics ingest quota increases may require Google Cloud Support
💡 Considerations:
  • Workloads concentrated in specific regions may experience quota usage changes
  • Set up alert policies to monitor per-region quota consumption for multi-region apps
  • Telemetry API quotas and Cloud Monitoring / Cloud Trace API quotas are independent -- manage both

🎯 Use Cases

🌍 UC1: Global Distributed Microservices Observability

Scenario: GKE clusters across Asia, Europe, and North America, each using OpenTelemetry Collector for metrics and trace collection.

YAML 
# OTel Collector DaemonSet config per region
exporters:
  otlp:
    endpoint: "telemetry.googleapis.com:443"
    headers:
      x-goog-user-project: "my-global-project"

service:
  pipelines:
    traces:
      exporters: [otlp]
    metrics:
      exporters: [otlp]
Effect: Each region gets independent 60,000 req/min (metrics) and 2.4 GB/min (traces) quotas, eliminating cross-region quota contention.

📡 UC2: High-Volume Trace Data Collection

Scenario: High-traffic API gateway in us-central1 generating large volumes of trace data. Previously hit request-count-based global quota limits.

Effect: Byte-based regional quota (2.4 GB/min) enables stable trace collection independent of span count or request count, managed purely by data volume.

💰 Pricing

Metrics ingested via Telemetry API are billed under the "Prometheus Samples Ingested" SKU (same as Google Cloud Managed Service for Prometheus). Traces are billed at Cloud Trace standard rates.
