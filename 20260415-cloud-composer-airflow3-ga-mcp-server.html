🔄 Before / After

❌ Before

  • 「Cloud Composer」の名称では Apache Airflow のマネージドサービスであることが直感的に伝わりにくい
  • Airflow 3 は Preview ステータスであり、本番環境での利用が推奨されない
  • スナップショットを使った Airflow 2 → 3 の移行、Workforce Identity Federation は未サポート
  • AI アプリから Cloud Composer 環境を操作するには REST API / gcloud CLI を個別に呼び出す必要がある

✅ After

  • 「Managed Service for Apache Airflow」で OSS マネージドサービスであることが一目瞭然
  • Airflow 3 が GA に昇格し、SLA 適用で本番ワークロードに安心して利用可能
  • DAG バージョニング、イベント駆動スケジューリング、Assets、新 UI を本番で活用
  • MCP サーバーにより AI エージェントから自然言語で環境管理、DAG 実行状況の確認が可能

🏷️ 主要機能 1: Managed Service for Apache Airflow への改名

OSS ファーストの姿勢を明確に

Cloud Composer は Managed Service for Apache Airflow に正式に改名されます。これは OSS ソリューションの成功と顧客の支持を踏まえた戦略的な変更です。

→Google Cloud が「最もオープンなクラウドエコシステム」であるコミットメントを強化
→サービスのポートフォリオにおける位置づけを直感的に理解可能
→「Managed Service for Apache Kafka」と同様の命名規則に統一
→Apache Airflow コミュニティとの連携をさらに深める意思表示

🚀 主要機能 2: Airflow 3 GA

Cloud Composer 3 において Airflow 3 が 一般提供 (GA) ステータスに昇格。本番環境での SLA が適用されます。

完全にリデザインされた新しいユーザーインターフェース

📜 DAG バージョニング

💡 airflowctl コマンドラインツール: 新しい CLI ツールにより、コマンドラインからの環境管理がさらに便利になりました。

🤖 主要機能 3: Cloud Composer リモート MCP サーバー (Preview)

Model Context Protocol (MCP) に準拠したリモートサーバーを提供。AI アプリケーションから Cloud Composer に接続し、自然言語で環境管理が可能になります。

MCP サーバーの主な機能

→Cloud Composer 環境の管理操作
→実行済み DAG Run の詳細取得
→Airflow タスクの情報取得
🤖 対応 AI クライアント: Gemini CLI、ChatGPT、Claude、カスタム AI アプリケーション。IAM による認証・認可、Model Armor によるセキュリティ保護に対応。

📜 ユースケース 2: DAG バージョニングを活用した CI/CD パイプライン

シナリオ: データエンジニアリングチームが DAG の変更をバージョン管理し、特定バージョンの実行結果を追跡

Python 
from airflow.decorators import dag, task
from datetime import datetime

@dag(
    schedule="@daily",
    start_date=datetime(2026, 4, 15),
    catchup=False,
)
def etl_pipeline():
    @task
    def extract():
        # データ抽出ロジック
        return {"source": "bigquery", "rows": 10000}

    @task
    def transform(data: dict):
        # データ変換ロジック
        return {"transformed_rows": data["rows"]}

    @task
    def load(data: dict):
        # データロードロジック
        print(f"Loaded {data['transformed_rows']} rows")

    raw = extract()
    transformed = transform(raw)
    load(transformed)

etl_pipeline()

効果: DAG の各バージョンが自動的に追跡され、変更前後の実行結果を比較できる。問題発生時にどのバージョンの変更が原因かを特定しやすい

📦 ユースケース 3: Assets を活用したデータパイプラインの連携

シナリオ: 複数の DAG が生成するデータの依存関係を宣言的に管理し、上流データの更新を自動検知して下流処理をトリガー

Python 
from airflow.sdk import Asset, DAG, task

# Asset の定義
sales_data = Asset("gs://my-bucket/sales/daily/")
report_data = Asset("gs://my-bucket/reports/daily/")

# 上流 DAG: 売上データを生成
with DAG(
    dag_id="produce_sales_data",
    schedule="@daily",
):
    @task(outlets=[sales_data])
    def generate_sales():
        # 売上データを生成して Cloud Storage に保存
        pass

# 下流 DAG: 売上データの更新を検知してレポートを生成
with DAG(
    dag_id="generate_report",
    schedule=[sales_data],  # Asset ベースのスケジューリング
):
    @task(outlets=[report_data])
    def create_report():
        # 売上データからレポートを生成
        pass

効果: DAG 間の依存関係が Asset として明示的に定義され、上流処理の完了を自動検知して下流処理が開始される。不要な待機時間が削減され、データの鮮度が向上

⚠️ 制限事項・注意点

制限事項

✗DAG bundles (LocalDagBundle 以外) は未サポート。リモート DAG ストレージの利用は今後のアップデートを待つ必要がある
✗Edge Executor および他言語でのタスク実行は未サポート
✗MCP サーバーは Preview ステータスであり、本番環境では Pre-GA 条件が適用
✗Airflow 2 → 3 の既存環境インプレースアップグレードのサポート状況は要確認

考慮すべき点

📌名称変更対応: ドキュメント、社内手順書、IaC テンプレートでの名称更新が必要になる可能性。API エンドポイントや gcloud コマンドの変更有無を確認
📌Airflow 3 移行: DAG の互換性確認が必須。Airflow 3 での破壊的変更 (API 変更、非推奨機能の削除など) を事前に検証
📌MCP セキュリティ: 適切な IAM 権限設定が必要。過剰な権限付与は環境の不正操作リスクにつながるため、最小権限の原則を遵守

💰 料金

Cloud Composer 3 (Managed Service for Apache Airflow) の料金は、環境のコンピューティングリソース使用量に基づきます。Airflow 3 GA 環境と Airflow 2 環境の料金モデルに変更はありません。MCP サーバーの利用に関する追加料金は、Preview 時点では発表されていません。

📝 まとめ

今回のアップデートは、Google Cloud のワークフローオーケストレーション戦略における最大級の転換点です。

  • 改名: Cloud Composer → Managed Service for Apache Airflow で OSS ファーストの姿勢を明確化
  • Airflow 3 GA: 次世代ワークフロー管理の本番利用が可能に
  • MCP サーバー: AI アプリとワークフローオーケストレーションの融合という新しい運用モデル

推奨アクション: 既存の Cloud Composer ユーザーは、Airflow 3 への移行計画を策定し、MCP サーバーによる AI 統合の検証を開始することが推奨されます。
