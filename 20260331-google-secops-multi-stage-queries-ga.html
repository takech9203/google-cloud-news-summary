📋 概要

Google SecOps の YARA-L 2.0 マルチステージクエリが GA (一般提供) となった。あるクエリステージの出力を次のステージの入力として連結し、単一のモノリシックなクエリでは困難だった高度なデータ変換と検出ロジックの構築が可能になる。

Dashboards と Search の両方で利用可能であり、特別な有効化操作は不要。既存の全ユーザーが即座に利用開始できる。

🔄 Before / After

❌ Before (アップデート前)

  • 単一のモノリシックなクエリに複雑なロジックを詰め込む必要があった
  • ベースラインベースの異常検出を単一クエリで表現するのが困難
  • 複数データソースの集約結果をさらに加工すると可読性・保守性が低下
  • Pre-GA のため本番ワークロードにサポート・互換性の制約あり

✅ After (アップデート後)

  • 最大 4 つの名前付きステージ + ルートステージでクエリを分割
  • 前段ステージの集約結果を後続ステージで参照するパイプライン分析
  • クロスジョインで個別イベントと集約統計を比較する高度な検出パターン
  • GA により本番環境での利用に十分なサポートと安定性を保証

🏗️ アーキテクチャ図

マルチステージクエリでは、各ステージが前段のステージフィールドを参照して段階的にデータを変換する。最終的にルートステージで条件判定を行い、Search または Dashboard に結果を出力する。

⚡ 主要機能

🔗 名前付きステージによるクエリ分割

  • stage <名前> { } 構文で最大 4 つのステージを定義
  • 各ステージは前方のステージのみ参照可能
  • ルートステージは全名前付きステージの後に処理

📤 ステージフィールドによるデータ受け渡し

  • $<ステージ名>.<変数名> で前段の変数にアクセス
  • window_start / window_end でタイムスタンプ参照
  • データ型 (文字列・整数・浮動小数点) はステージ間で保持

🔀 クロスジョインによる統計比較

  • cross join で集約統計と個別イベントを結合
  • limit: 1 のステージと他データの組み合わせ
  • 異常検出やベースラインからの逸脱検知に有効

📊 Dashboards と Search の両対応

  • Search でリアルタイム脅威ハンティング
  • Dashboards で継続的な可視化・モニタリング
  • コンポジット検出ルールを補完する位置付け

💻 クエリ構文例

時間単位の集約結果から、3 時間以上継続して異常なトラフィックを記録した IP ペアを検出するクエリの例。

YARA-L 
stage hourly_stats {
    metadata.event_type = "NETWORK_CONNECTION"
    $source = principal.hostname
    $target = target.ip
    $source != ""
    $target != ""

    match:
        $source, $target by hour

    outcome:
        $count = count(metadata.id)
        $avg_recd_bytes = avg(network.received_bytes)
        $std_recd_bytes = stddev(network.received_bytes)

    condition:
        $avg_recd_bytes > 100 and $std_recd_bytes > 50
}

// ルートステージ: hourly_stats の出力を参照
$src_ip = $hourly_stats.source
$dst_ip = $hourly_stats.target

match:
    $src_ip, $dst_ip

outcome:
    $count = count_distinct($hourly_stats.window_start)

condition:
    $count > 3

🛠️ 設定方法

💡 前提条件: Google SecOps (Chronicle) のアクティブなサブスクリプションと、Search または Dashboards へのアクセス権限が必要です。特別な有効化操作は不要です。

ステップ 1: Search でマルチステージクエリを作成

Google SecOps コンソールにアクセスし、Investigation > Search に移動してクエリエディタにマルチステージクエリを入力して実行します。

YARA-L 
stage user_login_counts {
    $user = principal.user.userid
    metadata.event_type = "USER_LOGIN"
    security_result.action = "ALLOW"

    match:
        $user

    outcome:
        $login_count = count(metadata.id)
}

$login_count = $user_login_counts.login_count
$user = $user_login_counts.user

match:
    $login_count

outcome:
    $num_users = count($user)

ステップ 2: Dashboard ウィジェットとして構成

Dashboard に新しいウィジェットを追加し、マルチステージクエリの結果を可視化します。時間ウィンドウの設定により継続的なモニタリングを実現できます。

✅ メリット

🎯 ビジネス面

  • 脅威検出の高度化: APT やラテラルムーブメントなど複雑な攻撃パターンの検出
  • SOC 運用効率の向上: ベースライン比較で誤検出を低減しアラート精度を向上
  • 調査時間の短縮: 1 つのクエリで完結しインシデント調査のスピード向上

⚙️ 技術面

  • クエリの可読性向上: ステージごとにロジックを分離し構造を明確化
  • 段階的データ変換: データパイプラインのような柔軟な処理が可能
  • リアルタイム分析: Search でリアルタイムの結果を返却可能

⚠️ デメリット・制約事項

制限事項
  • Rules (検出ルール) では非サポート。Search と Dashboards に限定
  • すべてのマルチステージクエリは統計クエリとして動作。非集約イベントをそのまま返すことは不可
  • UDM やエンティティとのジョインはデータサイズにより性能低下の可能性
  • API サポートは EventService.UDMSearch API に限定 (SearchService.UDMSearch は非サポート)
考慮すべき点
  • ステージ間のウィンドウサイズ差が大きい場合、行数の爆発で性能問題が発生する可能性
  • outcome 変数の上限は 20 (オプトインで 50) に制限
  • 統計クエリ実行制限 (120 QPH) が適用。高頻度自動化には注意

💡 ユースケース

🌐 ユースケース 1: 異常なネットワーク接続の検出

シナリオ: SOC チームが、通常とは異なる高トラフィックを維持する IP ペアを検出したい。時間単位で集約した後、3 時間以上継続して高活動が続くペアのみをアラート対象とする。

効果: 単発の高トラフィックではなく、持続的な異常通信パターンのみを抽出し、C2 通信やデータ流出の兆候を高精度に検出。

YARA-L 
stage hourly_stats {
    metadata.event_type = "NETWORK_CONNECTION"
    $src_ip = principal.ip
    $dst_ip = target.ip
    $src_ip != ""
    $dst_ip != ""

    match:
        $src_ip, $dst_ip by hour

    outcome:
        $count = count(metadata.id)
        $avg_recd_bytes = avg(network.received_bytes)
        $std_recd_bytes = stddev(network.received_bytes)

    condition:
        $avg_recd_bytes > 100 and $std_recd_bytes > 50
}

$src_ip = $hourly_stats.src_ip
$dst_ip = $hourly_stats.dst_ip

match:
    $src_ip, $dst_ip

outcome:
    $count = count_distinct($hourly_stats.window_start)

condition:
    $count > 3

👤 ユースケース 2: 通常と異なるログイン頻度の検出

シナリオ: 全ユーザーの平均ログイン回数と比較して、異常に多い (または少ない) ログイン回数のユーザーを特定したい。

効果: クロスジョインで全体統計と個別ユーザーを比較するベースライン分析により、内部不正やアカウント侵害の兆候を早期に発見。

YARA-L 
stage user_login_counts {
    $user = principal.user.userid
    metadata.event_type = "USER_LOGIN"
    security_result.action = "ALLOW"

    match:
        $user

    outcome:
        $login_count = count(metadata.id)
}

stage total_users {
    outcome:
        $count = count($user_login_counts.user)
    limit: 1
}

cross join $total_users, $user_login_counts
$login_count = $user_login_counts.login_count
$user = $user_login_counts.user
$tot_users = $total_users.count

match:
    $login_count

outcome:
    $num_users = count($user)
    $frequency_percent = (count($user) / max($tot_users)) * 100

💰 料金

追加料金なし -- マルチステージクエリ機能自体に追加料金は発生しない。Google SecOps (Chronicle) のサブスクリプションに含まれる Search および Dashboards の機能として提供される。ただし、クエリの実行量やデータ取り込み量に応じた標準料金が適用される。

🌍 利用可能リージョン

Google SecOps が利用可能なすべてのリージョンでマルチステージクエリを使用できる。機能の有効化に特別な操作や設定は不要であり、既存の Google SecOps インスタンスで即座に利用可能。

🔗 関連サービス・機能

コンポジット検出ルール

マルチステージクエリを補完。ルールベースの検出にはコンポジット検出ルール、リアルタイム調査にはマルチステージクエリを使用。

UDM Search

マルチステージクエリの基盤。UDM に正規化されたデータに対してクエリを実行する。

Search Joins

マルチステージクエリ内でデータソース間のジョインを行い、複数ソースのコンテキストを統合。

YARA-L 2.0 検出ルール

同じ YARA-L 言語を使用するが、ルール (リアルタイム検出) ではマルチステージは非サポート。

Google SecOps Dashboards

マルチステージクエリ結果の可視化・モニタリングのためのダッシュボード機能。
