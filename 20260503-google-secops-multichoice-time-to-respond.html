🛡️ Google SecOps SOAR: MultiChoiceQuestion「Time to Respond」オプション強化

📋 概要

Google SecOps SOAR のプレイブックにおける MultiChoiceQuestion ステップの「Time to respond (応答時間)」超過時の動作に、より細かい制御オプションが追加されました。アナリストが指定時間内に回答しなかった場合のプレイブック実行フローを、より柔軟に設計できるようになります。

SOC (Security Operations Center) のアナリストやプレイブック設計者が、インシデント対応の自動化ワークフローにおけるタイムアウト処理を最適化することを目的としています。

❌ Before (従来)

  • 応答時間超過時は「プレイブック停止」のみ
  • タイムアウト時の自動エスカレーションが困難
  • 代替処理パスの定義にワークアラウンドが必要
  • プレイブック中断による対応遅延が発生

✅ After (今回)

  • 3つの選択肢から最適な対応を選択可能
  • 既存ブランチの自動選択で継続実行
  • タイムアウト専用ブランチで独自アクション定義
  • 24時間体制の対応漏れ防止

🏗️ アーキテクチャ図

MultiChoiceQuestion ステップにおける応答時間超過時の3つの処理パスを示します。

sequenceDiagram
    participant PB as Playbook Engine
    participant MCQ as MultiChoiceQuestion Step
    participant AN as Analyst
    participant TO as Timeout Handler

    PB->>MCQ: MultiChoiceQuestion 実行
    MCQ->>AN: 質問を提示 (回答待ち)

    alt アナリストが時間内に回答
        AN->>MCQ: 回答を選択
        MCQ->>PB: 選択されたブランチへ進行
    else Time to respond 超過
        MCQ->>TO: タイムアウト検知
        alt Stop Playbook
            TO->>PB: プレイブック停止
        else Select existing branch
            TO->>PB: 事前定義ブランチへ自動進行
        else Create new branch (NEW)
            TO->>PB: Time to respond exceeded ブランチへ進行
            PB->>PB: 専用アクション実行 (例: エスカレーション)
        end
    end
💡 ポイント: アナリストが時間内に回答した場合は通常フローが実行され、タイムアウト時は設定に応じて「停止」「既存ブランチ選択」「専用ブランチ実行」のいずれかが行われます。

⚡ 主要機能 (3つのオプション)

🛑 Stop Playbook (プレイブック停止) 既存

ユーザーが指定時間内に応答しない場合、プレイブックの実行を完全に停止します。重要な判断が必要で、人間の回答なしに進行すべきでないケースに適しています。

🔄 Select existing branch (既存ブランチの選択) NEW

タイムアウト時に、既に定義されている回答ブランチの1つを自動的に選択してプレイブックを続行します。デフォルトの安全なアクションが明確な場合に有効です（例: 「隔離する」をデフォルトとして自動選択）。

🆕 Create new branch (新規ブランチの作成) NEW

タイムアウト専用の「Time to respond exceeded」ブランチがプレイブックデザイナーに生成されます。通常の回答オプションとは異なる独自のアクション（マネージャーへのメール送信、チケット作成など）を定義可能です。

⏱️ ステップ 3: Time to respond の設定（詳細）

1. サイドドロワーの Settings タブをクリック
2. Time to respond オプションを ON にトグル
3. 日、時間、分で適切な待ち時間を指定
4. If Time to Respond exceeded セクションで以下のいずれかを選択:
    - Stop Playbook: プレイブックを停止
    - Select existing branch: 既存の回答ブランチから1つを選択
    - Create new branch: タイムアウト専用ブランチを新規作成

⚖️ メリットと考慮事項

✅ メリット

🚀 インシデント対応時間の短縮 - アナリスト不在時でも対応が停滞しない
📋 SLA 遵守の向上 - プレイブック停滞がなくなり対応完了率が改善
📧 エスカレーション自動化 - 管理者への自動通知やチケット起票が可能
🔧 設計の柔軟性向上 - ワークアラウンド不要でネイティブ実装可能
🔗 既存互換性 - Stop Playbook 動作はそのまま維持

⚠️ 考慮事項

🌐 段階的ロールアウト - 全リージョンで即時利用可能ではない
🔨 手動設定が必要 - タイムアウトブランチ内のアクションは別途設定
⚡ 誤設定リスク - 不適切なブランチ選択で意図しないアクション実行の可能性
⏰ タイムアウト設定 - 短すぎると検討中でもタイムアウト発動
📝 既存PB移行 - 現在の設定を見直し、新オプションの適切性を評価する必要あり

💡 ユースケース

🦠 ユースケース 1: マルウェア検出時の承認ワークフロー
シナリオ: エンドポイントでマルウェアが検出された際、アナリストに「端末を隔離するか」「調査を続行するか」を質問。アナリストが15分以内に応答しない場合、安全策として自動的に「隔離する」ブランチを選択。
✅ 効果: 夜間やアナリスト不在時でも感染端末が放置されるリスクを最小化。即座に隔離が実行され、後続の調査アクションも自動的に開始。
🎣 ユースケース 2: フィッシングメール報告のエスカレーション
シナリオ: フィッシングメールの報告があった際、Tier 1 アナリストに対応方法を質問。30分以内に回答がない場合、専用の「Time to respond exceeded」ブランチが実行され、Tier 2 マネージャーにメール通知とSlack通知を送信し、ケースの優先度を自動昇格。
✅ 効果: 未対応のフィッシングインシデントが長時間放置されることを防止。エスカレーションパスが標準化され、対応漏れのリスクを排除。
