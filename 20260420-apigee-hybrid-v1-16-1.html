📄 概要

Apigee hybrid v1.16.1 は、AI/LLM 関連のトークン管理ポリシーの hybrid 環境への対応、 Model Context Protocol (MCP) サポートの拡張、 セキュリティ脆弱性の包括的な修正を含む重要なアップデートです。

特に注目すべきは、LLMTokenQuota および PromptTokenLimit ポリシーが Apigee hybrid で利用可能になったことで、ハイブリッド環境で LLM API のトークン使用量を制御できるようになった点です。ParsePayload ポリシーの MCP メソッド対応拡張により、エージェントアプリケーションからのリクエスト処理も強化されました。

💡 Helm チャート統合: パッチリリースは Helm チャートと統合されたコンテナイメージで提供されるため、Helm upgrade コマンド実行時にイメージが自動更新されます。手動でのイメージ変更は通常不要です。

❌ アップデート前の課題

  • ParsePayload のサポートする MCP メソッドが限定的
  • ParsePayload 出力に PII が含まれる可能性
  • 複数コンポーネントに既知の CVE 脆弱性

  • PII 除去によりセキュリティとプライバシーが向上
  • 13+ コンポーネントの CVE 脆弱性を修正

🔷 主要機能

🤖 1. LLMTokenQuota / PromptTokenLimit ポリシーの hybrid 対応

Bug ID: 469900037

Apigee hybrid 環境で LLM API のトークン使用量を制御する 2 つのポリシーが利用可能になりました。

⚡ PromptTokenLimit ポリシー
LLM バックエンドへのトークンスパイクを防止するレート制限ポリシー。ユーザープロンプト内のトークン数を分単位・秒単位で制限します。SpikeArrest に類似していますが、リクエスト数ではなくトークン数を制限対象とします。
📊 LLMTokenQuota ポリシー
LLM API のトークン消費量に対するクォータ管理ポリシー。API プロダクト単位で、時間・日・月などの期間にわたるトークン消費量制限を適用できます。
⚠️ 注意: 両ポリシーのレート超過時には HTTP 429 (Too Many Requests) が返されます。

🔗 2. ParsePayload ポリシーの MCP メソッド拡張

Bug ID: 502577947

ParsePayload ポリシーが、より広範な Model Context Protocol (MCP) メソッドをサポートするように拡張されました。

  • ✔ システムレベルの必須メソッドに対するガバナンスバイパスを実装
  • ✔ tools/list や tools/call などの基本的な MCP オペレーションを制約なしに処理可能
  • ✔ JSON-RPC 2.0 形式のリクエストから論理的オペレーションを抽出し、条件付きルーティングが可能
💡 注: ParsePayload ポリシーは Apigee hybrid v1.16.1 以降でのみ利用可能です。

🔒 3. ParsePayload ポリシー出力からの PII 除去

Bug ID: 503029410

ParsePayload ポリシーの出力から個人識別情報 (PII) が除去され、セキュリティとプライバシーが向上しました。API プロキシのデバッグセッションやログにおいて、機密情報が意図せず露出するリスクが低減されます。

🔧 技術仕様

PromptTokenLimit ポリシー設定例

XML PromptTokenLimit ポリシー設定 
<PromptTokenLimit continueOnError="false" enabled="true" name="PTL-limitTokens-1">
  <DisplayName>Limit Prompt Tokens</DisplayName>
  <Rate>1000pm</Rate>
  <UserPromptSource>{jsonPath('$.contents[-1].parts[-1].text',request.content,true)}</UserPromptSource>
</PromptTokenLimit>

ParsePayload ポリシー設定例 (MCP)

XML ParsePayload MCP ポリシー設定 
<ParsePayload continueOnError="false" enabled="true" name="parse-payload-mcp">
  <Source>request</Source>
  <PayloadType>JSON-RPC-2.0</PayloadType>
  <Protocol>MCP</Protocol>
</ParsePayload>

PromptTokenLimit vs LLMTokenQuota の使い分け

項目 PromptTokenLimit LLMTokenQuota
目的 トークンスパイク防止 (運用トラフィック管理) トークン消費量のクォータ制限 (コスト管理)
制限対象 ユーザープロンプト内のトークン数 レスポンスを含むトークン消費量
適用期間 分単位・秒単位 (短期) 時間・日・月単位 (長期)
類似ポリシー SpikeArrest Quota
HTTP レスポンス 429 Too Many Requests 429 Too Many Requests

🎯 ユースケース

⚡ ユースケース 1: LLM API ゲートウェイでのトークンレート制限

シナリオ: 企業が Apigee hybrid 経由で Vertex AI や他の LLM サービスへの API アクセスを提供しており、特定のクライアントによるトークンの過剰消費を防止したい。

XML PromptTokenLimit 実装例 
<!-- プロキシの Request Flow に PromptTokenLimit を追加 -->
<PromptTokenLimit continueOnError="false" enabled="true" name="PTL-vertexai">
  <DisplayName>Vertex AI Token Rate Limit</DisplayName>
  <Identifier ref="developer.id"/>
  <Rate>5000pm</Rate>
  <UserPromptSource>{jsonPath('$.contents[-1].parts[-1].text',request.content,true)}</UserPromptSource>
  <UseEffectiveCount>true</UseEffectiveCount>
</PromptTokenLimit>
✨ 効果: 開発者ごとに 1 分あたり 5,000 トークンに制限することで、LLM バックエンドの過負荷を防止しつつ、公平なリソース配分を実現。

🔗 ユースケース 2: MCP エージェントリクエストの条件付きルーティング

シナリオ: エージェントアプリケーションが MCP プロトコルを使用して複数のツール/サービスにアクセスする際、リクエストの MCP メソッドに基づいて異なるバックエンドにルーティングしたい。

📈 メリットとデメリット

✅ メリット

ビジネス面

  • LLMTokenQuota で API プロダクト単位のトークン消費量制限が可能に
  • 13+ コンポーネントの CVE 修正でセキュリティ監査対応が容易に
  • PII 除去で GDPR 等のプライバシー規制準拠が改善

技術面

  • PromptTokenLimit で LLM バックエンドへのトークンスパイクを防止
  • MCP 準拠エージェントからのリクエスト処理がより柔軟に
  • Helm チャート統合で手動イメージ変更不要、アップグレード簡素化

⚠️ デメリット・制約事項

制限事項

  • ParsePayload は v1.16.1 以降のみ利用可能 (バックポートなし)
  • ParsePayload はメッセージストリーミング非対応
  • MCP in Apigee は hybrid 組織での利用は対象外 (hosted MCP のみ)

考慮すべき点

  • アップグレード前に Cassandra DB のバックアップを取得すること
  • アップグレード中に新しい環境を作成しないこと
  • ロールバックは以前のバージョンの Helm チャートで逆順実行
