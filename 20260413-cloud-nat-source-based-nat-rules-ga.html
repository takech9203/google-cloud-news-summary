Public NAT ゲートウェイにおけるソース IP ベースの NAT 制御が正式リリース
Cloud NAT GA Networking
2026-04-13

📋 概要

Cloud NAT の Public NAT ゲートウェイにおいて、IPv4 ソースベース NAT ルールが一般提供 (GA) となりました。 これまで Preview として提供されていたこの機能が正式にプロダクション環境での利用が推奨されるステータスに昇格し、SLA の対象となります。

ソースベース NAT ルールは、パケットのソース IP アドレスに基づいて NAT の動作を細かく制御する機能です。 従来はサブネット内の全トラフィックに同一の NAT IP アドレスセットが使用されていましたが、ソースベースルールにより、送信元 IP レンジに応じて異なる NAT IP アドレスを割り当てることが可能になります。

主な対象: ネットワークエンジニア、クラウドアーキテクト、セキュリティ管理者 (マルチテナント環境やサードパーティ接続での NAT IP 使い分けが必要なケース)

🔄 Before / After

❌ Before (アップデート前)

  • サブネット内の全 VM が同じ NAT IP アドレスセットを使用し、ソース IP に基づく使い分けが不可能
  • IP フォワーディング VM の転送パケットにデフォルト NAT ルールが適用されない
  • Preview ステータスのため SLA 対象外で、プロダクション利用に制約

✅ After (アップデート後)

  • GA に昇格し、SLA が適用されるプロダクション対応機能として利用可能
  • ソース IP アドレス範囲ごとに異なる NAT IP を割り当て可能
  • IP フォワーディング VM からの転送パケットに対するソース IP ベースの NAT IP マッピングが定義可能

⚡ 主要機能

🔧 ソースベース NAT ルール (GA)

  • パケットのソース IP アドレスに基づいて NAT 動作を制御
  • IPv4 アドレスのみサポート
  • Common Expression Language (CEL) 構文でルール条件を記述
  • ルール優先度 (0 ~ 65,000) により評価順序を制御

🔀 IP フォワーディング対応

  • IP フォワーディングが有効な VM からの転送パケットに対して、ソース IP に基づく NAT IP 割り当てが可能
  • デフォルトルールは転送パケットには適用されないため、転送パケットの NAT にはソースベースルールの作成が必須

📐 優先度ベースのルール評価

  • 各ルールに 0 (最高優先度) ~ 65,000 (最低優先度) の優先度を設定
  • デフォルトルールの優先度は 65001
  • 複数ルールにマッチする場合は最高優先度のルールが適用

CEL ルール式の例

CEL ソースベースルールの式の例 
# 単一 IP アドレスのマッチ
"source.ip == '10.0.0.25'"

# 複数 IP アドレスの OR マッチ
"source.ip == '10.0.0.25' || source.ip == '10.0.0.26'"

# IP レンジのマッチ
"inIpRange(source.ip, '10.0.2.0/24')"

# IP アドレスと IP レンジの組み合わせ
"source.ip == '10.0.0.25' || inIpRange(source.ip, '10.0.2.0/24')"

🛠️ 設定方法

前提条件

💡 前提条件:
  • Cloud NAT ゲートウェイが Public NAT として作成済みであること
  • NAT IP アドレスの割り当てが手動 (MANUAL_ONLY) に設定されていること
  • Endpoint-Independent Mapping (EIM) が無効であること
  • ソースベースルールに使用する外部 IP アドレスが予約済みであること

ステップ 1: Cloud NAT ゲートウェイの作成

Bash Cloud Router + NAT ゲートウェイの作成 
# Cloud Router の作成
gcloud compute routers create my-router \
  --network=my-network \
  --region=us-central1

# Cloud NAT ゲートウェイの作成 (手動 IP 割り当て)
gcloud compute routers nats create my-nat-gateway \
  --router=my-router \
  --region=us-central1 \
  --nat-all-subnet-ip-ranges \
  --nat-external-ip-pool=NAT_IP_ADDRESS_3

ステップ 2: ソースベース NAT ルールの追加

Bash ソースベース NAT ルールの作成 
# ルール 1: ソース IP レンジ 192.168.1.0/24 に対して NAT_IP_ADDRESS_1 を使用
gcloud compute routers nats rules create 100 \
  --nat=my-nat-gateway \
  --router=my-router \
  --region=us-central1 \
  --match="inIpRange(source.ip, '192.168.1.0/24')" \
  --source-nat-active-ips=NAT_IP_ADDRESS_1

# ルール 2: ソース IP レンジ 192.168.2.0/24 に対して NAT_IP_ADDRESS_2 を使用
gcloud compute routers nats rules create 200 \
  --nat=my-nat-gateway \
  --router=my-router \
  --region=us-central1 \
  --match="inIpRange(source.ip, '192.168.2.0/24')" \
  --source-nat-active-ips=NAT_IP_ADDRESS_2

ルール優先度 100 と 200 を設定し、ソース IP レンジに応じて異なる NAT IP アドレスを割り当てます。これらのルールにマッチしない VM 自身のトラフィックは、デフォルトルール (NAT_IP_ADDRESS_3) で処理されます。

💼 ビジネス面

🏢 マルチテナント対応: テナントごとに異なる NAT IP を割り当て、トラフィック分離と識別が可能
🛡️ SLA による安心感: GA により SLA が適用され、プロダクション環境で正式サポート
📜 コンプライアンス対応: 特定のソース IP レンジに専用外部 IP を割り当て、ホワイトリスト要件に対応

🔧 技術面

⚙️ きめ細かな NAT 制御: ソース IP に基づく NAT IP 使い分けでネットワークポリシーの柔軟性が向上
🔀 IP フォワーディング環境: NVA や SD-WAN ゲートウェイでの転送パケット NAT 制御が可能
📝 CEL による柔軟な条件定義: 直感的なルール式で複雑なマッチ条件も記述可能

⚠️ 制約事項・注意点

制限事項

  • ❌IPv4 アドレスのみサポート。IPv6 には非対応
  • ❌NAT IP アドレス割り当ては手動 (MANUAL_ONLY) のみ。自動割り当て不可
  • ❌Endpoint-Independent Mapping (EIM) 有効時はルール追加不可
  • ❌ソースベースとデスティネーションベースの条件を 1 つのルール内で組み合わせ不可
  • ❌IP CIDR レンジはルール間で重複不可 (1 パケットに 1 ルールのみ適用)
  • ❌デフォルトルールは転送パケットに適用されない。転送パケットには明示的ルール作成が必要

考慮すべき点

  • 各 NAT ルールに VM ごとの最小ポート数が割り当てられるため、ルール数と IP アドレス数のバランスに注意。IP アドレスが不足するとポート割り当てができずトラフィックがドロップされる
  • 同一ルール内で Premium Tier と Standard Tier の IP アドレスを混在させることは不可
  • 0.0.0.0/0 をソースまたはデスティネーションレンジとするルールは作成不可 (デフォルトルールで使用済み)
  • ルールの Active IP アドレスが空の場合、そのルールにマッチする新規接続はドロップされる

💡 ユースケース

🏭 ユースケース 1: ネットワーク仮想アプライアンスからの転送トラフィック NAT 制御

シナリオ: SD-WAN ゲートウェイとして動作する VM (IP フォワーディング有効) が複数拠点の転送トラフィックを処理。拠点ごとに異なる NAT IP を使用し、外部サービスの ACL 要件に対応。

Bash 拠点ごとの NAT IP 割り当て 
# 拠点 A (192.168.1.0/24) のトラフィックに NAT IP 1 を割り当て
gcloud compute routers nats rules create 100 \
  --nat=sdwan-nat --router=sdwan-router --region=us-central1 \
  --match="inIpRange(source.ip, '192.168.1.0/24')" \
  --source-nat-active-ips=SITE_A_NAT_IP

# 拠点 B (192.168.2.0/24) のトラフィックに NAT IP 2 を割り当て
gcloud compute routers nats rules create 200 \
  --nat=sdwan-nat --router=sdwan-router --region=us-central1 \
  --match="inIpRange(source.ip, '192.168.2.0/24')" \
  --source-nat-active-ips=SITE_B_NAT_IP

効果: 拠点ごとに固定の外部 IP が割り当てられ、SaaS の IP ホワイトリスト設定が容易に。送信元拠点の識別も可能。

🌐 ユースケース 2: Cloud WAN 連携でのサードパーティサービス接続

シナリオ: Cloud WAN を利用して複数のネットワークセグメントからサードパーティサービスに接続する際、セグメントごとに異なる NAT IP を割り当てて通信を制御。

効果: サードパーティサービス側で送信元 IP に基づくアクセス制御が可能となり、セキュリティ要件を満たしつつ柔軟なネットワーク構成を実現。
