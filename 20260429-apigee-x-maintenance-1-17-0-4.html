⚡ 主要機能

📅 スケジュールメンテナンスの自動適用

→メンテナンスウィンドウが設定されたインスタンスに対して、7〜21日以内にアップデートが自動適用
→アップグレード予定日は2営業日以内にメール通知で送付
→Week 1 設定: 少なくとも1週間前に通知 / Week 2 設定: 少なくとも2週間前に通知

🔒 セキュリティ修正

→CVE-2025-61726, CVE-2025-61728, CVE-2025-61730, CVE-2025-61731, CVE-2025-61732 の修正

🛡️ 除外条件による保護

→DNS 設定ミスがあるインスタンス (Known Issue 445936920) は自動アップデートの対象外
→削除済み Apigee Java ライブラリを使用しているインスタンスは対象外
→アップデートによる予期しない障害を防止

メンテナンスウィンドウの設定オプション

項目 詳細
メンテナンスウィンドウ 曜日 + 開始時刻 (UTC)
アップデート順序 Week 1 または Week 2
通知タイミング Week 1: 1週間前以上、Week 2: 2週間前以上
アップデート所要時間 数時間 (構成により異なる)
ウィンドウ数制限 インスタンスごとに1つのみ
複数インスタンスの間隔 同一組織内では12時間以上の間隔を推奨

メンテナンスウィンドウの設定 (API)

Bash メンテナンスウィンドウ設定例 
AUTH="Authorization: Bearer $(gcloud auth print-access-token)"
curl -X PATCH \
  -H "$AUTH" \
  -H "Content-Type: application/json" \
  -d '{

      "maintenanceWindows": [
        {
          "day": "SUNDAY",
          "startTime": {
            "hours": 23

        }
      ],
      "maintenanceChannel": "WEEK1"
    }
  }' \
  "https://apigee.googleapis.com/v1/organizations/ORG_ID/instances/INSTANCE_ID?updateMask=maintenanceUpdatePolicy.maintenanceWindows,maintenanceUpdatePolicy.maintenanceChannel"

🛠️ 設定方法

💡 前提条件: Apigee Organization Admin ロール (roles/apigee.admin) または apigee.instances.update 権限が必要です。
1

現在のメンテナンス設定とスケジュールの確認

API を呼び出して scheduledMaintenance フィールドで予定されたメンテナンスの開始時刻を確認します。

2

メンテナンス通知のオプトイン

Google Cloud コンソール > ユーザー設定 > コミュニケーション > 「Apigee, Maintenance window」のメール通知をオンに設定。各ユーザーが個別にオプトイン必要。

3

除外条件の確認

DNS 設定ミスや削除済み Java ライブラリの使用がないか確認し、該当する場合は事前に修正を実施。

  "https://apigee.googleapis.com/v1/organizations/ORG_ID/instances/INSTANCE_ID"

⚖️ メリット・デメリット

✅ メリット
  • 計画的なダウンタイム管理: トラフィックが少ない時間帯にスケジュール可能
  • 事前通知: 2営業日以内に通知でチーム間調整が可能
  • セキュリティ強化: 7件の CVE 脆弱性が修正
  • 段階的ロールアウト: Week 1/2 でステージング先行確認が可能
  • 自動適用: 手動でのバージョンアップ作業不要
⚠️ 制限事項・注意点
  • メンテナンス中はインスタンス作成・環境アタッチ・スケーリング操作が不可
  • 正確な所要時間は事前に見積もれない (通常は数時間)
  • DNS 設定ミスのインスタンスは手動対応が必要
  • 設定変更は進行中のメンテナンスには適用されない
  • 緊急時はウィンドウ外でもアップデートされる可能性あり

💡 ユースケース

🛒 ユースケース 1: 本番環境の段階的アップデート

シナリオ: EC サイト運営企業。トラフィックピークは平日日中、日曜深夜が最もトラフィックが少ない。

Text 段階的ロールアウト戦略 
開発環境:       メンテナンスウィンドウ未設定 (最初にアップデート適用)
ステージング環境: Week 1、日曜 23:00 UTC
本番環境:       Week 2、日曜 23:00 UTC

効果: 開発環境で先にアップデート適用 → 動作確認 → ステージング → 本番と段階的に展開。問題発見時に本番への影響を最小化。

🌐 ユースケース 2: メンテナンス通知による運用チームの対応

シナリオ: 複数リージョンに Apigee インスタンスを展開している企業が、メンテナンス予定を把握し、モニタリング強化の計画を立てたい。

効果: 2営業日以内の通知メールにより、運用チームがメンテナンス日時を把握し、監視体制の強化や関係者への事前連絡が可能。
