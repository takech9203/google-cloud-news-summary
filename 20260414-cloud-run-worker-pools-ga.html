☁️ Cloud Run: Worker Pools が GA に昇格

Cloud Run GA Worker Pools
2026-04-14

📄 概要

Cloud Run の Worker Pools 機能が General Availability (GA) に昇格した。 Worker Pools は Cloud Run Services、Cloud Run Jobs に続く第 3 のリソースタイプであり、 継続的なバックグラウンド処理に特化して設計されている。

ロードバランサーのエンドポイント/URL を持たず、メッセージキューからのメッセージ消費、ストリーム処理、ML モデルのサービング、 長時間バックグラウンドタスクなど、リクエスト駆動でないワークロードを Cloud Run 上でネイティブに実行できる。 これにより、サーバーレスの運用利便性を維持しつつ、従来は GKE や Compute Engine が必要だったバックグラウンド処理ワークロードを Cloud Run に統合できる。

🔄 Before / After

❌ Before
✗Cloud Run では HTTP リクエスト駆動の Services とバッチ実行の Jobs のみ利用可能
✗メッセージキューからのプル型消費やストリーム処理には GKE / Compute Engine が必要
✗Worker Pools は Preview 段階で、SLA の保証やプロダクションサポートが限定的
✅ After
✓Worker Pools が GA となり、SLA の保証を受けた本番環境での利用が可能に
✓Cloud Run で Services / Jobs / Worker Pools の 3 つのワークロードパターンをカバー
✓コンソール、gcloud CLI、Terraform、REST API による完全なライフサイクル管理
✓サイドカーコンテナ、GPU、VPC Direct Egress などエンタープライズ機能が利用可能

⚡ 主要機能

🔨 手動スケーリング

インスタンス数を --instances フラグで明示的に指定。インスタンス数を 0 に設定して Worker Pool を無効化可能。スケーリングモードの変更は新しいリビジョンを作成しない。

📈 CREMA 統合

外部メトリクス (Pub/Sub キュー深度、Prometheus メトリクスなど) に基づくオートスケーリングが可能。CREMA サービスを別途デプロイし、YAML でスケーリングポリシーを定義。

🎬 GPU サポート

Worker Pool インスタンスで GPU を利用可能。ゾーン冗長性オプション (オン/オフ) を選択でき、ML 推論やデータ処理など GPU 活用バックグラウンドワークロードに対応。

📦 サイドカーコンテナ

1 インスタンスあたり最大 10 コンテナをデプロイ可能。メインコンテナとサイドカー間は localhost ポートで通信。共有インメモリボリュームによるファイル共有もサポート。

🔒 VPC ネットワーク統合

VPC Direct Egress によるプライベートネットワークへの接続。VPC 内のリソース (Cloud SQL、Memorystore など) へのセキュアなアクセスが可能。

🛠️ 設定方法

ステップ 1: Worker Pool のデプロイ

Bash 基本的なデプロイ 
gcloud run worker-pools deploy my-worker-pool \
  --image us-docker.pkg.dev/my-project/my-repo/my-worker:latest \
  --region us-central1 \
  --instances 3 \
  --cpu 2 \
  --memory 4Gi

ステップ 2: Worker Pool の状態確認

Bash 一覧表示と詳細表示 
# Worker Pool の一覧表示
gcloud run worker-pools list --region us-central1

# 詳細表示
gcloud run worker-pools describe my-worker-pool --region us-central1

ステップ 3: スケーリングの調整

Bash インスタンス数の変更 / 無効化 
# インスタンス数の変更
gcloud run worker-pools update my-worker-pool \
  --instances 5

# Worker Pool の無効化 (インスタンス数を 0 に設定)
gcloud run worker-pools update my-worker-pool \
  --instances 0

ステップ 4: Terraform によるデプロイ

HCL Terraform リソース定義 
resource "google_cloud_run_v2_worker_pool" "default" {
  name     = "my-worker-pool"
  location = "us-central1"

  template {
    containers {
      image = "us-docker.pkg.dev/my-project/my-repo/my-worker:latest"
    }
  }

  scaling {
    scaling_mode         = "MANUAL"
    manual_instance_count = 3
  }
}

🎯 メリット

💼 ビジネス面

✓インフラ統合によるコスト削減: GKE クラスタや Compute Engine VM の別途管理が不要に
✓サーバーレス運用の拡大: HTTP、バッチ、バックグラウンドの全てを Cloud Run で統一運用
✓GA の信頼性: SLA 保証により、ミッションクリティカルなワークロードにも安心して採用可能

🛠️ 技術面

✓Cloud Run ネイティブの管理性: コンソール、gcloud CLI、Terraform、REST API で統一管理
✓GPU 対応: ML 推論やデータ処理を GPU でサーバーレス実行可能
✓Cloud Deploy 統合: カナリアデプロイを含む CI/CD パイプライン構築が可能
✓柔軟なリソース構成: 最大 8 vCPU、32 GiB メモリまで対応

⚠️ 制約事項・注意点

🚧 制限事項
✗Worker Pools はオートスケーリングをネイティブにサポートしていない。自動スケーリングには CREMA (Preview) との組み合わせが必要
✗ロードバランサーのエンドポイント/URL がないため、外部からの HTTP リクエストを直接受信できない
✗GPU Worker Pools はオートスケーリング対象外。GPU インスタンスは常時課金される
💡 考慮すべき点
→CPU/メモリの料金は Cloud Run Services や Jobs とは異なる料金体系で課金される
→ワークロード変動に応じたスケーリングには CREMA 等の外部スケーリング機構の導入が推奨される
→Worker Pool 名はリージョンとプロジェクト内でユニークである必要があり、既存 Service 名と重複不可

🚀 ユースケース

📦 ユースケース 1: Pub/Sub メッセージの非同期処理
E コマースプラットフォームで、注文確定後の処理 (決済確認、在庫更新、配送手配、メール通知) を Pub/Sub 経由で非同期に処理。Worker Pool が Pub/Sub サブスクリプションからメッセージをプルし、継続的にバックグラウンド処理を実行する。
Python Pub/Sub メッセージ消費の実装例 
import os
from google.cloud import pubsub_v1
from concurrent.futures import TimeoutError

PROJECT_ID = os.environ.get('PROJECT_ID')
SUBSCRIPTION_ID = os.environ.get('SUBSCRIPTION_ID')

subscriber = pubsub_v1.SubscriberClient()
subscription_path = f"projects/{PROJECT_ID}/subscriptions/{SUBSCRIPTION_ID}"

def callback(message):
    data = message.data.decode("utf-8")
    # 注文処理ロジックを実行
    process_order(data)
    message.ack()

streaming_pull_future = subscriber.subscribe(subscription_path, callback=callback)

with subscriber:
    streaming_pull_future.result()
✅ 効果: GKE クラスタを管理することなく、Cloud Run のサーバーレス環境でメッセージ駆動のバックグラウンド処理を実現。CREMA と組み合わせることで、キュー深度に応じた自動スケーリングも可能。
🎬 ユースケース 2: GPU を活用した ML 推論サーバー
画像認識や自然言語処理の ML モデルを Worker Pool 上にデプロイし、他のサービスからの推論リクエストをキュー経由で処理。GPU を利用して高速な推論を実現しつつ、Cloud Run の運用利便性を享受する。
✅ 効果: GPU 対応の Worker Pool により、GKE のノードプール管理なしに GPU ワークロードを運用可能。ゾーン冗長性オプションにより高可用性も確保。
📡 ユースケース 3: ストリームデータのリアルタイム処理
IoT デバイスからのセンサーデータを Managed Kafka を通じて受信し、Worker Pool でリアルタイムに集計・異常検知を行い、結果を BigQuery や Cloud Storage に書き込む。
✅ 効果: Kafka コンシューマーをサーバーレスで運用でき、インフラストラクチャの管理負荷を大幅に削減。VPC Direct Egress でプライベートネットワーク内のデータストアにもセキュアにアクセス可能。

📝 まとめ

Cloud Run Worker Pools の GA は、Cloud Run プラットフォームにとって重要なマイルストーンである。 Services (HTTP リクエスト処理)、Jobs (バッチ処理) に加え、 Worker Pools (継続的バックグラウンド処理) が正式にサポートされたことで、 Cloud Run は幅広いワークロードパターンをカバーするフルスペクトラムのサーバーレスコンピューティングプラットフォームとなった。

Pub/Sub メッセージ消費、ML 推論、ストリーム処理など、これまで GKE や Compute Engine が必要だったバックグラウンド処理ワークロードを検討しているユーザーは、Worker Pools の評価と導入を推奨する。
