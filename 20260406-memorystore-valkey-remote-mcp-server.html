🗄️ Memorystore for Valkey: Remote MCP Server

Memorystore for Valkey Preview MCP
2026-04-06

📋 概要

Google Cloud は Memorystore for Valkey のリモート MCP (Model Context Protocol) サーバーを Preview としてリリースしました。この MCP サーバーにより、LLM、AI アプリケーション、AI 対応の開発プラットフォームから Memorystore for Valkey インスタンスに接続し、Valkey クラスタの管理操作を MCP プロトコルを通じて実行できるようになります。

🤖 Valkey とは: Redis 互換のオープンソースプロジェクトに基づく水平スケーラブルなクラスタサービス。シャーディングによるデータ分散、ベクトル検索機能、クロスリージョンレプリケーション、Cluster Mode Enabled / Disabled の選択などの機能を持ちます。

🏛️ アーキテクチャ図

AI アプリケーションからのリクエストは、IAM / OAuth 2.0 による認証・認可を経て、Model Armor セキュリティスキャンを通過した後、MCP サーバーに到達します。MCP サーバーはリクエストを Memorystore for Valkey API の操作に変換し、クラスタに対する管理操作を実行します。

🚀 設定方法

前提条件

✅Google Cloud プロジェクトが作成済みであること
✅Memorystore API (memorystore.googleapis.com) が有効化されていること
✅必要な IAM ロール (Service Usage Admin, MCP Tool User, Memorystore Admin) が付与されていること

Step 1: MCP サーバーの有効化

Bash MCP サーバーの有効化 
# gcloud CLI beta コンポーネントのインストール
gcloud components install beta

# Memorystore for Valkey の MCP サーバーを有効化
gcloud beta services mcp enable memorystore.googleapis.com \
  --project=PROJECT_ID

Step 2: IAM ロールの付与

Bash IAM ロールの付与 
# MCP Tool User ロールの付与
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="user:USER_EMAIL" \
  --role="roles/mcp.toolUser"

# Memorystore 管理者ロールの付与
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="user:USER_EMAIL" \
  --role="roles/memorystore.admin"

Step 3: MCP クライアントの設定 (Claude Code)

JSON Claude Code 用設定 
{
  "mcpServers": {
    "memorystore-valkey": {
      "url": "https://memorystore.googleapis.com/mcp",
      "transport": "http",
      "auth": {
        "type": "google_credentials",
        "scopes": ["https://www.googleapis.com/auth/cloud-platform"]
      }
    }
  }
}

Step 4: MCP クライアントの設定 (Gemini CLI)

JSON Gemini CLI 用設定 
{
  "name": "memorystore-valkey",
  "version": "1.0.0",
  "mcpServers": {
    "memorystore-valkey": {
      "httpUrl": "https://memorystore.googleapis.com/mcp",
      "authProviderType": "google_credentials",
      "oauth": {
        "scopes": ["https://www.googleapis.com/auth/cloud-platform"]
      },
      "timeout": 30000,
      "headers": {
        "x-goog-user-project": "PROJECT_ID"
      }
    }
  }
}

AI アプリケーション基盤の統合管理

ベクトル検索機能を持つ Valkey インスタンスの管理を AI エージェントに統合することで、RAG パイプラインや LLM キャッシュの基盤管理が一元化される

⚠️ デメリット・制約事項

制限事項

⚠️Preview 機能であるため、本番環境での使用は推奨されない。Pre-GA Offerings Terms が適用
⚠️MCP ツール、パラメータ、説明、出力は予告なく変更・追加・削除される可能性がある
⚠️MCP ツールの出力は非決定的であり、同一の呼び出しでも内容やフォーマットが変わる可能性がある
⚠️Valkey のブロックされたコマンド (CONFIG、SHUTDOWN 等) は MCP サーバーからも制限される可能性がある

考慮すべき点

📌MCP サーバーは管理操作 (コントロールプレーン) に焦点。Valkey のデータ読み書き (GET/SET) やベクトル検索 (FT.SEARCH) とは区別が必要
📌AI エージェントによる破壊的操作 (削除、シャード数削減) のリスクがあるため、IAM 最小権限 + Human-in-the-Middle モード推奨
📌Memorystore for Valkey は Private Service Connect を使用したネットワーク接続が必要。管理操作とデータプレーンの接続要件は異なる

🎯 ユースケース

🔄 UC1: AI エージェントによる Valkey クラスタの水平スケーリング

トラフィック増加時に、AI エージェントが自然言語の指示でシャード数の変更やノードタイプの変更を実行。複数シャードにまたがるデータ再分散を伴う複雑な操作を安全に実行可能。

Text 実装例: 自然言語による指示 
ユーザー: "本番環境の Valkey インスタンス prod-cache の現在の
           シャード数とノードタイプを確認して、シャード数を
           10 から 20 に増やして"

AI エージェント:
  1. GetInstance で prod-cache のインスタンス情報を取得
  2. 現在のシャード数 (10) とノードタイプ (highmem-medium) を確認
  3. UpdateInstance でシャード数を 20 に変更
  4. 変更結果とリシャーディングの進行状況を報告

🔍 UC2: ベクトル検索基盤としての Valkey インスタンス管理

RAG パイプラインのベクトルストアとして Memorystore for Valkey を使用している環境で、AI エージェントがインスタンスの状態確認、レプリカ数の調整、バックアップの管理を実行。クエリスループット低下時にレプリカ数を増やして検索スループットを線形にスケール。

🌐 UC3: マルチサービス AI ワークフローによるデータ基盤の統合管理

AI エージェントが Memorystore for Valkey MCP、Memorystore for Redis MCP、Cloud SQL MCP、BigQuery MCP を組み合わせて、キャッシュ層、データベース層、分析層を横断的に構成確認・管理。Valkey (キャッシュ/ベクトル検索) と Redis (セッション管理/キャッシュ) の両方を統一的に管理可能。

📝 まとめ

Memorystore for Valkey リモート MCP サーバーの Preview リリースは、Google Cloud の MCP エコシステムにインメモリデータストア管理の領域を本格的に加える重要なアップデートです。水平スケーラブルなクラスタアーキテクチャ、ベクトル検索機能、クロスリージョンレプリケーションなどの先進的な機能を持つ Valkey は、AI エージェントによる管理の恩恵が特に大きいサービスです。

推奨アクション: Preview 段階のため、まずは非本番環境で AI エージェントを通じた Valkey クラスタ管理を試験的に実施し、IAM による最小権限の設定と Model Armor の有効化を組み合わせた安全な運用パターンの確立を推奨します。
