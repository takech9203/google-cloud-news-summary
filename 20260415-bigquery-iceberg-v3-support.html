❄️ BigQuery: Apache Iceberg v3 サポート

バイナリ削除ベクトル対応で削除パフォーマンスが大幅向上

BigQuery Apache Iceberg Preview
📅 2026-04-15

📋 概要

BigQuery の Apache Iceberg 外部テーブルが Iceberg バージョン 3 (v3) をサポートしました。 これにより、バイナリ削除ベクトル (binary deletion vectors) を含む Iceberg v3 形式のテーブルを BigQuery から直接クエリできるようになりました。

Apache Iceberg はペタバイト規模のデータテーブルをサポートするオープンソーステーブルフォーマットであり、 オブジェクトストアに保存された単一のデータコピーに対して複数のクエリエンジンからアクセスできます。 BigQuery が Iceberg v3 をサポートすることで、Google Cloud のオープンレイクハウス戦略がさらに強化され、 Spark、Trino、Flink などの OSS エンジンとの相互運用性が向上します。

⚠️ 注意: 本機能は現在 Preview ステータスです。本番環境での使用には注意が必要であり、SLA の適用外となる可能性があります。 フィードバックや質問は biglake-help@google.com まで。

🔄 Before / After 比較

❌ アップデート前

  • Iceberg v2 までのサポートに限定
  • v3 形式のテーブルにアクセス不可
  • merge-on-read 方式の position delete / equality delete
  • データファイルあたり最大 10,000 削除ファイル
  • 大量削除時にクエリパフォーマンスが低下
  • 他エンジンの v3 テーブルを BigQuery から読み取り不可

✅ アップデート後

  • Iceberg v3 形式のテーブルをクエリ可能
  • バイナリ削除ベクトルで効率的な行削除
  • ビットマップベースでコンパクトに削除行を追跡
  • 削除ファイル数の制限が大幅に緩和
  • 削除行スキップが高速化
  • Spark / Flink / Trino の v3 テーブルをシームレスに共有

⭐ 主要機能

📦 1. Iceberg バージョン 3 サポート

✓Iceberg v3 仕様 (Extended Types and Capabilities) に準拠したテーブルの読み取りが可能
✓v3 で導入されたバイナリ削除ベクトルが主要なサポート対象
✓既存の Iceberg v2 テーブル (merge-on-read 対応) も引き続きサポート

⚡ 2. バイナリ削除ベクトル (Binary Deletion Vectors)

✓Iceberg v3 で新たに導入された削除行追跡メカニズム
✓各データファイルに対してビットマップベースのバイナリベクトルで削除行を記録
✓データファイルの書き換え (rewrite) なしに行レベルの削除を効率的に処理
✓クエリ実行時に削除ベクトルを読み込んで対象行をスキップ、merge-on-read のオーバーヘッドを軽減

🌐 3. マルチエンジン相互運用性の強化

✓BigLake Metastore、AWS Glue Data Catalog、JSON メタデータファイルのいずれ経由でも Iceberg v3 テーブルにアクセス可能
✓Spark、Flink、Trino など OSS エンジンが生成した Iceberg v3 テーブルを BigQuery から直接クエリ可能
✓BigQuery Storage API 経由の Managed Spark / Serverless Spark アクセスでもアクセス制御ポリシーが適用

🛠️ 設定方法

📋 前提条件

✓BigQuery Connection API と BigQuery Reservation API が有効化済み
✓Cloud Storage バケットに Iceberg テーブルのデータ/メタデータが格納済み
✓Cloud リソース接続が作成済みで適切なアクセス権限が設定済み
✓IAM ロール: BigQuery Admin (roles/bigquery.admin) + Storage Object Admin (roles/storage.objectAdmin)

Step 1: Cloud リソース接続の作成

Bash bq コマンドで接続を作成 
# Cloud リソース接続の作成
bq mk --connection --connection_type='CLOUD_RESOURCE' \
  --project_id=PROJECT_ID \
  --location=REGION \
  CONNECTION_ID

Step 2: Iceberg 外部テーブルの作成 (JSON メタデータファイル方式)

SQL JSON メタデータファイル方式 
CREATE EXTERNAL TABLE PROJECT_ID.DATASET.TABLE_NAME
  WITH CONNECTION `PROJECT_ID.REGION.CONNECTION_ID`
  OPTIONS (
    format = 'ICEBERG',
    uris = ["gs://BUCKET/PATH/metadata/XXXXX.metadata.json"]
  );

Step 3: Iceberg 外部テーブルの作成 (BigLake Metastore 方式 - 推奨)

SQL BigLake Metastore 方式 (推奨) 
-- BigLake Metastore を使用する場合 (推奨)
-- メタデータの自動同期により手動更新が不要
CREATE EXTERNAL TABLE PROJECT_ID.DATASET.TABLE_NAME
  WITH CONNECTION `PROJECT_ID.REGION.CONNECTION_ID`
  OPTIONS (
    format = 'ICEBERG',
    uris = ["gs://BUCKET/PATH/metadata/iceberg.metadata.json"]
  );
💡 推奨: BigLake Metastore は統合されたマネージドメタストアであり、メタデータの自動同期を提供するため、手動更新が不要になります。

Step 4: クエリの実行

SQL Iceberg v3 テーブルへのクエリ例 
-- バイナリ削除ベクトルが自動的に適用される
SELECT *
FROM PROJECT_ID.DATASET.TABLE_NAME
WHERE partition_column = 'value'
LIMIT 100;

-- タイムトラベル (過去のスナップショットへのアクセス)
SELECT *
FROM PROJECT_ID.DATASET.TABLE_NAME
FOR SYSTEM_TIME AS OF '2026-04-14 00:00:00 UTC';

📈 メリット

💼 ビジネス面

→オープンレイクハウス戦略の加速: Iceberg v3 という最新のオープンフォーマットをサポートすることで、ベンダーロックインを回避しつつ BigQuery の分析能力を活用可能
→マルチエンジン環境の統合: Spark や Flink で書き込んだ Iceberg v3 データを BigQuery で分析可能。データのコピーや ETL パイプライン構築コストを削減
→将来への投資保護: Iceberg コミュニティが v3 を主軸に開発を進めており、早期採用で将来の移行コストを抑制

⚙️ 技術面

→クエリパフォーマンス向上: バイナリ削除ベクトルにより、従来の merge-on-read と比較して削除行スキップが高速化。大量の行削除が発生するテーブルで特に効果的
→ストレージ効率改善: コンパクトなバイナリベクトルにより、メタデータサイズが削減
→削除ファイル制限の緩和: v2 のデータファイルあたりの削除ファイル数制限が v3 で大幅に緩和

⚠️ デメリット・制約事項

🚫 制限事項

✗Preview ステータスのため、SLA 適用外の可能性あり
✗Iceberg v3 の拡張データ型 (ナノ秒 timestamp、unknown、variant、geometry、geography) は BigQuery で未サポート
✗Initial default values / テーブル暗号化キーは未サポート
✗VPC Service Controls 使用クエリはサポート外 (NO_MATCHING_ACCESS_LEVEL エラー)
✗Iceberg 外部テーブルは読み取り専用 (BigQuery から直接 Iceberg v3 形式での書き込み不可)

📝 考慮すべき点

⚠️merge-on-read のコスト: オンデマンド課金の場合、削除済み行を含む全論理バイト数 + 削除ベクトルファイルの読み込みバイト数が課金対象
⚠️メタデータ管理: JSON メタデータファイル方式の場合、テーブル更新時に手動で URI 更新が必要 (BigLake Metastore 推奨)
⚠️Preview の変更リスク: API やサポート範囲が GA までに変更される可能性あり
⚠️書き込み側の対応: バイナリ削除ベクトル使用には、書き込み側エンジン (Spark 等) も Iceberg v3 サポートが必要

🎯 ユースケース

🛒 UC1: マルチエンジンデータレイクでのリアルタイム分析

シナリオ: 大規模 EC サイトが Apache Spark で注文データを Iceberg v3 テーブルに書き込み、キャンセルや返品をバイナリ削除ベクトルで管理。ビジネスアナリストが BigQuery から同じデータをクエリして売上分析レポートを作成。

SQL BigQuery から Iceberg v3 テーブルを直接クエリ 
-- バイナリ削除ベクトルにより、キャンセル済み注文は自動的に除外
SELECT
  DATE(order_timestamp) AS order_date,
  product_category,
  SUM(order_amount) AS total_sales,
  COUNT(*) AS order_count
FROM myproject.datalake.orders_iceberg_v3
WHERE order_timestamp >= '2026-04-01'
GROUP BY order_date, product_category
ORDER BY order_date, total_sales DESC;

効果: データのコピーや ETL パイプラインなしに Spark 管理の Iceberg v3 テーブルを BigQuery から直接分析。大量キャンセル処理がある場合でもクエリパフォーマンスが安定。

🔒 UC2: GDPR / データプライバシー対応のデータ削除

シナリオ: GDPR の「忘れられる権利」に対応するため、Spark ジョブで定期的に対象ユーザーの行を Iceberg v3 テーブルから削除。BigQuery からのクエリでも削除が反映されていることを確認。

効果: バイナリ削除ベクトルにより、データファイル全体の書き換えなしに行レベル削除が記録され、BigQuery からのクエリでも削除が即座に反映。プライバシーコンプライアンス要件を満たしつつ、分析ワークロードへの影響を最小限に抑制。

🔄 UC3: 段階的な Iceberg v2 から v3 への移行

シナリオ: 既存の Iceberg v2 テーブルで削除ファイルの増加によるクエリパフォーマンス低下が課題。段階的に Iceberg v3 へ移行し、バイナリ削除ベクトルのメリットを享受したい。

効果: BigQuery は v2/v3 の両方をサポートするため、テーブル単位で段階的に v3 へ移行可能。移行後は削除ファイル管理のオーバーヘッドが軽減され、クエリパフォーマンスが向上。

スロット消費に基づいて課金
(スロット時間あたりの料金)

項目 課金対象
データファイル読み取り全論理バイト数 (削除済み行を含む)
削除ベクトルファイル読み取り削除行を特定するために読み込む論理バイト数
