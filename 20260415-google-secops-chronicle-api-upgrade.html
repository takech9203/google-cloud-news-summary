Google SecOps Chronicle API v1beta SOAR API 統合
2026-04-15
Google Cloud

📋 概要

Google Security Operations (SecOps) は、Chronicle API の大規模な統合とアップグレードを発表しました。本アップデートには 2 つの核心があります。

第一に、これまで独立して存在していたレガシー SOAR API のリソースが Chronicle API に統合され、単一の統一 API サーフェスが実現しました。第二に、Chronicle API の多数のリソースがアルファ (v1alpha) からベータ (v1beta) にアップグレードされました。このベータ昇格は Google の API Improvement Proposals (AIP-181) に基づく API 安定性のシグナルであり、本番環境での採用を可能にするものです。

✅ アップグレードの核心: SIEM + SOAR の全機能を Chronicle API v1beta 一本で操作可能に。80 以上のリソースがベータ昇格し、AIP-181 に基づく後方互換性が保証されました。Google SecOps プラットフォーム史上最大規模の API アップグレードです。

🔄 Before / After 比較

❌ Before (統合前)

  • 🔴 SOAR API と Chronicle API (旧 Backstory API) が別々に存在し、異なるエンドポイント・認証方式・データモデルを使い分ける必要があった
  • 🔴 アルファ版 API は互換性のない変更が予告なく行われる可能性があり、本番環境での採用にリスクがあった
  • 🔴 レガシー SOAR API は Swagger ベースの独自エンドポイントで、Google Cloud IAM や Monitoring との統合が困難
  • 🔴 SOAR API は API キーベースの認証に依存し、サービスアカウント認証との整合性がなかった

✅ After (統合後)

  • 🟢 Chronicle API 一本で SIEM + SOAR の全機能をプログラマティックに操作可能。開発・保守コストが大幅削減
  • 🟢 v1beta への昇格により後方互換性が保証され、本番環境で安心して採用可能
  • 🟢 Google Cloud IAM、Cloud Monitoring、Cloud Audit Logs とのシームレスな統合が実現
  • 🟢 AIP 標準に基づくリソース指向の一貫した設計により、クライアントライブラリを通じた開発が簡素化

⚡ 主要機能

🔗 レガシー SOAR API の統合

従来の SOAR API (siemplify-soar.com ドメイン) が Chronicle API (googleapis.com ドメイン) に統合。ケース管理、アラート処理、インテグレーション管理、コネクタ操作など全 SOAR 機能が Chronicle API から利用可能に。

⚠️ レガシー SOAR API は 2026 年 9 月 30 日まで並行稼働。早期移行を推奨。

📈 大規模ベータアップグレード

AIP-181 に基づく API 安定性の保証。80 以上のリソースが一斉にベータへ昇格し、本番環境での採用が正式サポート。リソース指向の URL 構造 (projects/{project}/locations/{location}/instances/{instance}/...) による一貫した設計。

🔐 Google Cloud ネイティブ認証

レガシー API キー認証から Google Cloud IAM ベースの認証に移行。サービスアカウント認証およびワークロード ID フェデレーションに対応。きめ細かなアクセス制御が IAM ロールで管理可能に。

🛒 Marketplace 連携更新

SentinelOneV2、CrowdStrike Falcon、ServiceNow、Zscaler、Mandiant Threat Intelligence の 5 つのインテグレーションが同時にアップデートされ、サードパーティ連携が強化。

🛠️ 設定方法

前提条件

✓Google Cloud プロジェクトで Chronicle API が有効化されていること
✓適切な IAM ロールが付与されたサービスアカウントが作成されていること
✓Google SecOps インスタンス ID を把握していること

ステップ 1: 認証の設定

Bash サービスアカウント作成と IAM ロール付与 
# サービスアカウントの作成
gcloud iam service-accounts create chronicle-api-client \
    --display-name="Chronicle API Client"

# 必要なロールの付与
gcloud projects add-iam-policy-binding PROJECT_ID \
    --member="serviceAccount:chronicle-api-client@PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/chronicle.editor"

# サービスアカウントキーの生成
gcloud iam service-accounts keys create key.json \
    --iam-account=chronicle-api-client@PROJECT_ID.iam.gserviceaccount.com

ステップ 2: v1beta エンドポイントへの移行

Bash v1alpha から v1beta への切り替え 
# 旧 v1alpha エンドポイント (移行前)
curl -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  "https://chronicle.us.rep.googleapis.com/v1alpha/projects/PROJECT_ID/locations/us/instances/INSTANCE_ID/cases"

# 新 v1beta エンドポイント (移行後)
curl -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  "https://chronicle.us.rep.googleapis.com/v1beta/projects/PROJECT_ID/locations/us/instances/INSTANCE_ID/cases"

ステップ 3: レガシー SOAR API からの移行

Bash SOAR API から Chronicle API への移行 
# 旧 SOAR API エンドポイント (移行前)
# https://xxxx.siemplify-soar.com/api/external/v1/cases

# 新 Chronicle API エンドポイント (移行後)
curl -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  "https://chronicle.us.rep.googleapis.com/v1beta/projects/PROJECT_ID/locations/us/instances/INSTANCE_ID/cases"

レガシー SOAR API の siemplify-soar.com ドメインから googleapis.com ドメインへ変更し、認証方式を API キーから IAM 認証に切り替えます。

💻 統一 API 使用例

統一 Chronicle API クライアントにより、単一の認証情報で SIEM 検索から SOAR ケース管理まで一貫した自動化パイプラインを構築できます。

Python 統一 Chronicle API による SOAR ワークフロー自動化 
from google.cloud import chronicle_v1beta

# 統一 Chronicle API クライアントの初期化
client = chronicle_v1beta.ChronicleServiceClient()
instance = f"projects/{project_id}/locations/us/instances/{instance_id}"

# 1. UDM 検索でイベントを取得 (旧 Backstory API 機能)
events = client.udm_search(instance=instance, query="ip = '10.0.0.1'")

# 2. ケースを作成 (旧 SOAR API 機能)
case = client.create_case(parent=instance, case={
    "display_name": "Suspicious activity from 10.0.0.1",
    "priority": "HIGH",
})

# 3. タスクを割り当て (旧 SOAR API 機能)
task = client.create_task(parent=instance, task={
    "display_name": "Investigate source IP",
    "assignee": "analyst@example.com",
})

💡 ユースケース

🔄 ユースケース 1: SOAR ワークフローの完全自動化

シナリオ: セキュリティ運用チームが、アラート検知からケース作成、調査、対応アクション実行までを統一 API で自動化したい。これまでは SIEM 系の操作に Chronicle API、SOAR 系の操作にレガシー SOAR API と、2 つの API を使い分ける必要があった。

効果: 単一の API クライアントと認証情報で、SIEM 検索から SOAR ケース管理まで一貫した自動化パイプラインを構築でき、開発・保守コストが削減される。

🤝 ユースケース 2: パートナー製品の本番統合

シナリオ: セキュリティベンダーが自社製品と Google SecOps を統合する製品を開発しているが、これまでアルファ API の互換性リスクにより本番リリースを見送っていた。

効果: v1beta への昇格により API の安定性が保証されるため、本番環境向けの製品統合を安心してリリースできる。AIP 標準に準拠した一貫したインターフェースにより、新しいリソースが追加された場合でも予測可能なパターンで対応できる。

✅ メリット

📊 ビジネス面

→開発・運用コストの削減: API サーフェスの一本化により、開発チームが習得・保守すべき API が 1 つに集約
→本番環境への安心した採用: ベータ安定性保証により、API の互換性リスクを気にせず本番ワークフローに組み込める
→エコシステムの拡大: 統一 API によりサードパーティ製品やカスタムツールとの統合が容易に

🔧 技術面

→Google Cloud ネイティブ統合: IAM による細かなアクセス制御、Cloud Monitoring、Cloud Audit Logs が標準利用可能
→AIP 標準準拠: リソース指向の URL 構造、標準的な CRUD 操作、ページネーション、フィールドマスク
→クライアントライブラリ: Python、Java、Go、Node.js などで開発を簡素化
→データレジデンシー: リージョナルエンドポイントでデータの処理場所を制御

⚠️ デメリット・制約事項

制限事項
  • レガシー SOAR API は 2026 年 9 月 30 日に完全停止 -- それまでにすべてのスクリプトとインテグレーションを移行する必要がある
  • v1alpha エンドポイントは引き続き利用可能だが、ベータにアップグレードされたリソースは v1beta の使用を強く推奨
  • Cloud Armor によるレート制限 (900 リクエスト/分/IP) はベータ API にも適用
  • 一部のレガシー SOAR API エンドポイントは Chronicle API に完全に 1:1 マッピングされない場合がある
💡 考慮すべき点
  • 既存の SOAR API を使用するカスタムスクリプトやサードパーティインテグレーションは、エンドポイント URL、認証方式、データモデルの 3 つの観点で移行作業が必要
  • SOAR の権限グループ管理が Google Cloud IAM に移行されるため、既存のアクセス制御設計の見直しが必要になる場合がある
  • ステージング環境でのテストを十分に行ったうえで本番環境への移行を推奨
  • Webhook の URL 更新も必要 -- レガシードメイン (siemplify-soar.com) から googleapis.com ドメインに変更が必要

📝 まとめ

今回の Chronicle API 統合とベータアップグレードは、Google SecOps プラットフォームの API 戦略における重要なマイルストーンです。レガシー SOAR API の統合により API サーフェスが一本化され、80 以上のリソースのベータ昇格により本番環境での安定した利用が正式にサポートされました。

セキュリティチームおよびパートナーには、レガシー SOAR API の廃止期限 (2026 年 9 月 30 日) を見据え、早期に Chronicle API v1beta への移行を開始することを強く推奨します。まずは API マイグレーションガイドを確認し、ステージング環境でのテストから着手するのがよいでしょう。
