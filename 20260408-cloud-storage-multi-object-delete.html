🪣 Cloud Storage: マルチオブジェクト削除 XML API

Cloud Storage XML API GA
2026-04-08
Cloud Storage

📋 概要

Google Cloud は、Cloud Storage の XML API においてマルチオブジェクト削除機能の一般提供 (GA) を開始しました。この新機能により、1 回のリクエストで最大 1,000 個のオブジェクトを一括削除できるようになります。

この機能は Amazon S3 互換のマルチオブジェクト削除 API (POST /?delete) と互換性があるため、S3 対応のツールやライブラリをそのまま Cloud Storage エンドポイントに向けて利用できます。S3 からの移行やマルチクラウド環境での運用において大きなメリットがあります。

💡 対象ユーザー: 大量のオブジェクト削除を頻繁に行うデータパイプライン運用者、S3 から Cloud Storage への移行を計画しているチーム、マルチクラウド環境でストレージ操作を統一したいユーザーにとって特に重要なアップデートです。

🔄 Before / After 比較

❌ Before (アップデート前)

  • XML API では 1 回のリクエストで 1 つのオブジェクトしか削除できない
  • 大量のオブジェクト削除に多数の HTTP 接続が必要
  • S3 の POST /?delete に相当する機能がなく、移行時にコードの書き換えが必要
  • 大量削除には JSON API バッチリクエストや Object Lifecycle Management など別アプローチが必要

✅ After (アップデート後)

  • 1 回のリクエストで最大 1,000 個のオブジェクトを一括削除
  • HTTP 接続数とオーバーヘッドが大幅に削減
  • S3 互換 API (POST /?delete) をそのまま利用可能
  • S3 対応ツールはエンドポイント変更のみで移行完了

🏗️ アーキテクチャ図

マルチオブジェクト削除 API は、クライアントからの単一の POST リクエストで複数のオブジェクトを一括削除し、各オブジェクトの削除結果を XML レスポンスとして返します。

sequenceDiagram
    participant Client as クライアント (S3 互換ツール)
    participant GCS as Cloud Storage XML API
    participant Bucket as Cloud Storage バケット

    Client->>GCS: POST /?delete HTTP/1.1
Host: BUCKET.storage.googleapis.com
[Delete XML Body: 最大 1,000 オブジェクト]
    GCS->>Bucket: オブジェクト 1 削除
    GCS->>Bucket: オブジェクト 2 削除
    GCS->>Bucket: オブジェクト N 削除
    GCS-->>Client: HTTP 200
[DeleteResult XML: 各オブジェクトの結果]

    Note over Client,Bucket: 従来: N 個のオブジェクト = N 回の DELETE リクエスト / 今回: N 個 = 1 回の POST (N 1,000 以下)

⚡ 主要機能

🗑️ マルチオブジェクト一括削除

→1 回の HTTP リクエストで最大 1,000 個のオブジェクトを削除可能
→XML API の POST /?delete エンドポイントを使用
→各オブジェクトの削除結果 (成功/失敗) がレスポンスに含まれる

🔗 Amazon S3 互換性

→Amazon S3 の Multi-Object Delete API と完全互換
→既存の S3 対応ツール (AWS CLI, Boto3, 各種 S3 SDK) のエンドポイント変更のみで利用可能
→HMAC キー認証による S3 互換認証に対応

⚙️ 既存ワークフローとの統合

→Cloud Storage の HMAC キーを設定し、エンドポイントを storage.googleapis.com に変更するだけで利用開始
→OAuth 2.0 Bearer トークンおよび HMAC キー (V4 署名) の両方に対応
→Quiet モードでエラーのみのレスポンスに切り替え可能

リクエストボディの形式

XML マルチオブジェクト削除リクエスト 
<?xml version="1.0" encoding="UTF-8"?>
<Delete>
  <Quiet>false</Quiet>
  <Object>
    <Key>object-key-1</Key>
  </Object>
  <Object>
    <Key>object-key-2</Key>
  </Object>
  <!-- 最大 1,000 個の Object 要素 -->
</Delete>

🛠️ 設定方法

前提条件: Cloud Storage API が有効化済み / 対象バケットへの storage.objects.delete 権限を持つアカウント / 認証情報 (OAuth 2.0 トークンまたは HMAC キー)

ステップ 1: HMAC キーの作成 (S3 互換ツール利用の場合)

Bash Cloud Storage の HMAC キーを作成 
# Cloud Storage の HMAC キーを作成
gcloud storage hmac create SERVICE_ACCOUNT_EMAIL

出力される Access ID と Secret を S3 互換ツールの認証情報として設定します。

ステップ 2: curl でのマルチオブジェクト削除

Bash curl でマルチオブジェクト削除リクエストを送信 
curl -X POST "https://BUCKET_NAME.storage.googleapis.com/?delete" \
  -H "Authorization: Bearer $(gcloud auth print-access-token)" \
  -H "Content-Type: application/xml" \
  -d '<?xml version="1.0" encoding="UTF-8"?>
<Delete>
  <Quiet>false</Quiet>
  <Object>
    <Key>path/to/object1.txt</Key>
  </Object>
  <Object>
    <Key>path/to/object2.txt</Key>
  </Object>
</Delete>'

Quiet 要素を true に設定すると、エラーが発生したオブジェクトのみがレスポンスに含まれ、レスポンスサイズが削減されます。

ステップ 3: AWS CLI での利用

Bash AWS CLI でのマルチオブジェクト削除 (Cloud Storage エンドポイント) 
# AWS CLI のエンドポイントを Cloud Storage に設定
aws s3api delete-objects \
  --bucket BUCKET_NAME \
  --delete '{"Objects":[{"Key":"object1.txt"},{"Key":"object2.txt"}]}' \
  --endpoint-url https://storage.googleapis.com

AWS CLI や Boto3 を使用する場合は、エンドポイント URL を https://storage.googleapis.com に変更し、認証情報に Cloud Storage の HMAC キーを使用します。

🐍 サンプルコード (Python / Boto3)

ETL パイプラインの中間ファイルを一括削除する実装例です。

Python Boto3 を使ったマルチオブジェクト削除 
import boto3

# Cloud Storage エンドポイントを指定
s3_client = boto3.client(
    's3',
    endpoint_url='https://storage.googleapis.com',
    aws_access_key_id='GOOG_ACCESS_KEY_ID',
    aws_secret_access_key='GOOG_SECRET'
)

# 最大 1,000 個のオブジェクトを一括削除
objects_to_delete = [{'Key': f'tmp/stage/{i}.parquet'} for i in range(1000)]
response = s3_client.delete_objects(
    Bucket='my-data-pipeline-bucket',
    Delete={'Objects': objects_to_delete}
)
✅ ポイント: S3 SDK のエンドポイントを storage.googleapis.com に変更し、HMAC キーを認証情報に設定するだけで、既存の S3 コードをそのまま利用できます。

✨ メリット

💼 ビジネス面

✓運用コスト削減 - 大量オブジェクト削除時のリクエスト数が最大 1/1,000 に削減され、API 呼び出しコストと時間を節約
✓S3 移行の加速 - S3 からの移行時に削除処理のコード変更が不要。移行プロジェクトの工数を削減
✓マルチクラウド対応 - S3 互換 API により、同一ツール・スクリプトで複数クラウドのストレージを操作可能

⚙️ 技術面

✓HTTP 接続数の削減 - 1,000 個のオブジェクト削除に必要な接続数が 1,000 から 1 に削減
✓処理時間の短縮 - ラウンドトリップ削減により、大量削除処理のスループットが向上
✓S3 互換エコシステムの活用 - 既存の S3 互換ツール・SDK・ライブラリをエンドポイント変更のみで利用

⚠️ 制約事項・注意点

制限事項

⚠️1 リクエストあたりの削除上限は 1,000 オブジェクト。超える場合は複数リクエストに分割が必要
⚠️XML API 経由でのみ利用可能。JSON API や gcloud CLI からは直接利用できない
⚠️10 万個以上の一括削除は、Object Lifecycle Management やコンソールからの一括削除の方が効率的な場合あり

考慮すべき点

🔍ソフトデリートが有効なバケットでは、削除オブジェクトがデフォルト 7 日間保持され、保持期間中のストレージ料金が発生する
🔍オブジェクトバージョニングが有効なバケットでは、削除操作の動作が異なるため注意が必要
🔍オブジェクトに保持ポリシーやオブジェクトホールドが設定されている場合、削除が失敗する可能性がある

🎯 ユースケース

📊 ユースケース 1: データパイプラインの一時ファイルクリーンアップ

シナリオ: ETL パイプラインで生成された中間ファイル (Parquet, CSV など) を処理完了後に一括削除する必要がある場合
✅ 効果: 1,000 回の個別リクエストが 1 回に集約され、クリーンアップ処理の所要時間とリクエストコストが大幅に削減される

🔄 ユースケース 2: S3 からの移行における既存ワークフローの再利用

シナリオ: Amazon S3 を利用中のシステムを Cloud Storage に移行する際、既存の S3 対応の削除スクリプトをそのまま利用したい場合
✅ 効果: エンドポイントと認証情報の変更のみで既存コードを再利用でき、移行の工数とリスクを最小化できる

📝 ユースケース 3: ログファイルの定期ローテーション

シナリオ: アプリケーションログを Cloud Storage に保存しており、保持期間を超えたログファイルを定期的に大量削除する場合
✅ 効果: 日次・週次のログ削除ジョブにおいて、数百~数千のログファイルを効率的に削除でき、ジョブの実行時間が短縮される

💰 料金情報

マルチオブジェクト削除 API のリクエスト料金は、Cloud Storage のオペレーション料金に基づきます。削除オペレーションは通常無料ですが、リクエストに関連する料金が発生する場合があります。

→ストレージクラスに応じたクラス A / クラス B オペレーション料金が適用
