📋 概要

Google Cloud Deployment Manager が 2026年3月31日 をもって廃止されたことに伴い、Cloud Marketplace 経由の Apigee Drupal Portal のデプロイおよび管理機能に重大な中断が発生しています。

現在、Google Cloud は Deployment Manager ベースのソリューションを Infrastructure Manager (Terraform ベース) に移行する作業を進めていますが、移行期間中は一部の機能が利用できない状態です。

影響を受ける対象: Apigee Developer Portal Kickstart を Cloud Marketplace 経由で運用している組織。特に VPC Service Controls 環境内でポータルをホストしている組織。

Before (課題)

  • Apigee Drupal Portal は Deployment Manager に依存した Marketplace ソリューションだった
  • Deployment Manager の廃止スケジュールに対し後継ソリューションの準備が間に合わなかった
  • Marketplace UI を通じたデプロイメント管理が Deployment Manager に密結合

After (改善)

  • Infrastructure Manager (Terraform ベース) への移行が進行中
  • Terraform HCL による構成管理、Git リポジトリとの連携が可能に
  • 移行完了まで個別リソースの直接管理で運用を継続可能

Impact -- 主要な影響

1. 新規デプロイの失敗

2026年4月1日以降、Cloud Marketplace の「Deploy」ボタンを使用した新規 Apigee Drupal Portal インスタンスのデプロイは失敗します。Deployment Manager API が廃止されているため、新しいデプロイメントの作成は不可能です。

2. 既存デプロイメントの管理制限

基盤リソース (VM, Cloud SQL, Filestore) は引き続き正常に動作しますが、Marketplace UI からの管理操作および gcloud deployment-manager コマンドによる操作は利用不可です。

3. ワークアラウンド: 直接管理

構成変更や管理タスクは、個別の Google Cloud リソースに対して直接実行する必要があります。各サービスのコンソールまたは CLI で個別に管理してください。

Deployment Manager vs Infrastructure Manager

項目 Deployment Manager (廃止) Infrastructure Manager (移行先)
構成言語 YAML + Jinja/Python テンプレート Terraform HCL
状態管理 内部管理 Cloud Storage に保存
ロールバック 以前のデプロイメントへのロールバック対応 以前の Terraform 構成を再デプロイ (手動プロセス)
プレビュー機能 デプロイ前の変更プレビュー CreatePreview で Terraform plan を確認
ソースコード管理 各種オプション Git リポジトリ、Cloud Storage バケット
Google Cloud 連携 各種サービスと連携 Terraform プロバイダー経由 (より広範なカバレッジ)

ワークアラウンド: 既存デプロイメントの手動管理

Step 1: 既存リソースの確認

Bash 
# Compute Engine インスタンスの確認
gcloud compute instances list --filter="name~drupal"

# Cloud SQL インスタンスの確認
gcloud sql instances list

# Filestore インスタンスの確認
gcloud filestore instances list

Step 2: Compute Engine (MIG) の管理

Bash 
# マネージドインスタンスグループの確認
gcloud compute instance-groups managed list

# インスタンスの詳細確認
gcloud compute instances describe INSTANCE_NAME --zone=ZONE

VM のスケーリングや構成変更は、Compute Engine コンソールまたは gcloud compute コマンドで直接実行します。

Step 3: Cloud SQL の管理

Bash 
# Cloud SQL インスタンスの詳細確認
gcloud sql instances describe INSTANCE_NAME

# バックアップの作成
gcloud sql backups create --instance=INSTANCE_NAME

Step 4: Drupal コードのバックアップ

Bash 
# Compute Engine インスタンスに SSH 接続
gcloud compute ssh INSTANCE_NAME --zone=ZONE

# コードのエクスポート (インスタンス内で実行)
/opt/apigee/scripts/export-code.sh

カスタマイズしたコードやモジュールは Filestore にバックアップされます。

メリットとデメリット

メリット

ビジネス面

  • 基盤リソースは影響を受けず、Developer Portal は引き続き稼働
  • 移行完了後は Terraform ベースのモダンな IaC 管理が可能に

技術面

  • Terraform の豊富なモジュール・プロバイダー・ツールチェーンを活用
  • Git リポジトリ連携で GitOps ワークフローが実現可能

デメリット・制約

  • 新規 Apigee Drupal Portal のデプロイが不可能
  • Marketplace UI からの一元管理が利用不可
  • gcloud deployment-manager コマンドが全て利用不可
  • 移行期間の終了時期は未定
  • 移行期間中は個別リソース管理が必要で運用負荷が増加
  • 新規プロジェクトでは手動リソースプロビジョニングが必要

ユースケース

ユースケース 1: 既存ポータルの運用継続

シナリオ: 既に Apigee Drupal Portal を Cloud Marketplace 経由でデプロイし、本番運用している組織

Bash 
# 現在のリソース状態を確認
gcloud compute instances list --filter="name~drupal"
gcloud sql instances list
gcloud filestore instances list

# 必要に応じて Cloud SQL のバックアップを定期実行
gcloud sql backups create --instance=PORTAL_SQL_INSTANCE
効果: 基盤リソースは影響を受けないため、ポータルは正常に稼働し続けます。ただし、スケーリングや構成変更は個別リソースを直接管理する必要があります。

ユースケース 2: 新規ポータルの代替デプロイ

シナリオ: 新規に Apigee Drupal Portal を構築する必要がある組織

対応: Cloud Marketplace の「Deploy」ボタンは使用できないため、個別リソースを手動でプロビジョニングするか、Drupal ホスティングパートナーの利用を検討します。

効果: 手動プロビジョニングの場合は運用負荷が増加しますが、VPC Service Controls が不要な場合は Drupal ホスティングパートナーの利用により運用オーバーヘッドを削減できます。

料金への影響

料金に直接的な影響はありません。 既存のリソース (Compute Engine, Cloud SQL, Filestore, Load Balancer) の料金は従来通り発生します。Deployment Manager 自体は無料サービスであったため、廃止による料金変更はありません。

Infrastructure Manager への移行完了後の料金体系については Infrastructure Manager の料金ページ を参照してください。
