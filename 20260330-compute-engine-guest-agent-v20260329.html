📋 概要

Guest Agent v20260329.00 がリリースされました。本バージョンでは、ISV テレメトリ収集を行う新しいローカル拡張 guest telemetry の追加、ローカルプラグインの デフォルト有効化、gRPC 通信における connection_type 設定 の導入が含まれています。

Guest Agent は Compute Engine インスタンスのゲスト環境の中核コンポーネントであり、ネットワーク設定、SSH アクセス管理、メタデータアクセス、テレメトリ収集を担います。バージョン 20250901.00 以降、モノリシック設計からプラグインベースのアーキテクチャへ移行しており、今回のリリースはこのモジュラー設計をさらに推進するものです。

🔄 Before / After 比較

❌ Before (v20260329 未満)

  • ローカルプラグインはデフォルトで無効 (enable_local_plugins = false)
  • ISV テレメトリデータを収集する標準的な仕組みがなかった
  • プラグインの接続タイプ (UDS/TCP) を設定で切り替える方法がなかった
  • gRPC クライアントが null の場合にエージェントが異常終了
  • HTTPS 認証情報セットアップがレディネスシグナルをブロック

✅ After (v20260329.00)

  • ローカルプラグインがデフォルトで有効 (enable_local_plugins = true)
  • guest telemetry 拡張で ISV テレメトリを自動収集
  • connection_type 設定で UDS/TCP から選択可能に
  • gRPC クライアント null でもエージェントが安全に動作継続
  • レディネスシグナルが迅速に送信されるよう改善

🛠️ アーキテクチャ図

✨ 新機能

📡 1. Guest Telemetry ローカル拡張

→インスタンス上で稼働する ISV (Independent Software Vendor) ソフトウェアに関するテレメトリデータを収集する新しいローカル拡張
→既存システムテレメトリ (エージェントバージョン、OS 情報、カーネル情報) に加え、ISV 固有の情報を収集
→ローカル拡張として動作し、VM Extension Manager を介さずにエージェントマネージャーが直接管理

🔌 2. enable_local_plugins のデフォルト値変更

→デフォルト値が false から true に変更
→ローカルにバンドルされた拡張 (guest telemetry を含む) が追加の設定なしで自動的に有効化
→必要に応じて false に設定することで無効化可能

🔗 3. connection_type 設定フラグの導入

→PluginConfig セクションに新しい connection_type フラグを追加
→UDS (Unix Domain Socket) と TCP の2種類の接続方式をサポート
→プラグインとエージェントマネージャー間の gRPC 通信方式を柔軟に選択可能

🔧 バグ修正

🛡️ gRPC クライアント null 時の異常終了防止

拡張機能の gRPC クライアントが null の場合に Guest Agent が異常終了する問題を修正。null チェックの追加により、エージェントの安定性が大幅に向上。

⚡ HTTPS 認証情報セットアップの処理順序改善

メタデータサーバーへの HTTPS アクセス用認証情報のセットアップ処理を移動。レディネスシグナル (systemd-notify --ready) の送信がブロックされなくなり、依存サービスの起動が高速化。

📝 コアプラグインの起動ログ追加

コアプラグインが起動時にログを出力するように改善。トラブルシューティング時のシグナルが改善され、起動シーケンスの可視性が向上。

🛠️ ローカルバンドル拡張のプロセス管理改善

ローカルにバンドルされた拡張がエージェントマネージャーの直接の子プロセスとして動作するように変更。プロセスライフサイクル管理の信頼性が向上し、孤児プロセスの発生を防止。

📚 設定方法

💡 前提条件: Compute Engine インスタンスが Google 提供のパブリック OS イメージを使用し、ゲスト環境がインストール済みであること。

ステップ 1: Guest Agent のバージョン確認

Bash Linux でのバージョン確認 
# ステータス確認
sudo systemctl status google-guest-agent-manager

# ログからバージョンを確認
journalctl -u google-guest-agent-manager | grep "GCE Agent Started"
PowerShell Windows でのバージョン確認 
# サービスステータス確認
Get-Service GCEAgentManager | Select-Object Status, DisplayName

# イベントログからバージョン確認
Get-EventLog -LogName Application -Source GCEGuestAgent -Newest 5

ステップ 2: Guest Agent の更新 (必要な場合)

Bash Linux (Debian/Ubuntu / RHEL) 
# Debian/Ubuntu
sudo apt-get update && sudo apt-get install google-guest-agent

# RHEL/CentOS/Rocky
sudo yum update google-guest-agent
PowerShell Windows (GooGet) 
googet -noconfirm install google-compute-engine-windows

ステップ 3: 接続タイプの設定 (オプション)

Bash 設定変更と再起動 
# 設定ファイルを編集
sudo vi /etc/default/instance_configs.cfg
# [PluginConfig] セクションに connection_type = UDS を追加

# Guest Agent を再起動
sudo systemctl restart google-guest-agent-manager

ステップ 4: テレメトリ収集の確認・無効化 (オプション)

Bash ISV テレメトリを無効化する場合 
gcloud compute instances add-metadata INSTANCE_NAME \
    --metadata disable-guest-telemetry=true \
    --zone ZONE

⚠️ デメリット・制約事項

⚠️ 注意: enable_local_plugins のデフォルト変更により、以前は無効だったローカルプラグインが自動的に有効化されます。セキュリティポリシーにより無効にしている場合は明示的に false を設定してください。
⚠️guest telemetry 拡張は ISV テレメトリを収集するため、データプライバシーの観点から組織のポリシーに応じて無効化の検討が必要
⚠️connection_type の UDS オプションは Linux 環境向け。Windows では TCP の使用が推奨
⚠️メタデータキー disable-guest-telemetry を true に設定している場合、システムテレメトリに加え ISV テレメトリも無効化される
⚠️自動更新パッケージ (google-compute-engine-auto-updater) を使用していない場合は手動での更新が必要

💡 ユースケース

🏭 ユースケース 1: ISV ソフトウェアの運用管理

シナリオ: 大規模な Compute Engine フリートで複数の ISV ソフトウェア (データベース、セキュリティツール、モニタリングエージェントなど) を運用し、各インスタンスで稼働している ISV の状況を把握したい。

効果: ISV テレメトリの自動収集により、フリート全体のソフトウェア構成の可視化が可能になり、ライセンスコンプライアンスの確認やセキュリティ監査が効率化される。

Bash テレメトリ収集の確認 
# ローカルプラグインが有効であることを確認 (デフォルトで有効)
grep enable_local_plugins /etc/default/instance_configs.cfg

# テレメトリ収集が有効であることを確認
gcloud compute instances describe INSTANCE_NAME \
    --zone ZONE \
    --format="value(metadata.items[key='disable-guest-telemetry'])"

🛡️ ユースケース 2: 高信頼性プロダクション環境

シナリオ: ミッションクリティカルなワークロードを実行するインスタンスで、Guest Agent の安定性が運用に直結する環境。

効果: gRPC クライアント null 時の異常終了防止、レディネスシグナルのブロック解消、起動ログの改善により、エージェントの信頼性が向上し、依存サービスの起動遅延やエージェント停止に起因する障害リスクが低減。

🔧 ユースケース 3: カスタムプラグイン開発環境

シナリオ: 独自のローカルプラグインを開発し、Guest Agent のプラグインベースアーキテクチャを活用してインスタンスのカスタム機能を実装したい。

効果: connection_type の UDS/TCP 選択により、カスタムプラグインの通信要件に応じた最適な接続方式を選択でき、開発の柔軟性が向上。

INI カスタムプラグイン向け設定例 
# /etc/default/instance_configs.cfg
[PluginConfig]
enable_local_plugins = true
# UDS を使用してローカル通信を最適化
connection_type = UDS
