☁️ 概要

BigQuery の EXPORT DATA ステートメントを使用して、Cloud リソース接続経由で BigQuery のデータを Spanner にエクスポートする機能が一般提供 (GA) となりました。

これはリバース ETL (逆 ETL) ワークフローを実現する機能であり、BigQuery の分析結果を Spanner のオペレーショナルデータベースに直接書き戻すことが可能になります。SQL のみでエンドツーエンドのデータパイプラインを構築できます。

🔄 Before / After 比較

❌ Before (これまで)

  • Dataflow や Cloud Functions など中間パイプラインの構築・管理が必要
  • 外部 ETL ツールやカスタムスクリプトの運用コストが高い
  • BigQuery と Spanner 間のリアルタイム連携が困難

✅ After (これから)

  • EXPORT DATA 一文で直接エクスポート可能
  • Cloud リソース接続によるセキュアなアクセス委任
  • 継続クエリと組み合わせでリアルタイム リバース ETL が実現

⚡ 主要機能

📤 EXPORT DATA ステートメント

SQL の EXPORT DATA で BigQuery から Spanner へ直接データ書き込み。format = 'CLOUD_SPANNER' を指定し、宛先の Spanner インスタンス・データベース・テーブルを URI で指定します。

🔐 Cloud リソース接続

接続作成時に BigQuery が自動生成するサービスアカウントに Spanner への書き込み権限を付与。WITH CONNECTION 句で接続を指定して利用します。

🔄 継続クエリ対応

BigQuery の継続クエリ (Continuous Queries) と組み合わせて、リアルタイムのストリーミングエクスポートが可能。Enterprise/Enterprise Plus エディションが必要です。

🔢 型変換の自動対応

BigQuery と Spanner 間のデータ型自動変換をサポート。BIGNUMERIC → NUMERIC、FLOAT64 → FLOAT32、INT64 → ENUM などに対応。

🔧 技術仕様

spanner_options の設定項目

項目 型 説明
table STRING 必須。宛先の Spanner テーブル名
change_timestamp_column STRING TIMESTAMP 型カラム名。最新の行更新を追跡するために使用
priority STRING 書き込みリクエストの優先度。LOW / MEDIUM (デフォルト) / HIGH
tag STRING リクエストタグ。Spanner モニタリング用。デフォルト: bq_export

必要な IAM ロール

ロール リソース 用途
roles/bigquery.dataViewer BigQuery テーブル データの読み取り
roles/bigquery.user プロジェクト エクスポートジョブの実行
roles/spanner.viewer Spanner インスタンス インスタンスパラメータの確認
roles/spanner.databaseUser Spanner データベース データの書き込み
roles/bigquery.connectionAdmin プロジェクト Cloud リソース接続の作成

📋 設定方法

📌 前提条件: BigQuery Enterprise/Enterprise Plus エディションのスロット予約が作成済み、エクスポート先の Spanner データベース・テーブルが作成済み、BigQuery Connection API が有効化されていること。

Step 1: Cloud リソース接続の作成

Bash 
bq mk --connection \
  --location=REGION \
  --project_id=PROJECT_ID \
  --connection_type=CLOUD_RESOURCE \
  CONNECTION_ID

接続を作成後、サービスアカウント ID を確認します。

Bash 
bq show --connection PROJECT_ID.REGION.CONNECTION_ID

Step 2: サービスアカウントへの権限付与

Bash 
gcloud spanner databases add-iam-policy-binding DATABASE_ID \
  --instance=INSTANCE_ID \
  --member="serviceAccount:SERVICE_ACCOUNT_ID" \
  --role="roles/spanner.databaseUser"

Step 3: EXPORT DATA の実行

SQL 
EXPORT DATA
  WITH CONNECTION `PROJECT_ID.REGION.CONNECTION_ID`
  OPTIONS (
    uri = "https://spanner.googleapis.com/projects/PROJECT_ID/instances/INSTANCE_ID/databases/DATABASE_ID",
    format = 'CLOUD_SPANNER',
    spanner_options = """{ "table": "TABLE_NAME" }"""
  )
AS
SELECT * FROM `PROJECT_ID.DATASET.SOURCE_TABLE`;

💰 ビジネス面

✓ 運用コスト削減 - 中間パイプライン (Dataflow, Cloud Functions) の構築・運用が不要
✓ Time-to-Value の短縮 - SQL のみでオペレーショナルシステムに分析結果を反映
✓ リアルタイムデータ活用 - 継続クエリでニアリアルタイム反映が可能

🔧 技術面

✓ シンプルなアーキテクチャ - ETL パイプラインを SQL 一文で実現
✓ セキュアなアクセス委任 - サービスアカウントベースで認証情報の直接管理不要
✓ スケーラビリティ - ノードあたり約 5 MiB/s で線形にスケール

⚠️ 制約事項・注意点

制限事項
  • BigQuery Enterprise / Enterprise Plus エディションのスロット予約が必要 (オンデマンド課金では利用不可)
  • 継続エクスポートには CONTINUOUS ジョブタイプの予約割り当てが必要
  • 同一 rowkey の複数行エクスポート時、Spanner には 1 行のみ書き込み (どの行かは保証されない)
  • エクスポートジョブの最大実行時間は 6 時間
考慮すべき点
  • リージョンをまたぐエクスポートはデータ抽出料金が発生 (同一リージョン配置を推奨)
  • HIGH 優先度設定は Spanner の他ワークロードに影響する可能性あり
  • 100 GB 超のエクスポートでは Spanner ノードと BigQuery スロットの増強が必要
  • 宛先テーブルの主キーに生成列が含まれる場合、クエリ内で対応する式を含めるとパフォーマンスが最適化される

🎯 ユースケース

🤖 ユースケース 1: ML 予測結果のアプリケーション反映

BigQuery ML で顧客のチャーン予測モデルを実行し、予測結果を Spanner に書き戻してリアルタイムアプリケーションで活用。追加のパイプライン構築なしでアプリケーション層に直接反映できます。

SQL 
EXPORT DATA OPTIONS (
  uri = "https://spanner.googleapis.com/projects/my-project/instances/my-instance/databases/my-db",
  format = 'CLOUD_SPANNER',
  spanner_options = """{ "table": "customer_churn_predictions" }"""
)
AS
SELECT
  customer_id,
  predicted_churn_probability,
  recommended_action,
  CURRENT_TIMESTAMP() AS prediction_timestamp
FROM ML.PREDICT(
  MODEL `my_dataset.churn_model`,
  TABLE `my_dataset.customer_features`
);

📊 ユースケース 2: リアルタイムダッシュボード用集計データ配信

BigQuery で大量のトランザクションデータを集計し、結果を Spanner に格納して低レイテンシのダッシュボードアプリケーションから参照。Spanner の低レイテンシ読み取りでアプリケーションユーザーにデータを提供できます。

SQL 
EXPORT DATA OPTIONS (
  uri = "https://spanner.googleapis.com/projects/my-project/instances/my-instance/databases/my-db",
  format = 'CLOUD_SPANNER',
  spanner_options = """{ "table": "daily_sales_summary" }"""
)
AS
SELECT
  region,
  product_category,
  CURRENT_DATE() AS report_date,
  SUM(sales_amount) AS total_sales,
  COUNT(DISTINCT customer_id) AS unique_customers
FROM `my_dataset.transactions`
WHERE DATE(transaction_timestamp) = CURRENT_DATE()
GROUP BY region, product_category;
